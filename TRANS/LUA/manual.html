1 - Вступ

Lua є мовою програмування для розширення, створена для підтримки загального процедурного програмування за допомогою можливостей опису даних. Lua також пропонує добру підтримку об’єктно-орієнтованого програмування, функціонального програмування, та програмування на основі даних. Lua призначений для використання в якості потужної, легковажної, вбудованої скриптової мови для любої програми, яка потребує її. Lua реалізована як бібліотека, написана на чистому C, командній підмножині стандартного C та C++.

Як мова розширення Lua не має уяви про головну програму "main": вона робить тільки вбудованою в деякій клієнт, що називається вбудовуючою програмою, або просто приймаюча програма. Приймаюча програма може викликати функції для виконання частин коду Lua, може писати та читати змінні Lua, та може реєструвати функції C, що будуть викликані з коду Lua. Через використання функцій C, Lua може додати в подоланні у великому диапазоні різних предметів, таким чином створюючи власні мови програмування, що поділяють синтаксичну оболонку. Розповсюдження Lua включає просту приймаючу програму з назвою lua, що використовує бібліотеку Lua для надання повного, окремого інтерпретатору Lua для інтерактивного або пакетного використання.

Lua є вільним програмним забезпеченням, та провадиться як звичайно без гарантії, як зазначено в ліцензії. Реалізація, описана в цьому довідникові, доступна на офіційному веб сайті Lua, www.lua.org.

Подібно до інших довідників, цей документ подекуди прісний. Щоб ознайомитись з дискусіями щодо дизайну Lua, передивіться технічні папери, доступні на сайті Lua. Для детального вступу в програмування на Lua дивіться книжку Roberto Programming in Lua.

2 - Базові концепції

Цей розділ описує базові концепції мови.

2.1 - Значення і типи

Lua є мовою з динамічною типізацією. Це означає, що змінні не мають типів, тільки значення. Немає визначень типів в мові. Всі значення несуть свій тип.

Всі значення в Lua є першокласними значеннями. Це означає, що всі значення можуть зберігатись в змінних, передаватись в якості аргументів в інші функції, та повертатись як результати.

Є вісім базових типів в Lua: nil, boolean, number, string, function, userdata, thread, та table. Тип nil має одне єдине значення, nil, чиє головна властивість є відрізнятись від любого іншого значення; він звичайно представляє відсутність корисного значення. Тип boolean має два значення, false та true. Обоє, nil та false призводять до умови false; любе інше значення робить його true. Тип number представляє обоє, цілі числа та дійсні числа (з плаваючою точкою). Тип string представляє незмінні послідовності байтів. Lua є 8-бітно чистим: рядки можуть містити любі 8-бітові значення, включаючи вбудовані нулі ('\0'). Lua також є кодування-агностичним; він не робить припущень щодо вмісту рядків.

Тип number використовує два внутрішні представлення, або два підтипи, один з назвою integer та інший з назвою float. Lua має явні правила коли використовувати кожне з цій представлень, та він також конвертує між ними автоматично, коли виникає потреба (див. §3.4.3). Таким чином, програміст може обрати здебільшого ігнорувати різницю між цілими та з плаваючою точкою, або здійснювати повний контроль над представленнями кожного числа. Стандартний Lua використовує 64-бітні цілі та з подвоєною точністю з плаваючою точкою (64-бітні), але ви можете також скомпілювати Lua таким чином, щоб він використовував 32-бітні цілі та/або з одинароною точністю з плаваючою точкою (32-бітні). Опція з 32 бітами для обох, цілих та з плаваючою точкою, є зокрема привабливою для малих машин та вбудованих систем. (Дивіться макро LUA_32BITS в файлі luaconf.h.)

Lua може викликати та маніпулювати функціями на Lua та функціями, написаними на  C (див. §3.4.10). Обоє представлені типом function.

Тип userdata запроваджено, щоб дозволити довільні дані C до зберігання в змінних Lua. Значення користувацьких даних представляє блок необробленої пам’яті. Є два типи користувацької пам’яті: повні користувацькі дані, що є об’єктом з блоком пам’яті, що керується Lua, та легкі користувацькі дані, що є просто значенням вказівника C. Користувацькі дані не мають попередньо визначених операцій в Lua, за винятком призначення та первірки ідентичності. Використовуючі metatables, програміст може визначити операції для значень повних користувацькіх даних (див. §2.4). Значення користувацькіх даних не можуть бути створені або модифіковані в Lua, тільки через C API. Це гарантує цілість даних, що належать зовнішній програмі.

Тип thread представляє незалежні потоки виконання, та використовується для реалізації coroutines (див. §2.6). Потоки Lua не пов’язані з потоками операційної системи. Lua підтримує копроцедури на всіх системах, навіть на тих, що природно не підтримують потоки.

Тип table реалізує асиціативні масиви, тобто масиви, що можуть бути індексовані не тільки числами, але любим значенням Lua, за винятком nil та NaN. (Not a Number є спеціальним значенням, що використовується для представлення невизначених або неможливих числових значень, таких, як 0/0.) Таблиці можуть бути гетерогенними, тобто вони можуть містити значення всіх типів (за виключенням nil). Кожний ключ зі значенням nil розглядається як такий, що не належить до таблиці. Навпаки, любий ключ, що не є частиною таблиці, має асоційоване значення nil.

Таблиці є єдиним механізмом структуризації даних в Lua, вони можуть бути використані для представлення звичайних масивів, послідовностей, таблиць символів, множин, записів, графів, дерев, тощо. Щоб представляти записи Lua використовує ім’я поля в якості індексу. Мова підтримує це репрезентацію, провадячи a.name як синтаксичний цукор для a["name"]. Є декілька зручних шляхів створити таблиці в Lua (див. §3.4.9).

Ми використовуємо термін послідовність для визначення таблиці, де набір всіх невід’ємних числових ключів дорівнює {1..n} для деякого невід’ємного цілого n, що називаєтсья довжиною послідовності (див. §3.4.7).

Подібно до індексів, значення полів таблиці може бути любого типу. Зокрема, оскільки функції є першокласними значеннями, поля таблиці можуть містити функції. Таким чином, таблиці можуть також утримувати методи (див. §3.4.11).

Індексація таблиць слідує визначенню безпосередньої рівності в мові. Вирази a[i] та a[j] означають той же елемент тоді і тільки тоді, якщо i та j безпосередньо рівні (тобто рівні без метаметодів). Зокрема, з плаваючою точкою з цілими значеннями рівні до відповідних цілих аналогів (тобто 1.0 == 1). Щоб уникнути неоднозначності, любе з плаваючою точкою з цілим значенням, що використовується як ключ, конвертується до відповідного цілого. Наприклад, якщо записати a[2.0] = true, справжній ключ, що буде вставлений в таблицю, буде ціле 2. (З іншого боку, 2 та "2" відрізняються в Lua, та таким чином посилаються на різні елементи таблиці).

Таблиці, функції, потоки, та (повні) користувацькі дані є о’бєктами: змінні насправді не містять ці значення, але тільки посилаються на них. Призначення, передача параметрів, та повернення з функції завжди маніпулюють посиланнями на такі значення; ці операції не передбачають жодного виду копіювання.

Бібліотечна функція type повертає рядок, що описує тип наданого значення (див. §6.1).

2.2 - Оточення та глобальне оточення

Як буде показаним в §3.2 та §3.3.3, кожне посилання на вільне ім’я (тобто ім’я, не прив’язане до жодної декларації) var буде синтаксично трансльоване в _ENV.var. Більше того, кожний клаптик буде скомпільований в досяжності зовнішньої локальної змінної з ім’ям  _ENV (див. §3.3.2), таким чином _ENV по собі ніколи не є вільним ім’ям в клаптику.

Не зважаючи на існування зовнішньої змінної _ENV та трансляції вільних імен, _ENV є повністю звичайним ім’ям. Зокрема, ви можете визначити нові змінні та параметри з цім ім’ям. Кожне посилання на вільне ім’я використовує _ENV, що видіме з цієї частини програми, згідно зі звичайними правилами видімості Lua (див. §3.5).

Кожна таблиця, що використовується як значення для _ENV називається оточенням.

Lua утримує окреме оточення, що називається глобальним оточенням. Це значення зберігаєтся як спеціальний індекс в реєстрі C (див. §4.5). В Lua глобальна змінна _G ініціалізована цім самим значенням. (_G ніколи не використовується внутрішньо).

Коли Lua завантажує клаптик, по замовчанню значення для його _ENV є глобальне оточення (див. load). Таким чином, по замовчанню, вільні імена в коді Lua посилаються на елементи в глобальному оточенні (і, таким чином, вони також називаються глобальними змінними). Більше того, всі стандартні бібліотеки завантажуються в глобальне оточення, та деякі функції оперують з тим оточенням. Ви можете використовувати load (або loadfile) для завантаження клаптика з іншого оточення. (В C ви маєте завантажити клаптик, та потім змінити значення його першого upvalue).

2.3 - Обробка помилок

Оскільки Lua є вбудованною мовою, всі дії Lua починають з виклику коду C в зовнішній програмі, що викликає функцію з бібліотеки Lua. (Коли ви використовуєте Lua окремо, застосування lua є цією програмою). Коли виникає помилка на протязі компіляції або виконання клаптика Lua, керування повертається до викликаючої програми, що може прийняти відповідні міри (такі як надрукувати повідомлення про помилку).

Код Lua може явно згенерувати помилку, викликавши функцію error. Якщо вам треба перехопити помилку в Lua, ви можете використати pcall або  xpcall для виклику даної функції в захищеному режимі.

Коли виникає помилка, об’єкт помилки (також відомий як повідомлення помилки) розповсюджується разом з інформацією щодо помилки. Lua по собі тільки генерує помилки, чий о’бєкт помилки є рядком, але програми можуть генерувати помилки любого значення в якості об’єкта помилки. Це покладається на програму Lua, або на зовнішню програму, хто буде обробляти такий об’єкт помилки.

Коли ви використовуєте xpcall або lua_pcall, ви можете встановити обробник повідомлення, що буде викликаний в випадку помилок. Ця функція визивається з оригінальним повідомленням про помилку, та повертає нове повідомлення про помилку. Вона викликаєтсья перед тим, як помилка розкрутить стек, так що вона може зібрати більше інформації щодо помилки, наприклад, інспектуючи стек та створюючи трасу стеку. Цей обробник повідомлень все ще захищений через захищений виклик; таким чином, помилка в обробнику повідомлень буде знову викликати обробник повідомлень. Якщо цей цикл триватиме дуже довго, Lua перерве його та поверне відповідне повідомлення.

2.4 - Метатаблиці та метаметоди

Кожне значення в Lua може мати метатаблицю. Ця метатаблиця є звичайною таблицею в Lua, що визначає поведінку оригінального значення для окремих операцій. Ви можете змінити деякі аспекти поведінки для значення, встановлюючі специфічні поля в метатаблиці. Наприклад, коли нечислове значення використовується в додаванні, Lua перевіряє функцію в полі "__add" в метатаблиці для цього значення. Якщо буде знайдена така, Lua викличе функцію для виконання додавання.

Ключі в метатаблиці походять від імен подій; відповідні значення називаються метаметодами. В попередньому випадку подія є "add", та метаметод є функція, що виконує додавання.

Ви можете запитати метатаблицю на предмет значень, використовуючи функцію  getmetatable.

Ви можете замінити метатаблиці для таблиць, використовуючи функцію setmetatable. Ви не можете змінити метатаблицю для інших типів з коду Lua (за винятком використання бібліотеки налагодження (§6.10)); ви мусите користуватись C API для цього.

Таблиці та повні користувацькі дані мають індивідуальні метатаблиці (хоча багато таблиць та користувацькіх даних можуть поділяти свої метатаблиці). Значення всіх інших типів поділяють одну метатаблицю для одно типу; тобто одна метатаблиця для всіх чисел, для всіх рядків, таке інше. По замовчанню, значення не має метатаблиці, але бібліотека рядків встановлює метатаблицю для типу рядків (див. §6.4).

Метатаблиця контроює, як об’єкт поводиться в аріфметичних операціях, бітових операціях, порядкових порівняннях, приєднанні, визначенні довжини, викликах та індексуванні. Метатаблиці також можуть визначати функцію, що повинна викликатись, коли користувацькі дані або таблиця збирається як сміття (§2.5).

Далі наводиться перелік подій, що контролюються метатаблицею. Кожна операція ідентифікується відповідним ім’ям події. Ключ для кожної події є рядок з іменем, якому передує префікс з двох підкраслень , '__'; наприклад, ключ для операції "add" є рядок "__add". Зауважте, що запити до метаметодів завжди необроблені; доступ до метаметодів ніколи не викликає інших метаметодів.

Для унарних операторів (зміна знаку, запит довжини та бітове ні), метаметод обчислюється та викликається з незначущим другим операндом, рівним першому. Цей додатковий операнд служить тільки для спрощення внутрішнього світу Lua (що робить роботу ціх операндів схожими на роботу двомісних операндів), та може бути виключено в майбутніх версіях. (Для більшості використань цей додатковий операнд є несуттєвим).

  • "add": операція +. Якщо операнд для додавання не є числом (або рядком, що приводиться до числа), Lua спробує викликати метаметод. Спершу Lua перевірить перший операнд (навіть якщо він задовільний). Якщо цей операнд не визначає метаметод для події "__add", тоді Lua перевірить другий операнд. Якщо Lua знайде метаметод, він викличе метаметод з двома операндами як аргументами, та результат виклику (приведений до одного значення) буде результатом операції. В іншому випадку буде ініційована помилка.
  • "sub": операція -. Поведінка подібна до операції "add".
  • "mul": операція *. Поведінка подібна до операції "add".
  • "div":  операція /. Поведінка подібна до операції "add".
  • "mod": операція %. Поведінка подібна до операції "add".
  • "pow":  операція ^ (піднесеня до ступіню). Поведінка подібна до операції "add".
  • "unm":  операція - (унарний мінус). Поведінка подібна до операції "add".
  • "idiv": операція // (округлене вниз ділення). Поведінка подібна до операції "add".
  • "band": операція & (побітове ТА). Поведінка подібна до операції "add", за винятком того, що Lua спробує метаметод, коли любий з операндів не є цілим або не призводиться до цілого(див. §3.4.3).
  • "bor": операція | (побітове АБО). Поведінка подібна до операції "band".
  • "bxor": операція ~ (побітове виключне АБО). Поведінка подібна до операції "band".
  • "bnot": операція ~ (побітове унарне НІ). Поведінка подібна до операції "band".
  • "shl": операція << (побітовий зсув вліво). Поведінка подібна до операції "band".
  • "shr": операція >> (побітовий зсув вправо). Поведінка подібна до операції "band".
  • "concat": операція .. (конкатенація). Поведінка подібна до операції "add", за винятком того, що Lua спробує метаметод, якщо любий з операндів не є рядком та не числом (що завжди призводиться до рядка).
  • "len": операція # (довжина). Якщо об’єкт не є рядком, Lua спобує його метаметод. Якщо метаметод знайдено,  Lua викликає його з об’єктом в якості аргумента, та результат виклику (завжди обмежений до одного значення) є результатом операції. Якщо немає метаметода, але об’єкт є таблицею, Lua використовує операцію довжини таблиці (див. §3.4.7). Інакше Lua викликає помилку.
  • "eq": операція == (рівність). Поведінка подібна до операції "add", за винятком того, що Lua спробує метаметод тільки в тому випадку, коли порівнювані значення обоє є таблицями або обоє є повними користувацькими даними, та вони не є примітивно рівними. Результат виклику завжди перетворюється на логічне значення.
  • "lt": операція < (менше ніж). Поведінка подібна до операції "add", за винятком того, що Lua спробує метаметод тільки в випадку, коли порівнювані значення не є обоє числами, та не є обоє рядками. Результат виклику завжди перетворюється на логічне значення.
  • "le": операція <= (менше або дорівнює). На відміну від інших операцій, операція менше або дорівнює може використовувати два різні події. Перше, Lua шукає метаметод "__le" в обох операндах, так само як в операції "lt". Якщо він не може знайти такий метаметод, тоді він спробує подію "__lt", вважаючи, що a <= b еквівалентне до not (b < a). Так само, як і інші операції порівняння, результат завжди буде лоогічним значенням. (Це використання події "__lt" може бути видаленим в майбутніх версіях; це також повільніше, ніж справжній метаметод "__le").
  • "index": Індексований доступ до table[key]. Ця подія виникає, коли table не є таблицею, або коли key не є присутнім в table. Метаметод шукається в table.

    Не зважаючи на назву, метаметод для цієї події може бути або функцією, або таблицею. Якщо це функція, вона викликається з table та  key в якості аргументів. Якщо це таблиця, дійсний результат буде індексація цієї таблиці за допомогою key. (Ця індексація звичайна, не спрощена, і, таким чином, може викликати інший метаметод).

  • "newindex": Індексоване присвоєння table[key] = value. Як і подія індексації, ця подія трапляється, коли table не є таблицею, або коли key не є присутнім в  table. Метаметод шукається в table.

    Як і індексація, метаметод для цієї події може бути або функцією, або таблицею. Якщо це функція, вона викликається з table, key, та value в якості аргументів. Якщо це таблиця, Lua виконує індексоване присвоєння до цієї таблиці, для ціх же ключа та значення. (Це присвоєння є звичаним, не спрощеним, в, таким чином, може викликати інший метаметод).

    Коли є метаметод "newindex", Lua не виконує примітивного присвоювання. (Якщо потрібно, метаметод може сам викликати rawset щоб виконати просте присвоювання).

  • "call": Операція func(args). Ця подія виникає, коли Lua намагається викликати нефункціональне значення (тобто func не є функцією). Метаметод шукається в func. Якщо він присутній, метаметод викликається з func в якості першого аргументу, за яким слідують вргументи оригінального виклику (args).

Є доброю практикою додавати всі потрібні метаметоди до таблиці перед встановленням її як метатаблиці для іншого об’єкту. Зокрема, метаметод "__gc" робить тільки в випадку, коли ви слідуєте цьому порядку (див. §2.5.1).

2.5 - Прибирання сміття

Lua виконує автоматичне управління пам’яттю. Це означає, що ви не маєте турбуватись щодо розміщення пам’яті для нових об’єктів, або вивільнення її, коли об’єкти більше не потрібні. Lua керує пам’яттю автоматично, з використанням прибирача сміття для збирання всіх мертвих об’єктів (тобто, об’єктів, що більше не досяжні з Lua). Вся пам’ять, використана Lua, є предметом для автоматичного управління: ярдки, таблиці, користувацькі дані, функції, потоки, внутрішні структури, таке інше.

Lua реалізує накопичувальний збиральник поміть-та-прибери. Він використовує два числа для контролю своїми циклами прибирання сміття: паузу прибирача сміття та кроковий множник прибирача сміття. Обоє є відсотковими точками в якості одиниць віміру (тобто, значення 100 означає внутрішнє значення 1).

Пауза збиральника сміття контролює, наскільки довго збиральник очікує перед початком нового циклу. Більші значення роблять збиральник менш агресивним. Значення меньше ніж 100 ожначають, що збиральник не буде чекати, щоб розпочати новий цикл. Значення 200 означає, що збиральник очікує, доки не буде використана вся пам’ять, перед тим, як розпочати новий цикл.

Кроковий множник збиральника сміття контролює відносну швидкість збиральника, по відношенню до виділення пам’яті. Більші значення роблять збиральник більше агресивним, але також збільшують розмір кожного кроку. Ви не повинні використовувати значення, меньші ніж 100, оскільки вони роблять збиральник дуже повільним, та можуть призвести до того, що збиральник ніколи не завершить цикл. По замовчанню стоїть 200, що збиральник виконується на "подвійній" швидкості розміщення пам’яті.

Якщшо ви встановили множник кроку до дуже великого значення (більше ніж 10% максимального числа байтів, що програма може використоувати), збиральник поводиться як збиральник світ-почекає. Якщо ви потім встановите паузу в 200, збиральник поводиметься як в старих версіях Lua, що робить повний збір кожного разу, коли Lua подвоює використання пам’яті.

Ви можете змінити ці числа, викликавши  lua_gc в  C або collectgarbage в Lua. Ви можете також використовувати ці функції для контролю збиральника напряму (тобто, для його зупинки та запуску).

2.5.1 - Метаметоди збиральника сміття

Ви можете встановити метаметоди збиральника сміття для таблиць, та, використовуючи C API, для повних даних користувача (див. §2.4). Ці метаметоди також називаються фіналізаторами. Фіналізатори дозволяють вам кооординувати збір сміття Lua з менеджментом зовнішніх ресурсів (таким, як закриття файлів, мережевих з’єднать та сеансів баз даних, або вивільнення вашої власної пам’яті).

Для о’бєкту (таблиці або користувацьких даних), щоб бути фіналізованими під час збору, ви маєте маркувати їх для фіналізації. Ви маркуєте об’єкт для фіналізації, коли ви встановлюєте його метатаблицю, та метатаблиця містить поле, індексоване рядком "__gc". Зауважте, що якщо ви встановите метатаблицю без поля __gc, та потім створите це поле в метатаблиці, об’єкт не буде помічений для фіналізації.

Коли маркований об’єкт стає сміттям, він не збирається безпоседеньо збиральником сміття. Замість цього Lua додає його до списку. Після збирання Lua проходить по цьому списку. Для кожного об’єкту в цьому списку він перевіряє, чи об’єкт має метаметод __gc: якщо це функція,  Lua викликає її з об’єктом в якості єдиного аргументу; якщо це не функція, Lua просто ігнорує його.

Наприкінці кожного циклу збору сміття, фіналізатори для об’єктів викликаються в  порядку, зборотньому відносно того, як об’єкти були марковані для фіналізації, серед тих, що зібрані в цьому циклі; тобто, перший фіналізатор, що буде викликаний, буде той, що асоційований з об’єктом, маркованим останнів і цій програмі. Виконання кожного фіналізатора може відбуватись в кожній точці, на протязі виконання звичайного кода.

Оскільки об’єкт, що був зібраний, мусить все ще використаний фіналізатором, цей об’єкт (та інші об’єкти, що досяжні тільки через нього) мусять бути відновлені Lua. Звичайно, це відновлення є тимчасовим, та пам’ять об’єкта буде вивільнена під час наступного циклу збирання сміття. Однак, якщо фіналізатор зберігає об’єкт в деякому глобальному місці (тобто, в глобальній змінній), тоді відновлення є постійним. Більше того, якщо фіналізатор маркує фіналізованій об’єкт знову для фіналізації, його фіналізатор буде викликаний знову в наступному циклі, де об’єкт буде надоступний. В жодному разі пам’ять об’єкта вивільняється, тільки в циклі GC, де об’єкт недосяжний та не помічений для фіналізації.

Коли ви закриваєте стан (див. lua_close), Lua викликає фіналізатори для всіх об’єктів, маркованих для фіналізації, слідуючи зворотньому порядку маркування. Якщо любий фіналізатор маркує об’єкти для збирання на протязі цієї фази, ці маркування не матимуть ефекту.

2.5.2 - Слабкі таблиці

Слабка таблиця це таблиця, чиї елементи є слабкими посиланнями. Слабке посилання ігнорується збиральником сміття. Іншими словами, якщо всі посилання на об’єкт є тільки слабкими посиланнями, тоді збиральник сміття ігнорує такі посилання.

Слабка таблиця може мати слабкі ключі, слабі значення, або обоє. Таблиця з слабкими значеннями дозволяє збирання значень, але запобігає збиранню ключів. Таблиця з обоїми, слабкими ключами та значеннями, дозволяє збирання обох, ключів та значень. В жодному разі, якщо або ключ, або значення буде зібраним, ціла пара буде видалена з таблиці. Слабкість таблиці контролюється полем __mode її метатаблиці. Якщо поле __mode є рядком, що містить символ 'k', ключі будуть слабкими. Якщо __mode містить  'v', значення таблиці будуть слабкими.

Таблиця з слабкими ключами та сильними значеннями також називається ефемерною таблицею. В ефемерній таблиці значення вважається доступним, тільки якщо ключ є доступним. Зокрема, якщо посилання на ключ іде тільки через значення, пара видаляється.

Люба зміна в слабкості таблиці може мати ефект тільки в наступному циклі збирання. Зокрема, якщо ви змінили слабкість в бік сильного режиму, Lua може все ще зібрати деякі елементи з таблиці перед тим, як зміни наберуть чинності.

Тільки об’єкти, що мають явну конструкцію, видаляються з слабкої таблиці. Значення, такі як числа та легкі функції C, не є предметом для збирання сміття, і, таким чином, не видаляються з слабкої таблиці (хіба що збираються асоційовані з ними значення). Хоча рядки є предметом для збирання, вони не мають явної конструкції, та також не видаляються зі слабких таблиць. 

Відновлені об’єкти (тобто об’єкти, що фіналізуються та об’єкти, доступні тільки під час фіналізації) мають спеціальну поведінку в слабких  таблицях. Вони видаляються з слабких значень перед виконанням своїх фіналізаторів, але видаляються зі слабких ключів тільки в наступне збирання, після виконання своїх фіналізаторів, коли ці об’єкти насправді вивільняються. Ця поведінка дозволяє фіналізатору мати доступ до властивостей, асоційованих з об’єктом через слабку таблицю.

Якщо слабка таблиця є серед відновлених об’єктів в циклі збирання, вона може залишитись не до кінця очищеною до наступного циклу.

2.6 - Копроцедури

Lua підтримує копроцедури, також відомі як колоборативна мультипоточність. Копроцедури в Lua представляють незалежний потік виконання. Однак на відміну від потоків в багатопоточних системах, копроцедура призупиняє своє виконання тільки викликаючи функцію yield.

Ви створюєте копроцедуру, викликаючи coroutine.create. Її єдиний аргумент є функція, що є головною функцією копроцедури. функція create тільки створює нову копроцедуру та повертає вказівник на неї (об’єкт типу thread); вона не запускає копроцедуру.

Ви виконуєте копроцедуру, викликаючи coroutine.resume. Коли ви вперше викликаєте  coroutine.resume, передаючі їй в першому аргументі потік, що повернула  coroutine.create, копроцедура стартує своє виконання, викликаючи головну функцію. Додаткові аргументи, передані до coroutine.resume, передаються як аргументи в функцію. Після того, як копроцедура розпочала своє виконання, вона робить до тих пір, доки вона не завершиться, або не yields.

Копроцедура може завершити своє виконання в два шляхи: звичайно, коли головна функція повертається (явно або неявно, після останньої інструкції); та незвичайно, коли виникла помилка. В випадку звичайного завершення coroutine.resume повертає  true, плюс любі значення, повернуті головною функцією копроцедури. В випадку помилок coroutine.resume повертає false плюс повідомлення про помилку.

Копроцедура відддає керування, викликаючи coroutine.yield. Коли вона робить це, відповідна  coroutine.resume беспосередньо повертає керування, навіть якщо передача відбувається у вкладеному виклику функції (тобто, не в головній функції, але в функції, прямо або непрямо викликаної з головної функції). В випадку передачі coroutine.resume також повертає true, плюс любі значення, передані в coroutine.yield. Наступного разу, коли ви відновите роботу тієї ж копроцедури, вона продовжить виконання в тій точці, де вона залишила виконання, та виклик coroutine.yield поверне любі додаткові аргументи, передані в coroutine.resume.

Подібно до coroutine.create, функція coroutine.wrap також створює копроцедуру, але замість повернути саму копроцедуру, вона повертає функцію, що, коли викликається, повертає копроцедуру. Любі аргументи, передані в цю функцію, ідуть як додаткові аргументи для coroutine.resume. coroutine.wrap повертає всі значення, повернуті  coroutine.resume, за винятком першого (логічного кода помилки). На відміну від coroutine.resume, coroutine.wrap не перехоплює помилок; любі помилки просуваються до викликача.

Як приклад того, як це робить, розглянемо наступний код: 

     function foo (a)
       print("foo", a)
       return coroutine.yield(2*a)
     end
     
     co = coroutine.create(function (a,b)
           print("co-body", a, b)
           local r = foo(a+1)
           print("co-body", r)
           local r, s = coroutine.yield(a+b, a-b)
           print("co-body", r, s)
           return b, "end"
     end)
     
     print("main", coroutine.resume(co, 1, 10))
     print("main", coroutine.resume(co, "r"))
     print("main", coroutine.resume(co, "x", "y"))
     print("main", coroutine.resume(co, "x", "y"))

Коли ви викликаєте це, ви отримуєте наступний вихід: 

     co-body 1       10
     foo     2
     main    true    4
     co-body r
     main    true    11      -9
     co-body x       y
     main    true    10      end
     main    false   cannot resume dead coroutine

Ви також можете створювати та маніпулювати копроцедурами через C API: дивіться функції  lua_newthread, lua_resume, та lua_yield.

3 - Мова

Це розділ описує лексику, синтаксис та семантику Lua. Іншими словами, цей розділ описує, які токени є допустимі, як їх можна комбінувати, та що означатимуть їх комбінації.

Конструкції мови будуть описані з використанням розширеної нотації BNF, де {a} означає 0 або більше a, та [a] ожначає опціональне a. Нетермінали показані як нетермінали, ключові слова показані як kword, та інші термінальні символи показані як  ‘=’. Повний синтаксис Lua може бути знайдений в §9 в кінці цього довідника.

3.1 - Лексичні домовленості

Lua є мовою вільної форми. Він ігнорує проміжки (включаючи нові рядки) та коментарі між лексичними елементами (токенами), за винятком як роздільників між іменами та ключовими словами.

Імена (також знані як ідентифікатори) в Lua можуть бути любими рядками з буков, цифр та підкреслень, що не починаються з цифри. Ідентифікатори використовуються для іменування змінних, полів таблиць та міток.

Наступні ключові слова зарезервовані, та не можуть бути використані як імена: 

     and       break     do        else      elseif    end
     false     for       function  goto      if        in
     local     nil       not       or        repeat    return
     then      true      until     while

Lua я чутливою до реєстру мовою: and є зарезервованим словом, але And та AND є два різних, прийнятних імені. В якості домовленості програми мають уникати створювання імен, що починають з підкреслення, за яким слідує одна або більше великих літер (наприклад, як  _VERSION).

Наступні рядки визначають інші токени: 

     +     -     *     /     %     ^     #
     &     ~     |     <<    >>    //
     ==    ~=    <=    >=    <     >     =
     (     )     {     }     [     ]     ::
     ;     :     ,     .     ..    ...

Літеральні рядки можуть бути обмежені співпадаючими одинарними або подвідйними лапками, та можуть містити наступні C-подібні послідовності виключень: '\a' (bell), '\b' (backspace), '\f' (form feed), '\n' (newline), '\r' (carriage return), '\t' (horizontal tab), '\v' (vertical tab), '\\' (backslash), '\"' (quotation mark [double quote]), and '\'' (apostrophe [single quote]). Зворотня коса, за якою слідує справжній новий рядок, призводить до нового рядка в літералі. Послідовність '\z' пропускає наступний блок проміжків, включаючи і переводи рядка; це, зокрема, корисно, для розбиття та відступів довгих літеральних рядків, без додавання нових рядків та проміжків у вміст рядка.

Рядки в Lua можуть містити любі 8-бітові значення, включаючи вбудовані нулі, що виглядає як '\0'. Більш загально, ми можемо задати любий байт в літеральному рядку за допомогою числового значення. Це може бути зроблене за допомогою послідовностей \xXX, де XX є послідовність ровно двох шістнадцятиричних цифр, або послідовність \ddd, деe ddd є послідовність до трьох десятичних цифр. (Зауважте, що якщо десятична послідовність стоїть перед цифрою, то вона має містити рівно три цифри).

Кодування UTF-8 символу Unicode може бути вставлене в літеральний рядок за допомогою послідовності виключення \u{XXX} (зауважте на обов’язкові дужки), де XXX є послідовність з одного або більше десятичних цифр, що представляють точку символьного коду.

Літеральні рядки також можуть бути визначені за допомогою довгого формату з використанням довгих дужок. Ми визначаємо відкриваючі довги дужки рівня n як відкриваючі прямокутні дужки, за якими слідує n знаків рівняння, за якими слідує інші прямокутні дужки. Таким чином, відкриваючі довги дужки рівня 0 записуються як [[, відкриваючі довги дужки рівня 1 записуються як [=[, і так далі. Закриваючі довгі дужки визначаються схожим чином; наприклад, закриваючі довгі дужки рівня 4 записуютсья як ]====]. Довгий літерал починається з відкриваючих довгих дужок любого рівня, за завершується першими ж закриваючими довгими дужками того ж рівня. Літерали в ціх дужках можуть міститись на декількох рядках, не інтерпретують жодних послідовностей-виключень, та ігнорують довгі дужки жодного іншого рівня. Любий тип послідовностей кінця рядка (перевід каретки, новий рядок, перевод каретки та новий рядок, або навпаки) перетворюються в простий новий рядок.

Кожний байт в літеральному рядку, що явно не отримує вплив від попередніх правил, представляє сам себе. Однак Lua відкриває файли для розбору в текстовому режимі, та системні файлові функції можуть мати проблеми з деякими контрольними символами. Так що безпечніше представляти нетекстові дані як квотований літерал з явними послідовностями виключень для нетекстових символів.

Для зручності, коли відкриваючі довгі дужки безпосередньо стоять перед новим рядком, новий рядок не включається в рядок. Як приклад, в системі з використанням ASCII (де літеру 'a' закодовано як 97, новий рядок закодовано як 10, та '1' як 49), п’ять літеральних рядків вказують на один і той же рядок: 

     a = 'alo\n123"'
     a = "alo\n123\""
     a = '\97lo\10\04923"'
     a = [[alo
     123"]]
     a = [==[
     alo
     123"]==]

Числова константа (або нумерал) може бути записаний з додатковою дробною частиною, та опціональною десятковою експонентою, маркованою літерами 'e' або 'E'. Lua також сприймає шеснядцятиричні константи, що починаються з 0x або 0X. Шістнадцятиричні константи також сприймають додаткову дробну частину, плюс додаткову бінарну експоненту, марковану літерами 'p' або 'P'. Числова константа з дробною точкою або експонентою вказує на число з плаваючою точкою, інакше значення ціле. Приклади допустимих цілих констант є  

     3   345   0xff   0xBEBADA

Приклади допустимих констант з плаваючою точкою

     3.0     3.1416     314.16e-2     0.31416E1     34e1
     0x0.1E  0xA23p-4   0X1.921FB54442D18P+1

Коментар починається з подвійного мінусу (--) десь за межами рядка. Якщо текст безпосередньо після -- не є відкриваючими дужками, коментар є коротким коментарем, що продовжується до кінця рядка. Інакше це довгий коментар, що продовжується до відповідних закриваючих дужок. Довгі коментарі часто використовуються для тимчасового відключення коду.

3.2 - Змінні

Змінні - це місця для зберігання значень. Є три типи змінних в Lua: глобальні змінні, локальні змінні, та поля таблиць.

Єдине ім’я може вказувати на глобальну змунну або на локальну змінну (або на формальний параметр функції, що є окремим випадком локальної змінної): 

	var ::= Name

Ім’я означає ідентифікатор, як визначено в §3.1.

Кожна змінна вважається глобальною, якщо вона явно не визначена як локальна (див. §3.3.7). Локальні змінні мають лексичну видимість: локальні змінні можуть бути вільно досяжні в функціях, визначених в їх полі зору (див. §3.5).

Перед першим присвоюванням змінної її значення є nil.

Прямокутні дужки використовуються для індексації таблиці: 

	var ::= prefixexp ‘[’ exp ‘]’

Значення доступу до полів таблиці може бути змінене через мететаблиці. Доступ до індексованої змінної t[i] еквівалентно виклику gettable_event(t,i). (Див. §2.4 для повного опису функції gettable_event. Ця функція не визначена або може бути викликана в Lua. Ми використовуємо її тут тільки для цілей пояснення).

Синтаксис var.Name є тільки синтаксичним цукором для var["Name"]: 

	var ::= prefixexp ‘.’ Name

Доступ до глобальної змінної x еквівалентно до _ENV.x. Через шлях, яким компілюються клаптики, _ENV ніколи не є глобальним ім’ям (див. §2.2).

3.3 - Твердження

Lua підтримує загалом звичайний набір тверджень, подібних до таких в Pascal або C. Цей набір включає присвоєння, керівні структури, виклики функцій та декларації змінних.

3.3.1 - Блоки

Блок є списком тверджень, що виконуються послідовно: 

	block ::= {stat}

Lua має пусті твердження, що дозволяє вам розділити твердження за допомогою крапки з комою, розпочати блок з крапки з комою, або записати крапку з комою двічі послідовно: 

	stat ::= ‘;’

Виклики функції та присвоєння можуть починатись з відкритих дужок. Ця можливість призводить до невизначеності в граматиці Lua. Візьмемо наступний фрагмент: 

     a = b + c
     (print or io.write)('done')

Граматика може дивитись на це в два способи: 

     a = b + c(print or io.write)('done')
     
     a = b + c; (print or io.write)('done')

Поточний аналізатор завжди дивиться на ці конструкції першим шляхом, інтерпретуючі відкриті дужки як початок аргументів для виклику. Щоб запобігати невизначеності, є гарною практикою завжди ставити крапку з комою перед твердженням, що починається з відкритих дужок: 

     ;(print or io.write)('done')

Блок може  бути явно обмежений для утворення одного твердження: 

	stat ::= do block end

Явні блоки корисні для контролю видімості визначень змінних. Явні блоки також часом використовуються для додавання твердження return посередині іншого блоку (див. §3.3.4).

3.3.2 - Клаптики

Одиниця компіляції в Lua називається клаптик. Синтаксично клаптик це просто блок: 

	chunk ::= block

Lua обробляє клаптик як тіло анонімної функції зі змінним числом аргументів (див. §3.4.11). Як такі, клаптики можуть визначати локальні змінні, отримувати аргументи та повертати значення. Більше того, такі анонімні функції компілюються в полі зору зовнішньої локлаьної змінної з назвою _ENV (див. §2.2). Результуюча функція завжди має _ENV в якості свого upvalue, навіть якщо вона не використовує це значення.

Клаптик може бути збережений в файлі, або в рядку в зовнішній програмі. Для виконання клаптика Lua спочатку завантажує його, передкомпілюючі код клаптика в інструкції віртуальної машини, та потім Lua виконує компільований код за допомогою інтерпретатора віртуальної машини.

Клаптики можуть також бути передкомпільованими в двійникову форму; дивіться  luac та string.dump щодо деталей. Програми в первинній та компільованій формі взаємозамінні; Lua автоматично визначає тип файлу та діє відповідно (див. load).

3.3.3 - Присвоєння

Lua дозволяє множинне присвоєння. Таким чином, синтаксис для присвоєння визначає список змінних злівого боку, та список значень з правого боку. Елементи в обох списках розілені комами: 

	stat ::= varlist ‘=’ explist
	varlist ::= var {‘,’ var}
	explist ::= exp {‘,’ exp}

Вирази дискутуються в  §3.4.

Перед присвоюванням список значень підлаштовується до довжини списку змінних. Якщо значень більше, ніж треба, зайві значеня відкидаються. Якщо значень меньше, ніж треба, список розширюється за допомогою стількох nil, скільки потрібно. Якщо список виразів закінчується виклком функції, тоді всі значення, повернуті цім викликом, доповнюють список значень, перед налаштуванням (за винятком коли виклик заключений в дужки; див. §3.4).

Твердження присвоювання спочатку обчислює вирази ,та тільки потім виконує присвоєння. Таким чином код 

     i = 3
     i, a[i] = i+1, 20

встановлює a[3] в 20, не впливаючи на a[4], оскільки i в a[i] обчислене (як 3) перед тим, як йому присвоєне 4. Подібно до цього, рядок 

     x, y = y, x

обмінює значення x та y, та

     x, y, z = y, z, x

циклічно перемішує значення x, y та z.

Значення присвоєння до глобальних змінних та полів таблиць може бути зміненим через мататаблиці. Присвоєння до індексованої змінної t[i] = val еквівалентне до  settable_event(t,i,val). (Див. §2.4 для повного опису функції settable_event. Ця функція не визначена та не викликається в Lua. Ми використовуємо її тільки для цілей пояснення).

Присвоєння до глобальної змінної  x = val еквівалентне до  присвоєння _ENV.x = val (див. §2.2).

3.3.4 - Структури управління

Структури управління if, while та repeat мають звичайне значення та знайомий синтаксис: 

	stat ::= while exp do block end
	stat ::= repeat block until exp
	stat ::= if exp then block {elseif exp then block} [else block] end

Lua також має твердження for, двох типів (див. §3.3.5).

Умовний вираз керівної структури може повертати любе значення. Обоє, false та nil, розпізнаються як false. Всі значення, інші від nil та false, визначені як true (зокрема, число 0 та пустий рядок є також true).

В циклі repeat-until внутрішній блок не завершується на ключовому слові until, але тільки після умови. Так що умова може посилатись на локальні змінні, визначені в блоку.

Твердження goto переньсить управління програмою на мітку. З синтаксичних міркувань, міткі в Lua також вважаються твердженнями: 

	stat ::= goto Name
	stat ::= label
	label ::= ‘::’ Name ‘::’

Мітка видима в цілому блоці, де вона визначена, за винятком вкладених блоків, де визначена мітка з тим же ім’ям, а також у вкладених функціях. Команда goto може перестрибнути до кожної видимої мітки, доки вона не входить до видимості локальної змінної.

Мітки та пусті твердження називаються порожніми твердженнями, оскільки вони не виконують дій.

Твердження break завершує виконання циклів while, repeat, або for, перестрибуючи до наступного твердження після циклу: 

	stat ::= break

break завершує самий внутрішній цикл.

Твердження return використовується для повертання значень з функції або клаптика (що є анонімною функцією). функції можуть повертати більше ніж одне значення, так що синтаксис твердження return є наступним  

	stat ::= return [explist] [‘;’]

Твердження return може бути тільки останнім твердженням в блоку. Якщо є дійсна потреба викликати return посередині блоку, тоді має бути використаний явний внутрішній блок, як в ідіомі do return end, оскільки тепер return є останнім твердженням в своєму (внутрішньому) блоці.

3.3.5 - Твердження for

Твердження for має дві форми: числову та загальну.

Числовий цикл for повторює блок коду, докі управляюча змінна перебігає значення арифметичної прогрессії. Він має наступний синтаксис: 

	stat ::= for Name ‘=’ exp ‘,’ exp [‘,’ exp] do block end

Block повторюється для name, починаючи зі значення першого exp, докі не дійде другого exp з кроком exp. Більш точно, твердження циклу for як

     for v = e1, e2, e3 do block end

еквівалентний до коду: 

     do
       local var, limit, step = tonumber(e1), tonumber(e2), tonumber(e3)
       if not (var and limit and step) then error() end
       var = var - step
       while true do
         var = var + step
         if (step >= 0 and var > limit) or (step < 0 and var < limit) then
           break
         end
         local v = var
         block
       end
     end

Зауважте наступне:

  • Всі три контрольні вирази обчислюються тільки один раз, перед початком циклу. Вони мають обчислюватись як числа.
  • var, limit та step є невидімими змінними. Імена, показані тут, наводяться тільки в цілях пояснення.
  • Якщо третій вираз (крок) відсутній, використовується крок 1.
  • Ви можете використовувати break та goto для виходу з циклу for.
  • Змінна циклу v є локальною до тіла циклу. Якщо вам потрібне її значення після циклу, присвоюйте її значення до іншої змінної перед виходу з цикла.

Загальне твердження циклу for робить зверх функцій, що називаються ітераторами. На кожній ітерації викликається функція ітератора для продукування нового значення, що завершується, коли це значення є nil. Загальне твердження циклу for має наступний синтаксис: 

	stat ::= for namelist in explist do block end
	namelist ::= Name {‘,’ Name}

Твердження for типу 

     for var_1, ···, var_n in explist do block end

еквівалентно коду: 

     do
       local f, s, var = explist
       while true do
         local var_1, ···, var_n = f(s, var)
         if var_1 == nil then break end
         var = var_1
         block
       end
     end

Зауважте наступне:

  • explist обчислюється тільки один раз. Це призводить до того, що функція iterator, state, та початкове значення для першої iterator variable встановлюються до входу в цикл.
  • f, s та var є невидимі змінні. Імена тут наведені тільки для цілей пояснення.
  • Ви можете перервати цикл використовуючи break.
  • Змінні циклу var_i є локальними відносно до циклу; ви не можете використовувати їх значення після завершення for. Якщо вам треба ці значення, присвоюйте їх перед виходом з циклу.

3.3.6 - Виклики функцій як твердження

Щоб дозволити можливі побічні ефекти, виклики функцій можуть бути виконані як твердження: 

	stat ::= functioncall

В цьому випадку всі повернуті значення відкидаються. Виклики функцій пояснюються в  §3.4.10.

3.3.7 - Локальні декларації

Локальні змінні можуть бути задекларовані будь-де в блоці. Декларація може вклюати початкове присвоювання: 

	stat ::= local namelist [‘=’ explist]

На данний момент початкове присвоювання має ту ж семантику, що і множинне присвоювання (див. §3.3.3). Інакше всі змінні ініціалізуються значенням nil.

Клаптик є також і блоком (див. §3.3.2), так що локальні змінні можуть бути задекларовані в клаптику за межами любого явного блоку.

Правила видимості локальних змінних пояснені в §3.5.

3.4 - Вирази

Базові вирази в Lua наступні: 

	exp ::= prefixexp
	exp ::= nil | false | true
	exp ::= Numeral
	exp ::= LiteralString
	exp ::= functiondef
	exp ::= tableconstructor
	exp ::= ‘...’
	exp ::= exp binop exp
	exp ::= unop exp
	prefixexp ::= var | functioncall | ‘(’ exp ‘)’

Нумерали та рядки-літерали пояснюються в §3.1; змінні пояснюються в §3.2; визначення функцій пояснюється в §3.4.11; виклики функцій пояснюються в §3.4.10; конструктори таблиць пояснюються в §3.4.9. Вирази змінної арності, що виглядають як три крапки ('...'), можуть використовуватись тільки напряму в відповідних функціях; вони пояснюються в §3.4.11.

Двомісні операції складаються з арифметичних операторів (див. §3.4.1), бітових операцій (див. §3.4.2), реляційних операцій (див. §3.4.4), логічних операцій (див. §3.4.5), та оператора конкатенації (див. §3.4.6). Унарні (одномісні) оператори складаються з унарного мінуса (див. §3.4.1), унарного бітового НІ (див. §3.4.2), та унарного логічного  not (див. §3.4.5), а також з унарного оператора довжини (див. §3.4.7).

Обоє, виклики функцій та вирази змінної арності можуть призвести до декількох аргументів. Якщо виклик функції використовується як твердження (див. §3.3.6), тоді повернутий єю список підлаштовується до довжини в нуль елементів, такими чином відкидаючи повернуті значення. Якщо вираз використовується як останній (або єдиний) елемент в списку виразів, тоді підлаштування не відбувається (за винятком, коли вираз заключений в дужки). В усіх інших контекстах Lua підлаштовує списко, що повертаєтсья, до одного елементу, або відкидаючи всі значення окрім першого, або додаючи одне значення nil, якщо немає інших значень.

Ось деякі приклади: 

     f()                -- підлаштовано до 0 результатів
     g(f(), x)          -- f() підлаштовано до 1 результату
     g(x, f())          -- g отримує x плюс всі результати від f()
     a,b,c = f(), x     -- f() підлаштована до 1 результату (c отримує nil)
     a,b = ...          -- a отримує перший параметр vararg, b отримує
                        -- другий (обоє, a та b, можуть отримати nil, якщо немає
                        -- відповідних параметрів vararg)
     
     a,b,c = x, f()     -- f() підлаштовується до 2 результатів
     a,b,c = f()        -- f() підлаштовується до 3 результатів
     return f()         -- повертає всі результати з f()
     return ...         -- повертає всі отримані параметри vararg
     return x,y,f()     -- повертає x, y, та ві результати з f()
     {f()}              -- створює список зі всіма результатами від f()
     {...}              -- створює список зі всіма параметрами vararg
     {f(), nil}         -- f() підлаштовується до 1 результату

Любі вирази, заключені в дужки, завжди повертають тільки одне значення. Таким чином, (f(x,y,z)) є завжди одним значенням, навіть якщо f повертає декілька значень. (Значення (f(x,y,z)) є першим значенням, що повертає f, або nil, якщо fне повертає жодних значень ).

3.4.1 - Аріфметичні операції

Lua підтримує наступні арифметичні оператори:

  • +: додавання
  • -: віднімання
  • *: множення
  • /: ділення з плаваючою точкою
  • //: ціле ділення
  • %: модуль
  • ^: піднесення в ступінь
  • -: унарний мінус

За винятком експоненти та ділення з плаваючою точкою, арифметичні оператори роблять наступним чином: якщо обоє операнди цілі, операція виконується з цілими, та результат буде цілим. Інакше, якщо обоє операнди є числами або рядками, що можуть бути перетворені на числа (див. §3.4.3), тоді вони будуть перетворені на числа з плаваючою точкою, операція виконується за звичайними правилами для арифметики з плаваючою точкою  (звичайно за стандартом IEEE 754), та результат буде з плаваючою точкою.

Експонента та плаваюче ділення (/) завжди конвертують свої операнди до плаваючох точки. Експонента використовує функцію ISO C pow, так що вона робить також і для нецілих експонент.

Ціле ділення (//) є діленням, що округлює в сторону мінус нескінченості, тобто менше від ділення операндів.

Модуль визначений як остаток від ділення, що округлює в сторону мінус нескінченості. 

В разі переповнення цілої арифметики, всі оператори огртаються кругом, відповідно до звичайних правил двокомпонентої арифметики. (Іншими словами, вони повертають унікальне представлення цілог, що є модулем по 264 до математичного результату).

3.4.2 - Бітові операції

Lua підтримує наступні тілові операції:

  • &: бітове ТА
  • |: бітове АБО
  • ~: бітове виключне АБО
  • >>: правий зсув
  • <<: лівий зсув
  • ~: унарне бітове НІ

Всі бітові операції перетворюють свої операнди на цілі (див. §3.4.3), оперують зі всіма бітами ціх цілих, та повертають ціле.

Обоє, правий та лівий зсув, заповлюють вакантні біти нулями. Негативні значення виконують зсув в зворотньому напрямку; сзув з абсолютним значенням, рівним або більшим до числа біт в цілому на виході дають нуль (всі біти висуваються).

3.4.3 - Обмеження та перетворення

Lua виконує деякі автоматичні перетворення між деякими типими та репрезентаціями під час виконання. Бітові операції завжди перетворюють плаваючі операнди до цілих. Експонента та плаваюче ділення завжди перетворюють цілі операнди на з плаваючою точкою. Всі інші операції, що застосовуються до змішаних чисел (цілих та з плаваючою точкою) перетворюють цілий операнд до плаваючого; це називається звичайним правилом. C API також перетворює обоє, цілі до плаваючіх, та плаваючі до цілих, коли це потрібно. Більше того, конкатенація сприймає числа в якосі аргументів, окрім рядків.

Lua також перетворює рядки до чисел, де очікується число.

В перетворенні цілих од плаваючих, якщо ціле число має точне значення як плаваюче, це і буде результатом. Інакше перетворення бере найближче меньше або більше репрезентативне значення.  Цей тип перетворення ніколи не схиблює.

Перетворення від плаваючеого до цілого перевіряє, чи плаваюче має точну репрезентацію як ціле (тобто, плаваюче має ціле значення, і воно знаходиться в межах репрезентацій цілого). Якщо це має місце, репрезентація буде результатом. Інакше перетворення схибить.

Перетворення з рядків до чисел перебігає наступним чином: Спочатку рядок перетворюється на ціле або плаваюче, слідуя синтаксису та правилам лексера Lua. (Рядок може мати передуючі або або слідуючі проміжки та знак). Потім отримане число (плаваюче або ціле) перетворюється до типу (плаваючого або цілого), потрібного в контексті (тобто для операції, як примушує до перетворення).

Перетворення з чисел до рядків використовує незаданий формат, що сприймається людиною. Для повного контролю над тим, як числа перетворюються в рядки використовуйте функцію format з бібліотеки рядків (див. string.format).

3.4.4 - Операції відношень

Lua підтримує наступні операції відношень:

  • ==: рівність
  • ~=: нерівність
  • <: менше ніж
  • >: більше ніж
  • <=: меньше або рівне
  • >=: більше або рівне

Ці оператори завжди продукують false або true.

Рівність (==) спершу порівнює типи операндів. Якщо типи різні, тоді результат є false. Інакше порівнюються значення операндів. Рядки порівнюються очевидним шляхом. Число рівні, якщо вони позначають одне і те ж математичне значення.

Таблиці, користувацькі дані та потоки порівнюються за посиланням: для об’єкти визнаються рівними, тільки якщо вони є тим же самим об’єктом. Кожного разу, коли ви створюєте новий об’єкт (таблицю, користувацькі дані або потік), цей новий об’єкт відмінний від любого попередньо існуючого об’єкту. Замикання з тим же посиланням завжди рівні. Замикання з визначенною різницею (інша поведінка, інше визначення) завжди відмінні.

Ви можете змінювати спосіб, яким Lua порівнює таблиці та користувацькі дані, виконуючі метаметод "eq" (див. §2.4).

Перевірка рівності не перетворює рядки до чисел, та навпаки. таким чином, "0"==0 ябчислюється як false, t[0] та t["0"] вказують на різні елементи таблиці.

Оператор ~= є прямою протилежністю рівності (==).

Порядок операторів працює наступним чином. Якщо обоє аргументів є числа, тоді вони порівнюються згідно до їх математичних значень (не зважаючи на підтип). Інакше, якщо обоє операндів є числа, вони порівнюються згідно до поточної локалі. Інакше Lua намагається викликати метаметод "lt" або "le" (див. §2.4). Порівняння a > b перетворюється на b < a та a >= b перетворюється на b <= a.

Слідуя стандарту IEEE 754, NaN вважається ні меншим, ні рівним, ні більшим до жодного значення (включаючи себе).

3.4.5 - Логічні оператори

Логічні оператори в Lua є and, or та not. Як контрольні структури, (див. §3.3.4), всі логічні оператори вважають обоє, false та nil як НІ, та все інше як ТАК.

Операція негації not завжди повертає false або true. Операнд кон’юкції and повертає перший аргумент, якщо це значення false або nil; інакше and повертає свій другий операнд. Операнд дез’юкції or повертає свій перший аргумент, якщо його значення відмінне від nil та false; інакше or повертає свій другий аргумент. Обоє and та or використовують коретке обчислення; тобто другий операнд обчислюється тільки в випадку необхідності. Осі деякі приклади: 

     10 or 20            --> 10
     10 or error()       --> 10
     nil or "a"          --> "a"
     nil and 10          --> nil
     false and error()   --> false
     false and nil       --> false
     false or nil        --> nil
     10 and 20           --> 20

(В цьому документі --> вказує на результат попередніх обчислень).

3.4.6 - Конкатенація

Оператор конкатенації рядків в Lua позначається довма крапками ('..'). Якщо обоє операндів є числами або рядками, тоді вони перетвороюються на рядки, згідно з описаним в §3.4.3. Інакше викликається метаметод  __concat (див. §2.4).

3.4.7 - Оператор довжини

Опеартор довжини позначається унарним префіксним оператором #. Довжина рядка є число його байтів (тобто, звичане значення довжини рядка, коли кожний символ є одним байтом).

Програма може можифікувати поведінку оператора довжини для кожного значення крім рядків через метаметод  __len (див. §2.4).

Якщо метаметод __len не наданий, довжина таблиці t визначена тільки в разі, коли таблиця є послідовністю, тобто набір позитивних числових ключів лежить еквівалента до {1..n} для деякого невід’ємного цілого n. В цьому випадку n є її довжиною. Зауважте, що таблиця типу 

     {10, 20, nil, 40}

не є послідовністю, оскільки вона має ключ 4, але не має ключа 3. (Таким чином, немає такого n, щоб множина  {1..n} була множиною позитивних числових ключів для такої таблиці). Зауважте, однак, що нечислові ключі не впливають на те, чи є таблиця послідовністю.

3.4.8 - Порядок обчислень

Порядок обчислень в Lua слідує наведеній нижче таблиці, від нижнього до верхнього пріорітету: 

     or
     and
     <     >     <=    >=    ~=    ==
     |
     ~
     &
     <<    >>
     ..
     +     -
     *     /     //    %
     унарні оператори (not   #     -     ~)
     ^

Як звичайно, ви можете використовувати дужки для зміни порядку обчислення виразу. Конкатенація ('..') та експонента ('^') є право-асоциативними. Всі інші двомісні операції є ліво-асоциативними.

3.4.9 - Конструктори таблиць

Конструктори таблиць - це вирази, що створюють таблиці. Кожного разу, коли обчислюється конструктор, створюється нова таблиця. Конструктор може бути використаний для створення пустої таблиці та ініціалізації деяких полів. Главний синтаксис для конструкторів наступний

	tableconstructor ::= ‘{’ [fieldlist] ‘}’
	fieldlist ::= field {fieldsep field} [fieldsep]
	field ::= ‘[’ exp ‘]’ ‘=’ exp | Name ‘=’ exp | exp
	fieldsep ::= ‘,’ | ‘;’

Кожне поле в формі [exp1] = exp2 додає новий елемент з ключем exp1 та значенням exp2. Поле в формі name = exp єквівалнтно до ["name"] = exp. Нарешті, поля в формі exp еквівалентні до [i] = exp, де i є послідовні цілі значення, починаючи з 1. Поля в інших форматах не впливають на цей перелік. Наприклад, 

     a = { [f(1)] = g; "x", "y"; x = 1, f(x), [30] = 23; 45 }

еквівалентно до 

     do
       local t = {}
       t[f(1)] = g
       t[1] = "x"         -- 1st exp
       t[2] = "y"         -- 2nd exp
       t.x = 1            -- t["x"] = 1
       t[3] = f(x)        -- 3rd exp
       t[30] = 23
       t[4] = 45          -- 4th exp
       a = t
     end

Порядок присвоєнь в конструкторі невизначений. (Цей порядок буде відповідним, тільки коли є повторювані ключі).

Якщо останнє поле в списку має форму exp, та вираз є викликом функції, або виразом змінної арності, тоді всі значення, що повертаються виразом, будуть послідовно додані до списку (див. §3.4.10).

Список полів може мати опціональний завершальний роздільник, як спрощення для згенерованого машиною кода.

3.4.10 - Виклики функцій

Виклик функції в Lua має такий синтаксис: 

	functioncall ::= prefixexp args

В виклику функції перший prefixexp та args обчислюються. Якщо значення prefixexp має тип function, тоді ця функція викликається з переданими аргументами. Інакше викликається метаметод "call", що має першим параметром значення prefixexp, за яким слідує аргументи, надані при виклику (див. §2.4).

Форма

	functioncall ::= prefixexp ‘:’ Name args

може бути використана для виклику "методів". Виклик v:name(args)є синтаксичним цукором для v.name(v,args), за винятком того, що v обчислюється тільки один раз.

Аргументи мають наступний синтаксис: 

	args ::= ‘(’ [explist] ‘)’
	args ::= tableconstructor
	args ::= LiteralString

Всі вирази аргументів обчислюються перед викликом. Виклик в формі f{fields} є синтаксичним цукором для  f({fields}); тобто, список аргументів є однією новою таблицею. Виклик в формі f'string' (або f"string", або f[[string]]) є синтаксичним цукором для f('string'); тобто, аргумент функції є один літеральний рядок.

Виклик в формі return functioncall називається хвостовим викликом. Lua реалізує відповідні хвостові виклики (або відповідну хвостову рекурсію): в хвостовому виклику функція, що викликається, використує елемент стеку викликаючої функції. Таким чином, немає обмеження на число вкладених хвостових викликів, що може виконати програма. Однак хвостовий виклик знищує любу інформацію налаштування щодо викликаючої функції. Зауважте, що хвостовий виклик відбувається тільки за певного синтаксису, де return має один виклик функції в якості аргументу; цей синтаксис робить так, що викликаюча функція повертає рівно те, що повертає викликана функція. Таким чином, жодний з наступних прикладів не є хвостовими викликами: 

     return (f(x))        -- результати обмежені до одного
     return 2 * f(x)
     return x, f(x)       -- додаткові результати
     f(x); return         -- результати відкинуті
     return x or f(x)     -- результати обмежені до одного

3.4.11 - Визначення функцій

Синтаксис для визначення функцій

	functiondef ::= function funcbody
	funcbody ::= ‘(’ [parlist] ‘)’ block end

Наступний синтаксичний цукор спрощує визначення функцій: 

	stat ::= function funcname funcbody
	stat ::= local function Name funcbody
	funcname ::= Name {‘.’ Name} [‘:’ Name]

Вираз

     function f () body end

транслюється в 

     f = function () body end

Вираз 

     function t.a.b.c.f () body end

транслюється в 

     t.a.b.c.f = function () body end

Вираз

     local function f () body end

транслюється в 

     local f; f = function () body end

не в таке

     local f = function () body end

( Це має значення тільки в випадку, коли тіло функції містить посилання на f).

Визначення функції є виконуваним виразом, чиє значення моє тип function. Коли Lua попередньо компілює клаптик, всі тела функцій також передкомпілюються. Тоді, коли Lua виконує визначення функції, функція інстанцінується (створюється її екземпляр) (або замикається). Цей екземпляр функції  (або замикання) є остаточним значенням виразу.

Параметри діються як локальні змінні, що ініціалізовані значення аргументів: 

	parlist ::= namelist [‘,’ ‘...’] | ‘...’

Коли функція викликається, список вргументів підлаштовується до довжини списку параметрів, за винятком коли функція змінної арності, що позначається трьома крапками ('...') в кінці списку параметрів. функція змінної арності не підлаштовує свої аргументи; замість цього вона збирає додаткові аргументи, то надає їх в функцію через змінний вираз, що також записується як три крапки. Значення виразу є список всіх справжніх додаткових аргументів, подібно до функції з декількома результатами. Якщо мульти-арний вираз використовується в іншому виразі або посередині списку виразів, тоді він повертає список, обмежений до одного елементу. Якщо вираз використовується як останній елемент в списку виразів, тоді підлаштування не відбувається (за винятком коли вираз заточений в дужки).

В якості прикладу подивітьсяна наступні визначення: 

     function f(a, b) end
     function g(a, b, ...) end
     function r() return 1,2,3 end

Потім ми робимо відображення аргументів на параметри та вираз мульті-арності: 

     ВИКЛИК           ПАРАМЕТРИ
     
     f(3)             a=3, b=nil
     f(3, 4)          a=3, b=4
     f(3, 4, 5)       a=3, b=4
     f(r(), 10)       a=1, b=10
     f(r())           a=1, b=2
     
     g(3)             a=3, b=nil, ... -->  (нічого)
     g(3, 4)          a=3, b=4,   ... -->  (нічого)
     g(3, 4, 5, 8)    a=3, b=4,   ... -->  5  8
     g(5, r())        a=5, b=1,   ... -->  2  3

Результати повертаються з використанням твердження return (див. §3.3.4). Якщо керування досягає кінця функції без того, щоб зустріти твердження return, тоді функція повертається без результату.

Є системно-залежне обмеження на число значень, що може повертати функція. Це обмеження гарантовано є більшим за 1000.

Синтаксис двох крапок використовується для визначення методів, тобто функцій, що мають додатковий неявний параметр self. Таким чином твердження

     function t.a.b.c:f (params) body end

є синтаксичним цукором для 

     t.a.b.c.f = function (self, params) body end

3.5 - Правила видимості

Lua є мовою із лексичним полем зору. Поле зору локальної змінної починається на першому твордженні після декларації, та втрачається після непустого твердження найбільш внутрішнього блоку, що містить визначення. Розглянемо наступний приклад: 

     x = 10                -- глобальна змінна
     do                    -- новий блок
       local x = x         -- нова змінна 'x', зі значенням 10
       print(x)            --> 10
       x = x+1
       do                  -- інший блок
         local x = x+1     -- інше 'x'
         print(x)          --> 12
       end
       print(x)            --> 11
     end
     print(x)              --> 10  (глобальна змінна)

Зауважте, що в декларації як local x = x, новий x, що декларується, ще не є в в полі зору, та друге x посилаєтсья на зовнішню змінну.

Завдяки правилам лексичного поля зору, локальні змінні можуть бути легко досяжні в функціях того ж поля зору. Локальна змінні, використана у внутрішній функції, у внутрішній функції називається upvalue, або зовнішня локальна змінна.

Зауважте, що кожне виконання твердження local визначає нову локальну змінну. розгляньте наступний приклад: 

     a = {}
     local x = 20
     for i=1,10 do
       local y = 0
       a[i] = function () y=y+1; return x+y end
     end

Цикл утворює десять замикань (тобто, десять екземплярів анонімної функції). Кожне з ціх замикань використовує іншу змінну y, тоді як всі вони поділяють одну x.

4 - Програмний інтерфейс застосунку

Цей розділ описує C API для Lua, тобто, набір функцій C, доступних для головної програми для взаємодії з Lua. Всі функції API та пов’язані типи та константи визначені в файлі заголовку lua.h.

Навіть коли ми використовуєм термін "function", кожна можливість в API може провадитись замість цього як макро. Коли не вказано інакше, всі макроси використовують кожний зі своїх аргументів тільки один раз (за винятком першого аргументу, що завжди є станом Lua), і, таким чином, не генерують жодних прихованих побічних ефектів.

Як в більшості C бібліотек, функції Lua API не перевіряють свої аргументи на допустимість або сумісність. Однак ви можете змінити цю поведінку, скомпільовавши Lua з визначеним макро LUA_USE_APICHECK.

4.1 - Стек

Lua використовує віртуальний стек для передачі значень до та від C. Кожний елемент в цьому стеку представляє значення Lua (nil, число, рядок, etc.).

Коли Lua викликає C, викликана функція отримує новий стек, що незалежний від попередніх стеків, та від стеків функцій C, що ще активні. Цей стек зпочатку містить всі вргументи для функції C, та це місце, де функція C заштовхує результати, що мають повернутись до викликаючого (див. lua_CFunction).

Для зручності більшисть операцій запитів в API не слідують суворій дисципліні стека. Замість цього вони можуть посилатись на любий елемент стеку, використовуючи індекс: позитивний індекс представляє абсолютну позиція в стеку (починаючи з 1); негативний індекс представляє зміщення відносно гори стеку. Більш конкретно, якщо стек має n елементів , тоді індекс 1 представляє перший елемент (тобто, елемент, що був перший заштовхнутий в стек), та індекс n представляє останній елемент; індекс -1 також представляє останній елемент (тобто, елемент на горі), та індекс -n представляє перший елемент.

4.2 - Розмір стеку

Коли ви взаємодієте з Lua API, ви відповідаєте за гарантію відповідності. Зокрема, ви відповідні за контроль переповнення стеку. Ви можете використовувати функцію lua_checkstack, щоб впевнитись, що стек досить просторий для заштовхування нових елементів.

Коли Lua викликає C, він переконується, що стек має простір для щонайменше LUA_MINSTACK додаткових слотів. LUA_MINSTACK визначене як 20, і, таким чином, звичайно вам не треба турбуватись про розмір стеку, окрім випадка, коли ваш код заштовхує елементи в стек в циклі.

Коли ви викликаєте функцію Lua без фіксованого числа результатів (див. lua_call), Lua впевняється, що стек має досить результатів для всіх результатів, але він не гарантує, що буде додаткове місце. Так що, перед заштовхуваням будь чого після такого виклику, ви повинні використати  lua_checkstack.

4.3 - Допустимі та прийнятні індекси

Кожна функція в API, що отримує індекси стеку, робить тільки з допустимими індексами або прийнятними індексами.

Допустимий (valid) індекс є індексом, що посилаєтся на позицію, що зберігає модифікуєме значення Lua. Це включає індекси стеку між 1 та верхівкою стеку (1 ≤ abs(index) ≤ top) плюс псевдо-індекси, що представляють деякі позиції, що доступні в коді C, але якіх немає в стеку. Псевдо-індекси використовуються для доступу до реєстру (див. §4.5) та зовнішніх локальних (upvalues) значень функцій C (див. §4.4).

Функції, що не потребують специфічної змінної позиції, але тільки значення (тобто, функції-запити), можуть бути викликані з прийнятними (acceptable). прийнятний інедкс може бути любим доступним індексом, та також може бути любим позитивним інексом після верхівки стеку в просторі, виділеному для стеку, тобто індекси за межами розміру стеку. (Зауважте, що 0 ніколи не є прийнятним індексом). За винятком коли вказане інше, фіункції в API роблять з прийнятними індексами.

Прийнятні індекси прислуговуються для запобігання додаткових перевірок з верхівкою стека, коли іде звертання до стеку. Наприклад, функнція C може опитувати свій третій аргумент без потреби попередньої перевірки, чи є 3 допустимим індексом.

Для функцій, що можуть бути викликані з прийнятними індексами, любий недопустимий індекс трактується так, ніби він містить значення віртуального типу LUA_TNONE, що поводиться як значення nil.

4.4 - Замикання C

Коли створюється функція C, можливо асоціювати деякі значення з нею, тобто, створювати замикання C (див. lua_pushcclosure); ці значення називаються значеннями замикання (upvalues) (або змінними замикання, або замкнутими змінними, прим. перекладача) та доступні в функції, коли вона викликається.

Коли функція C викликаєтся, ії значення замикання розміщені по специфічних псевдо-індексах. Ці псевдо-індекси продукуються макро lua_upvalueindex. Перше значення замикання, ініційоване з функцією, розташоване по індексу lua_upvalueindex(1), і так далі . Кожний доступ до lua_upvalueindex(n), де n більше, ніж число значень замикання для поточної функції (але не більше, ніж 256), продукує прийнятний, але недопустимий індекс.

4.5 - Реєстр

Lua провадить реєстр, попередньо визначену таблицю, що може використовуватись любим кодом C для зберігання любих значень Lua, які вона бажає зберігти. Таблиці реєстру завжди розташована по псевдо індексу LUA_REGISTRYINDEX. Кожна бібліотека може зберігати дані в цій таблиці, але вона має пілкуватись, та обирати індекси, що відрізняються від тих, що вже використовуються в інших бібліотеках, щоб уникнути колізій. Типово, вам потрібно використовувати ключі з рядків, що містять ім’я вашої бібліотеки, або легкі дані користувача, з адресою C об’єкту в вашому коді, або любий об’єкт Lua, створений в вашому коді. Як з іменами бібліотек, рядкі ключів, що починаються з підкреслення, за яким слідує велика літера, зарезервовані для Lua.

Числові ключі в реєстрі використовуються як механізм посилань (див. luaL_ref) та деякими попередньо визначеними значеннями.  Таким чином, цілі ключі не мають бути використані для інших цілей.

Коли ви створюєте новий стан Lua, його реєстр іде з деякими попередньо визначеними значеннями. Ці значення індексовані цілими ключами, та визначені як константи в lua.h. Визначені наступні константи:

  • LUA_RIDX_MAINTHREAD: По цьому індексу реєстр має головний потік стану. (Головний потік це той, що створений разом зі станом).
  • LUA_RIDX_GLOBALS: По цьому індексу реєстр має глобальне оточення.

4.6 - Обробка помилок в C

Внутрішньо Lua використовує можливості C longjmp для обробки помилок. (Lua буде використовувати виключення, якщо ви скомпілюєте його як C++; шукайте LUAI_THROW в початковому коді для отримання деталей). Коли Lua стикається з помилкою (такою, як помилка розміщення пам’яті, синтаксичні помилки, та помилки часу виконання), підіймається помилка; тобто виконується далекій перехід. Захищенне оточення використовує setjmp для встановлення точки відновлення; кожна помилка перестрибує на останню встановлену точку відновлення.

Якщо це відбувається за межами жодного захищеного оточення, Lua викликає функцію паніки (див. lua_atpanic), та потім викликає abort, таким чином виходячи з зовнішнього застосування. Ваша функція паніки може уникнути цього виходу, якщо ніколи не поверне керування (тобто, зробить довгий перехід на вашу власну точку відновлення за межами Lua).

функція паніки виконується так, якби це був обробник повідомлення (див. §2.3); зокрема, повідомлення помилки лежить на верхівці стека. Однак немає гарантій щодо стекового простору. Щоб заштовхати будь-що в стек, функція паніки мусить спочатку перевірити доступний простір (див. §4.2).

Більшість функцій API можуть викликати помилки, наприклад, через помилку розміщення пам’яті. Документація для кожної функції вказує, чи викликає вона помилки.

Зсередини функції C ви можете викликати помилку, викликаючи lua_error.

4.7 - Оброблення передачі керування в C

Зсередини Lua використовує можливість C longjmp для надання керування з копроцедури. Таким чином, якщо функція C foo викликає функцію API, та ця функція API поступається керуванням (прямо або непрямо, викликаючи іншу функцію, що поступається керуванням), Lua більше не може повернутися до foo, оскільки  longjmp видаляє його фрейм зі стеку C.

Щоб уникнути цей тип проблем, Lua викликає помилку, коли хтось намагається надати керування через виклик API, за винятком трьох функцій: lua_yieldk, lua_callk та lua_pcallk. Всі ці функції отримують функцію продовження (як параметр на ім’я k), щоб продовжити виконання після надання керування.

Нам треба встановити деяку термінологію для пояснення подовження. Ми маємо функцію C, що викликається з Lua, яку ми будемо називати оригінальною функцією. Ця оригінальна функція потім викликає одну з ціх функцій з C API, що ми будемо називати функція що викликається, та потім вона поступається поточним потоком. (Це може трапитись, коли функція що викликається є lua_yieldk, або коли функція, що викликається, є або lua_callk або lua_pcallk, та функція, що викликається ними, віддає керування).

Уявіть, що робочий потік поступається керуванням під час виконання викликаною функції. Після того, як потік відновлюється, він з часом дійде кінця викликаної функції. Однак, викликана функція не може повернутися до оригінальної функції, оскільки її фрейм в C стеку був зруйновнаий під час віддачі керування. Замість цього, Lua викликає функцію продовження, яка була отримана як аргумент до викликаної функції. Як пдіказує ім’я, функція продовження повинна продовжити завдання оригінальної функції.

В якості ілюстрації подивимось на наступну функцію: 

     int original_function (lua_State *L) {
       ...     /* код 1 */
       status = lua_pcall(L, n, m, h);  /* викликається Lua */
       ...     /* код 2 */
     }

Тепер ми ми бажаємо дозволити коду Lua, що викликаний в lua_pcall, віддати керування. Перше, ми можемо переписати нашу функцію подібним чином: 

     int k (lua_State *L, int status, lua_KContext ctx) {
       ...  /* код 2 */
     }
     
     int original_function (lua_State *L) {
       ...     /* код 1 */
       return k(L, lua_pcall(L, n, m, h), ctx);
     }

В коді вище з’явилась нова функціий k, що є функцією продовження (з типом lua_KFunction), що має робити все те ж, що робить оригінальна функція після виклику lua_pcall. Тепер ми маємо поінформувати Lua, що він має викликати k, якщо код Lua, що виконується lua_pcall, буде перервано в деякий спосіб (помилка або добровільна передача), так що ми переписуємо код як показано тут, замінюючи lua_pcall на lua_pcallk: 

     int original_function (lua_State *L) {
       ...     /* код 1 */
       return k(L, lua_pcallk(L, n, m, h, ctx2, k), ctx1);
     }

Зауважте на зовнішній, явний виклик до продовження: Lua буде викликати продовження тільки в коли треба, тобто, в випадку помилок або відновлення після передачі керування. Якщо викликана функція повертаєтсья нормально, без навіть передачі, lua_pcallk (та lua_callk) також повернуться звичайним чином. (Звичайно, замість виклику продовження в цьому разі, ви можете виконати аналогічну роботу напряму, в оригінальній функції).

Окрім стану Lua, функція продовження має два інших параметрів: фінальний статус виклику, плюс значення контексту (ctx), що було передано в оригіналі до lua_pcallk. (Lua не використовує це значення контексту; він тільки передає це значення від оригінальної функції до функції продовження). Для lua_pcallk, статус таке саме значення, що повинно повертатись з lua_pcallk, за винятком того, що це LUA_YIELD, коли виконується після передачі (замість LUA_OK). Для lua_yieldk та lua_callk, статус завжди LUA_YIELD, коли Lua викликає продовження. (Для ціх трьох функцій Lua не буде викликати продовження в випадку, коли помилок, оскільки вони не оброблюють помилок). Подібно, коли використовуєте lua_callk, вам треба викликати функцію продовження з LUA_OK в якості статусу. (Для lua_yieldk, немає великого сенсу в виклику функції продовження напряму, оскільки lua_yieldk звичайно не повертається).

Lua трактує функцію продовження якби це була оригінальна функція. Функція продовження отримує той же самий стек Lua від оригінальної функції, в тому ж стані, в якою викликана функція повернулась. (Наприклад, після lua_callk функція та її аргументи видаляються зі стеку, та замінюються результатами з виклику). Він також має ті ж замкнені значення. Що вона там не повертає, оброблюється Lua так, якби це було повернення з оригінальної функції.

4.8 - Функції та типи

Ось весь список функцій та типів з C API в алфавітному порядку. Кожна функція має індикатор, подібний до наступного: [-o, +p, x]

Перше поле, o, є тим, як багато елементів функція виштовхує зі стеку. Друге поле, p, є склькі багато елементів функція заштовхує в стек. (Кожна функція завжди заштовхує свої результати після того, як отримує звідти свої аргументи). Поле в формі x|y означає функції, що можуть заштовхувати (або виштовхувати) x або y елементів, залежно від ситуації; знак запитання '?' позначає, що ми не можемо знати, як багато елементів функція заштовхує та виштовхує, дивлячить тільки на її аргументи (тобто, пони можуть залежати від того, що вони мають в стеці). Третє поле, x, каже, чи може функція викликати помилки : '-' означає, що функція нікооли не викликає помилок; 'e' ожначає, що функція може створювати помилки; 'v' означає, що функція може викликати помилки по бажанню.

lua_Alloc

typedef void * (*lua_Alloc) (void *ud,
                             void *ptr,
                             size_t osize,
                             size_t nsize);

Тип функції розташування пам’яті, що використовується станами Lua. Функція розміщення має провадити функціональність, подібну до realloc, але не обов’язково однакову. Її аргуменит є ud, прозорий вказівник, що передається до lua_newstate; ptr, вказівник на блок, що розміщений/перерозміщений/звільнений; osize, оригінальний розмір блоку, або деякий код шодо того, що було розміщено; та nsize, новий розмір блоку.

Коли ptr не є NULL, osize є розміром блоку, на який вказує ptr, тобто розмір, наданий коли він був розміщений або перерозміщений.

Коли ptr є NULL, osize кодує тип об’єкту, що розміщує Lua. osize є одним з LUA_TSTRING, LUA_TTABLE, LUA_TFUNCTION, LUA_TUSERDATA, або  LUA_TTHREAD, тоді (та тільки тоді), коли Lua створює новий об’єкт цього типу. Коли osize є якімось іншим значенням, Lua розміщує пам’ять для чогось іншого.

Lua очікує наступну поведінку від функції розміщення:

Коли nsize є нуль, розміщення мусить поводитись як freeна повертати  NULL.

Коли nsize не нуль, розміщення мусить поводитись як realloc. Розміщення повертає NULL, якщо, та тільки в разі, коли вона не може задовільнити запит. Lua очікує, що розміщення ніколи не схибить, коли osize >= nsize.

Ось проста реалізація для функції розміщення. Вона використовується в зовнішній бібліотеці в  luaL_newstate. 

     static void *l_alloc (void *ud, void *ptr, size_t osize,
                                                size_t nsize) {
       (void)ud;  (void)osize;  /* не використовується */
       if (nsize == 0) {
         free(ptr);
         return NULL;
       }
       else
         return realloc(ptr, nsize);
     }

Зауважте, що Standard C гарантує, що free(NULL) не має жодного ефекту, та що realloc(NULL,size) є еквівалентом до malloc(size). Цей код очікує, що realloc не схибить при стисканні блоку. (Хоча Standard C не гарантує цю поведінку, здається, що це досить беспечне припущення).

lua_call

[-(nargs+1), +nresults, e] 

void lua_call (lua_State *L, int nargs, int nresults);

Викликає функцію.

Для виклику функції ви повинні використовувати настпний протокол: спершу, функція, що має бути викликана, заштовхується в стек; тоді, аргументи функції заштовхуються в прямому порядку; тобто, перший аргумент заштовхується першим. Нарешті, ви викликаєте lua_call; nargs є числом аргументів, що заштовхнуті в стек. Всі аргументи та значення функції виштовхуються зі стеку, коли функція викликана. Результати функції заштовхуються в стек при поверненні з функції. Число результатів підлаштовується до nresults, якщо nresults не є LUA_MULTRET. В цьому випадку всі результати функції заштовхуються один по одному. Lua пілкується, щоб значення, що повертаються, пасували до розміру простору стеку. Результати функції заштовхуються в стек в прямому порядку (перший результат заштовхується першим), так що після виклику останній результат знаходиться на горі стеку.

Любі помилки зсередини викликаної функції просуваються нагору (зо допомогою longjmp).

Наступний приклад показує, як зовнішня програма може виконати еквівалент до коду Lua: 

     a = f("how", t.x, 14)

А це в C: 

     lua_getglobal(L, "f");                  /* функція, що буде викликана */
     lua_pushliteral(L, "how");                       /* 1ший аргумент */
     lua_getglobal(L, "t");                    /* таблиця, що буде індексована */
     lua_getfield(L, -1, "x");        /* заштовхнути результат t.x (2гий аргумент) */
     lua_remove(L, -2);                  /* видалення 't' зі стеку */
     lua_pushinteger(L, 14);                          /* 3тій аргумент */
     lua_call(L, 3, 1);     /* викликати 'f' з трьома аргументами та 1 результатом */
     lua_setglobal(L, "a");                         /* встановити глобальну 'a' */

Зауважте, що код вище є збалансованим: в кінці стек повертається до оригінальної конфігурації. Це вважається гарною практикою програмування.

lua_CFunction

typedef int (*lua_CFunction) (lua_State *L);

Тип функцій C.

Щоб комунікувати відповідно з Lua, функція C має використовувати наступний протокол, що визначає шлях, яким передаються переметри та результати: функція C отримує свої аргументи від Lua в своєму стеку в прямому порядку (перший аргумент заштовхується першим). Таким чином, коли функція починається,  lua_gettop(L) повертає число аргументів, отриманих функцією. Перший аргумент (якщо він є) з індексом 1, та її останній аргумент з індексом lua_gettop(L). Для повертання значень до Lua, функція C тільки заштовхує їх в стек, в прямому порядку (перший результат заштовхується першим), та повертає число результатів. Кожне інше значення в стеку нижче результатів буде відповідним чином відкинуте Lua. Подібно до функції Lua, функція C, викликана з Lua також може мати декілька результатів.

Як приклад, наступна функція отримує змінне число числових аргументів, та повертає їх середнє значення та суму: 

     static int foo (lua_State *L) {
       int n = lua_gettop(L);    /* число аргументів */
       lua_Number sum = 0.0;
       int i;
       for (i = 1; i <= n; i++) {
         if (!lua_isnumber(L, i)) {
           lua_pushliteral(L, "incorrect argument");
           lua_error(L);
         }
         sum += lua_tonumber(L, i);
       }
       lua_pushnumber(L, sum/n);        /* перший результат */
       lua_pushnumber(L, sum);         /* другий результат */
       return 2;                   /* число результатів */
     }

Завантажує клаптик Lua без його виконання. Якщо немає помилок, lua_load заштовхує скомпільований клаптик як функцію Lua нагору стеку. Інакше туди заштовхується повідомлення про помилку.

Значення, що повертає lua_load є наступними:

  • LUA_OK: немає помилок;

Аргумент chunkname задає ім’я клаптика, що використовується для повідомлень помилок, та в інформації налаштування (див. §4.9).

lua_load автоматично розпізнає, чи є клаптик текстом чи двійниковим, та завантажує його відповідно (див. програму luac). Рядок mode робить як в функції load, з додаванням того, що значення NULL еквівалентне до  рядка "bt".

lua_load внутрішньо використовує стек, так що фукнція читання мусить завжди залишати стек незмінним, коли повертає керування.

Якщо результуюча функція має замкнені змінні, її перше замкнуте значення встановлюється в значення глобального оточення, що зберігається за індексом LUA_RIDX_GLOBALS в реєстрі (див. §4.5). Коли завантажується головні клаптики, це значення буде змінною _ENV (див. §2.2). Ініші замкнені змінні будуть мати значення nil.

              lua_typename(L, lua_type(L, -1)));
       /* видаляє 'value'; утримує 'key' для наступної ітерації */
       lua_pop(L, 1);

Обоє, nargs та nresults мають однакове значення, як в lua_call. Якщо не виникає помилок на протязі виклику, lua_pcall поводиться точно так же, як і lua_call. Однак, якщо виникає будь-яка помилка, lua_pcall перехоплює її, заштовхує єдине значення в стек (повідомлення помилки), та повертає код помилки. Подібно до  lua_call, lua_pcall завжди видаляє функцію та аргументи зі стека.

Якщо msgh є 0, тоді помилка, що повертається в стеку, є точно оригінальним повідомленням про помилку.  Інакше msgh є індексом стеку для обробника повідомлення. (Цей індекс не може бути псевдо-індексом). В випадку помилки часу виконання, ця функція буде викликана з  повідомленням помилки, та повернуте єю значення буде повідомленням, що повернуте в стеку від lua_pcall.

Звичайно, обробник повідомлення використовується для додавання додаткової інформації налаштування до повідомлення помилки, такого, як трасса стеку. така інформація не може бути зібрана після повернення з lua_pcall, оскільки тоді стек вже буде відкручений назад.

Функція lua_pcall повертає одну з наступних констант (визначених в lua.h):

  • LUA_OK (0): успіх.
  • LUA_ERRRUN: помилка часу виконання.
  • LUA_ERRMEM: помилка виділення паам’яті. Для таких помилок Lua не викликає обробника повідомлень.
  • LUA_ERRERR: помилка при виконанні обробника повідомлення.
  • LUA_ERRGCMM: помилка при виконані метаметода __gc. (Ця помилка типово не має відношення до функції, що викликається).

Однак, є один особливий випадок, коли ця функція викликається всередині перехоплювача рядка (див. §4.9). В цьому випадку lua_yieldk повинна викликатись без продовження (можливо, в формі lua_yield), та перехоплення повинно повернутись безпосередньо після виклику. Lua передасть управління, та коли копроцедура відновиться знову, вона буде продовжувати нормальне виконання (Lua) функції, що перемкнула перехоплення.

Ця функція може викликати помилку, якщо вона викликається з потоку з підвислим викликом C без фукнції продовження, або вона викликана з потоку, що не робить в відновленні (тобто, з головного потоку).

4.9 - Інтерфейс налаштування

Lua has no built-in debugging facilities. Instead, it offers a special interface by means of functions and hooks. This interface allows the construction of different kinds of debuggers, profilers, and other tools that need "inside information" from the interpreter.

lua_Debug

typedef struct lua_Debug {
  int event;
  const char *name;           /* (n) */
  const char *namewhat;       /* (n) */
  const char *what;           /* (S) */
  const char *source;         /* (S) */
  int currentline;            /* (l) */
  int linedefined;            /* (S) */
  int lastlinedefined;        /* (S) */
  unsigned char nups;         /* (u) number of upvalues */
  unsigned char nparams;      /* (u) number of parameters */
  char isvararg;              /* (u) */
  char istailcall;            /* (t) */
  char short_src[LUA_IDSIZE]; /* (S) */
  /* private part */
  other fields
} lua_Debug;

A structure used to carry different pieces of information about a function or an activation record. lua_getstack fills only the private part of this structure, for later use. To fill the other fields of lua_Debug with useful information, call lua_getinfo.

The fields of lua_Debug have the following meaning:

  • source: the name of the chunk that created the function. If source starts with a '@', it means that the function was defined in a file where the file name follows the '@'. If source starts with a '=', the remainder of its contents describe the source in a user-dependent manner. Otherwise, the function was defined in a string where source is that string.
  • short_src: a "printable" version of source, to be used in error messages.
  • linedefined: the line number where the definition of the function starts.
  • lastlinedefined: the line number where the definition of the function ends.
  • what: the string "Lua" if the function is a Lua function, "C" if it is a C function, "main" if it is the main part of a chunk.
  • currentline: the current line where the given function is executing. When no line information is available, currentline is set to -1.
  • name: a reasonable name for the given function. Because functions in Lua are first-class values, they do not have a fixed name: some functions can be the value of multiple global variables, while others can be stored only in a table field. The lua_getinfo function checks how the function was called to find a suitable name. If it cannot find a name, then name is set to NULL.
  • namewhat: explains the name field. The value of namewhat can be "global", "local", "method", "field", "upvalue", or "" (the empty string), according to how the function was called. (Lua uses the empty string when no other option seems to apply.)
  • istailcall: true if this function invocation was called by a tail call. In this case, the caller of this level is not in the stack.
  • nups: the number of upvalues of the function.
  • nparams: the number of fixed parameters of the function (always 0 for C functions).
  • isvararg: true if the function is a vararg function (always true for C functions).

[-(0|1), +(0|1|2), e]

Gets information about a specific function or function invocation.

To get information about a function invocation, the parameter ar must be a valid activation record that was filled by a previous call to lua_getstack or given as argument to a hook (див. lua_Hook).

To get information about a function you push it onto the stack and start the what string with the character '>'. (In that case, lua_getinfo pops the function from the top of the stack.) For instance, to know in which line a function f was defined, you can write the following code: 

     lua_Debug ar;
     lua_getglobal(L, "f");  /* get global 'f' */
     lua_getinfo(L, ">S", &ar);
     printf("%d\n", ar.linedefined);

Each character in the string what selects some fields of the structure ar to be filled or a value to be pushed on the stack:

  • 'n': fills in the field name and namewhat;
  • 'S': fills in the fields source, short_src, linedefined, lastlinedefined, and what;
  • 'l': fills in the field currentline;
  • 't': fills in the field istailcall;
  • 'u': fills in the fields nups, nparams, and isvararg;
  • 'f': Заштовхує в стек the function that is running at the given level;
  • 'L': Заштовхує в стек a table whose indices are the numbers of the lines that are valid on the function. (A valid line is a line with some associated code, that is, a line where you can put a break point. Non-valid lines include empty lines and comments.)

5 - The Auxiliary Library

The auxiliary library provides several convenient functions to interface C with Lua. While the basic API provides the primitive functions for all interactions between C and Lua, the auxiliary library provides higher-level functions for some common tasks.

All functions and types from the auxiliary library are defined in header file lauxlib.h and have a prefix luaL_.

All functions in the auxiliary library are built on top of the basic API, and so they provide nothing that cannot be done with that API. Nevertheless, the use of the auxiliary library ensures more consistency to your code.

Several functions in the auxiliary library use internally some extra stack slots. When a function in the auxiliary library uses less than five slots, it does not check the stack size; it simply assumes that there are enough slots.

Several functions in the auxiliary library are used to check C function arguments. Because the error message is formatted for arguments (e.g., "bad argument #1"), you should not use these functions for other stack values.

Functions called luaL_check* always raise an error if the check is not satisfied.

A string buffer allows C code to build Lua strings piecemeal. Its pattern of use is as follows:

  • First declare a variable b of type luaL_Buffer.
  • Then initialize it with a call luaL_buffinit(L, &b).
  • Then add string pieces to the buffer calling any of the luaL_add* functions.
  • Finish by calling luaL_pushresult(&b). This call leaves the final string on the top of the stack.

If you know beforehand the total size of the resulting string, you can use the buffer like this:

  • First declare a variable b of type luaL_Buffer.
  • Then initialize it and preallocate a space of size sz with a call luaL_buffinitsize(L, &b, sz).
  • Then copy the string into that space.
  • Finish by calling luaL_pushresultsize(&b, sz), where sz is the total size of the resulting string copied into that space.

During its normal operation, a string buffer uses a variable number of stack slots. So, while using a buffer, you cannot assume that you know where the top of the stack is. You can use the stack between successive calls to buffer operations as long as that use is balanced; that is, when you call a buffer operation, the stack is at the same level it was immediately after the previous buffer operation. (The only exception to this rule is luaL_addvalue.) After calling luaL_pushresult the stack is back to its level when the buffer was initialized, plus the final string on its top.

6 - Standard Libraries

The standard Lua libraries provide useful functions that are implemented directly through the C API. Some of these functions provide essential services to the language (e.g., type and getmetatable); others provide access to "outside" services (e.g., I/O); and others could be implemented in Lua itself, but are quite useful or have critical performance requirements that deserve an implementation in C (e.g., table.sort).

All libraries are implemented through the official C API and are provided as separate C modules. Currently, Lua has the following standard libraries:

  • basic library (§6.1);
  • coroutine library (§6.2);
  • package library (§6.3);
  • string manipulation (§6.4);
  • basic UTF-8 support (§6.5);
  • table manipulation (§6.6);
  • mathematical functions (§6.7) (sin, log, etc.);
  • input and output (§6.8);
  • operating system facilities (§6.9);
  • debug facilities (§6.10).

Except for the basic and the package libraries, each library provides all its functions as fields of a global table or as methods of its objects.

To have access to these libraries, the C host program should call the luaL_openlibs function, which opens all standard libraries. Alternatively, the host program can open them individually by using luaL_requiref to call luaopen_base (for the basic library), luaopen_package (for the package library), luaopen_coroutine (for the coroutine library), luaopen_string (for the string library), luaopen_utf8 (for the UTF8 library), luaopen_table (for the table library), luaopen_math (for the mathematical library), luaopen_io (for the I/O library), luaopen_os (for the operating system library), and luaopen_debug (for the debug library). These functions are declared in lualib.h.

6.1 - Basic Functions

The basic library provides core functions to Lua. If you do not include this library in your application, you should check carefully whether you need to provide implementations for some of its facilities.

assert (v [, message])

Calls error if the value of its argument v is false (i.e., nil or false); otherwise, returns all its arguments. In case of error, message is the error object; when absent, it defaults to "assertion failed!"

collectgarbage ([opt [, arg]])

This function is a generic interface to the garbage collector. It виконує different functions according to its first argument, opt:

  • "collect": виконує a full garbage-collection cycle. This is the default option.
  • "stop": stops automatic execution of the garbage collector. The collector will run only when explicitly invoked, until a call to restart it.
  • "restart": restarts automatic execution of the garbage collector.
  • "count": returns the total memory in use by Lua in Kbytes. The value has a fractional part, so that it multiplied by 1024 gives the exact number of bytes in use by Lua (except for overflows).
  • "step": виконує a garbage-collection step. The step "size" is controlled by arg. With a zero value, the collector will perform one basic (indivisible) step. For non-zero values, the collector will perform as if that amount of memory (in KBytes) had been allocated by Lua. Повертає true if the step finished a collection cycle.
  • "setpause": sets arg as the new value for the pause of the collector (див. §2.5). Повертає the previous value for pause.
  • "setstepmul": sets arg as the new value for the step multiplier of the collector (див. §2.5). Повертає the previous value for step.
  • "isrunning": returns a boolean that tells whether the collector is running (i.e., not stopped).

In case of errors, dofile propagates the error to its caller (that is, dofile does not run in protected mode).

error (message [, level])

Terminates the last protected function called and returns message as the error object. Function error never returns.

Usually, error adds some information about the error position at the beginning of the message, if the message is a string. The level argument specifies how to get the error position. With level 1 (the default), the error position is where the error function was called. Level 2 points the error to where the function that called error was called; and so on. Passing a level 0 avoids the addition of error position information to the message.

_G

A global variable (not a function) that holds the global environment (див. §2.2). Lua itself does not use this variable; changing its value does not affect any environment, nor vice versa.

If object does not have a metatable, returns nil. Otherwise, if the object's metatable has a "__metatable" field, returns the associated value. Otherwise, returns the metatable of the given object.

Повертає three values (an iterator function, the table t, and 0) so that the construction 

     for i,v in ipairs(t) do body end

will iterate over the key-value pairs (1,t[1]), (2,t[2]), ..., up to the first nil value.

load (chunk [, chunkname [, mode [, env]]])

Loads a chunk.

If chunk is a string, the chunk is this string. If chunk is a function, load calls it repeatedly to get the chunk pieces. Each call to chunk must return a string that concatenates with previous results. A return of an empty string, nil, or no value signals the end of the chunk.

If there are no syntactic errors, returns the compiled chunk as a function; otherwise, returns nil plus the error message.

If the resulting function has upvalues, the first upvalue is set to the value of env, if that parameter is given, or to the value of the global environment. Other upvalues are initialized with nil. (When you load a main chunk, the resulting function will always have exactly one upvalue, the _ENV variable (див. §2.2). However, when you load a binary chunk created from a function (див. string.dump), the resulting function can have an arbitrary number of upvalues.) All upvalues are fresh, that is, they are not shared with any other function.

chunkname is used as the name of the chunk for error messages and debug information (див. §4.9). When absent, it defaults to chunk, if chunk is a string, or to "=(load)" otherwise.

The string mode controls whether the chunk can be text or binary (that is, a precompiled chunk). It may be the string "b" (only binary chunks), "t" (only text chunks), or "bt" (both binary and text). The default is "bt".

Lua does not check the consistency of binary chunks. Maliciously crafted binary chunks can crash the interpreter.

loadfile ([filename [, mode [, env]]])

Similar to load, but gets the chunk from file filename or from the standard input, if no file name is given.

next (table [, index])

Allows a program to traverse all fields of a table. Its first argument is a table and its second argument is an index in this table. next returns the next index of the table and its associated value. When called with nil as its second argument, next returns an initial index and its associated value. When called with the last index, or with nil in an empty table, next returns nil. If the second argument is absent, then it is interpreted as nil. In particular, you can use next(t) to check whether a table is empty.

The order in which the indices are enumerated is not specified, even for numeric indices. (To traverse a table in numerical order, use a numerical for.)

The behavior of next is undefined if, during the traversal, you assign any value to a non-existent field in the table. You may however modify existing fields. In particular, you may clear existing fields.

     for k,v in pairs(t) do body end

will iterate over all key-value pairs of table t.

See function next for the caveats of modifying the table during its traversal.

In case of any error, pcall returns false plus the error message.

If index is a number, returns all arguments after argument number index; a negative number indexes from the end (-1 is the last argument). Otherwise, index must be the string "#", and select returns the total number of extra arguments it received.

setmetatable (table, metatable)

Sets the metatable for the given table. (You cannot change the metatable of other types from Lua, only from C.) If metatable is nil, removes the metatable of the given table. If the original metatable has a "__metatable" field, raises an error.

This function returns table.

tonumber (e [, base])

When called with no base, tonumber tries to convert its argument to a number. If the argument is already a number or a string convertible to a number, then tonumber returns this number; otherwise, it returns nil.

The conversion of strings can result in integers or floats, according to the lexical conventions of Lua (див. §3.1). (The string may have leading and trailing spaces and a sign.)

When called with base, then e must be a string to be interpreted as an integer numeral in that base. The base may be any integer between 2 and 36, inclusive. In bases above 10, the letter 'A' (in either upper or lower case) represents 10, 'B' represents 11, and so forth, with 'Z' representing 35. If the string e is not a valid numeral in the given base, the function returns nil.

tostring (v)

Receives a value of any type and converts it to a string in a human-readable format. (For complete control of how numbers are converted, use string.format.)

If the metatable of v has a "__tostring" field, then tostring calls the corresponding value with v as argument, and uses the result of the call as its result.

type (v)

Повертає the type of its only argument, coded as a string. The possible results of this function are "nil" (a string, not the value nil), "number", "string", "boolean", "table", "function", "thread", and "userdata".

_VERSION

A global variable (not a function) that holds a string containing the current interpreter version. The current value of this variable is "Lua 5.3".

xpcall (f, msgh [, arg1, ···])

This function is similar to pcall, except that it sets a new message handler msgh.

6.2 - Coroutine Manipulation

This library comprises the operations to manipulate coroutines, which come inside the table coroutine. See §2.6 for a general description of coroutines.

coroutine.create (f)

Creates a new coroutine, with body f. f must be a function. Повертає this new coroutine, an object with type "thread".

coroutine.isyieldable ()

Повертає true when the running coroutine can yield.

A running coroutine is yieldable if it is not the main thread and it is not inside a non-yieldable C function.

coroutine.resume (co [, val1, ···])

Starts or continues the execution of coroutine co. The first time you resume a coroutine, it starts running its body. The values val1, ... are passed as the arguments to the body function. If the coroutine has yielded, resume restarts it; the values val1, ... are passed as the results from the yield.

If the coroutine runs without any errors, resume returns true plus any values passed to yield (when the coroutine yields) or any values returned by the body function (when the coroutine terminates). If there is any error, resume returns false plus the error message.

coroutine.running ()

Повертає the running coroutine plus a boolean, true when the running coroutine is the main one.

coroutine.status (co)

Повертає the status of coroutine co, as a string: "running", if the coroutine is running (that is, it called status); "suspended", if the coroutine is suspended in a call to yield, or if it has not started running yet; "normal" if the coroutine is active but not running (that is, it has resumed another coroutine); and "dead" if the coroutine has finished its body function, or if it has stopped with an error.

coroutine.wrap (f)

Creates a new coroutine, with body f. f must be a function. Повертає a function that resumes the coroutine each time it is called. Any arguments passed to the function behave as the extra arguments to resume. Повертає the same values returned by resume, except the first boolean. In case of error, propagates the error.

coroutine.yield (···)

Suspends the execution of the calling coroutine. Any arguments to yield are passed as extra results to resume.

6.3 - Modules

The package library provides basic facilities for loading modules in Lua. It exports one function directly in the global environment: require. Everything else is exported in a table package.

require (modname)

Loads the given module. The function starts by looking into the package.loaded table to determine whether modname is already loaded. If it is, then require returns the value stored at package.loaded[modname]. Otherwise, it tries to find a loader for the module.

To find a loader, require is guided by the package.searchers sequence. By changing this sequence, we can change how require looks for a module. The following explanation is based on the default configuration for package.searchers.

First require queries package.preload[modname]. If it has a value, this value (which must be a function) is the loader. Otherwise require searches for a Lua loader using the path stored in package.path. If that also fails, it searches for a C loader using the path stored in package.cpath. If that also fails, it tries an all-in-one loader (див. package.searchers).

Once a loader is found, require calls the loader with two arguments: modname and an extra value dependent on how it got the loader. (If the loader came from a file, this extra value is the file name.) If the loader returns any non-nil value, require assigns the returned value to package.loaded[modname]. If the loader does not return a non-nil value and has not assigned any value to package.loaded[modname], then require assigns true to this entry. In any case, require returns the final value of package.loaded[modname].

If there is any error loading or running the module, or if it cannot find any loader for the module, then require raises an error.

package.config

A string describing some compile-time configurations for packages. This string is a sequence of lines:

  • The first line is the directory separator string. Default is '\' for Windows and '/' for all other systems.
  • The second line is the character that separates templates in a path. Default is ';'.
  • The third line is the string that marks the substitution points in a template. Default is '?'.
  • The fourth line is a string that, in a path in Windows, is replaced by the executable's directory. Default is '!'.
  • The fifth line is a mark to ignore all text after it when building the luaopen_ function name. Default is '-'.

package.cpath

The path used by require to search for a C loader.

Lua initializes the C path package.cpath in the same way it initializes the Lua path package.path, using the environment variable LUA_CPATH_5_3 or the environment variable LUA_CPATH or a default path defined in luaconf.h.

package.loaded

A table used by require to control which modules are already loaded. When you require a module modname and package.loaded[modname] is not false, require simply returns the value stored there.

This variable is only a reference to the real table; assignments to this variable do not change the table used by require.

package.loadlib (libname, funcname)

Dynamically links the host program with the C library libname.

If funcname is "*", then it only links with the library, making the symbols exported by the library available to other dynamically linked libraries. Otherwise, it looks for a function funcname inside the library and returns this function as a C function. So, funcname must follow the lua_CFunction prototype (див. lua_CFunction).

This is a low-level function. It completely bypasses the package and module system. Unlike require, it does not perform any path searching and does not automatically adds extensions. libname must be the complete file name of the C library, including if necessary a path and an extension. funcname must be the exact name exported by the C library (which may depend on the C compiler and linker used).

This function is not supported by Standard C. As such, it is only available on some platforms (Windows, Linux, Mac OS X, Solaris, BSD, plus other Unix systems that support the dlfcn standard).

package.path

The path used by require to search for a Lua loader.

At start-up, Lua initializes this variable with the value of the environment variable LUA_PATH_5_3 or the environment variable LUA_PATH or with a default path defined in luaconf.h, if those environment variables are not defined. Any ";;" in the value of the environment variable is replaced by the default path.

package.preload

A table to store loaders for specific modules (див. require).

This variable is only a reference to the real table; assignments to this variable do not change the table used by require.

package.searchers

A table used by require to control how to load modules.

Each entry in this table is a searcher function. When looking for a module, require calls each of these searchers in ascending order, with the module name (the argument given to require) as its sole parameter. The function can return another function (the module loader) plus an extra value that will be passed to that loader, or a string explaining why it did not find that module (or nil if it has nothing to say).

Lua initializes this table with four searcher functions.

The first searcher simply looks for a loader in the package.preload table.

The second searcher looks for a loader as a Lua library, using the path stored at package.path. The search is done as described in function package.searchpath.

The third searcher looks for a loader as a C library, using the path given by the variable package.cpath. Again, the search is done as described in function package.searchpath. For instance, if the C path is the string 

     "./?.so;./?.dll;/usr/local/?/init.so"

the searcher for module foo will try to open the files ./foo.so, ./foo.dll, and /usr/local/foo/init.so, in that order. Once it finds a C library, this searcher first uses a dynamic link facility to link the application with the library. Then it tries to find a C function inside the library to be used as the loader. The name of this C function is the string "luaopen_" concatenated with a copy of the module name where each dot is replaced by an underscore. Moreover, if the module name has a hyphen, its suffix after (and including) the first hyphen is removed. For instance, if the module name is a.b.c-v2.1, the function name will be luaopen_a_b_c.

The fourth searcher tries an all-in-one loader. It searches the C path for a library for the root name of the given module. For instance, when requiring a.b.c, it will search for a C library for a. If found, it looks into it for an open function for the submodule; in our example, that would be luaopen_a_b_c. With this facility, a package can pack several C submodules into one single library, with each submodule keeping its original open function.

All searchers except the first one (preload) return as the extra value the file name where the module was found, as returned by package.searchpath. The first searcher returns no extra value.

package.searchpath (name, path [, sep [, rep]])

Searches for the given name in the given path.

A path is a string containing a sequence of templates separated by semicolons. For each template, the function replaces each interrogation mark (if any) in the template with a copy of name wherein all occurrences of sep (a dot, by default) were replaced by rep (the system's directory separator, by default), and then tries to open the resulting file name.

For instance, if the path is the string 

     "./?.lua;./?.lc;/usr/local/?/init.lua"

the search for the name foo.a will try to open the files ./foo/a.lua, ./foo/a.lc, and /usr/local/foo/a/init.lua, in that order.

Повертає the resulting name of the first file that it can open in read mode (after closing the file), or nil plus an error message if none succeeds. (This error message lists all file names it tried to open.)

6.4 - String Manipulation

This library provides generic functions for string manipulation, such as finding and extracting substrings, and pattern matching. When indexing a string in Lua, the first character is at position 1 (not at 0, as in C). Indices are allowed to be negative and are interpreted as indexing backwards, from the end of the string. Thus, the last character is at position -1, and so on.

The string library provides all its functions inside the table string. It also sets a metatable for strings where the __index field points to the string table. Therefore, you can use the string functions in object-oriented style. For instance, string.byte(s,i) can be written as s:byte(i).

The string library assumes one-byte character encodings.

string.byte (s [, i [, j]])

Повертає the internal numeric codes of the characters s[i], s[i+1], ..., s[j]. The default value for i is 1; the default value for j is i. These indices are corrected following the same rules of function string.sub.

Numeric codes are not necessarily portable across platforms.

string.char (···)

Receives zero or more integers. Повертає a string with length equal to the number of arguments, in which each character has the internal numeric code equal to its corresponding argument.

Numeric codes are not necessarily portable across platforms.

string.dump (function [, strip])

Повертає a string containing a binary representation (a binary chunk) of the given function, so that a later load on this string returns a copy of the function (but with new upvalues). If strip is a true value, the binary representation may not include all debug information about the function, to save space.

Functions with upvalues have only their number of upvalues saved. When (re)loaded, those upvalues receive fresh instances containing nil. (You can use the debug library to serialize and reload the upvalues of a function in a way adequate to your needs.)

string.find (s, pattern [, init [, plain]])

Looks for the first match of pattern (див. §6.4.1) in the string s. If it finds a match, then find returns the indices of s where this occurrence starts and ends; otherwise, it returns nil. A third, optional numeric argument init specifies where to start the search; its default value is 1 and can be negative. A value of true as a fourth, optional argument plain turns off the pattern matching facilities, so the function does a plain "find substring" operation, with no characters in pattern being considered magic. Note that if plain is given, then init must be given as well.

If the pattern has captures, then in a successful match the captured values are also returned, after the two indices.

string.format (formatstring, ···)

Повертає a formatted version of its variable number of arguments following the description given in its first argument (which must be a string). The format string follows the same rules as the ISO C function sprintf. The only differences are that the options/modifiers *, h, L, l, n, and p are not supported and that there is an extra option, q. The q option formats a string between double quotes, using escape sequences when necessary to ensure that it can safely be read back by the Lua interpreter. For instance, the call 

     string.format('%q', 'a string with "quotes" and \n new line')

may produce the string: 

     "a string with \"quotes\" and \
      new line"

Options A, a, E, e, f, G, and g all expect a number as argument. Options c, d, i, o, u, X, and x expect an integer. Option q expects a string. Option s expects a string without embedded zeros; if its argument is not a string, it is converted to one following the same rules of tostring.

When Lua is compiled with a non-C99 compiler, options A and a (hexadecimal floats) do not support any modifier (flags, width, length).

string.gmatch (s, pattern)

Повертає an iterator function that, each time it is called, returns the next captures from pattern (див. §6.4.1) over the string s. If pattern specifies no captures, then the whole match is produced in each call.

As an example, the following loop will iterate over all the words from string s, printing one per line: 

     s = "hello world from Lua"
     for w in string.gmatch(s, "%a+") do
       print(w)
     end

The next example collects all pairs key=value from the given string into a table: 

     t = {}
     s = "from=world, to=Lua"
     for k, v in string.gmatch(s, "(%w+)=(%w+)") do
       t[k] = v
     end

For this function, a caret '^' at the start of a pattern does not work as an anchor, as this would prevent the iteration.

string.gsub (s, pattern, repl [, n])

Повертає a copy of s in which all (or the first n, if given) occurrences of the pattern (див. §6.4.1) have been replaced by a replacement string specified by repl, which can be a string, a table, or a function. gsub also returns, as its second value, the total number of matches that occurred. The name gsub comes from Global SUBstitution.

If repl is a string, then its value is used for replacement. The character % works as an escape character: any sequence in repl of the form %d, with d between 1 and 9, stands for the value of the d-th captured substring. The sequence %0 stands for the whole match. The sequence %% stands for a single %.

If repl is a table, then the table is queried for every match, using the first capture as the key.

If repl is a function, then this function is called every time a match occurs, with all captured substrings passed as arguments, in order.

In any case, if the pattern specifies no captures, then it behaves as if the whole pattern was inside a capture.

If the value returned by the table query or by the function call is a string or a number, then it is used as the replacement string; otherwise, if it is false or nil, then there is no replacement (that is, the original match is kept in the string).

Here are some examples: 

     x = string.gsub("hello world", "(%w+)", "%1 %1")
     --> x="hello hello world world"
     
     x = string.gsub("hello world", "%w+", "%0 %0", 1)
     --> x="hello hello world"
     
     x = string.gsub("hello world from Lua", "(%w+)%s*(%w+)", "%2 %1")
     --> x="world hello Lua from"
     
     x = string.gsub("home = $HOME, user = $USER", "%$(%w+)", os.getenv)
     --> x="home = /home/roberto, user = roberto"
     
     x = string.gsub("4+5 = $return 4+5$", "%$(.-)%$", function (s)
           return load(s)()
         end)
     --> x="4+5 = 9"
     
     local t = {name="lua", version="5.3"}
     x = string.gsub("$name-$version.tar.gz", "%$(%w+)", t)
     --> x="lua-5.3.tar.gz"

string.len (s)

Receives a string and returns its length. The empty string "" has length 0. Embedded zeros are counted, so "a\000bc\000" has length 5.

string.lower (s)

Receives a string and returns a copy of this string with all uppercase letters changed to lowercase. All other characters are left unchanged. The definition of what an uppercase letter is depends on the current locale.

string.match (s, pattern [, init])

Looks for the first match of pattern (див. §6.4.1) in the string s. If it finds one, then match returns the captures from the pattern; otherwise it returns nil. If pattern specifies no captures, then the whole match is returned. A third, optional numeric argument init specifies where to start the search; its default value is 1 and can be negative.

string.pack (fmt, v1, v2, ···)

Повертає a binary string containing the values v1, v2, etc. packed (that is, serialized in binary form) according to the format string fmt (див. §6.4.2).

string.packsize (fmt)

Повертає the size of a string resulting from string.pack with the given format. The format string cannot have the variable-length options 's' or 'z' (див. §6.4.2).

string.rep (s, n [, sep])

Повертає a string that is the concatenation of n copies of the string s separated by the string sep. The default value for sep is the empty string (that is, no separator). Повертає the empty string if n is not positive.

string.reverse (s)

Повертає a string that is the string s reversed.

string.sub (s, i [, j])

Повертає the substring of s that starts at i and continues until j; i and j can be negative. If j is absent, then it is assumed to be equal to -1 (which is the same as the string length). In particular, the call string.sub(s,1,j) returns a prefix of s with length j, and string.sub(s, -i) returns a suffix of s with length i.

If, after the translation of negative indices, i is less than 1, it is corrected to 1. If j is greater than the string length, it is corrected to that length. If, after these corrections, i is greater than j, the function returns the empty string.

string.unpack (fmt, s [, pos])

Повертає the values packed in string s (див. string.pack) according to the format string fmt (див. §6.4.2). An optional pos marks where to start reading in s (default is 1). After the read values, this function also returns the index of the first unread byte in s.

string.upper (s)

Receives a string and returns a copy of this string with all lowercase letters changed to uppercase. All other characters are left unchanged. The definition of what a lowercase letter is depends on the current locale.

6.4.1 - Patterns

Patterns in Lua are described by regular strings, which are interpreted as patterns by the pattern-matching functions string.find, string.gmatch, string.gsub, and string.match. This section describes the syntax and the meaning (that is, what they match) of these strings.

Character Class:

A character class is used to represent a set of characters. The following combinations are allowed in describing a character class:

  • x: (where x is not one of the magic characters ^$()%.[]*+-?) represents the character x itself.
  • .: (a dot) represents all characters.
  • %a: represents all letters.
  • %c: represents all control characters.
  • %d: represents all digits.
  • %g: represents all printable characters except space.
  • %l: represents all lowercase letters.
  • %p: represents all punctuation characters.
  • %s: represents all space characters.
  • %u: represents all uppercase letters.
  • %w: represents all alphanumeric characters.
  • %x: represents all hexadecimal digits.
  • %x: (where x is any non-alphanumeric character) represents the character x. This is the standard way to escape the magic characters. Any non-alphanumeric character (including all punctuation characters, even the non-magical) can be preceded by a '%' when used to represent itself in a pattern.
  • [set]: represents the class which is the union of all characters in set. A range of characters can be specified by separating the end characters of the range, in ascending order, with a '-'. All classes %x described above can also be used as components in set. All other characters in set represent themselves. For example, [%w_] (or [_%w]) represents all alphanumeric characters plus the underscore, [0-7] represents the octal digits, and [0-7%l%-] represents the octal digits plus the lowercase letters plus the '-' character.

    The interaction between ranges and classes is not defined. Therefore, patterns like [%a-z] or [a-%%] have no meaning.

  • [^set]: represents the complement of set, where set is interpreted as above.

For all classes represented by single letters (%a, %c, etc.), the corresponding uppercase letter represents the complement of the class. For instance, %S represents all non-space characters.

The definitions of letter, space, and other character groups depend on the current locale. In particular, the class [a-z] may not be equivalent to %l.

Pattern Item:

A pattern item can be

  • a single character class, which matches any single character in the class;
  • a single character class followed by '*', which matches zero or more repetitions of characters in the class. These repetition items will always match the longest possible sequence;
  • a single character class followed by '+', which matches one or more repetitions of characters in the class. These repetition items will always match the longest possible sequence;
  • a single character class followed by '-', which also matches zero or more repetitions of characters in the class. Unlike '*', these repetition items will always match the shortest possible sequence;
  • a single character class followed by '?', which matches zero or one occurrence of a character in the class. It always matches one occurrence if possible;
  • %n, for n between 1 and 9; such item matches a substring equal to the n-th captured string (див. below);
  • %bxy, where x and y are two distinct characters; such item matches strings that start with x, end with y, and where the x and y are balanced. This means that, if one reads the string from left to right, counting +1 for an x and -1 for a y, the ending y is the first y where the count reaches 0. For instance, the item %b() matches expressions with balanced parentheses.
  • %f[set], a frontier pattern; such item matches an empty string at any position such that the next character belongs to set and the previous character does not belong to set. The set set is interpreted as previously described. The beginning and the end of the subject are handled as if they were the character '\0'.

Pattern:

A pattern is a sequence of pattern items. A caret '^' at the beginning of a pattern anchors the match at the beginning of the subject string. A '$' at the end of a pattern anchors the match at the end of the subject string. At other positions, '^' and '$' have no special meaning and represent themselves.

Captures:

A pattern can contain sub-patterns enclosed in parentheses; they describe captures. When a match succeeds, the substrings of the subject string that match captures are stored (captured) for future use. Captures are numbered according to their left parentheses. For instance, in the pattern "(a*(.)%w(%s*))", the part of the string matching "a*(.)%w(%s*)" is stored as the first capture (and therefore has number 1); the character matching "." is captured with number 2, and the part matching "%s*" has number 3.

As a special case, the empty capture () captures the current string position (a number). For instance, if we apply the pattern "()aa()" on the string "flaaap", there will be two captures: 3 and 5.

6.4.2 - Format Strings for Pack and Unpack

The first argument to string.pack, string.packsize, and string.unpack is a format string, which describes the layout of the structure being created or read.

A format string is a sequence of conversion options. The conversion options are as follows:

  • <: sets little endian
  • >: sets big endian
  • =: sets native endian
  • ![n]: sets maximum alignment to n (default is native alignment)
  • b: a signed byte (char)
  • B: an unsigned byte (char)
  • h: a signed short (native size)
  • H: an unsigned short (native size)
  • l: a signed long (native size)
  • L: an unsigned long (native size)
  • j: a lua_Integer
  • J: a lua_Unsigned
  • T: a size_t (native size)
  • i[n]: a signed int with n bytes (default is native size)
  • I[n]: an unsigned int with n bytes (default is native size)
  • f: a float (native size)
  • d: a double (native size)
  • n: a lua_Number
  • cn: a fixed-sized string with n bytes
  • z: a zero-terminated string
  • s[n]: a string preceded by its length coded as an unsigned integer with n bytes (default is a size_t)
  • x: one byte of padding
  • Xop: an empty item that aligns according to option op (which is otherwise ignored)
  • ' ': (empty space) ignored

(A "[n]" means an optional integral numeral.) Except for padding, spaces, and configurations (options "xX <=>!"), each option corresponds to an argument (in string.pack) or a result (in string.unpack).

For options "!n", "sn", "in", and "In", n can be any integer between 1 and 16. All integral options check overflows; string.pack checks whether the given value fits in the given size; string.unpack checks whether the read value fits in a Lua integer.

Any format string starts as if prefixed by "!1=", that is, with maximum alignment of 1 (no alignment) and native endianness.

Alignment works as follows: For each option, the format gets extra padding until the data starts at an offset that is a multiple of the minimum between the option size and the maximum alignment; this minimum must be a power of 2. Options "c" and "z" are not aligned; option "s" follows the alignment of its starting integer.

All padding is filled with zeros by string.pack (and ignored by string.unpack).

6.5 - UTF-8 Support

This library provides basic support for UTF-8 encoding. It provides all its functions inside the table utf8. This library does not provide any support for Unicode other than the handling of the encoding. Any operation that needs the meaning of a character, such as character classification, is outside its scope.

Unless stated otherwise, all functions that expect a byte position as a parameter assume that the given position is either the start of a byte sequence or one plus the length of the subject string. As in the string library, negative indices count from the end of the string.

utf8.char (···)

Receives zero or more integers, converts each one to its corresponding UTF-8 byte sequence and returns a string with the concatenation of all these sequences.

utf8.charpattern

The pattern (a string, not a function) "[\0-\x7F\xC2-\xF4][\x80-\xBF]*" (див. §6.4.1), which matches exactly one UTF-8 byte sequence, assuming that the subject is a valid UTF-8 string.

utf8.codes (s)

Повертає values so that the construction 

     for p, c in utf8.codes(s) do body end

will iterate over all characters in string s, with p being the position (in bytes) and c the code point of each character. It raises an error if it meets any invalid byte sequence.

utf8.codepoint (s [, i [, j]])

Повертає the codepoints (as integers) from all characters in s that start between byte position i and j (both included). The default for i is 1 and for j is i. It raises an error if it meets any invalid byte sequence.

utf8.len (s [, i [, j]])

Повертає the number of UTF-8 characters in string s that start between positions i and j (both inclusive). The default for i is 1 and for j is -1. If it finds any invalid byte sequence, returns a false value plus the position of the first invalid byte.

utf8.offset (s, n [, i])

Повертає the position (in bytes) where the encoding of the n-th character of s (counting from position i) starts. A negative n gets characters before position i. The default for i is 1 when n is non-negative and #s + 1 otherwise, so that utf8.offset(s, -n) gets the offset of the n-th character from the end of the string. If the specified character is neither in the subject nor right after its end, the function returns nil.

As a special case, when n is 0 the function returns the start of the encoding of the character that contains the i-th byte of s.

This function assumes that s is a valid UTF-8 string.

6.6 - Table Manipulation

This library provides generic functions for table manipulation. It provides all its functions inside the table table.

Remember that, whenever an operation needs the length of a table, the table must be a proper sequence or have a __len metamethod (див. §3.4.7). All functions ignore non-numeric keys in the tables given as arguments.

table.concat (list [, sep [, i [, j]]])

Given a list where all elements are strings or numbers, returns the string list[i]..sep..list[i+1] ··· sep..list[j]. The default value for sep is the empty string, the default for i is 1, and the default for j is #list. If i is greater than j, returns the empty string.

table.insert (list, [pos,] value)

Inserts element value at position pos in list, shifting up the elements list[pos], list[pos+1], ···, list[#list]. The default value for pos is #list+1, so that a call table.insert(t,x) inserts x at the end of list t.

table.move (a1, f, e, t [,a2])

Moves elements from table a1 to table a2. This function виконує the equivalent to the following multiple assignment: a2[t],··· = a1[f],···,a1[e]. The default for a2 is a1. The destination range can overlap with the source range. The number of elements to be moved must fit in a Lua integer.

table.pack (···)

Повертає a new table with all parameters stored into keys 1, 2, etc. and with a field "n" with the total number of parameters. Note that the resulting table may not be a sequence.

table.remove (list [, pos])

Removes from list the element at position pos, returning the value of the removed element. When pos is an integer between 1 and #list, it shifts down the elements list[pos+1], list[pos+2], ···, list[#list] and erases element list[#list]; The index pos can also be 0 when #list is 0, or #list + 1; in those cases, the function erases the element list[pos].

The default value for pos is #list, so that a call table.remove(l) removes the last element of list l.

table.sort (list [, comp])

Sorts list elements in a given order, in-place, from list[1] to list[#list]. If comp is given, then it must be a function that receives two list elements and returns true when the first element must come before the second in the final order (so that not comp(list[i+1],list[i]) will be true after the sort). If comp is not given, then the standard Lua operator < is used instead.

The sort algorithm is not stable; that is, elements considered equal by the given order may have their relative positions changed by the sort.

table.unpack (list [, i [, j]])

Повертає the elements from the given list. This function is equivalent to 

     return list[i], list[i+1], ···, list[j]

By default, i is 1 and j is #list.

math.cos (x)

math.deg (x)

math.exp (x)

math.floor (x)

math.fmod (x, y)

math.huge

math.log (x [, base])

math.max (x, ···)

math.random ([m [, n]])

6.8 - Input and Output Facilities

The I/O library provides two different styles for file manipulation. The first one uses implicit file handles; that is, there are operations to set a default input file and a default output file, and all input/output operations are over these default files. The second style uses explicit file handles.

When using implicit file handles, all operations are supplied by table io. When using explicit file handles, the operation io.open returns a file handle and then all operations are supplied as methods of the file handle.

The table io also provides three predefined file handles with their usual meanings from C: io.stdin, io.stdout, and io.stderr. The I/O library never closes these files.

Unless otherwise stated, all I/O functions return nil on failure (plus an error message as a second result and a system-dependent error code as a third result) and some value different from nil on success. On non-POSIX systems, the computation of the error message and error code in case of errors may be not thread safe, because they rely on the global C variable errno.

io.close ([file])

Equivalent to file:close(). Without a file, closes the default output file.

io.flush ()

Equivalent to io.output():flush().

io.input ([file])

When called with a file name, it opens the named file (in text mode), and sets its handle as the default input file. When called with a file handle, it simply sets this file handle as the default input file. When called without parameters, it returns the current default input file.

In case of errors this function raises the error, instead of returning an error code.

io.lines ([filename ···])

Opens the given file name in read mode and returns an iterator function that works like file:lines(···) over the opened file. When the iterator function detects the end of file, it returns no values (to finish the loop) and automatically closes the file.

The call io.lines() (with no file name) is equivalent to io.input():lines("*l"); that is, it iterates over the lines of the default input file. In this case it does not close the file when the loop ends.

In case of errors this function raises the error, instead of returning an error code.

io.open (filename [, mode])

This function opens a file, in the mode specified in the string mode. It returns a new file handle, or, in case of errors, nil plus an error message.

The mode string can be any of the following:

  • "r": read mode (the default);
  • "w": write mode;
  • "a": append mode;
  • "r+": update mode, all previous data is preserved;
  • "w+": update mode, all previous data is erased;
  • "a+": append update mode, previous data is preserved, writing is only allowed at the end of file.

The mode string can also have a 'b' at the end, which is needed in some systems to open the file in binary mode.

io.output ([file])

Similar to io.input, but operates over the default output file.

io.popen (prog [, mode])

This function is system dependent and is not available on all platforms.

Starts program prog in a separated process and returns a file handle that you can use to read data from this program (if mode is "r", the default) or to write data to this program (if mode is "w").

io.read (···)

Equivalent to io.input():read(···).

io.tmpfile ()

Повертає a handle for a temporary file. This file is opened in update mode and it is automatically removed when the program ends.

io.type (obj)

Checks whether obj is a valid file handle. Повертає the string "file" if obj is an open file handle, "closed file" if obj is a closed file handle, or nil if obj is not a file handle.

io.write (···)

Equivalent to io.output():write(···).

file:close ()

Closes file. Note that files are automatically closed when their handles are garbage collected, but that takes an unpredictable amount of time to happen.

When closing a file handle created with io.popen, file:close returns the same values returned by os.execute.

file:flush ()

Saves any written data to file.

file:lines (···)

Повертає an iterator function that, each time it is called, reads the file according to the given formats. When no format is given, uses "l" as a default. As an example, the construction 

     for c in file:lines(1) do body end

will iterate over all characters of the file, starting at the current position. Unlike io.lines, this function does not close the file when the loop ends.

In case of errors this function raises the error, instead of returning an error code.

file:read (···)

Reads the file file, according to the given formats, which specify what to read. For each format, the function returns a string or a number with the characters read, or nil if it cannot read data with the specified format. (In this latter case, the function does not read subsequent formats.) When called without formats, it uses a default format that reads the next line (див. below).

The available formats are

  • "n": reads a numeral and returns it as a float or an integer, following the lexical conventions of Lua. (The numeral may have leading spaces and a sign.) This format always reads the longest input sequence that is a valid prefix for a numeral; if that prefix does not form a valid numeral (e.g., an empty string, "0x", or "3.4e-"), it is discarded and the function returns nil.
  • "a": reads the whole file, starting at the current position. On end of file, it returns the empty string.
  • "l": reads the next line skipping the end of line, returning nil on end of file. This is the default format.
  • "L": reads the next line keeping the end-of-line character (if present), returning nil on end of file.
  • number: reads a string with up to this number of bytes, returning nil on end of file. If number is zero, it reads nothing and returns an empty string, or nil on end of file.

The formats "l" and "L" should be used only for text files.

file:seek ([whence [, offset]])

Sets and gets the file position, measured from the beginning of the file, to the position given by offset plus a base specified by the string whence, as follows:

  • "set": base is position 0 (beginning of the file);
  • "cur": base is current position;
  • "end": base is end of file;

In case of success, seek returns the final file position, measured in bytes from the beginning of the file. If seek fails, it returns nil, plus a string describing the error.

The default value for whence is "cur", and for offset is 0. Therefore, the call file:seek() returns the current file position, without changing it; the call file:seek("set") sets the position to the beginning of the file (and returns 0); and the call file:seek("end") sets the position to the end of the file, and returns its size.

file:setvbuf (mode [, size])

Sets the buffering mode for an output file. There are three available modes:

  • "no": no buffering; the result of any output operation appears immediately.
  • "full": full buffering; output operation is performed only when the buffer is full or when you explicitly flush the file (див. io.flush).
  • "line": line buffering; output is buffered until a newline is output or there is any input from some special files (such as a terminal device).

For the last two cases, size specifies the size of the buffer, in bytes. The default is an appropriate size.

file:write (···)

Writes the value of each of its arguments to file. The arguments must be strings or numbers.

In case of success, this function returns file. Otherwise it returns nil plus a string describing the error.

6.9 - Operating System Facilities

This library is implemented through table os.

os.clock ()

Повертає an approximation of the amount in seconds of CPU time used by the program.

os.date ([format [, time]])

Повертає a string or a table containing date and time, formatted according to the given string format.

If the time argument is present, this is the time to be formatted (див. the os.time function for a description of this value). Otherwise, date formats the current time.

If format starts with '!', then the date is formatted in Coordinated Universal Time. After this optional character, if format is the string "*t", then date returns a table with the following fields: year (four digits), month (1-12), day (1-31), hour (0-23), min (0-59), sec (0-61), wday (weekday, Sunday is 1), yday (day of the year), and isdst (daylight saving flag, a boolean). This last field may be absent if the information is not available.

If format is not "*t", then date returns the date as a string, formatted according to the same rules as the ISO C function strftime.

When called without arguments, date returns a reasonable date and time representation that depends on the host system and on the current locale (that is, os.date() is equivalent to os.date("%c")).

On non-POSIX systems, this function may be not thread safe because of its reliance on C function gmtime and C function localtime.

os.difftime (t2, t1)

Повертає the difference, in seconds, from time t1 to time t2 (where the times are values returned by os.time). In POSIX, Windows, and some other systems, this value is exactly t2-t1.

os.execute ([command])

This function is equivalent to the ISO C function system. It passes command to be executed by an operating system shell. Its first result is true if the command terminated successfully, or nil otherwise. After this first result the function returns a string plus a number, as follows:

  • "exit": the command terminated normally; the following number is the exit status of the command.
  • "signal": the command was terminated by a signal; the following number is the signal that terminated the command.

When called without a command, os.execute returns a boolean that is true if a shell is available.

os.exit ([code [, close]])

Calls the ISO C function exit to terminate the host program. If code is true, the returned status is EXIT_SUCCESS; if code is false, the returned status is EXIT_FAILURE; if code is a number, the returned status is this number. The default value for code is true.

If the optional second argument close is true, closes the Lua state before exiting.

os.getenv (varname)

Повертає the value of the process environment variable varname, or nil if the variable is not defined.

os.remove (filename)

Deletes the file (or empty directory, on POSIX systems) with the given name. If this function fails, it returns nil, plus a string describing the error and the error code.

os.rename (oldname, newname)

Renames file or directory named oldname to newname. If this function fails, it returns nil, plus a string describing the error and the error code.

os.setlocale (locale [, category])

Sets the current locale of the program. locale is a system-dependent string specifying a locale; category is an optional string describing which category to change: "all", "collate", "ctype", "monetary", "numeric", or "time"; the default category is "all". The function returns the name of the new locale, or nil if the request cannot be honored.

If locale is the empty string, the current locale is set to an implementation-defined native locale. If locale is the string "C", the current locale is set to the standard C locale.

When called with nil as the first argument, this function only returns the name of the current locale for the given category.

This function may be not thread safe because of its reliance on C function setlocale.

os.time ([table])

Повертає the current time when called without arguments, or a time representing the local date and time specified by the given table. This table must have fields year, month, and day, and may have fields hour (default is 12), min (default is 0), sec (default is 0), and isdst (default is nil). Other fields are ignored. For a description of these fields, see the os.date function.

The values in these fields do not need to be inside their valid ranges. For instance, if sec is -10, it means -10 seconds from the time specified by the other fields; if hour is 1000, it means +1000 hours from the time specified by the other fields.

The returned value is a number, whose meaning depends on your system. In POSIX, Windows, and some other systems, this number counts the number of seconds since some given start time (the "epoch"). In other systems, the meaning is not specified, and the number returned by time can be used only as an argument to os.date and os.difftime.

os.tmpname ()

Повертає a string with a file name that can be used for a temporary file. The file must be explicitly opened before its use and explicitly removed when no longer needed.

On POSIX systems, this function also creates a file with that name, to avoid security risks. (Someone else might create the file with wrong permissions in the time between getting the name and creating the file.) You still have to open the file to use it and to remove it (even if you do not use it).

When possible, you may prefer to use io.tmpfile, which automatically removes the file when the program ends.

6.10 - The Debug Library

This library provides the functionality of the debug interface (§4.9) to Lua programs. You should exert care when using this library. Several of its functions violate basic assumptions about Lua code (e.g., that variables local to a function cannot be accessed from outside; that userdata metatables cannot be changed by Lua code; that Lua programs do not crash) and therefore can compromise otherwise secure code. Moreover, some functions in this library may be slow.

All functions in this library are provided inside the debug table. All functions that operate over a thread have an optional first argument which is the thread to operate over. The default is always the current thread.

debug.debug ()

Enters an interactive mode with the user, running each string that the user enters. Using simple commands and other debug facilities, the user can inspect global and local variables, change their values, evaluate expressions, and so on. A line containing only the word cont finishes this function, so that the caller continues its execution.

Note that commands for debug.debug are not lexically nested within any function and so have no direct access to local variables.

debug.gethook ([thread])

Повертає the current hook settings of the thread, as three values: the current hook function, the current hook mask, and the current hook count (as set by the debug.sethook function).

debug.getinfo ([thread,] f [, what])

Повертає a table with information about a function. You can give the function directly or you can give a number as the value of f, which means the function running at level f of the call stack of the given thread: level 0 is the current function (getinfo itself); level 1 is the function that called getinfo (except for tail calls, which do not count on the stack); and so on. If f is a number larger than the number of active functions, then getinfo returns nil.

The returned table can contain all the fields returned by lua_getinfo, with the string what describing which fields to fill in. The default for what is to get all information available, except the table of valid lines. If present, the option 'f' adds a field named func with the function itself. If present, the option 'L' adds a field named activelines with the table of valid lines.

For instance, the expression debug.getinfo(1,"n").name returns a name for the current function, if a reasonable name can be found, and the expression debug.getinfo(print) returns a table with all available information about the print function.

debug.getlocal ([thread,] f, local)

This function returns the name and the value of the local variable with index local of the function at level f of the stack. This function accesses not only explicit local variables, but also parameters, temporaries, etc.

The first parameter or local variable has index 1, and so on, following the order that they are declared in the code, counting only the variables that are active in the current scope of the function. Negative indices refer to vararg parameters; -1 is the first vararg parameter. The function returns nil if there is no variable with the given index, and raises an error when called with a level out of range. (You can call debug.getinfo to check whether the level is valid.)

Variable names starting with '(' (open parenthesis) represent variables with no known names (internal variables such as loop control variables, and variables from chunks saved without debug information).

The parameter f may also be a function. In that case, getlocal returns only the name of function parameters.

debug.getmetatable (value)

Повертає the metatable of the given value or nil if it does not have a metatable.

debug.getregistry ()

Повертає the registry table (див. §4.5).

debug.getupvalue (f, up)

This function returns the name and the value of the upvalue with index up of the function f. The function returns nil if there is no upvalue with the given index.

Variable names starting with '(' (open parenthesis) represent variables with no known names (variables from chunks saved without debug information).

debug.getuservalue (u)

Повертає the Lua value associated to u. If u is not a userdata, returns nil.

debug.sethook ([thread,] hook, mask [, count])

Sets the given function as a hook. The string mask and the number count describe when the hook will be called. The string mask may have any combination of the following characters, with the given meaning:

  • 'c': the hook is called every time Lua calls a function;
  • 'r': the hook is called every time Lua returns from a function;
  • 'l': the hook is called every time Lua enters a new line of code.

Moreover, with a count different from zero, the hook is called also after every count instructions.

When called without arguments, debug.sethook turns off the hook.

When the hook is called, its first parameter is a string describing the event that has triggered its call: "call" (or "tail call"), "return", "line", and "count". For line events, the hook also gets the new line number as its second parameter. Inside a hook, you can call getinfo with level 2 to get more information about the running function (level 0 is the getinfo function, and level 1 is the hook function).

debug.setlocal ([thread,] level, local, value)

This function assigns the value value to the local variable with index local of the function at level level of the stack. The function returns nil if there is no local variable with the given index, and raises an error when called with a level out of range. (You can call getinfo to check whether the level is valid.) Otherwise, it returns the name of the local variable.

See debug.getlocal for more information about variable indices and names.

debug.setmetatable (value, table)

Sets the metatable for the given value to the given table (which can be nil). Повертає value.

debug.setupvalue (f, up, value)

This function assigns the value value to the upvalue with index up of the function f. The function returns nil if there is no upvalue with the given index. Otherwise, it returns the name of the upvalue.

debug.setuservalue (udata, value)

Sets the given value as the Lua value associated to the given udata. udata must be a full userdata.

Повертає udata.

debug.traceback ([thread,] [message [, level]])

If message is present but is neither a string nor nil, this function returns message without further processing. Otherwise, it returns a string with a traceback of the call stack. The optional message string is appended at the beginning of the traceback. An optional level number tells at which level to start the traceback (default is 1, the function calling traceback).

debug.upvalueid (f, n)

Повертає a unique identifier (as a light userdata) for the upvalue numbered n from the given function.

These unique identifiers allow a program to check whether different closures share upvalues. Lua closures that share an upvalue (that is, that access a same external local variable) will return identical ids for those upvalue indices.

debug.upvaluejoin (f1, n1, f2, n2)

Make the n1-th upvalue of the Lua closure f1 refer to the n2-th upvalue of the Lua closure f2.

7 - Lua Standalone

Although Lua has been designed as an extension language, to be embedded in a host C program, it is also frequently used as a standalone language. An interpreter for Lua as a standalone language, called simply lua, is provided with the standard distribution. The standalone interpreter includes all standard libraries, including the debug library. Its usage is: 

     lua [options] [script [args]]

The options are:

  • -e stat: executes string stat;
  • -l mod: "requires" mod;
  • -i: enters interactive mode after running script;
  • -v: prints version information;
  • -E: ignores environment variables;
  • --: stops handling options;
  • -: executes stdin as a file and stops handling options.

After handling its options, lua runs the given script. When called without arguments, lua behaves as lua -v -i when the standard input (stdin) is a terminal, and as lua - otherwise.

When called without option -E, the interpreter checks for an environment variable LUA_INIT_5_3 (or LUA_INIT if the versioned name is not defined) before running any argument. If the variable content has the format @filename, then lua executes the file. Otherwise, lua executes the string itself.

When called with option -E, besides ignoring LUA_INIT, Lua also ignores the values of LUA_PATH and LUA_CPATH, setting the values of package.path and package.cpath with the default paths defined in luaconf.h.

All options are handled in order, except -i and -E. For instance, an invocation like 

     $ lua -e'a=1' -e 'print(a)' script.lua

will first set a to 1, then print the value of a, and finally run the file script.lua with no arguments. (Here $ is the shell prompt. Your prompt may be different.)

Before running any code, lua collects all command-line arguments in a global table called arg. The script name goes to index 0, the first argument after the script name goes to index 1, and so on. Any arguments before the script name (that is, the interpreter name plus its options) go to negative indices. For instance, in the call 

     $ lua -la b.lua t1 t2

the table is like this: 

     arg = { [-2] = "lua", [-1] = "-la",
             [0] = "b.lua",
             [1] = "t1", [2] = "t2" }

If there is no script in the call, the interpreter name goes to index 0, followed by the other arguments. For instance, the call 

     $ lua -e "print(arg[1])"

will print "-e". If there is a script, the script is called with parameters arg[1], ···, arg[#arg]. (Like all chunks in Lua, the script is compiled as a vararg function.)

In interactive mode, Lua repeatedly prompts and waits for a line. After reading a line, Lua first try to interpret the line as an expression. If it succeeds, it prints its value. Otherwise, it interprets the line as a statement. If you write an incomplete statement, the interpreter waits for its completion by issuing a different prompt.

In case of unprotected errors in the script, the interpreter reports the error to the standard error stream. If the error object is not a string but has a metamethod __tostring, the interpreter calls this metamethod to produce the final message. Otherwise, the interpreter converts the error object to a string and adds a stack traceback to it.

When finishing normally, the interpreter closes its main Lua state (див. lua_close). The script can avoid this step by calling os.exit to terminate.

To allow the use of Lua as a script interpreter in Unix systems, the standalone interpreter skips the first line of a chunk if it starts with #. Therefore, Lua scripts can be made into executable programs by using chmod +x and the #! form, as in 

     #!/usr/local/bin/lua

(Of course, the location of the Lua interpreter may be different in your machine. If lua is in your PATH, then 

     #!/usr/bin/env lua

is a more portable solution.)

8 - Incompatibilities with the Previous Version

Here we list the incompatibilities that you may find when moving a program from Lua 5.2 to Lua 5.3. You can avoid some incompatibilities by compiling Lua with appropriate options (див. file luaconf.h). However, all these compatibility options will be removed in the future.

Lua versions can always change the C API in ways that do not imply source-code changes in a program, such as the numeric values for constants or the implementation of functions as macros. Therefore, you should not assume that binaries are compatible between different Lua versions. Always recompile clients of the Lua API when using a new version.

Similarly, Lua versions can always change the internal representation of precompiled chunks; precompiled chunks are not compatible between different Lua versions.

The standard paths in the official distribution may change between versions.

8.1 - Changes in the Language

  • The main difference between Lua 5.2 and Lua 5.3 is the introduction of an integer subtype for numbers. Although this change should not affect "normal" computations, some computations (mainly those that involve some kind of overflow) can give different results.

    You can fix these differences by forcing a number to be a float (in Lua 5.2 all numbers were float), in particular writing constants with an ending .0 or using x = x + 0.0 to convert a variable. (This recommendation is only for a quick fix for an occasional incompatibility; it is not a general guideline for good programming. For good programming, use floats where you need floats and integers where you need integers.)

  • The conversion of a float to a string now adds a .0 suffix to the result if it looks like an integer. (For instance, the float 2.0 will be printed as 2.0, not as 2.) You should always use an explicit format when you need a specific format for numbers.

    (Formally this is not an incompatibility, because Lua does not specify how numbers are formatted as strings, but some programs assumed a specific format.)

  • The generational mode for the garbage collector was removed. (It was an experimental feature in Lua 5.2.)

8.2 - Changes in the Libraries

  • The bit32 library has been deprecated. It is easy to require a compatible external library or, better yet, to replace its functions with appropriate bitwise operations. (Keep in mind that bit32 operates on 32-bit integers, while the bitwise operators in Lua 5.3 operate on Lua integers, which by default have 64 bits.)
  • The Table library now respects metamethods for setting and getting elements.
  • The ipairs iterator now respects metamethods and its __ipairs metamethod has been deprecated.
  • Option names in io.read do not have a starting '*' anymore. For compatibility, Lua will continue to accept (and ignore) this character.
  • The following functions were deprecated in the mathematical library: atan2, cosh, sinh, tanh, pow, frexp, and ldexp. You can replace math.pow(x,y) with x^y; you can replace math.atan2 with math.atan, which now accepts one or two parameters; you can replace math.ldexp(x,exp) with x * 2.0^exp. For the other operations, you can either use an external library or implement them in Lua.
  • The searcher for C loaders used by require changed the way it handles versioned names. Now, the version should come after the module name (as is usual in most other tools). For compatibility, that searcher still tries the old format if it cannot find an open function according to the new style. (Lua 5.2 already worked that way, but it did not document the change.)
  • The call collectgarbage("count") now returns only one result. (You can compute that second result from the fractional part of the first result.)

8.3 - Changes in the API

  • Continuation functions now receive as parameters what they needed to get through lua_getctx, so lua_getctx has been removed. Adapt your code accordingly.
  • Function lua_dump has an extra parameter, strip. Use 0 as the value of this parameter to get the old behavior.
  • Functions to inject/project unsigned integers (lua_pushunsigned, lua_tounsigned, lua_tounsignedx, luaL_checkunsigned, luaL_optunsigned) were deprecated. Use their signed equivalents with a type cast.
  • Macros to project non-default integer types (luaL_checkint, luaL_optint, luaL_checklong, luaL_optlong) were deprecated. Use their equivalent over lua_Integer with a type cast (or, when possible, use lua_Integer in your code).

9 - The Complete Syntax of Lua

Here is the complete syntax of Lua in extended BNF. As usual in extended BNF, {A} means 0 or more As, and [A] means an optional A. (For operator precedences, see §3.4.8; for a description of the terminals Name, Numeral, and LiteralString, see §3.1.) 

	chunk ::= block

	block ::= {stat} [retstat]

	stat ::=  ‘;’ | 
		 varlist ‘=’ explist | 
		 functioncall | 
		 label | 
		 break | 
		 goto Name | 
		 do block end | 
		 while exp do block end | 
		 repeat block until exp | 
		 if exp then block {elseif exp then block} [else block] end | 
		 for Name ‘=’ exp ‘,’ exp [‘,’ exp] do block end | 
		 for namelist in explist do block end | 
		 function funcname funcbody | 
		 local function Name funcbody | 
		 local namelist [‘=’ explist] 

	retstat ::= return [explist] [‘;’]

	label ::= ‘::’ Name ‘::’

	funcname ::= Name {‘.’ Name} [‘:’ Name]

	varlist ::= var {‘,’ var}

	var ::=  Name | prefixexp ‘[’ exp ‘]’ | prefixexp ‘.’ Name 

	namelist ::= Name {‘,’ Name}

	explist ::= exp {‘,’ exp}

	exp ::=  nil | false | true | Numeral | LiteralString | ‘...’ | functiondef | 
		 prefixexp | tableconstructor | exp binop exp | unop exp 

	prefixexp ::= var | functioncall | ‘(’ exp ‘)’

	functioncall ::=  prefixexp args | prefixexp ‘:’ Name args 

	args ::=  ‘(’ [explist] ‘)’ | tableconstructor | LiteralString 

	functiondef ::= function funcbody

	funcbody ::= ‘(’ [parlist] ‘)’ block end

	parlist ::= namelist [‘,’ ‘...’] | ‘...’

	tableconstructor ::= ‘{’ [fieldlist] ‘}’

	fieldlist ::= field {fieldsep field} [fieldsep]

	field ::= ‘[’ exp ‘]’ ‘=’ exp | Name ‘=’ exp | exp

	fieldsep ::= ‘,’ | ‘;’

	binop ::=  ‘+’ | ‘-’ | ‘*’ | ‘/’ | ‘//’ | ‘^’ | ‘%’ | 
		 ‘&’ | ‘~’ | ‘|’ | ‘>>’ | ‘<<’ | ‘..’ | 
		 ‘<’ | ‘<=’ | ‘>’ | ‘>=’ | ‘==’ | ‘~=’ | 
		 and | or

	unop ::= ‘-’ | not | ‘#’ | ‘~’
