Завітайте до Jinja2

Jinja2 є сучаною та дружньою до дизайнерів мовою шаблонів для Python, змодельованою подібно до шаблонів Django. Він є швидким, широко уживаним, та безпечним, з опціональним середовищем виконання в пісочниці:

<title>{% block title %}{% endblock %}title>
<ul>
{% for user in users %}
  <li><a href="{{ user.url }}">{{ user.username }}a>li>
{% endfor %}
ul>

Можливості:

  • виконання в пісочниці
  • потужна автоматична система екранування HTML для запобігання XSS
  • наслідування шаблонів
  • компілюється до оптимального коду python на льоту
  • опціональна передчасна компіляція шаблонів
  • простота налаштування. Номери рядків виключення напряму вказують на коректний рядок в шаблоні.
  • конфігурація синтаксису

Вступ

Це документація до шаблонної мови загального призначення Jinja2. Jinja2 є бібліотекою для Python, що розроблена бути гнучкою, швидкою та безпечною.

Якщо ви маєте любі навички з іншими базованими на тексті мовами шаблонів, як Smarty або Django, ви повинні почуватись як дома з Jinja2. Він є одночасно дружнім для дизайнерів та розробників, притримуючись принципів Python, та додаючи функціональність, корисну для шаблонних оточень.

Передумови

Jinja2 робить з Python 2.6.x, 2.7.x та >= 3.3. Якщо ви використовуєте Python 3.2, ви можете використовувати старішу версію Jinja2 (2.6), бо підтримка Python 3.2 була припинена в версії Jinja2 2.7.

Якщо ви бажаєте використовувати клас PackageLoader, вам також знадобляться setuptools або distribute, встановлені під час виконання.

Встановлення

Ви маєте декілька шляхів для встановлення Jinja2. Якщо ви непевні, що робити, використовуйте Python egg або tarball.

Як Python egg (через easy_install)

Ви можете встановити саму останню версію Jinja2 з використанням easy_install або pip:

easy_install Jinja2
pip install Jinja2

Це встановить Jinja2 egg в каталог пакунків вашої інсталяції Python.

(Якщо ви встановлюєте з командного рядка Windows, пропустіть sudo, та переконайтесь, що виконуєте командний рядок як користувач з адміністративними правами).

З tarball реліза

  1. Завантажте самий останній tarball з сторінки завантаження
  2. Розпакуйте архів
  3. sudo python setup.py install

Зауважте, що ви маєте встановити або setuptools або distribute; бажано останній. 

Це встановить Jinja2 в каталог пакунків вашої інсталяції Python.

Встановлення версії для розробки

  1. Встановіть git
  2. git clone git://github.com/mitsuhiko/jinja2.git
  3. cd jinja2
  4. ln -s jinja2 /usr/lib/python2.X/site-packages

Як альтернатива до кроку 4, ви можете також зробити setup.py develop, що встановить пакунок через distribute в режимі розробника. Це також має таку перевагу, що скомпілюються розширення C.

Залежність MarkupSafe

З версії 2.7 Jinja2 залежить від модуля MarkupSafe. Якщо ви встановите Jinja2 через pip або easy_install, він буде встановлений автоматично.

Базове використання API

Цей роділ дає вам короткий вступ до Python API для шаблонів Jinja2.

Найбільш базовий шлях для створення шаблону та відображення його через Template. Однак це не є рекомендованим шляхом робити це, якщо ваші шаблони не завантажені з рядків, але з фалової системи або іншого джерела даних:

>>> from jinja2 import Template
>>> template = Template('Hello {{ name }}!')
>>> template.render(name='John Doe')
u'Hello John Doe!'

Створюючи примірник Template ви повертаєте новий об'єкт шаблона, що провадить метод render(), який потім викликається з словник або ключовими аргументами, що розширюють шаблон. Словник або ключові аргументи, передані до шаблону є так званий “контекст” шаблону.

Що ви можете бачити тут, це що Jinja2 внутрішньо використовує юнікод, та повертає значення рядка юнікод. Так що переконайтесь, що ваше застосування також використовує юнікод зсередини.

Експеріментальна підтримка Python 3

Jinja 2.7 привносить експирементальну підтримку Python >=3.3. Це означає, що всі юніт тести проходять на новій версії, але там все ще можуть бути малі помилки, та поведінка може бути неузгодженою. Якщо ви помітили якусь помилку, будь ласка надайте зворотній зв'язок в трекері Jinja bug tracker.

Також майте на увазі, що документація написана з Python 2 на думці, так що вам доведеться адаптувати показані приклади коду на синтаксис Python 3 самотужки.

API¶

Цей документ описує API до Jinja2, та не мову шаблонів. Він буде найбільше корисний як посилання до тих, хте реалізує інтерфейс шаблонів з застосуванням, та не для тих, хто створює шаблони Jinja2.

Основи

Jinja2 використовує центральний об'єкт, що називається шаблонним  Environment. Примірники цього класу використовуються для зберігання конфігурації та глобальних об'єктів, та використовуються для завантаження шаблонів з файлової системи, або іншого розташування. Навіть якщо ви створюєте шаблони з рядків, використовуючи конструктор класу Template, оточення створюється автоматично за вас, хоча і одне загальне.

Більшість застосувань будуть створювати один об'єкт Environment при ініціалізації застосування, та використовувати його для завантаження шаблонів. Однак в деяких випадках корисно мати декілька оточень одночасно, в різних конфігураціях, що ви використовуєте.

найпростіший шлях сконфігурувати Jinja2 для завантаження шаблонів для вашого застоування виглядає приблизно так:

from jinja2 import Environment, PackageLoader
env = Environment(loader=PackageLoader('yourapplication', 'templates'))

Це буде створювати шаблонне оточення з налаштуваннями по замовчанню, та завантажувач, що шукає шаблони в папці templates в пакунку python yourapplication. Доступні різні завантажувачі, та ви можете також написати ваш власний, якщо ви бажаєте завантажувати шаблони з бази даних або інших ресурсів.

Щоб завантажити шаблон з цього оточення, вам тільки треба викликати метод get_template(), що повертає завантажений Template:

template = env.get_template('mytemplate.html')

Щоб відобразити його з деякими змінними, просто викличте метод  render():

print template.render(the='variables', go='here')

Використання завантажувача шаблону, скоріше, ніж передача рядків до  Template або Environment.from_string() має багато переваг. Окрім того, що це простіше використовувати, це також дозволяє наслідування шаблонів.

Юнікод

Jinja2 використовує Unicode внутрішньо, що означає, що ви маєте передавати об'єкти Unicode до функцій відображення, або байтові рядки, що складаються тільки з ASCII символів. Додатково нові рядки нормалізуються до однієї послідовності нового рядка, що по замовчанню в стилі UNIX (\n).

Python 2.x підтримує два шляхи представлення об'єктів рядків. Один є тип str, та інший тип unicode, обоє з яких розширюють тип, що називаєтсья basestring. Нажаль, по замовчанню іде str, що не повинен використовуватись для зберігання інформаціі на базі текста, за винятком, коли використовуються тільки символи ASCII. Починаючи з  Python 2.6 можливо зробити unicode по замовчанню на рівні модулів, та в Python 3 це буде по замовчанню.

Щоб явно використовувати рядок Unicode, ви маєте перед рядковим літералом поставити літеру u: u'Hänsel undGretel sagen Hallo'. Таким чином Python буде зберігати рядок як Unicode, декодуючи рядок з кодуванням символів з поточного модуля Python. Якщо кодування не вказане, по замовчанню це буде ‘ASCII’, що означає, що ви не можете використовувати жодних не-ASCII ідентифікаторів.

Щоб встановити краще кодування модуля, додайте наступний коментар в першому або другому рядку модуля Python, з використанням літерала Unicode:

# -*- coding: utf-8 -*-

Ми рекомендуємо кодування utf-8 в якості кодування для модулів та шаблонів Python, тому що тоді можливо представити любий можливий символ Unicode в utf-8, та оскільки це зворотньо сумісно з ASCII. Для Jinja2 кодуванням шаблонів по замовчанню вважається utf-8.

Неможливо використовувати Jinja2 для обробки не-Unicode даних. Причиною є те, що Jinja2 вже використовує Unicode на рівні мови. Наприклад, Jinja2 трактує не-переносячий проміжок як допустимий проміжок в виразах, що потребує знання кодування або операцій з рядком Unicode.

Для додаткових деталей щодо Unicode в Python подивіться на чудову документацію з Unicode.

Інша важлива річ, це як Jinja2 обробляє рядкові літералив шаблонах. Наївна реалізація буде використовувати рядки Unicode для всіх рядкових літералів, але як з'ясувалось в минулому, це проблематично, бо деякі бібліотеки явно перевіряють тип на str. Наприклад, datetime.strftime не сприймає аргументів Unicode. Щоб повністю не ламати це,  Jinja2 повертає str для рядків, що вміщуються в  ASCII, та unicode для вього іншого:

>>> m = Template(u"{% set a, b = 'foo', 'föö' %}").module
>>> m.a
'foo'
>>> m.b
u'f\xf6\xf6'

Високорівневий API

Високорівневий API є API, що ви будете використовувати в застосуванні для завантаження та відображення шаблонів Jinja2. Низькорівневий API, з другого боку, корисний тільки якщо ви бажаєте копати глибше в Jinja2, або розробляти розширення.

class jinja2.Environment([options])

Головний компонент Jinja є Environment. Він містить важливі загальні змінні, як конфігурацію, фільтри, тести, глобальні та інше. Примірники цього класу можуть бути модифіковані, якщо вони не розділені, та якщо досі шаблон не був завантажений. Модифікації оточення після того, як був завантажений перший шаблон призведе до несподіваних ефектів та невизначеної поведінки.

Ось можливі параметри ініціалізації:

block_start_string
Рядок, що позначає початок блоку. По замовчанню '{%'.
block_end_string
Рядок, що позначає кінець блока. По замовчанню '%}'.
variable_start_string
Рядок, що позначає початок твердження друку. По замовчанню '{{'.
variable_end_string
Рядок, що позначає кінець твердження друку. По замовчанню '}}'.
comment_start_string
Рядок, що позначає початок коментаря. По замовчанню '{#'.
comment_end_string
Рядок, що позначає кінець коментаря. По замовчанню '#}'.
line_statement_prefix
Якщо надано, та це рядок, буде використовуватись як префікс для тверджень на базі рядка. Дивіться також Твердження рядка.
line_comment_prefix

Якщо надано, та це рядок, буде використовуватись як суфікс для тверджень на базі рядка. Дивіться також Твердження рядка.

Нове в версії 2.2.

trim_blocks
Якщо встановлене в True, перший новий рядок після блоку видаляється (блок, не тег змінної!). По замовчанню False.
lstrip_blocks
Якщо встановлене в True, початкові проміжки та табуляції відкидаються від початку рядка до блоку. По замовчанню False.
newline_sequence
Послідовність, що починає новий рядок. Має бути одне з '\r', '\n' або '\r\n'. По замовчанню є '\n', що є корисним замовчанням для систем Linux та OS X, а також для веб застосувань.
keep_trailing_newline

Зберігає ноаий рядок при відтворенні шаблонів. По замовчанню False, що призводить що поодинокого нового рядка, якщо присутній, буде відкинуто наприкінці шаблону.

Нове в версії 2.7.

extensions
Список озширень Jinja, що треба використовувати. Це може бути шляхами імпорту в вигляді рядків, або класами розширення. Для додаткової інформації дивіться документацію щодо розширень.
optimized
чи включити оптимізатор? По замовчанню True.
undefined
Undefined або його субклас, що використовується для представлення невизначених значеньв шаблоні.
finalize
Дещо, що виклакається, та може бути використане для обробки результату виразу змінної, перед самим виводом. Наприклад, дехто може тут неявно конвертувати None в пустий рядок. 
autoescape

Якщо встановлене в true, можливість автоматичного екранування XML/HTML включена по замовчанню. Для додаткової інформації про автоекранування дивіться Markup. Як до Jinja 2.4, це може бути callable, що отримує ім'я шаблона, та повинне повертати True або False, в залежності від того, чи треба включити автоекранування по замовчанню.

Змінено в версії 2.4: autoescape тепер може бути функцією

loader
Завантажувач шаблону для даного оточення.
cache_size

Розмір кешу. По замовчанню це 400, що означає, що якщо завантажуються білше ніж 400 шаблонів, завантажувач буде очищувати найдавніше використані шаблони. Якщо розмір кешу встановленио в 0, шаблони рекомпілюються весь час, якщо розмір кешу -1, кеш не буде очищуватись.

Змінено в версії 2.8: Розмір кешу було збільшено до 400 з 50.

auto_reload
Деякі завантажувачі завантажують шаблони з локацій, де джерело шаблону може змінюватись (як файлова система або база даних). Якщо auto_reload встановлене в True (замовчання), кожного разу, коли шаблон запитується, завантажувач перевіряє, чи джерело змінилось, та якщо так, перезавантажує шаблон. Для кращої продуктивності можливо відключити це. 
bytecode_cache

Якщо встановлений об'єкт кешу байткоду, цей об'єкт буде провадити кеш для внутрішнього байткоду Jinja, так що шаблони не будуть розбиратись, якщо вони не змінились.

Дивіться Кеш байткоду для додаткової інформації.

shared

Якщо шаблон був створений з використанням конструктора Template, оточення створюється автоматично. Ці оточення створюються як загальні оточення, що означає, що декілька шаблонів можуть мати те саме анонімне оточення. Для всіх загальних оточень цей атрибут є True, інакше False.

sandboxed

Якщо оточення знаходиться в пісочниці, цей атрибут True. Для режима пісочниці дивіться документацію для  SandboxedEnvironment.

filters

Словник фільтрів для цього оточення. Доки шаблон не був завантажений, є безпечним додавати нові фільтри або видаляти старі. Для власних фільтрів дивіться Власні фільтри. Для допустимих імен фільтрів дивіться Зауваження щодо ідентифікаторів.

tests

Словник тестових функцій для цього оточення. Доки шаблон не був завантажений, є безпечним змінювати цей словник. Для власних тестів дивіться Власні тести. Для допустимих імен тестів дивіться Зауваження щодо ідентифікаторів.

Словник глобальних змінних. Ці змінні завжди доступні в шаблоні. Доки шаблон не був завантажений, є безпечним змінювати цей словник. Для додаткових деталей дивіться Глобальний простір імен. Для допустимих імен об'єктів дивіться Зауваження щодо ідентифікаторів.

Клас, що використовується для генерації кода. Його не треба змінювати в більшості випадків, за винятком коли вам треба змінити код Python, до якого компілюється шаблон.

context_class

Контекст, використуваний для шаблонів. Його не треба змінювати в більшості випадків, за винятком коли вам треба модифікувати подробиці обробки змінних шаблону. Для деталей дивіться Context.

overlay([options])

Створює нове оверлейне оточення, що поділяє всі дані з поточним оточенням, за винятком кеша та перекритих атрибутів. Розширення не можуть бути видалені для оверлейного оточення. Оверлейне оточення автоматично отримує всі розширення оточення, до якого він прив'язаний, плюс опціонально додаткові розширення.

Створення оверлеїв повинно відбуватись після того, як початкове оточення повністю налаштоване. Наспавді не всі атрибути пов'язуються, деякі просто копіюються наверх, так що модифікація оригінального оточення може не пробитись назовні. 

undefined([hint, obj, name, exc])

Створює новий об'єкт Undefined для name. Це корисне для фільтрів або функцій, що можуть повертати невизначені об'єкти для деяких операцій. Всі параметри, за винятком hint, повинні бути проваджені як ключові параметри, для кращої читабельності. hint, якщо надане, використовується як повідомлення помилки для виключення, інакше повідомлення помилки буде згенеровано з obj та name автоматично. Виключення, що провадиться як exc підіймається, якщо зроблене щось зі згенерованим невизначеним об'єктом, що невизначений об'єкт не дозволяє. Виключення по замовчанню є UndefinedError. Якщо hint запроваджено, name можна не вказувати.

Найбільш загальний спосіб створити невизначений об'єкт є провадження одного імені:

return environment.undefined(name='some_name')

Це означає, що ім'я some_name не визначене. Якщо ім'я було від атрибута об'єкта, має сенс вказати, що невизначений об'єкт є холдером, щоб покращити повідомлення про помилку:

if not hasattr(obj, 'attr'):
    return environment.undefined(obj=obj, name='attr')

Для більш складного прикладу ви можете провадити підказку. Наприклад, фільтр first() створює невизначений об'єкт в цей спосіб:

return environment.undefined('no first item, sequence was empty')

Якщо це ім'я name або obj відоме (наприклад, тому що атрибут мав доступ), він має бути переданий до невизначеного об'єкту, навіть якщо провадиться власний hint. Це дає невизначеним об'єктам можливість покращити повідомлення про помилку.

add_extension(extension)

compile_expression(source, undefined_to_none=True)

Зручний допоміжний метод, що повертає callable, що приймає ключові аргументи, що з'являються як змінні в виразі. Якщо викликане, повертає результат виразу.

Це корисне, якщо застосування бажає використовувати ті самі правила, як Jinja в “файлах конфігурації” шаблона, або подібних ситуаціях.

Приклад використання:

>>> expr(foo=23)

>>> expr(foo=42)

>>> env.compile_expression('var')() is None
True
>>> env.compile_expression('var', undefined_to_none=False)()

compile_templates(target, extensions=None, filter_func=None, zip='deflated',log_function=None, ignore_errors=True, py_compile=False)

Шукає всі шаблони, що завантажувач може знайти, компілює їх, та зберігає в target. Якщо zip є None, замість файлу zip шаблони будуть збережені в каталозі. По замовчанню використовується zip. Щоб переключитись на алгоритм stored, zip може бути встановлене в 'stored'.

Якщо ви бажаєте, щоб помилки синтаксису шаблонів переривали компіляцію, ви можете встановити  ignore_errors в False, та ви будете отримувати переривання за синтаксичних помилок.

class jinja2.Template

Центральний об'єкт шаблону. Цей клас представляє скомпільований шаблон, та використовується для його обчислення.

Звичайно об'єкт шаблону генеруєтсья з Environment, але він також має конструктор, що робить можливим створити примірник шаблона напряму, використовуючи конструктор. Він сприймає ті ж аргументи, що і конструктор оточення, але неможливо вказати завантажувач.

Кожний об'єкт шаблону має декілька методів та членів, що гарантовано існують. Однак важливо, що об'єкт шаблону повинно розглядати як незмінний. Модифікації об'єкта не підтримуются.

Об'єкти шаблонів, створені через конструкутор, скоріше ніж через оточення, мають атрибут environment, що вказує на тимчасове оточення, що, можливо, є загальним з іншими шаблонами, створеними через конструктор, та сумісними налаштуваннями.

>>> template = Template('Hello {{ name }}!')
>>> template.render(name='John Doe') == u'Hello John Doe!'
True
>>> stream = template.stream(name='John Doe')
>>> next(stream) == u'Hello John Doe!'
True
>>> next(stream)
Traceback (most recent call last):
    ...
StopIteration
globals

Словник з глобальними змінними цього шаблону. Є небезпечним можифікувати цей словник, оскільки він може бути загальним з іншими шаблонами, або з оточенням, що завантажило шаблон.

name

Ім'я завантаження шаблона. Якщо шаблон був завантажений з рядка, це буде None.

filename

Ім'я файлу для шаблону в файловій системі, якщо він був завантажений звідти. Інакше це None.

render([context])

Цей метод сприймає ті ж аргументи, що і конструктор dict: словник, субклас словника, або деякий аргументи з ключовими словами. Якщо аргументів не надано, контекст буде пустий. Ці две виклики роблять одне і те ж:

template.render(knights='that say nih')
template.render({'knights': 'that say nih'})

Це повертатиме відображений шаблон як рядок юнікод.

generate([context])

Для любих великих шаблонів може бути корисним не малювати цілий шаблон за раз, але обчислити кожний вираз один за одинм, та отримувати частину за частиною. Цей метод в основному робить саме це, та повертає те що видає генератор, елемент за елементом, в вигляді юнікод рядків.

Він сприймає ті ж аргументи, що і render().

stream([context])

Робить точно як generate(), але повертає TemplateStream.

make_module(vars=None, shared=False, locals=None)

Цей метод робить як атрибут module, коли викликане без аргументів, але буде обчислювати шаблон при кожному виклику, скоріше, ніж кешувати його. Також можливо запровадити словник, що потім використовується як контекст. Аргументи ті ж самі, що і для метод new_context().

module

Шаблон як модуль. Це використовується для імпорту в рантаймі шаблону, але також корисне, якщо хтось бажає отримати доступ до експортованих змінних шаблону з рівня Python:

>>> str(t.module)

>>> t.module.foo() == u'42'

class jinja2.environment.TemplateStream

Потік шаблону робить досить подібно до звичайного генератора python, але він може буферизувати декілька елементів для зменшеннячисла загальних ітерацій. По замовчення вивод небуферизований, що означає, що для кожної небуферизованої інструкції в шаблоні буде отриманий один рядок юнікод.

Автоекранування

Нове в версії 2.4.

З Jinja 2.4 the переважним способом автоекранування є ввімкнення Розширення автокранування, та конфігурація розумних налаштувань по замовчанню для автоекранування. Це робить можливим вмикати та вимикати автоекранування на по-шаблонній основі (наприклад, для HTML та тексту).

Ось рекомендоване налаштування, що дозволяє автоекранування для шаблонів, що завершуються на '.html', '.htm'and '.xml', вимкненняйого для всіх інших розширень:

def guess_autoescape(template_name):
    if template_name is None or '.' not in template_name:
        return False
    ext = template_name.rsplit('.', 1)[1]
    return ext in ('html', 'htm', 'xml')

env = Environment(autoescape=guess_autoescape,
                  loader=PackageLoader('mypackage'),
                  extensions=['jinja2.ext.autoescape'])

При реалізації функції розпізнання автоекранування, переконайтесь, що ви тако жсприймаєте None в якості можливого імені шаблону. Це буде передаватись, коли генеруються шаблони з рядків.

В шаблонах поведінка може бути тимчасово змінена з використанням блоку autoescape (дивіться Розширення автоекранування).

Зауваження щодо ідентифікаторів

Jinja2 використовує звичайні правила для імен Python 2.x. Можливі ідентифікатори мають співпадати з шаблоном [a-zA-Z_][a-zA-Z0-9_]*. Фактично, не-ASCII символи наразі не підтримуються. Це обмеження можливо скоро буде прибране, як тільки юнікод ідентифікатори будуть повністю специфіковані в Python 3.

Фільтри та тасти шукаються в різних просторах імен, та мають дещо рязних модифікований синтаксис ідентифікаторів. Фільтри та тести можуть містити крапки, щоб групувати фільтри та тести за темою. Для прикладу, є досить прийнятним, додати функцію в словник фільтрів, та назвати його to.unicode. Регулярний вираз для ідентифікаторів фільтру  та тесту є [a-zA-Z_][a-zA-Z0-9_]*(\.[a-zA-Z_][a-zA-Z0-9_]*)*`.

Невизначені типи

Ці класи можуть бути використані як невизначені типи. Конструктор Environment бере параметр undefined, що може бути одним з ціх класів, або власного суб-класу Undefined. Кожного разу, коли шаблон не в змозі знайти ім'я або отриматти доступ до атрибуту, створюється та повертається один з ціх об'єктів. Деякі операції на невизначених значеннях допустимі, інші даватимуть збій.

Найближчий до звичайної поведінкої Python є StrictUndefined, що недозволяє всі операції, окрім тестування, чи це невизначений об'єкт.

Коли викликаєтся з любими аргументами, цей метод підіймає _undefined_exception з повідомленням про помилку, згенерованим від невизначених підказок, що зберігаютсья на невизначеному об'єкту.

>>> str(foo)

>>> not foo
True

  ...

Невизначений об'єкт, що лається при друкові та ітерації, так само, як і за логічних перевірках, та всіх різновидах порівнянь. Іншими словами, ви не можете нічого робити з ним, за винятком перевірки, що він визначений, з використанням перевірки defined.

Є також функції-фабрики, що може декорувати невизначені об'єкти для реалізації журналювання при збоях:

jinja2.make_logging_undefined(logger=None, base=None)

Надається об'єкт журналювання, повертає новий невизначений клас, що буде журналювати певні відмови. Він буде записувати ітерації та друк. Якщо журналювальник не наданий, буде створений по замовчанню.

Приклад:

logger = logging.getLogger(__name__)
LoggingUndefined = make_logging_undefined(
    logger=logger,
    base=Undefined
)

Нове в версії 2.8.

Параметри:
  • logger – журналювальник, що треба використовувати. Якщо не наданий, буде створений по замовченню.
  • base – базовий клас для додавання до нього функціональності журналювання. По замовчанню Undefined.

Невизначені об'єкти створені за допомогою виклика undefined.

Реалізація

Об'єкти Undefined реалізовані перекриттям спеціальних методів __underscore__. Наприклад, клас по замовчанню Undefined реалізує __unicode__ таким чином, що вн повертає пустий рядок, однак __int__ та інші все ще схиблюють з виключенням. Щоб зробити конвертацію до int повертаючи 0, ви можете реалізувати вашу власну:

class NullUndefined(Undefined):
    def __int__(self):
        return 0
    def __float__(self):
        return 0.0

Щоб відключити цей метод, просто перекрийте його, та підійміть  _undefined_exception. Оскільки це дуже загальна ідіома в невизначених об'єктах, існує допоміжний метод _fail_with_undefined_error(), що робить підняття помилки автоматично. Ось клас, що робить як регулярнний Undefined, але затикається на ітерації:

class NonIterableUndefined(Undefined):
    __iter__ = Undefined._fail_with_undefined_error

Контекст

class jinja2.runtime.Context

Контекст шаблона містить змінні шаблона. Він зберігає значення, передані до шаблону, та також імена, які експортує шаблон. Створення примірників однаково, не підтримується, та не корисне, тому що вони створюються автоматично, на різних стадіях обчислення шаблону, та не повинні створюватись власноруч.

Контекст є незмінним. Модифікації parent не мають відбуватись, та модифікації vars дозволені тільки зі згенерованого коду коду шаблону. Фільтри шаблону  та глобальні функції, помічені як contextfunction(), отримують активний контекст, переданий чк перший аргумент, та дозволений доступ до контексту тільки для читання.

Контекст шаблону підтримує операції словника тільки для читання (get, keys, values, items, iterkeys,itervalues, iteritems, __getitem__,  __contains__). Додатково є метод resolve(), що не схиблює з KeyError, але повертає об'єкт Undefined для відсутніх змінних.

parent

Словник тільки для читання, глобальні змінні, які шукає шаблон. Вони можуть походити з іншого Context, з Environment.globals або Template.globals, або вказує на словник, створений комбінацією глабальних змінних, переданих до функції відображення. Він не може бути змінений.

vars

Локальні змінні шаблону. Цей список містить функції оточення та контексту з поля зору parent, так само, як локальні модифікації та експортовані змінні з шаблону. Шаблон буде модифікувати цей словник на протязі обчислення шаблону, але функції фільтрів та контексту не можуть модифікувати його.

environment

Оточення, що завантажує шаблон.

exported_vars

Цей набір містить всі імена, що експортує шаблон. Значення для імен знаходяться в словнику vars. Щоб отримати копію експортованих змінних як словника можна використати  get_exported().

name

Завантажене ім'я шаблону, що володіє контекстом.

blocks

Словник з поточним відображенням блоків в шаблоні. Ключі в цьому словникові є імена блоків, та значення - список зареєстрованих блоків. Останній елемент в кожномусписку є поточний активний блок (останній в ланцюжку наслідування).

eval_ctx

Поточний Контекст обчислення.

call(callable, *args, **kwargs)

Виклик callable з запровадженими аргументами та ключовими аргументами, але з підстановкою  активного контексту або оточення, як першого аргументу, якщо callable є contextfunction() або environmentfunction().

get_all()

Повертає копію повного контексту як словника, включаючи експортовані змінні.

get_exported()

Отримує новий словник з експортованими змінними.

resolve(key)

Шукає змінну, як __getitem__ або get, але повертає об'єкт Undefined з ім'ям шо шукаєтсья.

Реалізація

Контекст є незмінним з тих самих причин, що локальні змінні фрейма  Python є незмінними в функціях. Обоє, Jinja2 та Python, не використовують локальні контексту/фрейма в якості сховища даних для змінних, але тільки як первинне джерело даних.

Коли шаблон отримує доступ до змінної, що не визначає шаблон, Jinja2 шукає змінну в контексті, після чого змінна трактується якби вона була визначена в шаблоні.

Завантажувачі

Завантажувачі відповідальні за завантаження шаблонів з ресурсів, таких, як файлова система. Оточення буде тримати компільовані модулі в пам'яті, як Python sys.modules. Однак на відміну від sys.modules, цей кеш обмежений в розмірі по замовчанні, та шаблони автоматично перезавантажуються. Всі завантажувача є субкласами  BaseLoader. Якщо ви бажаєте створити ваш сласний завантажувач, почніть з  BaseLoader, та перекрийте get_source.

class jinja2.BaseLoader

Базовий клас для всіх завантажувачів. Створіть його субклас, та перекрийте get_source, для реалізації власного механізму завантаження. Оточення продвадить метод get_template, що викликає метод завантажувача load для отримування об'єкта Template.

Дуже простий приклад для завантажувача, що шукає шаблоні в файловій системі, виглядає таким чином:

from jinja2 import BaseLoader, TemplateNotFound
from os.path import join, exists, getmtime

class MyLoader(BaseLoader):

    def __init__(self, path):
        self.path = path

    def get_source(self, environment, template):
        path = join(self.path, template)
        if not exists(path):
            raise TemplateNotFound(template)
        mtime = getmtime(path)
        with file(path) as f:
            source = f.read().decode('utf-8')
        return source, path, lambda: mtime == getmtime(path)
get_source(environment, template)

Отримує джерело шаблона, ім'я файла та допоміжник перезавантажувача для шаблона. Отримує оточення та ім'я шаблону, та має повертати кортеж в формі (source,filename, uptodate), або викликати помилку TemplateNotFound, якщо не може знайти шаблон.

Частина джерела source повернутого кортежу має бути джерелом шаблону в вигляді юнікод рядка або байтовим рядком ASCII. Ім'я файла filename повинно бути ім'ям файла в файловій системі, якщо шаблон завантажений звідти, або None. Ім'я файла використовується в python для трасувань, якщо не використовується розширення завантажувача.

Останній елемент в кортежі є функція uptodate. Якщо автоматичне перезавантаження ввімкнено, вона завжди викликається для перевірки, чи шаблон змінено. Жодних аргументів не передається, так щофункція має зберігати старий стан деінде (наприклад в замиканні). Якщо вона повертає False, шаблон буде перезавантажений.

load(environment, name, globals=None)

Завантажує шаблон. Цей метод шукає шаблон в кеші, або завантажує його, викликаючи get_source(). Субкласи не повинні перекривати цей метод, як завантажувачі, робляци з колекціями інших завантажувачів (таких як PrefixLoader або ChoiceLoader) не будуть викликати цей метод, але get_source напряму.

Ось перелік вбудованих завантажувачів, що провадить Jinja2:

class jinja2.FileSystemLoader(searchpath, encoding='utf-8', followlinks=False)

Завантажує шаблони з системи. Цей завантажувач може знайти шаблони в теках файлової системи, та є переважним способом їх завантаження.

Завантажувач бере шлях до шаблонів як рядок, або, якщо треба вказати декілька локацій, їх список, що потім переглядається в наданому порядку:

>>> loader = FileSystemLoader('/path/to/templates')
>>> loader = FileSystemLoader(['/path/to/templates', '/other/path'])

По замовчанню кодування шаблонів 'utf-8', що може бути змінене встановленням параметра encoding на щось інше.

Щоб слідувати символічним посиланням, встановіть параметр followlinks в True:

>>> loader = FileSystemLoader('/path/to/templates', followlinks=True)

Змінено в версії 2.8+: Доданий параметр followlinks.

class jinja2.PackageLoader(package_name, package_path='templates', encoding='utf-8')

Завантажує шаблони з яєць python eggs або пакунків. Він побудований з ім'ям пакунка python, та шляха до шаблонів в пакунку:

loader = PackageLoader('mypackage', 'views')

Якщо шлях пакунка не наданий, передбачається 'templates'.

По замовчанню кодування шаблону 'utf-8', що може бути змінено встановленням encoding в щось інше. Через природу яєць, перезавантаження шаблонів можливе, тільки якщо пакунки завантажені з файлової системи, та не zip файла.

class jinja2.DictLoader(mapping)

Завантажує шаблон зі словника python. Йому передається словник з юнікод рядків, прив'язаних до імен шаблонів. Цей завантажувач корисний для юніт тестів:

>>> loader = DictLoader({'index.html': 'source here'})

Оскільки авто перезавантаження рідко корисне, воно відключене по замовчанню.

class jinja2.FunctionLoader(load_func)

Завантажувач, що отримує функцію, що виконує завантаження. Функція отримує ім'я шаблона, та має повертати або юнікод рядок з джерелом шаблона, кортеж в формі (source, filename,  uptodatefunc), або None, якщо шаблон не існує.

>>> def load_template(name):
...     if name == 'index.html':
...         return '...'
...
>>> loader = FunctionLoader(load_template)

uptodatefunc є функцією, що викликається, якщо включене авто перезавантаження, та має повертати True, якщо шаблон досі актуальний. Для додаткових деталей дивіться  BaseLoader.get_source(), що повертає те саме значення.

class jinja2.PrefixLoader(mapping, delimiter='/')

Завантажувач, що отримує словник завантажувачів, де кожний завантажувач прикріплений до префіксу. Префікс обмежений від шаблону косою по замовчанню, що може бути змінене  встановленням аргументу delimiter на щось інше:

loader = PrefixLoader({
    'app1':     PackageLoader('mypackage.app1'),
    'app2':     PackageLoader('mypackage.app2')
})

Завантаження 'app1/index.html' завантажує файл з пакунку app1, завантаження app2/index.html' візьме файл з другого пакунку.

class jinja2.ChoiceLoader(loaders)

Цей завантажувач робить як PrefixLoader, тільки коли префікс на вказаний. Якщо шаблон не може бути знайдений одним завантажувачем, буде спробуваний наступний.

>>> loader = ChoiceLoader([
...     FileSystemLoader('/path/to/user/templates'),
...     FileSystemLoader('/path/to/system/templates')
... ])

Це корисне, якщо ви бажаєте дозволити користувачам перекривати вбудовані шаблони, на такі з іншого розташування.

class jinja2.ModuleLoader(path)

Цей завантажувач завантажує шаблони з попередньо скомпільованих шаблонів.

Приклад використання:

>>> loader = ChoiceLoader([
...     ModuleLoader('/path/to/compiled/templates'),
...     FileSystemLoader('/path/to/templates')
... ])

Кеш байткодів

Jinja 2.1 та вище підтримує зовнішнє кешування байткодів. Кеші байткодів роблять можливим зберігати згенерований байткод в файловій системі або іншому розташуванні, щоб уникнути розбору шаблона при першому застосуванні.

Це особливо корисне, якщо ви маєте веб застосування, що ініціюється при першому запиті, та Jinja компілює багато шаблонів за раз, що уповільнює застосування.

Для використання кешу байткодів, створіть його примірник, та передайте його в Environment.

class jinja2.BytecodeCache

Щоб реалізувати ваш власний кеш байткодів, ви маєте створити субклас цього класу, та перекрити load_bytecode() та dump_bytecode(). Обоє з ціх методів передаються в  Bucket.

Дуже простий кеш байткодів, що зберігає байткоди в файловій системі:

from os import path

class MyCache(BytecodeCache):

    def __init__(self, directory):
        self.directory = directory

    def load_bytecode(self, bucket):
        filename = path.join(self.directory, bucket.key)
        if path.exists(filename):
            with open(filename, 'rb') as f:
                bucket.load_bytecode(f)

    def dump_bytecode(self, bucket):
        filename = path.join(self.directory, bucket.key)
        with open(filename, 'wb') as f:
            bucket.write_bytecode(f)

Більш складна версія кешу, що базується на на файловій системі, є частиною Jinja2.

clear()

Очищує кеш. Цей метод не використовується в Jinja2, але має бути реалізований, щоб дозволити застосуванням  очищувати кеш байткодів, з використанням певного оточення.

dump_bytecode(bucket)

Субкласи мають перекривати цей метод, щоб записувати байткод з бакета назад до кешу. Якщо неможливо це зробити, він не має тихо схибити, але підняти виключення.

load_bytecode(bucket)

Субкласи мають перекривати цей метод, щоб завантажити байткод в бакет. Якщо неможливо знайти код в кеші для бакета, нічого не треба робити.

class jinja2.bccache.Bucket(environment, key, checksum)

Бакети використовуються для зберігання байткода для одного шаблона. Він створюється та ініціюється кешем байткода, та передається до функцій завантаження.

Бакети отримують внутрішню контрольну суму від призначеного кеша, та використовують його для автоматичного відхилення простроченого матеріалу з кеша. Окремий субклас кеша байткода не має турбуватись про прострочений код.

environment

Environment, що створив бакет.

key

Унікальний ключ кеша для цього бакета

code

Байткод, якщо завантажений, або None.

bytecode_from_string(string)

Завантаження байткода з рядка.

bytecode_to_string()

Повертає байткод як рядок.

load_bytecode(f)

Завантажує байткод з файла, або схожого на файл об'єкта.

reset()

Скидає бакет (відвантажує байткод).

write_bytecode(f)

Скидає байткод в файл, або схожий на файл об'єкт.

Вбудовані кеші байткода:

class jinja2.FileSystemBytecodeCache(directory=None,pattern='__jinja2_%s.cache')

Кеш байткода, що зберігає байткод в файловій системі. Він сприймає два аргументи: каталог, де зберігаються елементи кешу, та рядок шаблону, що використовується для побудови імені файла.

Якщо каталог не вказаний, обирається каталог по замовчанню. На Windows це тимчасовий каталог користувача, на UNIX системах каталог створюється від імені користувача в системному тимчасовому каталозі.

Шаблон може використовуватись для декількох окремих кешів, оперуючих на тому ж каталозі. Шаблон по замовчанню  '__jinja2_%s.cache'. %s замішується на ключ кеша.

>>> bcc = FileSystemBytecodeCache('/tmp/jinja_cache', '%s.cache')

Цей кеш байткодів підтримує очистку кешу з використанням метода clear.

class jinja2.MemcachedBytecodeCache(client, prefix='jinja2/bytecode/', timeout=None,ignore_memcache_errors=True)

Цей клас реалізує кеш байткода, що використовує кеш memcache для зберігання інформації. Він не примушує специфічну бібліотеку memcache (tummy memcache або cmemcache), але буде сприймати любий клас, що провадить мінімальний потрібний інтерфейс.

Бібліотеки, сумісні з цім класом:

  • werkzeug.contrib.cache
  • python-memcached
  • cmemcache

(На жаль, інтерфейс кеша django несумісний, оскільки він не підтримує зберігання двоїчних даних, тільки юнікод. Ви можете, однак, передати підлеглий клієнт кеша до кеша байткода, що доступний як django.core.cache.cache._client.)

Мінімальний інтерфейс для клієнта, переданого до конструктора, наступний:

class MinimalClientInterface
set(key, value[, timeout])

Зберігає байткод в кеші. value є рядок та timeout таймаут ключа. Якщо таймаут не запроваджено, таймаут по замовчанню, або його відсутність буде впроваджено. Якщо є, то це ціле число секунд, скільки буде існувати елемент кеша .

get(key)

Повертає значення для ключа кеша. Якщо елемент не існує, кеш повертає значення None.

Інші аргументи до конструктора є префікс для всіх ключів, що додається справжнім ключом кеша, та таймаут для байткода в системі кешування. Ми рекомендуємо високий (або відсутній) таймаут.

Цей кеш байткода не підтримує очищення використуваних елементів в кеші. Метод очистки є функцією без призначення.

Нове в версії 2.7: Додана підтримка для ігнорування помилок  memcache через параметр ignore_memcache_errors.

Утіліти

Ці допоміжні функції та класи корисні, якщо ви додаєте власні фільтри або функції для оточення Jinja2.

jinja2.environmentfilter(f)

Декоратор для створення залежних від оточення фільтрів. Поточний Environment передається до фільтра як перший аргумент.

jinja2.contextfilter(f)

Декоратор для створення залежних від контексту фільтрів. Поточний Context буде переданий як перший аргумент.

jinja2.evalcontextfilter(f)

Декоратор для створення eval-контекст залежних фільтрів. Об'єкт eval конектст передається як перший аргумент. Для додаткової інформації щодо eval контексту дивіться Контексти обчислення.

Нове в версії 2.4.

jinja2.environmentfunction(f)

Цей декоратор може бути використаний для відмітки функції або метода як callable оточення. Цей декоратор робить точно як декоратор contextfunction(), за винятком що його перший аргумент є активний Environment, а не контекст.

jinja2.contextfunction(f)

Цей декоратор може біти використаний для помітки функції або метода як callable контексту. Йому передається активний Context як перший аргумент, коли він викликається з шаблону. Це корисно, якщо функція хоче отримати доступ до контексту або функцій, що провадяться на об'єкті контексту. Наприклад, функція, що повертає впорядкований список змінних шаблона, що експортує поточний шаблон, може виглядати так:

@contextfunction
def get_exported_names(context):
    return sorted(context.exported_vars)
jinja2.evalcontextfunction(f)

Цей декоратор може бути використаний для помітки функції або метода як callable eval контекста. Це подібне до contextfunction(), але замість передачі контекста, передається об'єкт контекста обчислення. Для додаткової інформації щодо eval контекста дивіться Контекст обчислення.

Нове в версії 2.4.

jinja2.escape(s)

Конвертує символи &, <, >, ', та " в рядку s на HTML-безпечні послідовності. Використовуйте її, якщо вам треба відобразити текст, що може містит такі символи в HTML. Ця функція не буде екранувати о'єкти, що мають HTML представлення, як вже екрановані дані.

Повертає значення як рядок Markup.

jinja2.clear_caches()

Jinja2 утримує внутрішні кеші для оточень та лексерів. Вони використовуються так, що Jinja2 не повинна перестворювати оточення та лексери кожного разу. Звичано ви не повинні турбуватись про це, але якщо ви вимірюєте споживання пам'яті, ви можете забажати очистити кеші.

jinja2.is_undefined(obj)

Перевіряє, чи переданий об'єкт невизначений. Це не робить нічого більшого, ніж виконує перевірку примірника на Undefined, але виглядає краще. Це може бути використане для власних фільтрів або тестів, що бажають реагувати на невизначені змінні. Наприклад, власний фільтр по замовчанню може виглядати таким чином:

def default(var, default=''):
    if is_undefined(var):
        return default
    return var
class jinja2.Markup([string])

Помічає рядок, як безпечний для вставки в вивід HTML/XML без необхідності екранування. Це реалізує інтерфейс __html__, що використовують пара фреймворків та веб застосувань. Markup є прямим субкласом unicode, та провадить всі методи unicode, так що екранує всі передані аргументи та завжди повертає Markup.

Функція escape повертає об'єкти розмітки, так що подвійне екранування не відбувається.

Конструктор класу Markup може бути використаний для трьох різних речей: коли надається об'єкт юнікод, він вважається безпечним. Коли переданий об'єкт має HTML представлення (що має метод __html__), використовується це представлення. Інакше переданий об'єкт конвертується до рядка юнікод, що вважається безпечним:

>>> Markup("Hello World!")
Markup(u'Hello World!')
>>> class Foo(object):
...  def __html__(self):
...   return 'foo'
...
>>> Markup(Foo())
Markup(u'foo')

Якщо ви бажаєте, щоб переданий об'єкт трактувався як небезпечний, ви можете використовувати метод класу escape() для створення об'єкту Markup:

>>> Markup.escape("Hello World!")
Markup(u'Hello World!')

Оператори на рядку розміки познані щодо розмітки, що означає, що всі аргументи проходять через функцію escape():

>>> em = Markup("%s")
>>> em % "foo & bar"
Markup(u'foo & bar')
>>> strong = Markup("%(text)s")
>>> strong % {'text': 'hacker here'}
Markup(u'hacker here')
>>> Markup("Hello ") + ""
Markup(u'Hello ')
classmethod escape(s)

Екранує рядок. Робить як escape(), з тією різницею, що для субкласів Markup ця функція буде повертати коректний субклас.

striptags()

Відекранює розмітку в рядок text_type, та виключає всі теги. Це також розрішує відомі сутності HTML4 та XHTML. Проміжки нормалізуються до одного:

>>> Markup("Main »  About").striptags()
u'Main \xbb About'
unescape()

Відекранює розмітку знову, до рядка text_type. Це також розрішує відомі сутності HTML4 та XHTML:

>>> Markup("Main » About").unescape()
u'Main \xbb About'

Зауваження

Клас Jinja2 Markup є сумісним з, щонайменьше, Pylons та Genshi. Очікується, що більше фреймворків та рушійників шаблонів будуть сприймати концепцію __html__.

Виключення

exception jinja2.TemplateError(message=None)

Базовий клас для всіх помилок шаблонів.

exception jinja2.UndefinedError(message=None)

Підіймається, якщо шаблон намагається оперувати Undefined.

exception jinja2.TemplateNotFound(name, message=None)

Підіймається, якщо шаблон ні існує.

exception jinja2.TemplatesNotFound(names=(), message=None)

Як TemplateNotFound, але підіймається, якщо обране декілька шаблонів. Є субкласом до виключення TemplateNotFound, таким чином перехоплення цього буде перехоплювати обоє.

Неве в версії 2.2.

exception jinja2.TemplateSyntaxError(message, lineno, name=None,filename=None)

Підіймається, щоб сказати користувачеві, що шаблон має проблему.

message

Повідомлення про помилку як байтрядок utf-8.

lineno

Номер рядка, де виникла помилка.

name

Ім'я завантаження для шаблона, як рядок юнікода.

filename

Ім'я файла, звідки завантажений шаблон, як байтрядок в кодуванні файлової системи (найбільш вірогідно utf-8 або mbcs на  Windows).

Причиною, чому ім'я фала та повідомлення про помилку є байтовими рядками, але не юнікодом, полягає в тому, що Python 2.x не використовує юнікод для виключень та журналу, так само, як і компілятор. Це зміниться в Python 3.

exception jinja2.TemplateAssertionError(message, lineno, name=None,filename=None)

Як і помилка синтаксису шаблона, але покриває випадки, коли дещо в шаблоні викликає помилку під час компіляції, що не обов'язково викликане синтаксичною помилкою. Однак це прямий субклас до TemplateSyntaxError, та має ті ж атрибути.

Власні фільтри

Власні фільтри є звичайними фукнціями Python, що сприймають ліву сторону фільтру як перший аргумент, та надані до фільтра аргумента як додаткові агрументи або ключові аргументи.

Наприклад, функція фільтра {{ 42|myfilter(23) }} буде викликана як  myfilter(42, 23). Цей приклад є простим фільтром, що може бути застосований до об'єкта дати, щоб відформатувати її:

def datetimeformat(value, format='%H:%M / %d-%m-%Y'):
    return value.strftime(format)

Ви можете зареєструвати його на оточенні шаблону, оновивши словник filters для оточення:

environment.filters['datetimeformat'] = datetimeformat

В самому шаблоні ви можете використовувати його наступним чином:

written on: {{ article.pub_date|datetimeformat }}
publication date: {{ article.pub_date|datetimeformat('%d-%m-%Y') }}

Фільтри також можуть бути передані в поточний контекст шаблона, або оточення. Це корисне, якщо фільтр бажає повернути невизначене значення, або перевірити поточне налаштування autoescape. Для цієї цілі існують три декоратори:  environmentfilter(),  contextfilter() та evalcontextfilter().

Ось простий приклад фільтра, що переводить текстові переводи рядка в  переноси та параграфи HTML, та помічає повернуте значення як безпечний рядок HTML, якуо ввімнене автоекранування:

import re
from jinja2 import evalcontextfilter, Markup, escape

_paragraph_re = re.compile(r'(?:\r\n|\r|\n){2,}')

@evalcontextfilter
def nl2br(eval_ctx, value):
    result = u'\n\n'.join(u'
%s
' % p.replace('\n', Markup('
\n'))
                          for p in _paragraph_re.split(escape(value)))
    if eval_ctx.autoescape:
        result = Markup(result)
    return result

Фільтри контексту роблять таким же чином, за винятком що перший аргумент є поточним активним Context, але не оточенням.

Контекст обчислення

Контекст обчислення (скорочено eval контекст, або eval ctx) є новим об'єктом, введеним в Jinja 2.4, що робить можливим активувати або відключити скомпільовані можливості під час виконання.

Наразі він використовується тільки для підключення та відключення автоматичного екранування, але він може бути використаний також для розширень.

В попередніх версіях Jinja фільтри та функції були марковані як callables оточення, щоб перевіряти статус автоекранування з оточення. Замість цього в нових версіях заохочуєтся перевіряти налаштування з контексту обчислення.

Попередні версії:

@environmentfilter
def filter(env, value):
    result = do_something(value)
    if env.autoescape:
        result = Markup(result)
    return result

В нових весіях ви можете або використовувати contextfilter(), та мати доступ до контексту обчислення з актуального контексту, або використовувати evalcontextfilter(), що напряму передає контекст обчислення до функції:

@contextfilter
def filter(context, value):
    result = do_something(value)
    if context.eval_ctx.autoescape:
        result = Markup(result)
    return result

@evalcontextfilter
def filter(eval_ctx, value):
    result = do_something(value)
    if eval_ctx.autoescape:
        result = Markup(result)
    return result

Контекст обчислення не має бути модифікований під час виконання. Модифікації мають відбуватись в anodes.EvalContextModifier та nodes.ScopedEvalContextModifier з розширення, не на самому об'єкті контексту обчислення.

class jinja2.nodes.EvalContext(environment, template_name=None)

Утримує інформацію часу обчислення. В розширеннях до неї можуть приєднатись власні атрибути.

autoescape

True або False, в залежності від того, чи активне автоекранування.

volatile

True, якщо компілятор не може деяке значення пі дчас компіляції. Під час виконання це завжди має бути False.

Власні тести

Тести роблять як фільтри, але для тестів немає змоги отримати доступ до оточення або контексту, та, таким чином, вони не можуть утіорювати ланцюжки. Значення, що повертає тест, має бути True або False. Призначення тесту - надати розробникам шаблонів можливість виконати перевірки типів та сумісності.

Ось приклад тесту, що перевіряє, чи є змінна простим числом:

import math

def is_prime(n):
    if n == 2:
        return True
    for i in xrange(2, int(math.ceil(math.sqrt(n))) + 1):
        if n % i == 0:
            return False
    return True

Ви можете зареєструвати його на шаблонному оточенні, оновлюючи словник tests на оточенні:

environment.tests['prime'] = is_prime

Розробник шаблону після цього може використовувати тести таким чином:

{% if 42 is prime %}
    42 is a prime number
{% else %}
    42 is not a prime number
{% endif %}

Глобальний простір імен

Змінні, що зберігаються в словникові Environment.globals є особливими, тому що вони доступні для імпортованих шаблонів, також, навіть якщо вони імпортовані без контексту. Це місце, де ви можете покласти змінні та функції, що доступні весь час. Додатково, існує Template.globals, що є тими змінними, що доступні до окремого шаблону для всіх викликів render().

Низкорівневий API

Низькорівневий API оголює функціональність, що може бути корисною для розуміння деяких деталей реалізації, цілей налаштування, або просунутої підтримки розширень. Якщо ви не знаєте, що робите, ми не рекомендуємо використовувати ці виклики.

Environment.lex(source, name=None, filename=None)

Робить лексичний аналіз джерельного коду, та повертає генератор, що дає токени, як кортежі, в формі(lineno, token_type, value). Це може бути корисним для розробки розширень та налаштування шаблонів.

Це не виконеє обробку. Якщо ви бажаєте застосувати обробку розширень, вам треба фільтрувати джерело через метод preprocess().

Environment.parse(source, name=None, filename=None)

Розбирає джерельний код, та повертає абстрактне синтаксичне дерево. Це дерево з вузлів використовується компілятором для перетворення шаблону на виконавчий джерельний- або байткод. Це корисне для налаштування або виділення інформації з шаблону.

Якщо ви розробляєте розширення Jinja2, це дає вам гарний огляд згенерованого дерева вузлів.

Environment.preprocess(source, name=None, filename=None)

Передобробка джерела з усіма розширеннями. Це викликається автоматично для всіх методів розбору та компіляції, але не для lex(), оскільки там ви бажаєте тільки токенузувати дійсне джерело.

Template.new_context(vars=None, shared=False, locals=None)

Створює новий Context для цього шаблону. Запроваджені змінні будуть передані до шаблона. По замовчанню глобальні будуть додані до контексту. Якщо shared встановлене в True, дані передаються як є, без додавання глобальних змінних.

locals може бути словником для локальних змінних для внутрішнього використання.

Template.root_render_func(context)

Це низькорівнева функція відтворення. Вона отримує Context, що має бути створений через new_context() того ж шаблону або сумісного шаблону. Ця функція відтворення генерується компілятором з коду шаблону, та повертає генератор, що видає рядки юнікод.

Якщо в коді шаблона трапляється виключення, рушій шаблону не буде переписувати виключення, а передать оригінальне. Фактично, ця функція повинна викликатись тільки з  викликів render() / generate() / stream().

Template.blocks

Словник відтворювачів блоків. Кожна з функцій  робить точно як root_render_func() з тими ж обмеженнями.

Template.is_up_to_date

Цей атрибут є False, якщо є новіша версія шаблону, інакше True.

Зауваження

Низькорівневе API крихке. Майбутні версії Jinja2 будуть намагатись не змінювати це в зворотньо несумісний спосіб, але модифікації в ядрі Jinja2 можуть просвічуватись. Наприклад, якщо Jinja2 вводить новий вузол AST в останніх версіях, вони можуть повертатись parse().

Мета API

Нове в версії 2.2.

Мета API повертає деяку інформацію щодо абстрактного синтаксичного дерева, що може допомогти застосуванням реалізувати більш потужні шаблонні концепції. Всі функції меат API оперують на абстрактному синтаксичному дереві, що повертає метод Environment.parse().

jinja2.meta.find_undeclared_variables(ast)

Повертає набір всіх змінних в AST, що будуть переглядатись з контекста під час виконання. Оскільки під час виконання невідомо, які змінні будуть використані, в залежності від шляха виконання під час роботи, будуть повернуті всі змінні.

>>> from jinja2 import Environment, meta
>>> env = Environment()
>>> ast = env.parse('{% set foo = 42 %}{{ bar + foo }}')
>>> meta.find_undeclared_variables(ast) == set(['bar'])
True

Реалізація

Внутрішньо генератор коду використовуєтсья для пошуку незадекларованих змінних. Це гарно знати, оскільки генератор може генерувати TemplateAssertionError під час компіляції, та, фактично, ця функція наразі може підіймати це виключення, також.

jinja2.meta.find_referenced_templates(ast)

Шукає всі посилання на шаблони в AST. Це буде повертати ітератор по всіх жорстко закодованих шаблонах, вставлення та імпорти. Якщо використовується динамічне наслідування або вставлення, буде повернено None.

>>> from jinja2 import Environment, meta
>>> env = Environment()
>>> ast = env.parse('{% extends "layout.html" %}{% include helper %}')
>>> list(meta.find_referenced_templates(ast))
['layout.html', None]

Ця функція корисна для відстежування залежностей. Наприклад, якщо ви бажаєте перебудувати веб сайт після зміни розташування шаблона.

Note

The Jinja2 sandbox alone is no solution for perfect security. Especially for web applications you have to keep in mind that users may create templates with arbitrary HTML in so it’s crucial to ensure that (if you are running multiple users on the same server) they can’t harm each other via JavaScript insertions and much more.

Also the sandbox is only as good as the configuration. We strongly recommend only passing non-shared resources to the template and use some sort of whitelisting for attributes.

Also keep in mind that templates may raise runtime or compile time errors, so make sure to catch them.

Operator Intercepting

Нове в версії 2.6.

For maximum performance Jinja2 will let operators call directly the type specific callback methods. This means that it’s not possible to have this intercepted by overridingEnvironment.call(). Furthermore a conversion from operator to special method is not always directly possible due to how operators work. For instance for divisions more than one special method exist.

With Jinja 2.6 there is now support for explicit operator intercepting. This can be used to customize specific operators as necessary. In order to intercept an operator one has to override the SandboxedEnvironment.intercepted_binops attribute. Once the operator that needs to be intercepted is added to that set Jinja2 will generate bytecode that calls theSandboxedEnvironment.call_binop() function. For unary operators the unary attributes and methods have to be used instead.

The default implementation of SandboxedEnvironment.call_binop will use theSandboxedEnvironment.binop_table to translate operator symbols into callbacks performing the default operator behavior.

This example shows how the power (**) operator can be disabled in Jinja2:

from jinja2.sandbox import SandboxedEnvironment


class MyEnvironment(SandboxedEnvironment):
    intercepted_binops = frozenset(['**'])

    def call_binop(self, context, operator, left, right):
        if operator == '**':
            return self.undefined('the power operator is unavailable')
        return SandboxedEnvironment.call_binop(self, context,
                                               operator, left, right)

Make sure to always call into the super method, even if you are not intercepting the call. Jinja2 might internally call the method to evaluate expressions.

Пісочниця Jinja2 сама по особі не є рішенням для ідеальної безпеки. Особливо для веб застосувань ви повинні мати на увазі, що користувачі можуть створювати шаблони з довільним HTML, так що є критичним переконатись, (якщо ви маєте декілька користувачів на тому самому сервері), що вони не можуть зашкодити один одному через вставку JavaScript та багато чого іншого.

Також пісочниця є такою гарною, як її конфігурація. Ми наполегливо рекомендуємо передавати тільки не-загальні ресурси в шаблони, та використовувати деякий різновид білих списків для атрибутів.

Також майте на увазі, що шаблони можуть підіймати помилки часу компіляції або виконання, так що переконайтесь, що відловлюєте їх.

Перехоплення операторів

Нове в версії 2.6.

Для максимальної продуктивності Jinja2 буде дозволяти операторам напряму викликами специфічні для типу методи зворотнього зв'язку. Це означає, що неможливо перехопити це, перекриваючи Environment.call(). Більше того, перетворення з оператора до спеціального метода не завжди напряму можливе, через те, як роблять оператори. Наприклад, для ділення існує більше одного спеціального метода.

Починаючи з Jinja 2.6, тепер є підтримка для явного перехоплення операторів. Це може бути використане для налаштування специфічних операторів, по мірі необхідності. Щоб перехопити оператор треба перекрити атрибут SandboxedEnvironment.intercepted_binops. Коли оператор, що треба перехопити, доданий до цього набору, Jinja2 буде генерувати байткод, що викликає функцію SandboxedEnvironment.call_binop(). Для унарних операторів замість цього треба використовувати унарні атрибути та методи.

Реалізація по замовчанню SandboxedEnvironment.call_binop буде використовувати SandboxedEnvironment.binop_table, щоб транслювати символи оператора в зворотні виклики, виконуючи поведінку оператора по замовчанню.

Приклад показує, як оператор ступіня (**) може бути відключений в Jinja2:

from jinja2.sandbox import SandboxedEnvironment


class MyEnvironment(SandboxedEnvironment):
    intercepted_binops = frozenset(['**'])

    def call_binop(self, context, operator, left, right):
        if operator == '**':
            return self.undefined('the power operator is unavailable')
        return SandboxedEnvironment.call_binop(self, context,
                                               operator, left, right)

Переконайтесь, що завжди викликаєте супер метод, навіть якщо ви не перехоплюєте виклик. Jinja2 може внутрішньо викликати метод для обчислення виразів.

Template Designer Documentation

This document describes the syntax and semantics of the template engine and will be most useful as reference to those creating Jinja templates. As the template engine is very flexible, the configuration from the application can be slightly different from the code presented here in terms of delimiters and behavior of undefined values.

Synopsis

A Jinja template is simply a text file. Jinja can generate any text-based format (HTML, XML, CSV, LaTeX, etc.). A Jinja template doesn’t need to have a specific extension: .html, .xml, or any other extension is just fine.

A template contains variables and/or expressions, which get replaced with values when a template is rendered; and tags, which control the logic of the template. The template syntax is heavily inspired by Django and Python.

Below is a minimal template that illustrates a few basics using the default Jinja configuration. We will cover the details later in this document:


<html lang="en">
<head>
    <title>My Webpagetitle>
head>
<body>
    <ul id="navigation">
    {% for item in navigation %}
        <li><a href="{{ item.href }}">{{ item.caption }}a>li>
    {% endfor %}
    ul>

    <h1>My Webpageh1>
    {{ a_variable }}

    {# a comment #}
body>
html>

The following example shows the default configuration settings. An application developer can change the syntax configuration from {% foo %} to <% foo %>, or something similar.

There are a few kinds of delimiters. The default Jinja delimiters are configured as follows:

  • {% ... %} for Statements
  • {{ ... }} for Expressions to print to the template output
  • {# ... #} for Comments not included in the template output
  • #  ... ## for Line Statements

Variables

Template variables are defined by the context dictionary passed to the template.

You can mess around with the variables in templates provided they are passed in by the application. Variables may have attributes or elements on them you can access too. What attributes a variable has depends heavily on the application providing that variable.

You can use a dot (.) to access attributes of a variable in addition to the standard Python__getitem__ “subscript” syntax ([]).

The following lines do the same thing:

{{ foo.bar }}
{{ foo['bar'] }}

It’s important to know that the outer double-curly braces are not part of the variable, but the print statement. If you access variables inside tags don’t put the braces around them.

If a variable or attribute does not exist, you will get back an undefined value. What you can do with that kind of value depends on the application configuration: the default behavior is to evaluate to an empty string if printed or iterated over, and to fail for every other operation.

Implementation

For the sake of convenience, foo.bar in Jinja2 does the following things on the Python layer:

  • check for an attribute called bar on foo (getattr(foo, 'bar'))
  • if there is not, check for an item 'bar' in foo (foo.__getitem__('bar'))
  • if there is not, return an undefined object.

foo['bar'] works mostly the same with a small difference in sequence:

  • check for an item 'bar' in foo. (foo.__getitem__('bar'))
  • if there is not, check for an attribute called bar on foo. (getattr(foo, 'bar'))
  • if there is not, return an undefined object.

This is important if an object has an item and attribute with the same name. Additionally, theattr() filter only looks up attributes.

Filters

Variables can be modified by filters. Filters are separated from the variable by a pipe symbol (|) and may have optional arguments in parentheses. Multiple filters can be chained. The output of one filter is applied to the next.

For example, {{ name|striptags|title }} will remove all HTML Tags from variablename and title-case the output (title(striptags(name))).

Filters that accept arguments have parentheses around the arguments, just like a function call. For example: {{ listx|join(', ') }} will join a list with commas (str.join(', ',listx)).

The List of Builtin Filters below describes all the builtin filters.

Tests

Beside filters, there are also so-called “tests” available. Tests can be used to test a variable against a common expression. To test a variable or expression, you add is plus the name of the test after the variable. For example, to find out if a variable is defined, you can do name isdefined, which will then return true or false depending on whether name is defined in the current template context.

Tests can accept arguments, too. If the test only takes one argument, you can leave out the parentheses. For example, the following two expressions do the same thing:

{% if loop.index is divisibleby 3 %}
{% if loop.index is divisibleby(3) %}

The List of Builtin Tests below describes all the builtin tests.

Comments

To comment-out part of a line in a template, use the comment syntax which is by default set to{# ... #}. This is useful to comment out parts of the template for debugging or to add information for other template designers or yourself:

{# note: commented-out template because we no longer use this
    {% for user in users %}
        ...
    {% endfor %}
#}

Whitespace Control

In the default configuration:

  • a single trailing newline is stripped if present
  • other whitespace (spaces, tabs, newlines etc.) is returned unchanged

If an application configures Jinja to trim_blocks, the first newline after a template tag is removed automatically (like in PHP). The lstrip_blocks option can also be set to strip tabs and spaces from the beginning of a line to the start of a block. (Nothing will be stripped if there are other characters before the start of the block.)

With both trim_blocks and lstrip_blocks enabled, you can put block tags on their own lines, and the entire block line will be removed when rendered, preserving the whitespace of the contents. For example, without the trim_blocks and lstrip_blocks options, this template:

<div>
    {% if True %}
        yay
    {% endif %}
div>

gets rendered with blank lines inside the div:

<div>

        yay

div>

But with both trim_blocks and lstrip_blocks enabled, the template block lines are removed and other whitespace is preserved:

<div>
        yay
div>

You can manually disable the lstrip_blocks behavior by putting a plus sign (+) at the start of a block:

<div>
        {%+ if something %}yay{% endif %}
div>

You can also strip whitespace in templates by hand. If you add a minus sign (-) to the start or end of a block (e.g. a For tag), a comment, or a variable expression, the whitespaces before or after that block will be removed:

{% for item in seq -%}
    {{ item }}
{%- endfor %}

This will yield all elements without whitespace between them. If seq was a list of numbers from1 to 9, the output would be 123456789.

If Line Statements are enabled, they strip leading whitespace automatically up to the beginning of the line.

By default, Jinja2 also removes trailing newlines. To keep single trailing newlines, configure Jinja to keep_trailing_newline.

Note

You must not add whitespace between the tag and the minus sign.

valid:

{%- if foo -%}...{% endif %}

invalid:

{% - if foo - %}...{% endif %}

Escaping

It is sometimes desirable – even necessary – to have Jinja ignore parts it would otherwise handle as variables or blocks. For example, if, with the default syntax, you want to use {{ as a raw string in a template and not start a variable, you have to use a trick.

The easiest way to output a literal variable delimiter ({{) is by using a variable expression:

{{ '{{' }}

For bigger sections, it makes sense to mark a block raw. For example, to include example Jinja syntax in a template, you can use this snippet:

{% raw %}
    
    
    {% for item in seq %}
        
  • {{ item }}
    • 
    {% endfor %}
    

{% endraw %}

Line Statements

If line statements are enabled by the application, it’s possible to mark a line as a statement. For example, if the line statement prefix is configured to #, the following two examples are equivalent:

<ul>
# for item in seq
    <li>{{ item }}li>
# endfor
ul>

<ul>
{% for item in seq %}
    <li>{{ item }}li>
{% endfor %}
ul>

The line statement prefix can appear anywhere on the line as long as no text precedes it. For better readability, statements that start a block (such as for, if, elif etc.) may end with a colon:

# for item in seq:
    ...
# endfor

Note

Line statements can span multiple lines if there are open parentheses, braces or brackets:

<ul>
# for href, caption in [('index.html', 'Index'),
                        ('about.html', 'About')]:
    <li><a href="{{ href }}">{{ caption }}a>li>
# endfor
ul>

Since Jinja 2.2, line-based comments are available as well. For example, if the line-comment prefix is configured to be ##, everything from ## to the end of the line is ignored (excluding the newline sign):

# for item in seq:
    <li>{{ item }}li>     ## this comment is ignored
# endfor

Template Inheritance

The most powerful part of Jinja is template inheritance. Template inheritance allows you to build a base “skeleton” template that contains all the common elements of your site and definesblocks that child templates can override.

Sounds complicated but is very basic. It’s easiest to understand it by starting with an example.

Base Template

This template, which we’ll call base.html, defines a simple HTML skeleton document that you might use for a simple two-column page. It’s the job of “child” templates to fill the empty blocks with content:


<html lang="en">
<head>
    {% block head %}
    <link rel="stylesheet" href="style.css" />
    <title>{% block title %}{% endblock %} - My Webpagetitle>
    {% endblock %}
head>
<body>
    <div id="content">{% block content %}{% endblock %}div>
    <div id="footer">
        {% block footer %}
        © Copyright 2008 by <a href="http://domain.invalid/">youa>.
        {% endblock %}
    div>
body>
html>

In this example, the {% block %} tags define four blocks that child templates can fill in. All the block tag does is tell the template engine that a child template may override those placeholders in the template.

Child Template

A child template might look like this:

{% extends "base.html" %}
{% block title %}Index{% endblock %}
{% block head %}
    {{ super() }}
    <style type="text/css">
        .important { color: #336699; }
    style>
{% endblock %}
{% block content %}
    <h1>Indexh1>
    <p class="important">
      Welcome to my awesome homepage.
    p>
{% endblock %}

The {% extends %} tag is the key here. It tells the template engine that this template “extends” another template. When the template system evaluates this template, it first locates the parent. The extends tag should be the first tag in the template. Everything before it is printed out normally and may cause confusion. For details about this behavior and how to take advantage of it, see Null-Master Fallback.

The filename of the template depends on the template loader. For example, theFileSystemLoader allows you to access other templates by giving the filename. You can access templates in subdirectories with a slash:

{% extends "layout/default.html" %}

But this behavior can depend on the application embedding Jinja. Note that since the child template doesn’t define the footer block, the value from the parent template is used instead.

You can’t define multiple {% block %} tags with the same name in the same template. This limitation exists because a block tag works in “both” directions. That is, a block tag doesn’t just provide a placeholder to fill - it also defines the content that fills the placeholder in the parent. If there were two similarly-named {% block %} tags in a template, that template’s parent wouldn’t know which one of the blocks’ content to use.

If you want to print a block multiple times, you can, however, use the special self variable and call the block with that name:

<title>{% block title %}{% endblock %}title>
<h1>{{ self.title() }}h1>
{% block body %}{% endblock %}

Super Blocks

It’s possible to render the contents of the parent block by calling super. This gives back the results of the parent block:

{% block sidebar %}
    <h3>Table Of Contentsh3>
    ...
    {{ super() }}
{% endblock %}

Named Block End-Tags

Jinja2 allows you to put the name of the block after the end tag for better readability:

{% block sidebar %}
    {% block inner_sidebar %}
        ...
    {% endblock inner_sidebar %}
{% endblock sidebar %}

However, the name after the endblock word must match the block name.

Block Nesting and Scope

Blocks can be nested for more complex layouts. However, per default blocks may not access variables from outer scopes:

{% for item in seq %}
    <li>{% block loop_item %}{{ item }}{% endblock %}li>
{% endfor %}

This example would output empty 

  •  items because item is unavailable inside the block. The reason for this is that if the block is replaced by a child template, a variable would appear that was not defined in the block or passed to the context.

    Starting with Jinja 2.2, you can explicitly specify that variables are available in a block by setting the block to “scoped” by adding the scoped modifier to a block declaration:

    {% for item in seq %}
    <li>{% block loop_item scoped %}{{ item }}{% endblock %}li>
{% endfor %}

    When overriding a block, the scoped modifier does not have to be provided.

    • Template Objects

    Changed in version 2.4.

    If a template object was passed in the template context, you can extend from that object as well. Assuming the calling code passes a layout template as layout_template to the environment, this code works:

    {% extends layout_template %}

    Previously, the layout_template variable had to be a string with the layout template’s filename for this to work.

    HTML Escaping

    When generating HTML from templates, there’s always a risk that a variable will include characters that affect the resulting HTML. There are two approaches:

    1. manually escaping each variable; or
    2. automatically escaping everything by default.

    Jinja supports both. What is used depends on the application configuration. The default configuration is no automatic escaping; for various reasons:

    • Escaping everything except for safe values will also mean that Jinja is escaping variables known to not include HTML (e.g. numbers, booleans) which can be a huge performance hit.
    • The information about the safety of a variable is very fragile. It could happen that by coercing safe and unsafe values, the return value is double-escaped HTML.

    Working with Manual Escaping

    If manual escaping is enabled, it’s your responsibility to escape variables if needed. What to escape? If you have a variable that may include any of the following chars (>, <, &, or ") youSHOULD escape it unless the variable contains well-formed and trusted HTML. Escaping works by piping the variable through the |e filter:

    {{ user.username|e }}

    Working with Automatic Escaping

    When automatic escaping is enabled, everything is escaped by default except for values explicitly marked as safe. Variables and expressions can be marked as safe either in:

    1. the context dictionary by the application with MarkupSafe.Markup, or
    2. the template, with the |safe filter

    The main problem with this approach is that Python itself doesn’t have the concept of tainted values; so whether a value is safe or unsafe can get lost.

    If a value is not marked safe, auto-escaping will take place; which means that you could end up with double-escaped contents. Double-escaping is easy to avoid, however: just rely on the tools Jinja2 provides and don’t use builtin Python constructs such as str.format or the string modulo operator (%).

    Jinja2 functions (macros, super, self.BLOCKNAME) always return template data that is marked as safe.

    String literals in templates with automatic escaping are considered unsafe because native Python strings (str, unicode, basestring) are not MarkupSafe.Markup strings with an __html__attribute.

    List of Control Structures

    A control structure refers to all those things that control the flow of a program - conditionals (i.e. if/elif/else), for-loops, as well as things like macros and blocks. With the default syntax, control structures appear inside {% ... %} blocks.

    For

    Loop over each item in a sequence. For example, to display a list of users provided in a variable called users:

    <h1>Membersh1>
<ul>
{% for user in users %}
  <li>{{ user.username|e }}li>
{% endfor %}
ul>

    As variables in templates retain their object properties, it is possible to iterate over containers like dict:

    <dl>
{% for key, value in my_dict.iteritems() %}
    <dt>{{ key|e }}dt>
    <dd>{{ value|e }}dd>
{% endfor %}
dl>

    Note, however, that Python dicts are not ordered; so you might want to either pass a sortedlist of tuple s – or a collections.OrderedDict – to the template, or use the dictsortfilter.

    Inside of a for-loop block, you can access some special variables:

    VariableDescription
    loop.indexThe current iteration of the loop. (1 indexed)
    loop.index0The current iteration of the loop. (0 indexed)
    loop.revindexThe number of iterations from the end of the loop (1 indexed)
    loop.revindex0The number of iterations from the end of the loop (0 indexed)
    loop.firstTrue if first iteration.
    loop.lastTrue if last iteration.
    loop.lengthThe number of items in the sequence.
    loop.cycleA helper function to cycle between a list of sequences. See the explanation below.
    loop.depthIndicates how deep in deep in a recursive loop the rendering currently is. Starts at level 1
    loop.depth0Indicates how deep in deep in a recursive loop the rendering currently is. Starts at level 0

    Within a for-loop, it’s possible to cycle among a list of strings/variables each time through the loop by using the special loop.cycle helper:

    {% for row in rows %}
    <li class="{{ loop.cycle('odd', 'even') }}">{{ row }}li>
{% endfor %}

    Since Jinja 2.1, an extra cycle helper exists that allows loop-unbound cycling. For more information, have a look at the List of Global Functions.

    Unlike in Python, it’s not possible to break or continue in a loop. You can, however, filter the sequence during iteration, which allows you to skip items. The following example skips all the users which are hidden:

    {% for user in users if not user.hidden %}
    <li>{{ user.username|e }}li>
{% endfor %}

    The advantage is that the special loop variable will count correctly; thus not counting the users not iterated over.

    If no iteration took place because the sequence was empty or the filtering removed all the items from the sequence, you can render a default block by using else:

    <ul>
{% for user in users %}
    <li>{{ user.username|e }}li>
{% else %}
    <li><em>no users foundem>li>
{% endfor %}
ul>

    Note that, in Python, else blocks are executed whenever the corresponding loop did not break. Since Jinja loops cannot break anyway, a slightly different behavior of the else keyword was chosen.

    It is also possible to use loops recursively. This is useful if you are dealing with recursive data such as sitemaps or RDFa. To use loops recursively, you basically have to add the recursivemodifier to the loop definition and call the loop variable with the new iterable where you want to recurse.

    The following example implements a sitemap with recursive loops:

    <ul class="sitemap">
{%- for item in sitemap recursive %}
    <li><a href="{{ item.href|e }}">{{ item.title }}a>
    {%- if item.children -%}
        <ul class="submenu">{{ loop(item.children) }}ul>
    {%- endif %}li>
{%- endfor %}
ul>

    The loop variable always refers to the closest (innermost) loop. If we have more than one level of loops, we can rebind the variable loop by writing {% set outer_loop = loop %} after the loop that we want to use recursively. Then, we can call it using {{ outer_loop(...) }}

    If

    The if statement in Jinja is comparable with the Python if statement. In the simplest form, you can use it to test if a variable is defined, not empty or not false:

    {% if users %}
<ul>
{% for user in users %}
    <li>{{ user.username|e }}li>
{% endfor %}
ul>
{% endif %}

    For multiple branches, elif and else can be used like in Python. You can use more complexExpressions there, too:

    {% if kenny.sick %}
    Kenny is sick.
{% elif kenny.dead %}
    You killed Kenny!  You bastard!!!
{% else %}
    Kenny looks okay --- so far
{% endif %}

    If can also be used as an inline expression and for loop filtering.

    Macros

    Macros are comparable with functions in regular programming languages. They are useful to put often used idioms into reusable functions to not repeat yourself (“DRY”).

    Here’s a small example of a macro that renders a form element:

    {% macro input(name, value='', type='text', size=20) -%}
    <input type="{{ type }}" name="{{ name }}" value="{{
        value|e }}" size="{{ size }}">
{%- endmacro %}

    The macro can then be called like a function in the namespace:

    <p>{{ input('username') }}p>
<p>{{ input('password', type='password') }}p>

    If the macro was defined in a different template, you have to import it first.

    Inside macros, you have access to three special variables:

    varargs
    If more positional arguments are passed to the macro than accepted by the macro, they end up in the special varargs variable as a list of values.
    kwargs
    Like varargs but for keyword arguments. All unconsumed keyword arguments are stored in this special variable.
    caller
    If the macro was called from a call tag, the caller is stored in this variable as a callable macro.

    Macros also expose some of their internal details. The following attributes are available on a macro object:

    name
    The name of the macro. {{ input.name }} will print input.
    arguments
    A tuple of the names of arguments the macro accepts.
    defaults
    A tuple of default values.
    catch_kwargs
    This is true if the macro accepts extra keyword arguments (i.e.: accesses the special kwargsvariable).
    catch_varargs
    This is true if the macro accepts extra positional arguments (i.e.: accesses the specialvarargs variable).
    caller
    This is true if the macro accesses the special caller variable and may be called from a calltag.

    If a macro name starts with an underscore, it’s not exported and can’t be imported.

    Call

    In some cases it can be useful to pass a macro to another macro. For this purpose, you can use the special call block. The following example shows a macro that takes advantage of the call functionality and how it can be used:

    {% macro render_dialog(title, class='dialog') -%}
    <div class="{{ class }}">
        <h2>{{ title }}h2>
        <div class="contents">
            {{ caller() }}
        div>
    div>
{%- endmacro %}

{% call render_dialog('Hello World') %}
    This is a simple dialog rendered by using a macro and
    a call block.
{% endcall %}

    It’s also possible to pass arguments back to the call block. This makes it useful as a replacement for loops. Generally speaking, a call block works exactly like a macro without a name.

    Here’s an example of how a call block can be used with arguments:

    {% macro dump_users(users) -%}
    <ul>
    {%- for user in users %}
        <li><p>{{ user.username|e }}p>{{ caller(user) }}li>
    {%- endfor %}
    ul>
{%- endmacro %}

{% call(user) dump_users(list_of_user) %}
    <dl>
        <dl>Realnamedl>
        <dd>{{ user.realname|e }}dd>
        <dl>Descriptiondl>
        <dd>{{ user.description }}dd>
    dl>
{% endcall %}

    Filters

    Filter sections allow you to apply regular Jinja2 filters on a block of template data. Just wrap the code in the special filter section:

    {% filter upper %}
    This text becomes uppercase
{% endfilter %}

    Assignments

    Inside code blocks, you can also assign values to variables. Assignments at top level (outside of blocks, macros or loops) are exported from the template like top level macros and can be imported by other templates.

    Assignments use the set tag and can have multiple targets:

    {% set navigation = [('index.html', 'Index'), ('about.html', 'About')] %}
{% set key, value = call_something() %}

    Block Assignments

    New in version 2.8.

    Starting with Jinja 2.8, it’s possible to also use block assignments to capture the contents of a block into a variable name. This can be useful in some situations as an alternative for macros. In that case, instead of using an equals sign and a value, you just write the variable name and then everything until {% endset %} is captured.

    Example:

    {% set navigation %}
    <li><a href="/">Indexa>
    <li><a href="/downloads">Downloadsa>
{% endset %}

    The navigation variable then contains the navigation HTML source.

    Extends

    The extends tag can be used to extend one template from another. You can have multiple extendstags in a file, but only one of them may be executed at a time.

    See the section about Template Inheritance above.

    Blocks

    Blocks are used for inheritance and act as both placeholders and replacements at the same time. They are documented in detail in the Template Inheritance section.

    Include

    The include statement is useful to include a template and return the rendered contents of that file into the current namespace:

    {% include 'header.html' %}
    Body
{% include 'footer.html' %}

    Included templates have access to the variables of the active context by default. For more details about context behavior of imports and includes, see Import Context Behavior.

    From Jinja 2.2 onwards, you can mark an include with ignore missing; in which case Jinja will ignore the statement if the template to be included does not exist. When combined withwith or without context, it must be placed before the context visibility statement. Here are some valid examples:

    {% include "sidebar.html" ignore missing %}
{% include "sidebar.html" ignore missing with context %}
{% include "sidebar.html" ignore missing without context %}

    New in version 2.2.

    You can also provide a list of templates that are checked for existence before inclusion. The first template that exists will be included. If ignore missing is given, it will fall back to rendering nothing if none of the templates exist, otherwise it will raise an exception.

    Example:

    {% include ['page_detailed.html', 'page.html'] %}
{% include ['special_sidebar.html', 'sidebar.html'] ignore missing %}

    Changed in version 2.4: If a template object was passed to the template context, you can include that object using include.

    Import

    Jinja2 supports putting often used code into macros. These macros can go into different templates and get imported from there. This works similarly to the import statements in Python. It’s important to know that imports are cached and imported templates don’t have access to the current template variables, just the globals by default. For more details about context behavior of imports and includes, see Import Context Behavior.

    There are two ways to import templates. You can import a complete template into a variable or request specific macros / exported variables from it.

    Imagine we have a helper module that renders forms (called forms.html):

    {% macro input(name, value='', type='text') -%}
    <input type="{{ type }}" value="{{ value|e }}" name="{{ name }}">
{%- endmacro %}

{%- macro textarea(name, value='', rows=10, cols=40) -%}
    <textarea name="{{ name }}" rows="{{ rows }}" cols="{{ cols
        }}">{{ value|e }}textarea>
{%- endmacro %}

    The easiest and most flexible way to access a template’s variables and macros is to import the whole template module into a variable. That way, you can access the attributes:

    {% import 'forms.html' as forms %}
<dl>
    <dt>Usernamedt>
    <dd>{{ forms.input('username') }}dd>
    <dt>Passworddt>
    <dd>{{ forms.input('password', type='password') }}dd>
dl>
<p>{{ forms.textarea('comment') }}p>

    Alternatively, you can import specific names from a template into the current namespace:

    {% from 'forms.html' import input as input_field, textarea %}
<dl>
    <dt>Usernamedt>
    <dd>{{ input_field('username') }}dd>
    <dt>Passworddt>
    <dd>{{ input_field('password', type='password') }}dd>
dl>
<p>{{ textarea('comment') }}p>

    Macros and variables starting with one or more underscores are private and cannot be imported.

    Changed in version 2.4: If a template object was passed to the template context, you can import from that object.

    Import Context Behavior

    By default, included templates are passed the current context and imported templates are not. The reason for this is that imports, unlike includes, are cached; as imports are often used just as a module that holds macros.

    This behavior can be changed explicitly: by adding with context or without context to the import/include directive, the current context can be passed to the template and caching is disabled automatically.

    Here are two examples:

    {% from 'forms.html' import input with context %}
{% include 'header.html' without context %}

    Note

    In Jinja 2.0, the context that was passed to the included template did not include variables defined in the template. As a matter of fact, this did not work:

    {% for box in boxes %}
    {% include "render_box.html" %}
{% endfor %}

    The included template render_box.html is not able to access box in Jinja 2.0. As of Jinja 2.1,render_box.html is able to do so.

    Expressions

    Jinja allows basic expressions everywhere. These work very similarly to regular Python; even if you’re not working with Python you should feel comfortable with it.

    Literals

    The simplest form of expressions are literals. Literals are representations for Python objects such as strings and numbers. The following literals exist:

    “Hello World”:
    Everything between two double or single quotes is a string. They are useful whenever you need a string in the template (e.g. as arguments to function calls and filters, or just to extend or include a template).
    42 / 42.23:
    Integers and floating point numbers are created by just writing the number down. If a dot is present, the number is a float, otherwise an integer. Keep in mind that, in Python, 42 and42.0 are different (int and float, respectively).
    [‘list’, ‘of’, ‘objects’]:

    Everything between two brackets is a list. Lists are useful for storing sequential data to be iterated over. For example, you can easily create a list of links using lists and tuples for (and with) a for loop:

    <ul>
{% for href, caption in [('index.html', 'Index'), ('about.html', 'About'),
                         ('downloads.html', 'Downloads')] %}
    <li><a href="{{ href }}">{{ caption }}a>li>
{% endfor %}
ul>
    (‘tuple’, ‘of’, ‘values’):
    Tuples are like lists that cannot be modified (“immutable”). If a tuple only has one item, it must be followed by a comma (('1-tuple',)). Tuples are usually used to represent items of two or more elements. See the list example above for more details.
    {‘dict’: ‘of’, ‘key’: ‘and’, ‘value’: ‘pairs’}:
    A dict in Python is a structure that combines keys and values. Keys must be unique and always have exactly one value. Dicts are rarely used in templates; they are useful in some rare cases such as the xmlattr() filter.
    true / false:
    true is always true and false is always false.

    Note

    The special constants true, false, and none are indeed lowercase. Because that caused confusion in the past, (True used to expand to an undefined variable that was considered false), all three can now also be written in title case (True, False, and None). However, for consistency, (all Jinja identifiers are lowercase) you should use the lowercase versions.

    Math

    Jinja allows you to calculate with values. This is rarely useful in templates but exists for completeness’ sake. The following operators are supported:

    +
    Adds two objects together. Usually the objects are numbers, but if both are strings or lists, you can concatenate them this way. This, however, is not the preferred way to concatenate strings! For string concatenation, have a look-see at the ~ operator. {{ 1 + 1 }} is 2.
    -
    Subtract the second number from the first one. {{ 3 - 2 }} is 1.
    /
    Divide two numbers. The return value will be a floating point number. {{ 1 / 2 }} is {{0.5 }}. (Just like from __future__ import division.)
    //
    Divide two numbers and return the truncated integer result. {{ 20 // 7 }} is 2.
    %
    Calculate the remainder of an integer division. {{ 11 % 7 }} is 4.
    *
    Multiply the left operand with the right one. {{ 2 * 2 }} would return 4. This can also be used to repeat a string multiple times. {{ '=' * 80 }} would print a bar of 80 equal signs.
    **
    Raise the left operand to the power of the right operand. {{ 2**3 }} would return 8.

    Comparisons

    ==
    Compares two objects for equality.
    !=
    Compares two objects for inequality.
    >
    true if the left hand side is greater than the right hand side.
    >=
    true if the left hand side is greater or equal to the right hand side.
    <
    true if the left hand side is lower than the right hand side.
    <=
    true if the left hand side is lower or equal to the right hand side.

    Logic

    For if statements, for filtering, and if expressions, it can be useful to combine multiple expressions:

    and
    Return true if the left and the right operand are true.
    or
    Return true if the left or the right operand are true.
    not
    negate a statement (see below).
    (expr)
    group an expression.

    Note

    The is and in operators support negation using an infix notation, too: foo is not bar andfoo not in bar instead of not foo is bar and not foo in bar. All other expressions require a prefix notation: not (foo and bar).

    Other Operators

    The following operators are very useful but don’t fit into any of the other two categories:

    in
    Perform a sequence / mapping containment test. Returns true if the left operand is contained in the right. {{ 1 in [1, 2, 3] }} would, for example, return true.
    is
    Performs a test.
    |
    Applies a filter.
    ~

    Converts all operands into strings and concatenates them.

    {{ "Hello " ~ name ~ "!" }} would return (assuming name is set to 'John') HelloJohn!.

    ()

    Call a callable: {{ post.render() }}. Inside of the parentheses you can use positional arguments and keyword arguments like in Python:

    {{ post.render(user, full=true) }}.

    . / []
    Get an attribute of an object. (See Variables)

    If Expression

    It is also possible to use inline if expressions. These are useful in some situations. For example, you can use this to extend from one template if a variable is defined, otherwise from the default layout template:

    {% extends layout_template if layout_template is defined else 'master.html' %}

    The general syntax is  something> if  is true> else something else>.

    The else part is optional. If not provided, the else block implicitly evaluates into an undefined object:

    {{ '[%s]' % page.title if page.title }}

    abs(number)

    Return the absolute value of the argument.

    attr(obj, name)

    Get an attribute of an object. foo|attr("bar") works like foo.bar just that always an attribute is returned and items are not looked up.

    See Notes on subscriptions for more details.

    batch(value, linecount, fill_with=None)

    A filter that batches items. It works pretty much like slice just the other way round. It returns a list of lists with the given number of items. If you provide a second parameter this is used to fill up missing items. See this example:

    <table>
{%- for row in items|batch(3, ' ') %}
  <tr>
  {%- for column in row %}
    <td>{{ column }}td>
  {%- endfor %}
  tr>
{%- endfor %}
table>
    capitalize(s)

    Capitalize a value. The first character will be uppercase, all others lowercase.

    center(value, width=80)

    Centers the value in a field of a given width.

    default(value, default_value=u'', boolean=False)

    If the value is undefined it will return the passed default value, otherwise the value of the variable:

    {{ my_variable|default('my_variable is not defined') }}

    This will output the value of my_variable if the variable was defined, otherwise'my_variable is not defined'. If you want to use default with variables that evaluate to false you have to set the second parameter to true:

    {{ ''|default('the string was empty', true) }}

    range([start, ]stop[, step])

    Extensions

    The following sections cover the built-in Jinja2 extensions that may be enabled by an application. An application could also provide further extensions not covered by this documentation; in which case there should be a separate document explaining said extensions.

    i18n

    If the i18n extension is enabled, it’s possible to mark parts in the template as translatable. To mark a section as translatable, you can use trans:

    <p>{% trans %}Hello {{ user }}!{% endtrans %}p>

    To translate a template expression — say, using template filters, or by just accessing an attribute of an object — you need to bind the expression to a name for use within the translation block:

    <p>{% trans user=user.username %}Hello {{ user }}!{% endtrans %}p>

    If you need to bind more than one expression inside a trans tag, separate the pieces with a comma (,):

    {% trans book_title=book.title, author=author.name %}
This is {{ book_title }} by {{ author }}
{% endtrans %}

    Inside trans tags no statements are allowed, only variable tags are.

    To pluralize, specify both the singular and plural forms with the pluralize tag, which appears between trans and endtrans:

    {% trans count=list|length %}
There is {{ count }} {{ name }} object.
{% pluralize %}
There are {{ count }} {{ name }} objects.
{% endtrans %}

    By default, the first variable in a block is used to determine the correct singular or plural form. If that doesn’t work out, you can specify the name which should be used for pluralizing by adding it as parameter to pluralize:

    {% trans ..., user_count=users|length %}...
{% pluralize user_count %}...{% endtrans %}

    It’s also possible to translate strings in expressions. For that purpose, three functions exist:

    • gettext: translate a single string
    • ngettext: translate a pluralizable string
    • _: alias for gettext

    For example, you can easily print a translated string like this:

    {{ _('Hello World!') }}

    To use placeholders, use the format filter:

    {{ _('Hello %(user)s!')|format(user=user.username) }}

    For multiple placeholders, always use keyword arguments to format, as other languages may not use the words in the same order.

    Changed in version 2.5.

    If newstyle gettext calls are activated (Newstyle Gettext), using placeholders is a lot easier:

    {{ gettext('Hello World!') }}
{{ gettext('Hello %(name)s!', name='World') }}
{{ ngettext('%(num)d apple', '%(num)d apples', apples|count) }}

    Note that the ngettext function’s format string automatically receives the count as a numparameter in addition to the regular parameters.

    Expression Statement

    If the expression-statement extension is loaded, a tag called do is available that works exactly like the regular variable expression ({{ ... }}); except it doesn’t print anything. This can be used to modify lists:

    {% do navigation.append('a string') %}

    Loop Controls

    If the application enables the Loop Controls, it’s possible to use break and continue in loops. When break is reached, the loop is terminated; if continue is reached, the processing is stopped and continues with the next iteration.

    Here’s a loop that skips every second item:

    {% for user in users %}
    {%- if loop.index is even %}{% continue %}{% endif %}
    ...
{% endfor %}

    Likewise, a loop that stops processing after the 10th iteration:

    {% for user in users %}
    {%- if loop.index >= 10 %}{% break %}{% endif %}
{%- endfor %}

    Note that loop.index starts with 1, and loop.index0 starts with 0 (See: For).

    With Statement

    New in version 2.3.

    If the application enables the With Statement, it is possible to use the with keyword in templates. This makes it possible to create a new inner scope. Variables set within this scope are not visible outside of the scope.

    With in a nutshell:

    {% with %}
    {% set foo = 42 %}
    {{ foo }}           foo is 42 here
{% endwith %}
foo is not visible here any longer

    Because it is common to set variables at the beginning of the scope, you can do that within thewith statement. The following two examples are equivalent:

    {% with foo = 42 %}
    {{ foo }}
{% endwith %}

{% with %}
    {% set foo = 42 %}
    {{ foo }}
{% endwith %}

    Autoescape Extension

    New in version 2.4.

    If the application enables the Autoescape Extension, one can activate and deactivate the autoescaping from within the templates.

    Example:

    {% autoescape true %}
    Autoescaping is active within this block
{% endautoescape %}

{% autoescape false %}
    Autoescaping is inactive within this block
{% endautoescape %}

    After an endautoescape the behavior is reverted to what it was before.
