5.1 Визначення фрагменту документу SVG: елемент ‘svg’

5.1.1 Огляд

 Фрагмент документу SVG складається з довільної кількісті елементів SVG, що містяться в елементі ‘svg’.

Фрагмент документу SVG може бути від пустого фрагменту (тобто без жодного вмісту всередині елементу ‘svg’), до дуже простого фрагменту документу SVG, що містить поодинокий графічний елемент SVG, такий як ‘rect’, і до складної, глибоко вкладеної колекції контейнерних елементів та графічних елементів.

Фрагмент документу SVG може міститись окремо, як самодостатній файл або ресурс, в якому випадку фрагмент документу SVG є SVG документом, або він може бути вбудований як фрагмент в батьківський документ XML.

Наступний приклад показує простий вміст SVG, вбудований як фрагмент в батьківський XML документ. Зауважте використання просторів імен XML для наголосу на тому, що елементи ‘svg’ та ‘ellipse’ належать до простору імен SVG:

<?xml version="1.0" standalone="yes"?>
<parent xmlns="http://example.org"
        xmlns:svg="http://www.w3.org/2000/svg">
   <!-- тут розташовано батьківський контент -->
   <svg:svg width="4cm" height="8cm" version="1.1">
      <svg:ellipse cx="2cm" cy="4cm" rx="2cm" ry="1cm" />
   </svg:svg>
   <!-- ... -->
</parent>

Цей приклад показує дещо складніший (містить декілька прямокутників) самостійний, самодостатній SVG документ:

<?xml version="1.0" standalone="no"?>
<!DOCTYPE svg PUBLIC "-//W3C//DTD SVG 1.1//EN" 
  "http://www.w3.org/Graphics/SVG/1.1/DTD/svg11.dtd">
<svg width="5cm" height="4cm" version="1.1"
     xmlns="http://www.w3.org/2000/svg">
  <desc>Чотири окремих прямокутника
  </desc>
    <rect x="0.5cm" y="0.5cm" width="2cm" height="1cm"/>
    <rect x="0.5cm" y="2cm" width="1cm" height="1.5cm"/>
    <rect x="3cm" y="0.5cm" width="1.5cm" height="2cm"/>
    <rect x="3.5cm" y="3cm" width="1cm" height="0.5cm"/>

  <!-- Відображує контур полотна з використанням елемента 'rect' -->
  <rect x=".01cm" y=".01cm" width="4.98cm" height="3.98cm"
        fill="none" stroke="blue" stroke-width=".02cm" />

</svg>

Елементи ‘svg’ можуть з'являтись посередені вмісту SVG. Цей механізм, завдяки якому фрагменти документів SVG можуть бути вбудовані в інші фрагменти документів SVG.

Інше застосування елементів ‘svg’ всередені вмісту SVG є встановлення нового оглядового порту. (Дивіться Встановлення нового порту огляду).

В усіх випадках для сумісності з рекомендацією Простори імен в XML [XML-NS], визначення простіру імен SVG має бути впроваджено таким чином, щоб всі елементи SVG були ідентифіковані як ті, що належать до простору імен SVG. Наступне є можливими шляхами впровадження декларації простору імен. Атрибут ‘xmlns’ без префіксу простору імен може бути заданим на елементі ‘svg’, що означає, що SVG є простором імен по замовчанню для всіх елементів в полі зору елемента з атрибутом ‘xmlns’:

<svg xmlns="http://www.w3.org/2000/svg" …>
  <rect …/>
</svg>

Якщо простір імен заданий як атрибут ‘xmlns’ (тобто xmlns:svg="http://www.w3.org/2000/svg"), тоді відповідний простір імен не є простором по замовчанню, і, таким чином, префікс простору імен має бути явно бути наданий елементам:

<svg:svg xmlns:svg="http://www.w3.org/2000/svg" …>
  <svg:rect …/>
</svg:svg>

Префікси простору імен можуть бути вказані на елементах-предках (проілюстровано в прикладі нижче). Для додаткової інформації посилайтесь до рекомендацій Простори імен в XML [XML-NS].

5.2 Групування: елемент ‘g’

5.2.1 Overview

Елемент ‘g’ є контейнерний елемент для групування разом графічних елементів.

Групуючі конструкції, коли використовуються разом з елементами ‘desc’ та ‘title’, провадять інформацію щодо структури та семантики документу. Документи, що мають розвинену структуру, можуть бути намальовані графічно, прочитані вголос, в вигляді шрифту брайля, і таким чином впровадити доступність.

Група елементів, так само як і окремі елементи, можуть отримати ім'я за допомогою атрибута ‘id’. Іменовані групи потрібні для декількох цілей, таких, як анімація та повторно використані об'єкти.

Приклад:

<?xml version="1.0" standalone="no"?>
<!DOCTYPE svg PUBLIC "-//W3C//DTD SVG 1.1//EN" 
  "http://www.w3.org/Graphics/SVG/1.1/DTD/svg11.dtd">
<svg xmlns="http://www.w3.org/2000/svg"
     version="1.1" width="5cm" height="5cm">
  <desc>Дві групи, кожна з двох прямокутників</desc>
  <g id="group1" fill="red">
    <rect x="1cm" y="1cm" width="1cm" height="1cm"/>
    <rect x="3cm" y="1cm" width="1cm" height="1cm"/>
  </g>
  <g id="group2" fill="blue">
    <rect x="1cm" y="3cm" width="1cm" height="1cm"/>
    <rect x="3cm" y="3cm" width="1cm" height="1cm"/>
  </g>

  <!-- Показати контур полотна з використанням елементу 'rect' -->
  <rect x=".01cm" y=".01cm" width="4.98cm" height="4.98cm"
        fill="none" stroke="blue" stroke-width=".02cm"/>
</svg>

Показати цей приклад як SVG (тільки в переглядачах, що підтримують SVG)

Елемент ‘g’ може містити другі елементи ‘g’, вкладені в нього на довільну глибину. Таким чином можливе наступне:

<?xml version="1.0" standalone="no"?>
<!DOCTYPE svg PUBLIC "-//W3C//DTD SVG 1.1//EN" 
  "http://www.w3.org/Graphics/SVG/1.1/DTD/svg11.dtd">
<svg xmlns="http://www.w3.org/2000/svg"
     version="1.1" width="4in" height="3in">
  <desc>Groups can nest</desc>
  <g>
     <g>
       <g>
       </g>
     </g>
   </g>
</svg>

Любий елемент, що не міститься в ‘g’, розглядається (щонайменше концептуально) так, якби він був би у своїй власній групі.

5.3 Визначення вмісту для повторного використання та елемент ‘defs’

5.3.1 Огляд

SVG дозволяє графічним елементам бути визначеними для подальшого використання. Щоб зробити це інтенсивно використовуються посилання IRI [RFC3987] на ці інші об’єкти. Наприклад, щоб заповнити прямокутник лінійним градієнтом, ви спочатку визначаєте елемент ‘linearGradient’, та надаєте йому ID, як оце:

<linearGradient id="MyGradient">...</linearGradient>

Потім ви посилаєтесь на лілійний градієнт, як на значення властивості ‘fill’ для прямокутника, так:

<rect style="fill:url(#MyGradient)"/>

Деякі типи елементів, такі як градієнти, не будуть продукувати графічний результат саі по собі. Вони можуть, таким чином, роташовуватись будь-де у вмісті. Однак, іноді, є бажаним визначити графічний об’єкт, та запобігти тому, щоб він був безпосередньо намальований, він буде створений тільки для того, щоб на нього зробити посилання будь-де інде. Щоб зробити це, та дозволити зручне групування визначеного вмісту, SVG провадить елемент ‘defs’.

Є рекомендованим, що коли це можливо, об’єкти, на які є посилання, слід визначати в рамках елементів  ‘defs’. Серед елементів, що ми на них завжди посилаємось, є такі: ‘altGlyphDef’, ‘clipPath’, ‘cursor’, ‘filter’, ‘linearGradient’, ‘marker’, ‘mask’, ‘pattern’, ‘radialGradient’ та ‘symbol’. Визначення ціх елементів всередині елементу ‘defs’ пропагує розуміння вмісту SVG, і таким чином просуває доступність.

Елемент ‘defs’ є елементом-контейнером для елементів, доступних через посилання. З причин розумілості та доступності, є рекомендованим, щоб коли це є можливим, елементи, на які є посилання, були визначені в ‘defs’.

Модель вмісту для ‘defs’ є такою ж, як і для елементу ‘g’; таким чином, любий елемент, що може бути вкладений в ‘g’, може також буди вкладений в ‘defs’, та навпаки.

Елементи, що містяться в ‘defs’, не малюються безпосередньо; вони виключені щоб стати частиною дерева малювання, саме так, якби елементи ‘defs’ були в елементі ‘g’, та властивість ‘display’ була встановлена в none. Зауважте, однак, що вкладені в ‘defs’ елементи завжди присутні в джерелі дерева і, як слідство, можуть завжди бути об’єктом посилання з других елементів; таким чином, значеня властивості ‘display’ для елементу ‘defs’, або кожного з вкладених, не забороняє цім елементам бути частиною посилання з інших елементів.

Щоб надати деяким користувацьким агентам SVG можливість реалізувати ефективну роботу в потокових оточеннях, створювачі SVG вмісту заохочуються розміщувати всі елементи, що є цілями для посилань через локальні IRI за допомогою елементу ‘defs’, що є прямим потомком одного з послідовнікив об’єкта, на який посилаються. Наприклад:

<?xml version="1.0" standalone="no"?>
<!DOCTYPE svg PUBLIC "-//W3C//DTD SVG 1.1//EN" "http://www.w3.org/Graphics/SVG/1.1/DTD/svg11.dtd">
<svg width="8cm" height="3cm"
     xmlns="http://www.w3.org/2000/svg" version="1.1">
  <desc>Local URI references within ancestor's 'defs' element.</desc>
  <defs>
    <linearGradient id="Gradient01">
      <stop offset="20%" stop-color="#39F" />
      <stop offset="90%" stop-color="#F3F" />
    </linearGradient>
  </defs>
  <rect x="1cm" y="1cm" width="6cm" height="1cm" 
        fill="url(#Gradient01)"  />

  <!-- Show outline of canvas using 'rect' element -->
  <rect x=".01cm" y=".01cm" width="7.98cm" height="2.98cm"
        fill="none" stroke="blue" stroke-width=".02cm" />

</svg>

Переглянути приклад як SVG (тільки для браузерів, що підтримують SVG)

В наведеному вищу документі лінійний градієнт визначається за допомогою елементу  ‘defs’, що є безпосереднім потомком елелменту ‘svg’, що, в свою чергу є предком елементу ‘rect’, що посилається на лінійний градієнт. Таким чином, наведений вище документ задовільняє рекомендаціям.

Коли поточний фрагмент документу SVG малюється як SVG на візуальному носії, елемнети ‘desc’ та ‘title’ не малюються як частина графіки. Це легко досягається за допомогою іншого (можливо користувацького) листа стилів. Для глибокіх ієрархій та для послідуючих посилань в елементі ‘use’ , іноді буває бажаним дозволити користувачу контроювати, як глибоко вони можуть занюрюватись в текст опису.

В відповідних фрагментах документу SVG, любі елементи ‘title’ мають бути першими дочірніми елементами своїх батьків. Зауважте, що ті реалізації, що використовують ‘title’ для відображення підказок, часто будуть робити це, якщо ‘title’ дійсно є першим дочірнім елементом своїх батьків.

Наступний приклад демонструє це. В типовому випадку користувацькій агент SVG не буде малювати елементи ‘desc’ та ‘title’, та буде відмальовувати залишок вмісту елементу ‘g’.

<?xml version="1.0" standalone="no"?>
<!DOCTYPE svg SYSTEM "http://www.w3.org/Graphics/SVG/1.1/DTD/svg11.dtd">
<svg xmlns="http://www.w3.org/2000/svg"
     version="1.1" width="4in" height="3in">
  <g>
    <title>Company sales by region</title>
    <desc>
      This is a bar chart which shows 
      company sales by region.
    </desc>
    <!-- Bar chart defined as vector data -->
  </g>
</svg>

Елементи опису та заголовку можуть містити маркований текст з інших проторів імен. Ось приклад:

<?xml version="1.0" standalone="yes"?>
<svg xmlns="http://www.w3.org/2000/svg"
     version="1.1" width="4in" height="3in">
  <desc xmlns:mydoc="http://example.org/mydoc">

    <mydoc:para>The global description uses markup from the 
      <mydoc:emph>mydoc</mydoc:emph> namespace.</mydoc:para>
  </desc>
  <g>
    <!-- the picture goes here -->
  </g>
</svg>

Автори мають завжди провадити дочірній елемент ‘title’ для найбільш зовнішнього елементу svg в окремому документі SVG. Дочірній елелмент ‘title’ для елементу ‘svg’ прислуговується цілям ідентифікації вмісту даного фрагменту документу SVG. Оскільки користувачі часто використовують документи за межами контексту, авторам слід провадити контекст-обізнані заголовки. Таким чином, замість такого заголовка, як "Введення", що не надає багато обізнаності на контекстуалному тлі, авторам замість цього слід надавати такі заголовки, як "Введення до середньовічного бджільництва". Задля причин доступності, користувацькі агенти мають завжди робити вміст дочірнього елементу ‘title’ для найбільш зовнішнього елементу svg доступним для користувача. Механізм зробити це залежить від агенту користувача (такі, як захоплення, промова).

Визначення DTD для багатьох елементів SVG (особливо контейнерів та текстових елементів) не накладає обмежень на розміщення або число під-елелментів ‘desc’ та ‘title’. Ця гнучкість тільки тимчасова можливість, так що  утворюється узгоджена модель вмісту для контейнерних елементів, оскільки деякі контейнерні елементи в SVG дозволяють змішаний вміст, та оскільки існують правила змішаного вмісту для XML ([XML10], розділ 3.2.2), що не дозволяють бажані обмеження. Представлення майбутніх версій мови SVG можуть використовувати більш виразні представлення DTD, що дозволять більш обмежені правиила для змішаного вмісту . Суворо рекомендовано, щоб якнайбільше один елемент  ‘desc’, та щонайбільш один елемент ‘title’ з’являлися в якості дочірнього для кожного елементу, та що ці елелементи з’являлись перед кожним іншим дочврнім елементом (можливо, за винятком елементів ‘metadata’), або вмісту текстових даних. Якщо користувач потребує обрати між багатьма елементами ‘desc’ або ‘title’ для обробки (тобто обрати рядок, що треба використати як підказку), користувацькій агент має обрати перший зустрічний.

Любий елемент ‘svg’, ‘symbol’, ‘g’, графічний елемент або інший ‘use’ є потенційним елементом шаблону, що може бути повторно використаним (тобто, може біти утворений новий екземпляр) в документі SVG через застосування елементу ‘use’. Елемент ‘use’ посилаєтсья на інший елемент та вказує, що графічний вміст цього елементу буде вставлено/відмальовано в даній точці документу.

На відміну від ‘image’, елемент ‘use’ не може посилатись на цілі файли.

Елемент  ‘use’ має опціональні атрибути, ‘x’, ‘y’, ‘width’ та ‘height’, що використовуються для відображення графічного вмісту елемента, на який посилаються, в прямокутний регіон в поточній координатній системі.

Ефект елементу ‘use’ такий, якби вміст елемента, на який є посилання, було глибоко клоноване в окреме нерозкрите дерево DOM, що має елемент ‘use’ в якості батьківського, та всі елементи, вкладені в ‘use’ в якості цого наслідків вищого рівня. Оскільки клоноване дерево DOM є нерозкритим, SVG Document Object Model (DOM) містить тільки елемент ‘use’ та його атрибути. SVG DOM не показує вміст посиланих елементів як дочірніх до елементу ‘use’.

Для користувацькіх агентів, що підтримують стилі за допомогою CSS, концептуальне глибоке клонування посиланих елементів в нерозкрите дерево DOM, також копіює любі значення властивостей, що утворились від CSS каскадування ([CSS2], глава 6) на посиланому елементі, та його вмісті. CSS2 селектори можуть бути застосовані до оригінального (тобто, посиланого) елементу, оскільки вони є частиною формальної структури документу. Селектори CSS2 не можуть бути застосовані до  (концептуально) клонованого дерева DOM, оскільки його вміст не є частиною формальної структури документу.

Наслідування властивостей, одначе, робить таким чином, якби посиланий елемент був текстуально включений як глибоко клонований потомок елементу ‘use’. Посиланий елемент наслідує властивості від елементу ‘use’ та предків елементу ‘use’. Екземпляр посиланого елементу не наслідує властивості від оригінальних батьків посиланого елементу.

Якщо атрибути присвоєні посиланому елементу, тоді нагальною ціллю для події буде об’єкт SVGElementInstance в "дереві екземпляру", що відповідає даному посиланому елементу.

Обробка подій для нерозкритого дерева робить так, якби посиланий елемент був текстуально включений як глибоко клонований дочірній елемент елементу ‘use’, за винятком тих подій, що дасилаються до об’єктів SVGElementInstance. Ціль події та атрибути currentTarget встановлюються в SVGElementInstance, що відповідає цілі та поточним елементам цілі в посиланому піддереві. Подія поширюється через розкриті та нерозкриті частини дерева, в тому ж стилі, як це відбувається в звичайному дереві документу: зпершу ідучи від кореневого елементу до елементу ‘use’, та тоді через невідкриті елементи дерева в фазі захоплення, за чим слідує цільова фаза на цілі події, та потім підіймається назад через нерозкрите дерево до елементу use, та потім ще далі через звичайне дерево, до кореневого елементу на фазі спливання бульбашки.

Елемент, та всі його відповідні об’єкти  SVGElementInstance поділяють список слухача подій. Атрибут currentTarget події може бути використаний для визначення, через якій об’єкт був визваний слухач подій. 

Поведінка влистивості ‘visibility’ відповідає цій моделі наслідування влоастивостей. Таким чином, вказуючи 'visibility:hidden' для елементу ‘use’ не гарантує, що посиланий вміст не буде намальовано. Якщо елемент  ‘use’ вказує 'visibility:hidden', та елемент, на який він посилається, вказує 'visibility:hidden' або 'visibility:inherit', тоді цей один елемент буде прихованим. Однак, якщо посиланий елемент замість цього вказує 'visibility:visible', тоді цей елемент буде видимим, навіть якщо елемент ‘use’ вказує 'visibility:hidden'.

Анімації на посиланому елементі будуть спричиняти те, що екземпляри також будуть анімовані.

Елемент ‘use’ має той же візуальний ефект, якби елемент був ‘use’ був заміщений наступним згенерованим вмістом:

  • Якщо елемент ‘use’ посилається на елемент ‘symbol’

    В згенерованому вмісті ‘use’ буде замінено на ‘g’, де всі атрибути з елементу ‘use’, за винятком ‘x’, ‘y’, ‘width’, ‘height’ та ‘xlink:href’ пересилаються до згенерованого елементу ‘g’. Додаткове перетворення translate(x,y) додається в кінці (тобто, зправа) до атрибуту ‘transform’ на згенерованому ‘g’, де x та y представляють значення атрибутів ‘x’ та ‘y’ елементу ‘use’. Посиланий  ‘symbol’ та його вміст глибоко клонується в згенероване дерево, з тим винятком, що ‘symbol’ замінюється на ‘svg’. Цей згенерований ‘svg’ буде завжди мати явні значення для атрибутів ‘width’ та ‘height’. Якщо атрибути ‘width’ та/або ‘height’ провадяться для елементу ‘use’, тоді ці атрибути будуть передані до згенерованого ‘svg’. Якщо атрибути ‘width’ та/або ‘height’ не задані, згенерований елемент ‘svg’ буде використовувати значення '100%' для ціх атрибутів.

  • Якщо елемент ‘use’ посилається на елементn ‘svg’

    В згенерованому вмісті ‘use’ буде замінено на ‘g’, де всі атрибути з елементу ‘use’, за винятком ‘x’, ‘y’, ‘width’, ‘height’ та ‘xlink:href’ пересилаються до згенерованого елементу ‘g’. Додаткове перетворення translate(x,y) додається в кінці (тобто, зправа) до атрибуту ‘transform’ на згенерованому ‘g’, де x та y представляють значення атрибутів ‘x’ та ‘y’ елементу ‘use’. Посиланий  ‘svg’ та його вміст глибоко клонується в згенероване дерево, Якщо атрибути ‘width’ та/або ‘height’ провадяться для елементу ‘use’, тоді ці значення будуть перекривати відповідні атрибути ‘svg’ в згенерованому дереві.

  • Інакше:

    В згенерованому вмісті ‘use’ буде заміщеним на ‘g’, де всі атрибути від елементу ‘use’, за винятком  ‘x’, ‘y’, ‘width’, ‘height’ та ‘xlink:href’передаються в згенерований елемент ‘g’. Додаткове перетворення translate(x,y)додається в кінці (тобто, зправа) до атрибуту ‘transform’ на згенерованому ‘g’, де x та y представляють значення атрибутів ‘x’ та ‘y’ елементу ‘use’. Посиланий об’єкт та його вміст глибоко клонований в згенероване дерево.

Для користувацькіх агентів, що підтримують стилі за довомогою CSS, згенерований елемент ‘g’ переноситься разом з "каскадованими" значеннями властивосі для елелменту ‘use’, що є результатом  CSS каскадування ([CSS2], глава 6). Елемент ‘image’ вказує, що вміст цілого файлу буде відмальовано в наданому прямокутнику в поточній користувацькій системі координат. елемент ‘image’ може посилатись на файл растрового зображення, такого як PNG або JPEG, або на файли з типом MIME "image/svg+xml". Щоб задовільняти SVG, переглядачі мають щонайменше підтримувати формати файлів PNG, JPEG та SVG.

Результат обробки ‘image’ завжди є чотири-канальний RGBA результат. Коли елемент ‘image’ посилається на файл растрового зображення, таке як PNG або JPEG, що мають тільки три канали (RGB), тоді буде виконане перетворення об’єкту до 4-канального зображення RGBA, з альфа каналом встановленим в 1. Для одноканального растрового зображення, ефект буде такий, якщо б об’єкт був конвертований в 4-канальне зображення RGBA, де єдиний канал з посилаємого об’єкту використовується для обчислення трьох каналів кольору, та альфа канал встановлений в 1. 

Елемент ‘image’ встановлює нове поле перегляду для посилаємого файлу, як пояснюється в Встановлення нового поля перегляду. Границі нового поля перегляду визначаютсья через атрибути ‘x’, ‘y’, ‘width’ та ‘height’. Розміщення та маштабування поилаємого зображення контролюється атрибутом  ‘preserveAspectRatio’ елементу ‘image’.

Коли елемент ‘image’ посилається на зображення SVG, властивості ‘clip’ та ‘overflow’ на кореневому елементі в посилаємому зображенні SVG ігноруються (таким же чином, як ігноруються атрибути ‘x’, ‘y’, ‘width’ та ‘height’). Інакше як значення ‘preserveAspectRatio’ елементу ‘image’ починається з 'defer', атрибут ‘preserveAspectRatio’ на кореневому елементі зображення SVG також ігнорується (дивіться ‘preserveAspectRatio’ для деталей ). Замість цього, атрибут ‘preserveAspectRatio’ для посилаємого елелменту ‘image’ визначає, як вміст SVG зображення підганаяється в поле перегляду та властивості  ‘clip’ та ‘overflow’ для елементу ‘image’ визначає, як вміст SVG зображення обрізається (або ні) відносно до поля перегляду.

Значення атрибуту ‘viewBox’ використовується, коли обчислюється атрибут ‘preserveAspectRatio’, визначений для посилаємого вмісту. Для вмісту, що явно ідентифікує viewBox (тобто, файл SVG з атрибутом ‘viewBox’ для найбільш зовнішнього елементу svg), має використовуватись це значення. Для більшості растрового вмісту (PNG, JPEG) мають бути використані границі зображення (тобто, елелмент  ‘image’ має неявний ‘viewBox’ виду '0 0 raster-image-width raster-image-height'). Коли значення явно не надане (тобто, файл SVG без атрибуту ‘viewBox’ для найбільш зовнішнього елементу svg) атрибут ‘preserveAspectRatio’ ігнорується, та для малювання використовується тільки перенос згідно атрибутів  ‘x’ & ‘y’ поля перегляду.

Наприклад, якщо зображення посилається на PNG або JPEG, та preserveAspectRatio="xMinYMin meet", тоді відношення сторін растру буде збережено  (що означає, що фактор маштабу від координат зображення до простору користувацькіх координат, будуть однакові для X та Y), растр буде збільшений наскількі це можливо, при тому впевняючись, що весь растр пасує до поля перегляду, та що верхній/лівий кут растру має бути вірівняним з верхнім/лівим кутом поля перегляду, як визначено атрибутами ‘x’, ‘y’, ‘width’ та ‘height’ для елементу ‘image’.  Якщо значення ‘preserveAspectRatio’ було 'none', тоді відношення сторін зображення не буде збереженим. Зображення буде підлагоджене таким чином, що верхній/лівий кутом растру точно вирівняно по координатах (‘x’, ‘y’) нижній/правий кут растру точно віривняно по координатах (‘x’+‘width’, ‘y’+‘height’).

Ресурс, на який посилається елемент ‘image’ представляє окремий документ, що генерує своє власне дерево розбору та об’єктну модель документу (якщо ресрсом є XML). Таким чином в зображенні немає наслідування властивостей.

На відміну від ‘use’, елемент ‘image’ не може посилатись на елементи в SVG файлі.

5.8 Умовне виконання

5.8.1 Огляд умовного виконання

SVG має елемент ‘switch’ разом з атрибутами  ‘requiredFeatures’, ‘requiredExtensions’ та ‘systemLanguage’, що забезпечує можливість змінювати перегляд, в залежності від можливостей даного користувацького агенту або мови користувача.

Атрибути ‘requiredFeatures’, ‘requiredExtensions’ та ‘systemLanguage’ діють як перевірки та повертають true або false. ‘switch’ малює свого першого нащадка, для якого всі ці перевірки дають true. Якщо окремий атрибут не вказаний, вважається що значення є true.

Подібно до властивості ‘display’, атрибути умовного виконання впливають тільки на пряме малювання елемнетів, і не заважають елементам бути ціллю посилання з інших елементів (наприклад через ‘use’).

Як наслідок:

  • Атрибути ‘requiredFeatures’, ‘requiredExtensions’ та ‘systemLanguage’ впливають на елементи ‘a’, ‘altGlyph’, ‘foreignObject’, ‘textPath’, ‘tref’ та ‘tspan’.
  • Атрибути ‘requiredFeatures’, ‘requiredExtensions’ та ‘systemLanguage’ не будуть мати ефекту на елементи ‘mask’, ‘clipPath’ та ‘pattern’.
  • Атрибути ‘requiredFeatures’, ‘requiredExtensions’ та ‘systemLanguage’ не стосуються до елементів  ‘defs’ та ‘cursor’, оскільки вони не є частиною дерева малювання.
  • Атрибути ‘requiredFeatures’, ‘requiredExtensions’ та ‘systemLanguage’ впливають на елементи  ‘animate’, ‘animateColor’, ‘animateMotion’, ‘animateTransform’ та ‘set’. Якщо умовне виконання для ціх елементів анімації зазнає невдачі, анімація ніколи не буде ввімкнена.

5.8.3 Атрибут ‘requiredFeatures’

Визначення requiredFeatures:

requiredFeatures = list-of-features
Значення є список рядків-можливостей, з індивідувальними значеннями, поділених проміжками. Визначає, чи названі можливості підтримуються користувацькім агентом. Дозволяються тільки рядки можливостей, вказані як Рядки можливостей. Якщо всі надані можливості підтримуються, тоді атрибут обчислюється до true; інакше поточний елемент та його нащадки пропускаються та, таким чином, не малюються.
Анімується: ні .

Якщо атрибут не присутній, він неявно повертає значення "true". Якщо null-рядок або пустий рядок передається в атрибут ‘requiredFeatures’, атрибут повертає "false".

‘requiredFeatures’ часто використовується разом з елементом ‘switch’. Якщо ‘requiredFeatures’ використовується в іншій ситуації, тоді він представляє простий перемикач для цього елементу, чи треба його малювати, чи ні.

5.8.4 Атрибут ‘requiredExtensions’

Атрибут ‘requiredExtensions’ визначає список потрібних розширень мови. Розширення мови є можливостями користувацького агенту, що виходять за межі набору можливостей цієї спеціфікації. Кожне розширення ідентифікується IRI посиланням.

Визначення requiredExtensions:

requiredExtensions = list-of-extensions
Значення є списком IRI посилань, що ідентифікують необхідні розширення, розширені проміжками. Визначає, чи будуть всі поіменовані розширення підтримуватись користувацькім агентом. Якщо всі перелічені розширення підтримуються, тоді атрибут обчислюється як true; в іншому разі поточний елемент та його нащадки пропускаються, і таким чином не малюються.
Анімується: ні.

Якщо дане IRI посилання саме  містить проміжок, він має бути виключений.

Якщо атрибут не присутній, тоді він неявно повертає значення "true". Якщо null-рядок або пустий рядок передається в атрибут ‘requiredExtensions’, атрибут повертає "false".

‘requiredExtensions’ часто використовується разом з елементом ‘switch’. Якщо ‘requiredExtensions’ використовується в інших ситуаціях, він представляє простий перемикач для даного елемента, чи треба малювати елемент, чи ні.

Імена IRI для розширень повинні містити інформацію щодо версії, наприклад, "http://example.org/SVGExtensionXYZ/1.0", так що автори скриптів можуть відрізнити різні версії даного розширення.

5.8.5 Атрибут ‘systemLanguage’

Значення атрибуту є розділений комами список наіменувань мов, як визначено в BCP 47 [BCP47].

Обчислюється до "true", якщо одна з мов, вказаних як перевага користувача, точно співпадає з однією з мов, наданих як значення цього параметра, або якщо одна з мов, вказаних як перевага користувача, співпадає з префіксом однієї з мов, наданих як значення цього параметру, тобто першої частини, що за ним іде символ "-".

В інакшому випадку обчислюється до "false".

Зауваження: Це використання правила порівняння префіксів не має на увазі, що всі теги так призначені до мов, що коли користувач розуміє мову з окремим тегом, з цього завжди випливатиме, що користувач розумітиме і всі мови, до яких цей тег буде префіксом. 

Правило префіксів тільки дозволяє використовувати теги префіксів, якщо це можливо.

Зауваження з реалізації: коли ви робите вибір лінгвістичних переваг доступним для користувача, реалізації повинні також брати до уваги факт, що користувачі не будуть добре знайомі з описаними вище деталями порівняння мов, та мають надати відповідну допомогу. Як приклад, користувачі можуть розраховувати, що обираючи "en-gb", вони будуть розуміти кожний документ англійською, навіть якщо British English не доступний. Інтерфейс користувача для встановлення переваг має направляти користувача на додавання "en", щоб отримати найкращу поведінку співпадіння.

Багато мов МОЖУТЬ бути перелічені для вмісту, що призначено для багатьох аудиторій. Наприклад, вміст, що представлено одночасно в оригіналі на мові маорі, та англійською, може виглядати так:

<text systemLanguage="mi, en"><!-- content goes here --></text>

Однак тільки тому, що багато мов є присутніми в об’єкті, де присутня перевірка атрибуту ‘systemLanguage’ , це не означатиме, що він буде призначений для багатьох мовних аудиторій. Прикладом може бути початковий варіант, як "A First Lesson in Latin," що ясно налаштований на англомовну аудієнцію. В цьому випадку атрибут ‘systemLanguage’ буде включати тільки "en".

Зауваження для авторів: Автори мають розуміти, що якщо декілька альтернативних об’єктів мов заключені в ‘switch’, та жодний з них не співпадає, це може призвести до ситуацій, коли зовсім ніякого вмісту не буде намальовано. Таким чином рекомендовано включити вибір "спіймай-усіх" в кінці, так що  ‘switch’ буде поводитись прийнятно в жодному разі.

Для ‘systemLanguage’ атрибут: Анімується: ні.

Якщо атрибут відсутній, повертається неявне значення "true". Якщо null-рядок або пустий рядок передається в ‘systemLanguage’, атрибут повертає "false".

‘systemLanguage’ часто використовується разом з елементом ‘switch’. Якщо ‘systemLanguage’ використовується в іншиї ситуаціях, тоді він представляє простий перемикач для даного елемента, чи треба його малювати, чи ні.

5.8.6 Застосування атрибутів перевірки

Наступний список описує застосування атрибутів перевірки до елементів, що напряму не викликають малювання.

  • Тестові атрибути не мають ефекту до елементів ‘mask’, ‘clipPath’, ‘linearGradient’, ‘radialGradient’ та ‘pattern’. Атрибути перевірки на елементах, на які є посилання, не мають дії на малювання елементів, що посилаються на елемент.
  • Атрибути перевірки не мають ефекту до елементів ‘defs’ та ‘cursor’, оскільки вони не є частиною дерева малювання.
  • Елементи анімації  (‘animate’, ‘animateMotion’, ‘animateTransform’, ‘animateColor’ та ‘set’) ніколи не спрацюють, якщо вони містять атрибути перевірки, що обчислюються як false.

5.9 Визначення, коли зовнішні ресурси необхідні для належного малювання

Документи часто посилаються на, та використовують вміст інших файлів  (та інших веб ресурсів) як частину малювання. В деяких випадках автори можуть вказати, що окремий ресурс є потрібним, щоб документ розглядався як корекстний.

Атрибут ‘externalResourcesRequired’ достпуний для всіх контейнерних елементів, та для всіх документів, що потенційно можуть посилатись на зовнішні ресурси. Він вказує, коли ресурси, на які є посилання, та що не є частиною поточного документу, є потрібними для належного малювання окремого контейнерного або графічного елементу.

Визначення атрибуту:

externalResourcesRequired = "false | true"
false
(Значення по замовчанню) Вказує, що ресурси, зовнішні по відношенню до поточного документу є опціональніми. Малювання документу може продовжуватись, навіть якщо зовнішній ресурс недосяжний для поточного елементу та його послідовників.
true
Вказує на те, що ресурсі, зовнішні для поточного документу, є необхідними. Якщо зовнішній ресурс недоступний, послідовна обробка призупиняється, подія документу SVGLoad не відбувається, та часовий відлік анімації не починається, доки цей ресурс, та всі інші ресурси не будуть доступними, будуть розібрані, та готові до малювання. Якщо на запрошеному ресурсі відбувається подія таймауту, тоді документ набуває помилкового стану (дивіться Обробка помилок). Документ залишаєтся в помилковому стані до того, як всі ресурси не будуть доступні.

Цей атрибут застосовується для всіх типів посилань на ресурси, включаючи сторінки стилів, властивості кольорів (дивіться Дескриптори кольорових профілів) та шрифти, вказані через IRI посилання з використанням елементу ‘font-face’ або специфікації CSS @font-face. Зокрема, якщо елемент встановлює externalResourcesRequired="true", тоді всі листи стилів повинні бути доступні, оскільки кожний лист стилів може впливати на малювання елементу. 

Атрибут ‘externalResourcesRequired’ не наслідується (в сенсі наслідування значень атрибутів), але якщо він встановлений на контейнерний елемент, його значення буде застосовано для всіх елементів в контейнері.

Оскільки встановлення externalResourcesRequired="true" на контейнерному елементі буде мати ефект на відключенні прогресивного малювання для вмісту цього контейнеру, якщо цей контейнер включає елементи, що посилаються на зовнішні ресурси, інструменти для генерації SVG вмісту застерігаються від простого встановлення externalResourcesRequired="true" для  найбільш зовнішнього елементу svg на універсальній основі. Замість цього краще вказати externalResourcesRequired="true" на тих окремих графічних елементах або контейнерних елементах, що особисто потребують доступність зовнішніх ресурсів щоб бути відмальовані належним чином.

Для ‘externalResourcesRequired’: Анімується: ні.

5.11 DOM інтерфейси

5.11.1 Інтерфейс SVGDocument

Коли елемент ‘svg’ безпосередньо вбудований як компонент документу з іншого простору імен, наприклад, коли елемент ‘svg’ вбудований в XHTML документ [XHTML], тоді об’єкт SVGDocument не існує; замість цього кореневий об’єкт ієрархії доку буде об’єктом Document іншого типу, такий як об’єкт HTMLDocument.

Однак, об’єкт SVGDocument буде навпаки, існувати, коли кореневий елемент ієрархії XML є ‘svg’елементом , як коли продивляється окремий SVG файл (тобто вміст з типом  MIME "image/svg+xml"). В цьому випадку об’єкт SVGDocument буде кореневим об’єктом ієрархії об’єктної моделі документу.

В випадку, коли SVG документ вбудований за посиланням, наприклад, як коли документ XHTML містить елемент ‘object’, чий атрибут ‘href’ посилається на SVG документ (тобто, на документ, чий тип MIME є "image/svg+xml", та  чий кореневий елемент є ‘svg’), буде використовуватись дві різні ієрархії DOM. Перша DOM ієрархія буде для об’єкта, що містить посилання (тобто, XHTML документ). Друга DOM ієрархія буде для SVG документу, на який є посилання. В цій другій ієрархії DOM, кореневим документом ієрархії об’єктної моделі буде об’єкт SVGDocument.

Інтерфейс SVGDocument містить перелік атрибутів та методів, подібний до інтерфейсу HTMLDocument, що описаний в главі Document Object Model (HTML) Level 1 специфікації [DOM1].

viewport (readonly SVGRect)

Позиція та розмір поля перегляду (явного або неявного), що відповідає цьому елементу ‘svg’. Коли користувацький агент насправді малює вміст, тоді позиція та розмір представляють дійсні значення при малюванні. Розиція та розмір є безвимірні значення в координатній системі батьківського елементу. Якщо немає батьківського елементу (тобто, елемент ‘svg’ представляє корінь дерева документу), та якщо цей SVG документ вбудований як частина іншого документу (тобто, через HTML елемент ‘object’), тоді позиція та розмір є безвимірними значеннями в координатній системі батьківського документу. (Якщо той використовує розташування CSS або XSL, тоді безрозмірні значення представляють одиниці пікселів для поточного поля перегляду CSS або XSL, як описано в специфікації CSS2). Якщо батьківський елемент не має координатної системи, тоді користувацький агент повинен запропонувати логічні значення по замовчанню для цього атрибуту.

Об’єкт SVGRect є тільки для читання.

На системах, де неможливо визначити розмір пікселя, буде запроваджено підходящі розміри пікселя по замовчанню.

Коли користувацькій агент дійсно знає розмір в "екранних одиницях", цей атрибут буде виражати цю інформацію; в іншому випадку користувацькій агент буде провадити підходяще значення, таке як .28mm.

useCurrentView (readonly boolean)
Первинний перегляд (перед збільшенням та кадруванням) поточного самого внутрішнього фрагменту документа SVG може бути або "standard"-перегляд (базоване на атрибутах елемента  ‘svg’, таких як  ‘viewBox’) або "custom"-перегляд (гіперпосилання на окремий ‘view’ або інший елемент - дивіться Зв’язування вмісту SVG: IRI фрагменти та SVG перегляди). Якщо первинний перегляд є "standard", тоді цей атрибут є false. Якщо первинний перегляд є "custom", тоді цей атрибут дорівнює true.
currentView (readonly SVGViewSpec)

Визначення первинного перегляду (перед збільшенням та кадруванням) поточного самого внутрішнього фрагменту документу SVG. Значення залежить від ситуації:

  • Якщо первиний перегляд є "standard", тоді:
    • значення для viewBox, preserveAspectRatio та zoomAndPan для currentView будуть співпадати зі значеннями для відповідних DOM атрибутів, що напряму на SVGSVGElement
    • значення для transform та viewTarget для currentView будуть null
  • Якщо первинний перегляд є посиланням на елемент ‘view’, тоді:
    • значення для viewBox, preserveAspectRatio та zoomAndPan для currentView будуть відповідати атрибутам для наданого елемента ‘view’
    • значення для transform та viewTarget для currentView буде null
  • Якщо первинний перегляд був посиланням на інший елемент (відмінний від ‘view’, тоді) :
    • значення для viewBox, preserveAspectRatio та zoomAndPan для currentView буде співпадати зі значеннями для відповідних DOM атрибутів SVGSVGElement, напряму для найближчого батьківського елементу ‘svg’
    • значення для transform для currentView буде null
    • viewTarget для currentView буде представляти ціль посилання
  • Якщо первиннй перегляд був посиланням на фрагмент SVG документу з використанням специфікації ідентифікатору перегляду SVG (наприклад, #svgView(...)), тоді:
    • значення для viewBox, preserveAspectRatio, zoomAndPan, transform та viewTarget для currentView буде відповідати значенням для ідентифікатору фрагменту SVG

currentScale (float)
Для найбільш зовнішнього елементу svg, цей атрибут вказує поточний фактор маштабування, відносно до первинного перегляду, що приймає до уваги користувацькі операції збільшення та кадрування, як описано в розділі Збільшення та кадрування. DOM атрибути currentScale та currentTranslate еквівалентні матриці 2x3 [a b c d e f] = [currentScale 0 0 currentScale currentTranslate.x currentTranslate.y]. Якщо дозволене  "збільшення" (тобто zoomAndPan="magnify"), тоді ефект є такий, якби додаткове перетворення було розміщено на самий зовнішний рівень SVG фрагменту документу (тобто, за межі найбільш зовнішнього елементу svg).

Коли застосовується для ‘svg’ елементу, що не є найбільш зовнішнім елементом svg, поведінка цього атрибуту не визначена. 

currentTranslate (readonly SVGPoint)
Для найбільш зовнішнього елементу svg, відповідний фактор перетворення, що враховує користувацьке "збільшення".

Коли застосовується для ‘svg’ елементу, що не є найбільш зовнішнім елементом svg, поведінка цього атрибуту не визначена. 

Операції:
unsigned long suspendRedraw(in unsigned long maxWaitMilliseconds)
Приймає значення тайм-ауту, що вказує, що перемалювання не має відбуватись, доки не:
  1. викликається відповідний unsuspendRedraw(),
  2. викликаєтсья unsuspendRedrawAll(), або
  3. таймер викликає тайм-аут.

В оточеннях, де не підтримується інтерактивність (таких, як друк), перемальовка не повинна призупинятись. Виклики suspendRedraw() та unsuspendRedraw() мають, але не обов’язково, робитись в збалансованих парах.

Щоб затримати дії перемальовування по мірі виникнення пакетів змін SVG DOM, просувайте зміни до SVG DOM за допомогою виклику наступного методу:

suspendHandleID = suspendRedraw(maxWaitMilliseconds);

та слідуйте за змінами за допомогою наступного метода:

unsuspendRedraw(suspendHandleID);

Зауважте, що декільки викликів suspendRedraw можуть бути застосовані одночасно, та що кожний такий виклик методу трактується незалежно від інших викликів методу suspendRedraw.

Параметри:
  1. unsigned long maxWaitMilliseconds
    Кількість часу в мілісекундах, що треба утримуватись перед малюванням на пристрої. Значення, більші ніж 60 секунд будуть усічені до 60 секунд.
Повертає:
Число, що діє як унікальний ідентифікатор для окремого виклику suspendRedraw(). Це значення має передаватись як параметр в відповідний виклик методу unsuspendRedraw().
void unsuspendRedraw(in unsigned long suspendHandleID)
Відміняє вказаний suspendRedraw(), що вказується унікальним ідентифікатором ID призупиненого запиту, що повертається попереднім викликом suspendRedraw().
Параметри:
  1. unsigned long suspendHandleID
    Число, що діє як унікальний ідентифікатор для бажаного виклику suspendRedraw(). Число має бути числом, що повертається з попереднього виклику suspendRedraw(). Якщо при запиті unsuspendRedraw() передати недійсне значення ID, воно буде тихо проігнороване.
void unsuspendRedrawAll()
Відміняє всі поточні активні виклики методу suspendRedraw(). Цей метод є найбільш корисним в самому кінці набору викликів SVG DOM, щоб впевнитись, що всі очікувані виклики методів  suspendRedraw() будуть відмінені.

void pauseAnimations()

5.11.9 Інтерфейс SVGElementInstance

Для кожного елементу ‘use’, SVG DOM підтримує тіньове дерево ("дерево екземплярів") об’єктів типу SVGElementInstance. SVGElementInstance представляє один вузол в дереві екземплярів. Кореневий об’єкт в дереві екземплярів доступний через атрибут instanceRoot об’єкту SVGUseElement для відповідного елементу ‘use’.

Якщо елемент ‘use’ посилається на простий графічний елемент, такий як ‘rect’, тоді це тільки один об’єкт SVGElementInstance, та відповідний атрибут correspondingElement на цьому об’єкті SVGElementInstance є SVGRectElement, що відповідає цьому елементу ‘rect’, на яке є посилання .

Якщо елемент ‘use’ посилається на елемент ‘g’, який містить два елементи ‘rect’, тоді дерево екземплярів містить три об’єкти SVGElementInstance, кореневий об’єкт SVGElementInstance, чий  об’єкт whose correspondingElement є SVGGElement для ‘g’, та ще два об’єктра-потомка SVGElementInstance, кожний з яких має correspondingElement, що є SVGRectElement.

Якщо об’єкт, на який є посилання, сам є ‘use’, або якщо є вкладені ‘use’ піделементи в об’єкті, на який є посилання, дерево буде містити рекурсивні розширення непрямих посилань, що сформує форму повного дерева. Наприклад, якщо елемент ‘use’ посилається на ‘g’, та ‘g’ сам містить ‘use’, та цей ‘use’ посилаєтсья на ‘rect’, тоді дерево екземплярів є для оригінального (зовнішнього) ‘use’ буде складатись з ієрархії об’єктів SVGElementInstance, як наведено нижче:

SVGElementInstance #1 (parentNode=null, firstChild=#2, correspondingElement is the 'g')
  SVGElementInstance #2 (parentNode=#1, firstChild=#3, correspondingElement is the other 'use')
    SVGElementInstance #3 (parentNode=#2, firstChild=null, correspondingElement is the 'rect')
interface SVGElementInstance : EventTarget {
  readonly attribute SVGElement correspondingElement;
  readonly attribute SVGUseElement correspondingUseElement;
  readonly attribute SVGElementInstance parentNode;
  readonly attribute SVGElementInstanceList childNodes;
  readonly attribute SVGElementInstance firstChild;
  readonly attribute SVGElementInstance lastChild;
  readonly attribute SVGElementInstance previousSibling;
  readonly attribute SVGElementInstance nextSibling;
};
Атрибути:
correspondingElement (readonly SVGElement)
Відповідний елемент, до якого цей об’єкт є екземпляторм. Наприклад, якщо елемент ‘use’ посилається на елемент ‘rect’, тоді створюється SVGElementInstance, з його correspondingElement встановленим в об’єкт SVGRectElement для елементу ‘rect’.
correspondingUseElement (readonly SVGUseElement)
Відповідний елемент ‘use’, до якого належить цей SVGElementInstance. Коли елемент ‘use’ вкладені один в одний (тобто ‘use’ посилається на інший ‘use’, що посилається на графічний елемент, такий як ‘rect’), тоді correspondingUseElement є найбільш зовнішній ‘use’ (тобто, який непрямо посилається на ‘rect’, не той, що містить пряме посилання).
parentNode (readonly SVGElementInstance)
Батьківський вузол цього SVGElementInstance в дереві екземплярів. Всі об’єкти мають  SVGElementInstance батьківський, за винятком SVGElementInstance, що відповідає елемент, на який напряму вказує елемент ‘use’. В останньому випадку значення parentNode є  null.
childNodes (readonly SVGElementInstanceList)
SVGElementInstanceList, що містить всі нащадки цього SVGElementInstance в дереві екземплярів. Якщо немає нащадків, тоді SVGElementInstanceList не містить елементів (тобто пустий список).
firstChild (readonly SVGElementInstance)
Перший нащадок цього SVGElementInstance в дереві елементів. Якщо немає такого SVGElementInstance, повертає null.
lastChild (readonly SVGElementInstance)
Останній нащадок цього SVGElementInstance в дереві елементів. Якщо немає такого SVGElementInstance, повертає null.
previousSibling (readonly SVGElementInstance)
SVGElementInstance безпосередньо перед цім SVGElementInstance. Якщо немає такого SVGElementInstance, повертає null.
nextSibling (readonly SVGElementInstance)
SVGElementInstance безпосередньо після цього SVGElementInstance. Якщо немає такого SVGElementInstance, повертає null.

5.11.13 Інтерфейс GetSVGDocument

Інтерфейс провадить доступ до SVG документу, вбудованого за посиланням в іншу DOM-базовану мову. Очікується, що цей інтерфейс реалізований для DOM об’єктів, що дозволяють таке посилання на документ SVG, таким чином, як це робиться для об’єкту DOM Element, що відповідає елементу HTML ‘object’. Такі DOM об’єкти також часто потрібні для реалізації EmbeddingElement, визначеного в специфікації Window [WINDOW].

Цей інтерфейс застарілий, та може бути вилучений в наступних версіях специфікації SVG. Автори заохочуються використовувати атрибут на інтерфейсі EmbeddingElement для отримання SVG документу, на яке є посилання, якщо цей інтерфейс доступний.

interface GetSVGDocument {
  SVGDocument getSVGDocument();
};
Операції:
SVGDocument getSVGDocument()

Цей метод має повертати об’єкт Document, вбудований як вбудований елемент, або null, якщо немає документу.

Зауважте, що це еквівалентно до отримання значення атрибута EmbeddingElement::contentDocument вбудованого елемента, якщо інтерфейс EmbeddingElement також реалізований. Автору рекомендовано перевіряти, що повернутий Document є документ ‘svg’, замість ніж сподіватись, що це завжди буде так само.

Повертає
Об’єкт Document для об’єкт, на яке є посилання, або null, якщо намає документу.
