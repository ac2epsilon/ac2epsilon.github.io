Класи та наслідування

Класи

Класи в Kotlin декларуються з використанням ключового слова class:

class Invoice {
}

Декларація класу складається з імені класу, заголовка класу (що вказує його параметри типу, первинний конструктор, тощо) та тіло класу, оточене фігурними дужками. Обоє, заголовок та тіло опціональні; якщо клас не має тіла, фігурні дужки можуть бути випущені.

class Empty

Конструктори

Клас в Kotlin може мати первинний конструктор, та один або більше вторинних конструкторів. Первинний конструктор є частиною заголовка класу: він іде після імені класу (та опціональні переметри типу).

class Person constructor(firstName: String) {
}

Якщо первинний конструктор не має ніяких анотацій або модифікатороів видимості, ключове слово constructor може бути випущене:

class Person(firstName: String) {
}

Первинний конструктор не може містити ніякого коду. Код ініціалізації може бути розміщений в блоках ініціалізатора, що мають префікс з ключового слова init:

class Customer(name: String) {
    init {
        logger.info("Customer initialized with value ${name}")
    }
}

Зауважте, що параметри первинного конструктора можуть використовуватись в блоках ініціалізатора. Вони також можуть бути використані в ініціалізаторах властивостей, декларованих в тілі класу:

class Customer(name: String) {
    val customerKey = name.toUpperCase()
}

Фактично, для декларації властивостей та їх ініціалізації в первинному конструкторі Kotlin має стислий синтаксис:

class Person(val firstName: String, val lastName: String, var age: Int) {
    // ...
}

Майже вв той же спосіб, що і звичайні властивості, властивості, декларовані в первинному конструкторі можуть бути змінними (var) або тільки для читання (val).

Якщо конструктор має анотації або модифікатори видимості, ключове слово constructor обов'язкове, та модифікатори ідуть перед ним:

class Customer public @Inject constructor(name: String) { ... }

Щодо деталей дивіться Модифікатори видимості.

Вторинні конструктори

Клас також може декларувати вторинні конструктори, що мають префікс constructor:

class Person {
    constructor(parent: Person) {
        parent.children.add(this)
    }
}

Якщо клас має первинний конструктор, кожний вторинний конструктор потребує делегувати до первинного конструктора, або напряму, або через інші вторинні конструктор(и). Делегація до іншого конструктора того ж класу робиться з використанням ключового слова this:

class Person(val name: String) {
    constructor(name: String, parent: Person) : this(name) {
        parent.children.add(this)
    }
}

Якщо не-абстрактний клас не декларує жодних конструкторів (первинних або вторинних), він буде мати згенерованний первинний конструктор без аргументів. Видимість конструктора буде публічною. Якщо ви не бажаєте, щоб ваш клас мав публічний конструктор, вам треба декларувати пустий первинний конструктор з видимістю не-по-замовчанню:

class DontCreateMe private constructor () {
}

ПРИМІТКА: На JVM, якщо всі параметри первинного конструктора мають значення по замовчанню, компілятор буде генерувати додатковий конструктор без параметрів, що буде використовувати значення по замовчанню. Це спрощує використання Kotlin з бібліотеками, такими, як Jackson або JPA, що створюють примірники класу через конструктори без параметрів.

class Customer(val customerName: String = "")

Створення примірників класів

Щоб створити примірник класу, и викликаємо конструктор, так, якби це була звичайна функція:

val invoice = Invoice()

val customer = Customer("Joe Smith")

Зауважте, що Kotlin не має ключового слова new.

Створення примірників вкладених, внутрішніх та анонімних внутрішніх класів описане в розділі Вкладені класи.

Члени класів

Класи можуть містити:

  • Конструктори та блоки ініціалізаторів
  • Функції
  • Властивості
  • Вкладені та внутрішні класи
  • Декларацію об'єкту

Наслідування

Всі класи в Kotlin мають загальний суперклас Any, що є супертипом по замовчанню для класу, якщо супертип не декларовано:

class Example // Неявно наслідує від Any

Any не є java.lang.Object; зокрема, він не має жодних членів, крім equals(), hashCode() та toString(). Будь ласка, посилайтесь на розділ Взаємодія з Java для додаткових деталей.

Щоб декларувати явний супертип, ми ставимо тип після двох крапок в заголовку класа:

open class Base(p: Int)

class Derived(p: Int) : Base(p)

Якщо клас має первинний конструктор, базовий тип може (та мусить) бути ініціалізований прямо тут, використовуючи параметри первинного конструктора.

Якщо клас не має первинного конструктора, тоді кожний вторинний конструктор має ініціалізувати базовий тип, використовуючи ключове слово super, або делегувати до іншого конструктора, що робить це. Зауважте, що в цьому випадку різні вторинні конструктори можуть викликати різні конструктори базового типа:

class MyView : View {
    constructor(ctx: Context) : super(ctx)

    constructor(ctx: Context, attrs: AttributeSet) : super(ctx, attrs)
}

Анотація open на класі має протилежне значення до Java final: вона дозволяє іншим наслідувати від цього класа. По замовчанню всі класи в Kotlin є фінальними, що відповідає Effective Java, пункт 17: Розробляйте та документуйте для наслідування, або інакше забороніть її.

Перекриття методів

Як ми згадували раніше, ми схильні робити речі явними в Kotlin. Та, на відміну від Java, Kotlin вимагає явні анотації для перекриваних членів (ми називаємо їх відкритими) та для перекрить:

open class Base {
    open fun v() {}
    fun nv() {}
}
class Derived() : Base() {
    override fun v() {}
}

Анотація override необхідна для Derived.v(). Якщо ми пропустимо її, компілятор буде скаржитись. Якщо не буде анотації open на функції, такої як Base.nv(), декларування методу з такою ж сигнатурою в субкласі нелегальне, чи з override, чи без нього. В фінальному класі (тобто класі без анотації open), відкриті методи заборонені.

Метод, відмічений override, сам по собі є відкритим, тобто може бути перекритий в субкласі. Якщо ви бажаєте заборонити перекриття, використовуйте final:

open class AnotherDerived() : Base() {
    final override fun v() {}
}

Перекриття властивостей

Перекриття властивостей робить в подібний спосіб до перекриття методів; властивості, що декларовані в суперкласі, та перевизначені в похідному класі, мусять бути відмічені як override, та повинні мати сумісний тип. Кожна декларована властивість може бути перекритою властивістю з ініціалізатором, або властивістю з методом геттера.

open class Foo {
    open val x: Int get() { ... }
}

class Bar1 : Foo() {
    override val x: Int = ...
}

Також ви можете перекрити властивість val властивістю var, але не навпаки. Це дозволяється, бо властивість val в основі декларує метод геттера, та перекриття її як var додатково декларує метод-сеттер в похідному класі.

Зауважте, що ви можете використовувати ключове слово override як частину декларації властивості в первинному конструкторі. 

interface Foo {
    val count: Int
}

class Bar1(override val count: Int) : Foo

class Bar2 : Foo {
    override var count: Int = 0
}

Виклик реалізації суперкласа

Код в похідному класі може викликати функції свого суперкласу та мати доступ до властивостей, використовуючи ключове слово super:

open class Foo {
    open fun f() { println("Foo.f()") }
    open val x: Int get() = 1
}

class Bar : Foo() {
    override fun f() { 
        super.f()
        println("Bar.f()") 
    }
    
    override val x: Int get() = super.x + 1
}

У внутрішньому класі доступ до суперкласа зовнішнього класа робиться за допомогою ключового слова super, кваліфікованого іменем зовнішнього класу: super@Outer:

class Bar : Foo() {
    override fun f() { /* ... */ }
    override val x: Int get() = 0
    
    inner class Baz {
        fun g() {
            super@Bar.f() // Викликає реалізацію Foo f()
            println(super@Bar.x) // Використовує реалізацію Foo геттара x
        }
    }
}

Правило перекриття

В Kotlin наслідування реалізації регулюється наступним правилом: якщо клас наслідує багато реалізацій того ж самого члена від своїх безпосередніх суперкласів, він має перекривати цей член, та провадити свою власну реалізацію (можливо, використовуючи один з перекритих варіантів). Щоб вказати супертип, від якого була взята наслідувана реалізація, ми використовуємо super, кваліфікований ім'ям супертипа в кутових дужках, тобто super<Base>:

open class A {
    open fun f() { print("A") }
    fun a() { print("a") }
}

interface B {
    fun f() { print("B") } // члени інтерфейсу 'open' по замовченню
    fun b() { print("b") }
}

class C() : A(), B {
    // Компілятор вимагає перекриття f():
    override fun f() {
        super<A>.f() // виклик A.f()
        super<B>.f() // виклик B.f()
    }
}

Це добре наслідувати від обох, A та B, а ми не маємо проблем з a() та b(). оскільки C наслідує тільки одну реалізацію кожної з ціх функцій. Але для f() ми маємо дві реалізації, наслідувані C, і, таким чином, змушені перекритиf() в C, та запровадити нашу власну реалізацію, що виключає неоднозначність.

Абстрактні класи

Клас та деякі з його членів можуть бути декларовані як abstract. Абстрактний член не має реалізації в своєму класі. Зауважте, що нам не треба анотувати абстрактний клас або функцію як open – це зрозуміло і без пояснень.

Ми можемо перекрити не-абстрактний відкритий член абстрактним

open class Base {
    open fun f() {}
}

abstract class Derived : Base() {
    override abstract fun f()
}

Об'єкти-супутники

В Kotlin, на відміну від Java або C#, класи не мають статичних методів. В більшості випадків рекомендовано просто використовувати функції рівня пакунку.

Якщо вам треба написати функцію, що може бути викликана без наявності екземпляра класу, але потребує доступ до нутрощів класу (наприклад, метод-фабрику), ви можете записати її як член декларації об'єкта в середині класа.

Навіть більш точно, якщо ви декларуєте о'бєкт-супутник в вашому класі, ви зможете викликати його члени з тим же синтаксисом, як викликаються статичні методи в Java/C#, використовуючи тільки ім'я класу в якості кваліфікатора.

Properties and Fields

Declaring Properties

Classes in Kotlin can have properties. These can be declared as mutable, using the var keyword or read-only using the val keyword.

class Address {
    var name: String = ...
    var street: String = ...
    var city: String = ...
    var state: String? = ...
    var zip: String = ...
}

To use a property, we simply refer to it by name, as if it were a field in Java:

fun copyAddress(address: Address): Address {
    val result = Address() // there's no 'new' keyword in Kotlin
    result.name = address.name // accessors are called
    result.street = address.street
    // ...
    return result
}

Getters and Setters

The full syntax for declaring a property is

var <propertyName>[: <PropertyType>] [= <property_initializer>]
    [<getter>]
    [<setter>]

The initializer, getter and setter are optional. Property type is optional if it can be inferred from the initializer (or from the getter return type, as shown below).

Examples:

var allByDefault: Int? // error: explicit initializer required, default getter and setter implied
var initialized = 1 // has type Int, default getter and setter

The full syntax of a read-only property declaration differs from a mutable one in two ways: it starts with valinstead of var and does not allow a setter:

val simple: Int? // has type Int, default getter, must be initialized in constructor
val inferredType = 1 // has type Int and a default getter

We can write custom accessors, very much like ordinary functions, right inside a property declaration. Here's an example of a custom getter:

val isEmpty: Boolean
    get() = this.size == 0

A custom setter looks like this:

var stringRepresentation: String
    get() = this.toString()
    set(value) {
        setDataFromString(value) // parses the string and assigns values to other properties
    }

By convention, the name of the setter parameter is value, but you can choose a different name if you prefer.

Since Kotlin 1.1, you can omit the property type if it can be inferred from the getter:

val isEmpty get() = this.size == 0  // has type Boolean

If you need to change the visibility of an accessor or to annotate it, but don't need to change the default implementation, you can define the accessor without defining its body:

var setterVisibility: String = "abc"
    private set // the setter is private and has the default implementation

var setterWithAnnotation: Any? = null
    @Inject set // annotate the setter with Inject

Backing Fields

Classes in Kotlin cannot have fields. However, sometimes it is necessary to have a backing field when using custom accessors. For these purposes, Kotlin provides an automatic backing field which can be accessed using the field identifier:

var counter = 0 // the initializer value is written directly to the backing field
    set(value) {
        if (value >= 0) field = value
    }

The field identifier can only be used in the accessors of the property.

A backing field will be generated for a property if it uses the default implementation of at least one of the accessors, or if a custom accessor references it through the field identifier.

For example, in the following case there will be no backing field:

val isEmpty: Boolean
    get() = this.size == 0

Backing Properties

If you want to do something that does not fit into this "implicit backing field" scheme, you can always fall back to having a backing property:

private var _table: Map<String, Int>? = null
public val table: Map<String, Int>
    get() {
        if (_table == null) {
            _table = HashMap() // Type parameters are inferred
        }
        return _table ?: throw AssertionError("Set to null by another thread")
    }

In all respects, this is just the same as in Java since access to private properties with default getters and setters is optimized so that no function call overhead is introduced.

Compile-Time Constants

Properties the value of which is known at compile time can be marked as compile time constants using the const modifier. Such properties need to fulfil the following requirements:

  • Top-level or member of an object
  • Initialized with a value of type String or a primitive type
  • No custom getter

Such properties can be used in annotations:

const val SUBSYSTEM_DEPRECATED: String = "This subsystem is deprecated"

@Deprecated(SUBSYSTEM_DEPRECATED) fun foo() { ... }

Late-Initialized Properties

Normally, properties declared as having a non-null type must be initialized in the constructor. However, fairly often this is not convenient. For example, properties can be initialized through dependency injection, or in the setup method of a unit test. In this case, you cannot supply a non-null initializer in the constructor, but you still want to avoid null checks when referencing the property inside the body of a class.

To handle this case, you can mark the property with the lateinit modifier:

public class MyTest {
    lateinit var subject: TestSubject

    @SetUp fun setup() {
        subject = TestSubject()
    }

    @Test fun test() {
        subject.method()  // dereference directly
    }
}

The modifier can only be used on var properties declared inside the body of a class (not in the primary constructor), and only when the property does not have a custom getter or setter. The type of the property must be non-null, and it must not be a primitive type.

Accessing a lateinit property before it has been initialized throws a special exception that clearly identifies the property being accessed and the fact that it hasn't been initialized.

Delegated Properties

The most common kind of properties simply reads from (and maybe writes to) a backing field. On the other hand, with custom getters and setters one can implement any behaviour of a property. Somewhere in between, there are certain common patterns of how a property may work. A few examples: lazy values, reading from a map by a given key, accessing a database, notifying listener on access, etc.

Such common behaviours can be implemented as libraries using delegated properties.
