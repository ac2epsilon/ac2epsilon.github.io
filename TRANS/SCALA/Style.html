Керівництво зі стилю Scala

Про цей документ

Цей документ призначений окреслити деякі базові стилістичні направляючі Scala, яким треба слідувати з більшим або меншим завзяттям. Коли це можливо, ця інструкція намагаєтсья пояснити, чому заохочуєтся певний стиль, та як це співвідноситься з іншими альтернативами. Як з усіма інструкціями, розглядайте цей документ як перелік правил, що можна порушувати. Напевне, існуюсть випадки, коли перевага надається іншим стилям, ніж тим, що надані тут.

Відступи

Відступи повинні слідувати "домовленості двох проміжків”. Таким чином, замість відступати так:

  1. // не вірно!
  2. class Foo {
  3.     def bar = ...
  4. }

ви повинні відступати так:

  1. // вірно!
  2. class Foo {
  3.   def bar = ..
  4. }

Мова Scala заохочує вражаючу кількість вкладених меж та логічних блоків (значень функцій та тому подібне). Зробіть собі послугу, та не штрафуйте себе синтаксично, відкриваючи новий блок. Прийшовши з Java, цей стиль потребує часу, щоб до нього звикнути, але він вартий зусиль. 

Переноси рядків

Є випадки, коли один вираз досягає довжини, коли він стає нечитаємий, якщо обмежити його одним рядком (звичайно ця довжина десь більше 80 символів). В таких випадках переважний  підхід є просто розділити вираз на декілька виразів, присвоївши проміжні результати до значень. Однак це не завжди практичне рішення.

Коли абсолютно необхідно перенести вираз на більше ніж один рядок, кожний наступний рядок повинен бути зсунутий на два проміжки від першого. Також пам'ятайте, що Scala потребує, щоб кожний "розрив рядка” мав або незакриті дужки, або завершувався на інфіксний метод, де правий параметр не наданий:

  1. val result = 1 + 2 + 3 + 4 + 5 + 6 +
  2.   7 + 8 + 9 + 10 + 11 + 12 + 13 + 14 +
  3.   15 + 16 + 17 + 18 + 19 + 20

Без цього заключного метода Scala буде виводити крапку з комою на кінці рядка, там де повинен бути перенос, скидаючи компіляцію, іноді навіть без такої дещиці, як попередження.

Методи з багатьма аргументами

Коли викликається метод, що сприймає багато аргументів (біля п'яти, або більше), часто необхідно перенести виклик метода на декілька рядків. В таких випадках розташуйте кожний аргумент на власний рядок, зсунутий на два проміжки від поточного рівня відступу:

  1. foo(
  2.   someVeryLongFieldName,
  3.   andAnotherVeryLongFieldName,
  4.   "this is a string",
  5.   3.1415)

Таким чином всі параметри вирівняні, але ви не повинні пере-вирівнювати їх, якщо ви пізніше зміните ім'я метода.

Треба добре прикласти зусиль, щоб уникати таких викликів в районі довжини рядка. Більш специфічно, такого виклика треба уникати, якщо кожний параметр буде треба відступити на більше 50 проміжків, щоб досягти вірівнювання. В таких випадках сам виклик треба перемістити на наступний рядок, та зсунути на два проміжки:

  1. // вірно!
  2. val myOnerousAndLongFieldNameWithNoRealPoint = 
  3.   foo(
  4.     someVeryLongFieldName,
  5.     andAnotherVeryLongFieldName,
  6.     "this is a string",
  7.     3.1415)

  9. // не вірно!
  10. val myOnerousAndLongFieldNameWithNoRealPoint = foo(someVeryLongFieldName,
  11.                                                    andAnotherVeryLongFieldName,
  12.                                                    "this is a string",
  13.                                                    3.1415)

Але краще просто спробувати уникати жодного метода, що приймає більше двох або трьох параметрів!

Домовленості іменування

Загалом кажучи, Scala використовує іменування в "верблюжому реєстрі”. Тобто кожне слово починається з великої літери, можливо, окрім першого:

  1. UpperCamelCase
  2. lowerCamelCase

Підкреслення в іменах (_) насправді не заборонені компілятором, але суворо не заохочуються, оскільки вони мають особливе значення в синтаксисі Scala. (Але дивіться нижче щодо прикладів).

Класи/трейти

Класи повинні іменуватись в верхньому верблюжому реєстрі:

  1. class MyFairLady

Це імітує домовленість по іменуванню для Java класів.

Об'єкти

Імена об'єктів подібні до імен класів (верхній верблюжий реєстр).

Виключення тільки для імітації пакунку або функції. Це не загальне правило. Приклад:

  1. object ast {
  2.   sealed trait Expr

  4.   case class Plus(e1: Expr, e2: Expr) extends Expr
  5.   ...
  6. }

  8. object inc {
  9.   def apply(x: Int): Int = x + 1
  10. }

Пакунки

Пакунки Scala повинні слідувати домовленості по іменуванню пакунків Java:

  1. // невірно!
  2. package coolness

  4. // вірно! покладає в поле зору тільки coolness._
  5. package com.novell.coolness

  7. // вірно! покладає обоє, novell._ та coolness._ в поле зору
  8. package com.novell
  9. package coolness

  11. // вірно, для обєкту пакунка com.novell.coolness
  12. package com.novell
  13. /**
  14.  * Провадить класи, пов'язані з coolness
  15.  */
  16. package object coolness {
  17. }

root

Іноді необхідно повністю кваліфіковані імпорти з використанням _root_. Наприклад, якщо інший netв полі зору, тоді доступ до net.liftweb буде виглядати так:

  1. import _root_.net.liftweb._

Не зловживайте _root_. Загалом, розрішення вкладених пакунків є гарною річчю, та дуже корисні для зменшення безладу імпортів. Використання _root_ не тільки невілює ці вигоди, але також вводить додатковий безлад в вигляді себе.

Методи

Текстуальні (літерні) імена методів повинні бути в нижньому верблюжому реєстрі:

  1. def myFairMethod = ...

Цей розділ не є вичерпним керівництвом по ідіоматичним методам іменування Scala. Більше інфорації може бути знайдене в розділі викликів методів.

Аксесори/мутатори

Scala не слідує домовленості Java префіксів set/get до мутаторів та аксесорів (відповідно). Замість цього використовується наступна домовленість:

  • Для аксеворів властивостей ім'я метода повинне бути ім'ям властивості.
  • В деяких випадках допустиме поперед поставити  "`is`” для логчсного аксесора  (наприклад, isEmpty). Це повинно бути тільки в випадку, коли не надається відповідного мутатора. Будь ласка, зауважте, що домовленість Lift додавати "_?” до логічних аксесоров є нестандартною, та не використовується за межами фреймворку Lift.

  • Для мутаторів ім'я метода повинно бути ім'ям властивості з доданим "_=”. Доки відповідний аксесор для цієї ж властивості визначений на оточующому типі, ця домовленість дозволятиме синтаксис мутаторів, що відзеркалює присвоєння. Зауважте, що це не тільки домовленість, але також вимога мови. 

    1. class Foo {

    3.   def bar = ...

    5.   def bar_=(bar: Bar) {
    6.     ...
    7.   }

    9.   def isBaz = ...
    10. }

    12. val foo = new Foo
    13. foo.bar             // accessor
    14. foo.bar = bar2      // mutator
    15. foo.isBaz           // boolean property

На жаль, ці домовленості впадають в протидію з Java домовленостями іменувати приватні поля, інкапсульовані аксесорами та мутаторами, згідно з властивістю, що вони представляють. Наприклад:

  1. public class Company {
  2.     private String name;

  4.     public String getName() {
  5.         return name;
  6.     }

  8.     public void setName(String name) {
  9.         this.name = name;
  10.     }
  11. }

В Scala немає різниці між полями та методами. Фактично, поля повністю іменуються та контролюються компілятором. Якщо ви бажаєте адаптувати Java домовленість для bean гетерів та сетерів в Scala, це, скоріше, досить просте кодування:

  1. class Company {
  2.   private var _name: String = _

  4.   def name = _name

  6.   def name_=(name: String) {
  7.     _name = name
  8.   }
  9. }

Хоча угорська нотація жахливо потворна, вона має перевагу усунення неоднозначності змінної _nameбез захаращення ідентифікатора. Підкреслення стоїть в префіксній позиції, скоріше, ніж в суфіксній, щоб уникниту небезпеки помилково набрати name _ замість name_. З потужним використанням інтерфейсу типів Scala, такі помілки можуть потенційно призвести до дуже заплутаної помилки.

Зауважте, що Java парадігма гетерів/сетерів часто використовувалась для обходу відсутності першокласної підтримки для властивостей та прив'язок. В Scala є бібліотеки, що підтримують властивості та прив'язки. По домовленості використовується незмінне посилання та клас властивості, що містить свої влсні сетери та гетери. Наприклад:

  1. class Company {
  2.   val string: Property[String] = Property("Initial Value")

Дужки

На відміну від Ruby, Scala додає значущості до того, був метод декларований з дужками, або ні  (стосується тільки методів нульової арності). Наприклад:

  1. def foo1() = ...

  3. def foo2 = ...

Є різні методи під час компіляції. Тоді як foo1може бути викликаний з дужками, або без дужок, foo2 не може бути викликаний з дужками.

Таким чином, насправді це досить важливо, щоб були переглянуті відповідні поради, коли доцільно декларувати метод без дужок, а коли ні.

Методи, що діють як асесори різного сорту (енкапсулюють поле або логічну властивість) повинні декларуватись без дужок, за винятком, якщо вони мають побічні ефекти. Хоча Ruby та Lift використовують ! щоб вказати на це, використання круглих дужок переважне  (будь ласка, зауважте, що флюїдні API та внутрішні DLS мають тенденцію ламати настанови, надані нижче, на користь синтаксиса. Такі виключення не повинні розглядатись як сильні порушення, бо, часом, ці правила не застосовуються. В DSL синтаксис слідує бути першорядним над домовленостями).

Більше того, місце виклику повиннно слідувати декларації; якщо декларація з дужками, виклик має бути з дужками. Хоча існує спокуса зберігти декілька символів, та якщо ви слідуєте цім настановам, ваш код буде значно більш читабельний та підтримуваний.

  1. // не змінює стан, викликайте як birthdate
  2. def birthdate = firstName

  4. // оновлює наш внутрішній стан, викликаєтся як age()
  5. def age() = {
  6.   _age = updateAge(birthdate)
  7.   _age
  8. }

Символічні імена методів

Уникайте! Не зважаючи на ступінь, до якої Scala спрощує це поле розробки API, визначення методів з символічними іменами не повинно бути сприйнято легковажно, зокрема коли символи по собі є нестандартними (наприклад, >>#>>). Як головне правило, символічні імена методів мають два випадки використання:

  • Спеціфічні для мови предметного домену (наприклад, actor1 ! Msg)
  • Логічні математичні операції (наприклад, a + b або c :: d)

В першому випадку символічні імена можуть бути використані безкарно, доки синтаксис дійсно надає вигоди. Однак, в курсі стандартного дизайну API, символічні імена методів повинні бути суворо зарезервовані для чисто фунціональних операцій. Таким чином, допустиме визначити метод >>= для поєднання монад, але не допустиме визначити метод << для запису в вихіднй потік. Перше є математичною гарно визначеним, та вільне від побічних ефектів. Для останнього не вірне ні те, не інше. 

Як головне правило, символічні імена методів повинні бути гарно зрозумілими, та само-документованими по натурі. Правило великого пальця наступне: якщо вам треба пояснити, що робить метод, тоді він повинен мати справжнє, описове ім'я, а не послідовність символів. Існують деякі дуже рідкісні випадки, коли допустиме ввести нові символічні імена методів. Існують шанси, що ваш API не оидн з ціх випадків!

Визначення методів з символічними іменами розглядаються як просунута можливість Scala, щоб бути викорстаними тільки тими, хто гарно розбирається в усіх підводних каменях. Без догляду, надмірне використання символічних імен методів може просто трансформувати навіть найпростіший код на символічний суп. 

Константи, значення, змінні та методи

Імена констант повинні бути в верхньому верблюжому реєстрі. Подібно до Java static final членів, якщо член є фінальним, незмінним, та він належить до об'єкту пакунку або об'єкту, він може розглядатись як константа:

  1. object Container {
  2.   val MyConstant = ...
  3. }

Значення Piв пакунку scala.math є іншим прикладом такої константи.

Імена методів, значень та змінних повинні бути в малому верблюжому реєстрі:

  1. val myValue = ...
  2. def myMethod = ...
  3. var myVariable

Параметри типу

Для простих патаметрів типу має використовуватись одна велика літера (з англійської абетки), починаючи з A (це відрізняється від Java домовленості, де початок іде з T). Наприклад:

  1. class List[A] {
  2.   def map[B](f: A => B): List[B] = ...
  3. }

Якщо параметр типу має більш спеціфічне значення, треба використовувати описове ім'я, слідуя домовленості щодо іменування класів (на відміну від стиля всі-великі):

  1. // Вірно
  2. class Map[Key, Value] {
  3.   def get(key: Key): Value
  4.   def put(key: Key, value: Value): Unit
  5. }

  7. // Навірно; не викорстовуйте всі-великі
  8. class Map[KEY, VALUE] {
  9.   def get(key: KEY): VALUE
  10.   def put(key: KEY, value: VALUE): Unit
  11. }

Якщо поле зору параметра типу досить мале, заміть довгого змістовного імені може використовуватись мнемоніка:

  1. class Map[K, V] {
  2.   def get(key: K): V
  3.   def put(key: K, value: V): Unit
  4. }

Параметри типу вищих ґатунків та параметризовані

Параметри вищого ґатунку теоретично не відрізняються від звичайних параметрів типів (за винятком того, що їх ґатунок щанайменьше *=>*, скоріше ніж просто *). Домовленості іменування загалом однакові, однак бажано використовувати описові імена, скоріше, ніж одну літеру, для ясності:

  1. class HigherOrderMap[Key[_], Value[_]] { ... }

Одна літера формує (іноді) допустиме ім'я для фундаментальних концепцій, використовуваних в кодовій базі, таких, як F[_] для Functor та M[_] для Monad.

В таких випадках фундаментальна концепція повинна бути досить гарно відомою, та зрозумілою для команди, або мати трикратні докази, як в коди нижче:

  1. def doSomething[M[_]: Monad](m: M[Int]) = ...

Тут прив'язка типу: Monad пропонує потрібний доказ, щоб поінформувати читача, що цей M[_] є типом Monad.

Анотації

Анотації, такі як @volatile, повинні бути в нижньому верблюжому реєстрі:

  1. class cloneable extends StaticAnnotation

Домовленість використовується у всій бібліотеці Scala, навіть не зважаючи, що це не сумісне з іменуванням анотацій Java.

Зауваження: ця домовленість застосовується, навіть коли використовуються псевдоними типів на анотаціях. Наприклад, при використанні JDBC:

  1. type id = javax.persistence.Id @annotation.target.field
  2. @id
  3. var id: Int = 0

Спеціальне зауваження щодо стислості

Оскільки корені Scala в функціональних мовах, є досить нормальним для локальних імен бути дуже короткими:

  1. def add(a: Int, b: Int) = a + b

Це може бути поганою практикою для мов, як Java, але це добра практика в  Scala. Ця домовленість робить, бо вірно написані методи Scala є досить куцими, що охоплюють один вираз, та рідко займають більше декількох рядків. Використовуються тільки декілька локальних імен  (включаючи параметри), так що немає потреби вводити довгі, описові імена. Ця домовленість значно покращує краткість більшість тестів Scala. Це, в свою чергу, покращує читабельність, бо більше виразів ідуть до одного рядка, та аргументи метода мають описові імена.

Ця домовленість застосовується тільки до параметрів дуже простих методів (та локальних полів для дуже простих класів); все в публічному інтерфейсі повинно бути описовим. Також занотуйте, що імена аргументів тепер є частиною публічного API класа, оскільки користувачі можуть використовувати іменовані параметри в викликах метода.

Типи

Виведення

Використовуйте вивід типів, коли можливе, але на перше місце ставьте ясність, та схиляйтесь до явного в публічних API.

Ви повинні майже ніколи не анотувати тип приватного поля або локальної змінної, бо їх тип буде зазвичай безпосередньо випливати з його значення:

  1. private val name = "Daniel"

Однак, ви можете побажати все ще  відображати тип, де присвоєне значення має складну або неочевидну форму. 

Але публічні методи повинні мати явну анотацію типу. Вивід типу може зруйнувати інкапсуляцію в ціх випадкахє, оскільки це залежить від внутрішнього метода та деталей класа. Без явного типа зміна внутрішнього в методі або val може змінити публічний API класа без попереджень, потенційно ламаючи клієнтський код. Явні анотації типу можуть допомогти покращити час компіляції.

Значення функцій

Значення функцій підтримують спеціальний випадок вивода типу, який сам по собі вартий розголосу:

  1. val ls: List[String] = ...
  2. ls map (str => str.toInt)

В випадках, коли Scala вже знає тип значення функції, що ми декларуємо, не мамає потреби анотувати параметри (в цьому випадку, str). Це дуже корисний вивід, що повинний бути використаний, де тільки можливо. Зауважте, що неявні перетворення, що оперують на значеннях функцій, зведуть цей вивід до нуля, змушуючи явні анотації типів параметрів.

Анотації

Анотації типів повинні бути сформавані згідно до наступного шаблона:

  1. value: Type

Цей стиль адаптований в більшості стандартної бібліотеки Scala, та у всіх прикладах Martin Odersky. Проміжок між значенням та типом допомагає оку в акуратному розбору синтаксису. Причина для розміщення двокрапки в кінці значення, скоріше, ніж на початку типа є в уникненню неоднозначності, в випадках, подібних до наступного:

  1. value :::

Це, насправді, валідний Scala, що декларує значення типу ::. Очевидно, що анотація з префіксом-двокрапкою значно затуманює речі. 

Приписування

Приписування типу часто плутається з анотацією типу, тому що синтаксис в Scala ідентичний. Наступне є прикладами приписування:

  • Nil: List[String]
  • Set(values: _*)
  • "Daniel": AnyRef

Приписування в основному тільки кастинг, виконанний під час компіляції для потреб перевірки типа. Його використання не є загальним, але часом трапляється при нагоді. Найбільш частий випадок приписування є виклик метода зі змінним числом аргументів з одним параметром Seq. Це робиться приписуванням типу _*  (як в другому прикладі вище).

Приписування дотримуєстья домовленостей анотацій типів; за двокрапкою слідує проміжок.

Функції

Типи функцій мають бути декларовані з проміжком між типом параметра, стрілкою, та повернутим типом:

  1. def foo(f: Int => String) = ...

  3. def bar(f: (Boolean, Double) => List[String]) = ...

Дужки повинні бути опущені, коли це потрібно (тобто в методах арності-1, таких як Int => String).

Арність-1

Scala має особливий синтаксис для декларування типів функцій арності-1. Наприклад:

  1. def map[B](f: A => B) = ...

Кажучи більш конкретно, дужки можуть бути опущені для параметру типу. Таким чином, ми не декларуєте fяк тип(A) => B, тому це буде без потреби балакучим. Розглянемо більш екстремальний приклад:

  1. // невірно!
  2. def foo(f: (Int) => (String) => (Boolean) => Double) = ...

  4. // вірно!
  5. def foo(f: Int => String => Boolean => Double) = ...

Опускаючи дужки ми зберігли цілі шість символів, та драматично покращили читабельність вираза типу.

Структурні типи

Структурні типи повинні декларуватись на одному рядку, якщо вони мають менше 50 символів в довжину. Інакше вони повинні бути розділені між декількома рядками, та (зазвичай)  присвоєні власним псевдоніма типів:

  1. // невірно!
  2. def foo(a: { def bar(a: Int, b: Int): String; val baz: List[String => String] }) = ...

  4. // вірно!
  5. private type FooParam = {
  6.   val baz: List[String => String]
  7.   def bar(a: Int, b: Int): String
  8. }

  10. def foo(a: FooParam) = ...

Простіші структурні типи (до 50 символів) можуть бути задекларовані та використані одночасно (інлайн):

  1. def foo(a: { val bar: String }) = ...

Коли ви декларуєте структурі типи інлайн, кожний член повинен відділятись крапкою з комою та одним проміжком, відкриваюча фігурна дужка повинна мати проміжок після себе, тоді як для закриваючої фігурної дужки проміжок іде перед нею (як демонструють обоє приклади вище).

Структурі типи реалізовані за допомогою рефлексії під час виконання, так що природно є менш продуктивними, ніж нормальні типи. Розробники повинні схилятись до використання номінальних типів, тільки якщо структурні типи не провадять явної вигоди. 

Вкладені блоки

Фігурні дужки

Відкриваючі фігурні дужки ({) мають бути на тому ж рядку, що і декларація, що вони представляють:

  1. def foo = {
  2.   ...
  3. }

Технично парсер Scala не підтримує нотацію в GNU-стилі з відкриваючими дужками на рядку, що слідує за декларацією. Однак парсер не є жахливо передбачуваним, коли має справу з цім стилем, через шлях, яким реалізоване виведення крапки з комою. Багато головної болі буде збережено, коли просто слідувати домовленості щодо фігурних дужок, показаної вище.

Дужки

В рідких випадках, коли блоки дужок переходять на інший рядок, відкриваючі та закриваючі дужки повинні іти без проміжків, та, загалом, знаходитись на тому ж рядкові, що і їхній вміст (в стилі Lisp):

  1. (this + is a very ++ long *
  2.   expression)

Дужки також прислуговувуються для відключення вивідення крапки з комою, та, таким чином, дозволяють розробнику починати рядки з операторів, до чого деякі схильні:

  1. (  someCondition
  2. || someOtherCondition
  3. || thirdCondition
  4. )

В цьому випадку прийнятні заключні дужки на наступному рядку, з естетичних причин.

Декларації

Класи

Конструктри класів/об'єкті/трейтів повинні декларуватись на одному рядку, за винятком, коли рядок стає "дуже довгим" (більше 100 символів). В цьому випадку розташовуйте кожний аргумент конструктора у власному рядку, з зсувом в чотири проміжки:

  1. class Person(name: String, age: Int) {
  2. }

  4. class Person(
  5.     name: String,
  6.     age: Int,
  7.     birthdate: Date,
  8.     astrologicalSign: String,
  9.     shoeSize: Int,
  10.     favoriteColor: java.awt.Color) {
  11.   def firstMethod: Foo = ...
  12. }

Якщо клас/об'єкт/трейт розширює будь-що, застосовується це ж загальне правило: розташовуйте його в один рядок, за виключенням випадку, коли це займає більше 100 символів, та потім сзувайте на чотири проміжки для кожного елементу на власному рядку, та на два проміжки для розширень; це дає візуальне поділення між аргументами конструктора, та розширеннями:

  1. class Person(
  2.     name: String,
  3.     age: Int,
  4.     birthdate: Date,
  5.     astrologicalSign: String,
  6.     shoeSize: Int,
  7.     favoriteColor: java.awt.Color) 
  8.   extends Entity
  9.   with Logging
  10.   with Identifiable
  11.   with Serializable {
  12. }

Впорядкування елементів класа

Всі члени класа/об'єкта/трейта повинні бути задекларовані, переміжаючись порожніми рядками. Одне виключення до цього правила є varта val. Вони можуть бути декларовані без пустих рядків, але тільки якщо жодне з полів не має Scaladoc, та якщо всі з полів мають прості визначення (в один рядок, до 20 символів):

  1. class Foo {
  2.   val bar = 42
  3.   val baz = "Daniel"

  5.   def doSomething(): Unit = { ... }

  7.   def add(x: Int, y: Int): Int = x + y
  8. }

Поля повинні передувати методам в полі зору. Одне виключення є випадок, коли val має блочне визначення (більше ніж один вираз), або виконує операції, що можуть розглядатись "методо-подібно" (як обчислення довжини List). В таких випадках нетривіальний val може бути задекларований пізніше в файлі, ніж диктує логічний порядок членів. Це правило стосується тільки до val та lazy val! Буде дуже складним відслідити змінені псевдоніми, якщо декларації var розкидані по вьому файлу класа.

Методи

Методи повинні декларуватись згідно наступного шаблону:

  1. def foo(bar: Baz): Bin = expr

Методи з параметрами по замовчанню повинні декларуватись аналогічним чином, з проміжком з кожного боку знаку рівності:

  1. def foo(x: Int = 6, y: Int = 7): Int = x + y

Ви повинні задавати тип повернення для всіх публічних методів. Зробіть так, щоб їх документація перевірялась компілятором. Це також допомогає зберігати двійникову сумісність перед лицем зміни типу інтерфесу (зміни до реалізації методу можуть поширитись на тип результату, якщо він виводиться).

Локальні або приватні методи можуть не вказувати свій тип результату:

  1. private def foo(x: Int = 6, y: Int = 7) = x + y

Процедурний синтаксис

Уникайте процедурного синтаксису, бо він має тенденцію бути спантеличливим, ціною дуже малої вигоди в розмірі.

  1. // не робіть цього
  2. def printBar(bar: Baz) {
  3.   println(bar)
  4. }

  6. // замість цього пишить так
  7. def printBar(bar: Bar): Unit = {
  8.   println(bar)
  9. }

Модифікатори

Модифікатори методів повинні надаватись в наступному порядку (коли вони взагалі застосовуються):

  1. Анотації, кожна на окремому рядку
  2. Модифікатор перекриття (override)
  3. Модифікатор доступу (protected, private)
  4. Модифікатор фінальності (final)
  5. def
  1. @Transaction
  2. @throws(classOf[IOException])
  3. override protected final def foo() { 
  4.   ...
  5. }

Тіло

Коли тіло метода складається з одного вираза, що є меньше 30 (або десь біля цього) символів, воно повинно надаватись в одному рядку з методом:

  1. def add(a: Int, b: Int): Int = a + b

Коли тіло метода є одним виразом, довшим ніж 30 (або близько цього) симолів, але все ще меньше ніж 70 (або близько цього) символів, воно повинне надаватись в наступному рядку, зсунуте на два проміжка:

  1. def sum(ls: List[String]): Int =
  2.   ls.map(_.toInt).foldLeft(0)(_ + _)

Різниця між ціма двома випадками є дещо штучна. Кажучи загалом, ви маєте обрати, який стиль є більш читабельний в кожному окремому випадку. Наприклад, декларація вашого метода може бути дуже довгою, тоді як вираз тіла може бути досить коротке. В такому випадку може бути більш читабельним перенести вираз на наступний рядок, скоріше, ніж робити рядок декларації задовгим. 

Коли тіло метода не може бути стисло виражене в одному рядку, або воно має нефункціональну природу (деякі змінні стани, локальні, або ще щось), тіло має бути заточене в фігурні дужки:

  1. def sum(ls: List[String]): Int = {
  2.   val ints = ls map (_.toInt)
  3.   ints.foldLeft(0)(_ + _)
  4. }

Методи, що містять один вираз match, повинні бути задекларовані наступним чином:

  1. // вірно!
  2. def sum(ls: List[Int]): Int = ls match {
  3.   case hd :: tail => hd + sum(tail)
  4.   case Nil => 0
  5. }

Не таким чином:

  1. // невірно!
  2. def sum(ls: List[Int]): Int = {
  3.   ls match {
  4.     case hd :: tail => hd + sum(tail)
  5.     case Nil => 0
  6.   }
  7. }

Декілька списків параметрів

Загалом, ви повинні використовувати декілька списків параметрів, якщо для цього є досить поважні причини. Ці методи (або подібно задекларовані функції) мають більш балакучі декларації та синтаксис, та складніші для розуміння меньш досвідченими Scala розробниками.

Є три головні причини, коли ви маєте це робити:

  1. Для побіжного API

    Декілька списків параметрів дозволяють вам створювати ваші власні "керівні структури":

    1. def unless(exp: Boolean)(code: => Unit): Unit = if (!exp) code
    2. unless(x < 5) { 
    3.   println("x was not less than five")
    4. }

  2. Неявні параметри

    При використанні неявних параметрів, за використання ключового слова implicit, воно застосовується до цілого списка параметрів. Таким чином, якщо ви бажаєте, щоб неявними були тільки деякі параметри, ви повинні використовувати декілька спискіа параметрів.

  3. Для виведення типу

    Коли викликається метод з використанням тільки деяких зі списків параметрів, механізм виводу типу може дозволити простіший синтаксис при виклику залишку списків параметрів. Розглянемо згортання:

    1. def foldLeft[B](z: B)(op: (A,B) => B): B
    2. List("").foldLeft(0)(_ + _.length)

    4. // Якщо замість цього написатит так:
    5. def foldLeft[B](z: B, op: (B, A) => B): B
    6. // вам доведеться вказувати типи
    7. List("").foldLeft(0, (b: Int, a: String) => a + b.length)
    8. List("").foldLeft[Int](0, _ + _.length)

Для складних DSL, або для імен типів, що задовгі, може бути складним розмістити цілу сигнатуру на одному рядку. В таких випадках вирівняйте відкриваючу дужку кожного списку параметрів, по одному списку на рядок, (тобто ви не маєте покладати їх на одному рядку, розміщуйте кожний список в окремому рядку):

  1. protected def forResource(resourceInfo: Any)
  2.   (f: (JsonNode) => Any) 
  3.   (implicit urlCreator: URLCreator, configurer: OAuthConfiguration): Any = {
  4.   ...
  5. }

Функції вищого порядку

Має сенс пам'ятати, що коли декларується функції вищого порядку, Scala дозволяє дещо кращий синтаксис для таких функцій в точці виклику, якщо функціональний параметр карований як останній аргумент. Наприклад, ось функція foldl в SML:

  1. fun foldl (f: ('b * 'a) -> 'b) (init: 'b) (ls: 'a list) = ...

В Scala переважний стиль прямо протилежний:

  1. def foldLeft[A, B](ls: List[A])(init: B)(f: (B, A) => B): B = ...

Розміщуючи функціональний параметр останнім, ми дозволяємо синтаксис виклику таким чином:

  1. foldLeft(List(1, 2, 3, 4))(0)(_ + _)

Значення функції в функції не замкнене в дужки; воно синтаксично досить відімкнене від самої функції (foldLeft). Цьому стилю має надаватись перевага, задля стислості та прозорості. 

Поля

Поля повинні слідувати правилам декларації для методів, приймаючи спеціальну нотацію впорядкування модифікаторів доступу та домовленості щодо анотацій.

Ліниві змінні повинні використовувати ключове словоlazyпрямо перед val:

  1. private lazy val foo = bar()

Функціональні значення

Scala провадить декілька різних синтаксичних опцій для декларування функціональних значень. Наприклад, наступні декларації є повністю еквівалентними:

  1. val f1 = ((a: Int, b: Int) => a + b)
  2. val f2 = (a: Int, b: Int) => a + b
  3. val f3 = (_: Int) + (_: Int)
  4. val f4: (Int, Int) => Int = (_ + _)

З усіх ціх стилів (1) та (4) є бажаними за будь-яких умов. (2) виглядає коротшим в цьому прикладі, але коли функціональне значення займає декілька рядків, цей синтаксис стає вкрай громіздким. Подібно, (3) є виразним, але зарозумілим. Для нетренованого ока є складинм дешифрувати той факт, що це функціональне значення для звичайного складання. 

Коли використовуються виключно стилі (1) та (4), стає дуже простим віднайти місця в первинному коді, де використовуються функціональні значення. Обоє стилі використовують дужки, оскільки вони виглядають зрозуміло в одному рядку.

Проміжки

Не має бути проміжка між дужками, та кодом, якій вони обіймають. Фігурні дужки повинні відокремлюватись від кода в них розривом в один проміжок, щоб надати візуально навантаженим дужками "простір для дихання".

Функції з декількох виразів

Більшість функціональних значень менш тривіальні, ніж приклади, наведені вище. Багато містять більш ніж один вираз. В таких випадках часто є більш читабельним розділити функціональне значення на декілька рядків. Коли це трапляється, треба використовувати тільки стиль (1), замінюючи дужки на фігурні дужки. Стиль (4) стає вкрай складним для відстеження, коли в дужках замкнено дуже багато кода. Сама декларація повинна вільно слідувати стилю декларації методів, з відкиваючою дужкою на тому ж рядку, що і присвоєння або виклик, тоді як замикаючі дужки на власному рядку, що безпосередньо слідує за останнім рядком функції. Параметри повинні бути на тому ж рядку, що і відкриваюча дужка, так само, як і "стрілка" (=>):

  1. val f1 = { (a: Int, b: Int) =>
  2.   val sum = a + b
  3.   sum
  4. }

Як зазначалось вище, значення функцій повинні задіяти вивід типів, коли це тільки можливе.

Структури керування

Всі струтури керування повинні бути записані з проміжком, що ідуть за ключовим словом:

  1. // вірно!
  2. if (foo) bar else baz
  3. for (i <- 0 to 10) { ... }
  4. while (true) { println("Hello, World!") }

  6. // невірно!
  7. if(foo) bar else baz
  8. for(i <- 0 to 10) { ... }
  9. while(true) { println("Hello, World!") }

Фігурні дужки

Фігурні дужки повинні опускатись в випадках, коли керуюча структура представляє часто функціональну операцію, та всі гілки структури керування (має відношення if/else) є виразами з одного рядка. Пам'ятайте наступні настанови:

  • if - Не вказуйте дужки, якщо ви маєте твердження else. Інакше оточіть вміст в фігурні дужки, навіть якщо вміст складається тільки з одного рядка.
  • while - Ніколи не забувайте про фігурні дужки (while не може використовуватись в повністю функціональний спосіб).
  • for - Не вказуйте дужки, коли ви маєте yield. Інакше заточіть вміст в фігурні дужки, навіть якщо вміст складається лише з одного рядка.
  • case - Завжди опускатйе дужки для цієї конструкції.
  1. val news = if (foo)
  2.   goodNews()
  3. else
  4.   badNews()

  6. if (foo) {
  7.   println("foo was true")
  8. }

  10. news match {
  11.   case "good" => println("Good news!")
  12.   case "bad" => println("Bad news!")
  13. }

Осяжності

Scala має можливість предсавляти for-осяжності з більше ніж одним генератором (звичайно, з більше ніж одним символом <-). В таких випадках є два альтернативних синтаксиса, що можуть використовуватись:

  1. // невірно!
  2. for (x <- board.rows; y <- board.files) 
  3.   yield (x, y)

  5. // вірно!
  6. for {
  7.   x <- board.rows
  8.   y <- board.files
  9. } yield (x, y)

Хоча останній синтаксис більш балакучий, він звичайно визнається як більш простіший для читання, та більш "маштабований” (в тому значенні, що він не приховує, коли складність осяжності зростає). Вам треба схилятись до цієї форми для всіх for-осяжностей, що мають більше одного генератора. Осяжності з тільки одним генератором  (як for (i <- 0 to 10) yield i) повинні використовувати першу форму (з дужками, не з фігурними дужками).

Виключення до цього правила є for-осяжності, яким бракує частини yield. В таких випадках конструкція насправді є швидше циклом, ніж функціональною осяжністю, та, загалом, є більш читабельне вистроїти генератори один за одним між дужками, ніж використовувати  синтаксично-збентежливу конструкцію } { :

  1. // невірно!
  2. for {
  3.   x <- board.rows
  4.   y <- board.files
  5. } {
  6.   printf("(%d, %d)", x, y)
  7. }

  9. // вірно!
  10. for (x <- board.rows; y <- board.files) {
  11.   printf("(%d, %d)", x, y)
  12. }

Нарешті, for осяжності схильні до зціплених викликів map, flatMap, та filter, та це може ускладнити читання (це одне з призначень розширених for осяжностей).

Тривіальні умови

Є деякі ситуації, коли корисно створити короткий вираз if/else для вкладеного використання в більшому виразі. В Java цей різновид випадків традиційно обробляється тримісним оператором (?/:), синтаксичним пристроєм, якого бракує в Scala. В таких ситуаціях (та, насправді, кожного разу, коли ви маєте вкрай короткий вираз if/else) допускається розмістити гілки "then” та "else” в один рядок, разом з ключовими словами  if та else:

  1. val res = if (foo) bar else baz

Ключем тут є читабельність, що не страждає від пересування обох гілок в один рядок з if/else. Зауважте, що цей стиль ніколи не повинен використовуватись в імперативних виразах if, та також не слід використовувати фігурні дужки.

Виклик методів

Загально кажучи, виклик методів в Scala слідує домовленостям Java. Іншими словами, не повинно бути проміжка між цілюю виклика та крапкою (.), а також проміжка між крапкою та іменем метода, а також не повинно бути проміжка між ім'ям метода та роздільниками аргументів (дужками). Кожний аргумент повиннен відокремлюватись поодиноким комою, за якою слідує проміжок (,):

  1. foo(42, bar)
  2. target.foo(42, bar)
  3. target.foo()

Як щодо версії 2.8, Scala тепер має підтримку для поіменованих параметрів. Іменовані параметри в виклику метода повинні трактуватись як звичайні параметри (відповідно розділені проміжками та комами), з проміжком з обох сторонін знакак рівняння:

  1. foo(x = 6, y = 7)

Хоча цей стиль створює візуальну неоднозначність між іменованими параметрами, та присвоєнням змінних, альтернатива (без проміжків коло знака рівняння) призводить до кода, що може бути дуже важко читати, зокрема для нетривіальних виразів для актуалів.

Арність-0

Scala дозволяє не вказувати дужки для методів з арністю-0 (без аргументів):

  1. reply()

  3. // те ж саме, що і

  5. reply

Однак цей синтаксис повинен використовуватись тільки коли метод не має побічних ефектів (чисто функціональний). Іншими словами, може бути допустимим опустити дужки при викликуqueue.size, але не для виклику println(). Ця домовленість відзеркалює домовленість визначення методів, надану вище.

Якщо прискіпливо дотримуватись цієї домовленості можливо драматично покращити читабельність кода, та значно спростити розуміння з першого погляду найбільш базових операцій будь-якого метода. Чиніть супротив квапливості в відкиданні дужок, тільки щоб зберігти два символи!

Суфіксна нотація

Scala дозволяє методи з арністю-0 бути викликаними в суфіксній нотації:

  1. names.toList

  3. // те саме, що і

  5. names toList // Небезпечно, не використовуйте!

Цей стиль небезпечний, та не має використовуватись. Оскільки крапка з комою є опціональні, компілятор буде намагатись трактувати це як інфіксний метод, якщо він зможе, потенційно захоплюючи терм з наступного рядка.

  1. names toList
  2. val answer = 42        // не зкомпілюється!

Це може призвести до неочікуваних помилок компіляції, в кращому випадку, та до компіляції збійного кода, в гіршому. Хоча синтаксис використовується в деяких DSL, він має бути визнаний як заборонений, та його слід уникати. 

В Scala 2.10 використання суфіксних операторів буде призводити до попереджень компілятора.

Арність-1

Scala має спеціальний синтаксис для виклику методів з арністю-1 (один аргумент):

  1. names.mkString(",")

  3. // те ж саме, що і

  5. names mkString ","

Цей синтаксис формально відомий як "інфіксна нотація". Його варто застосовувати тільки для чисто функціональних методів (методів, що не мають побічних ефектів) - як mkString - або методів, що приймають функції як параметри, як foreach:

  1. // вірно!
  2. names foreach (n => println(n))
  3. names mkString ","
  4. optStr getOrElse ""

  6. // невірно!
  7. javaList add item

Функції вищого порядку

Як зазначалось, методи, що приймають фукнції як параметри (як map або foreach) повинні викликатись за допомогою інфіксної нотації. Також можливо викликати такі методи наступним шляхом:

  1. names.map (_.toUpperCase)     // невірно!

Цей шлях не є прийнятним стандартом! Причина уникати цього стиля є ситуації, коли більш ніж один виклик потрібно зцепити разом:

  1. // невірно!
  2. names.map (_.toUpperCase).filter (_.length > 5)

  4. // вірно!
  5. names map (_.toUpperCase) filter (_.length > 5)

Обоє варіанти працюють, але перший експлуатує вкрай неінтуітивну дірку в граматиці Scala. Підвираз (_.toUpperCase).filter, якщо його розглядати окремо, виглядає для всього світу так, якби ми викликали метод filterна значенні функції. Однак насправді ми викликаємо filterна результаті метода map, що приймає значення функції як параметр. Цей синтаксис бентежить, та часто вводить у відчай в  Ruby, але він прямо уникаєтся в Scala.

Символічні методи/оператори

Методи з символічними іменами завжди повинні викликатись з використанням інфіксної нотації, з проміжками, що відокремлють ціль, символічний метод та параметр:

  1. // вірно!
  2. "daniel" + " " + "Spiewak"

  4. // невірно!
  5. "daniel"+" "+"spiewak"

Здебільшого ця ідіома слідує за синтаксичними домовленостями Java та Haskell.

Символічні методи, що приймають більше одного параметра  (вони існують!) повинн все ще викликатись в інфіксній нотації, відокремлені проміжками:

  1. foo ** (bar, baz)

Однак такі методи досить рідкісні, та їх треба уникати під час  розробки API.

Нарешті, використання /: та :\ треба уникати, на користь до більш явних методів foldLeftта foldRight на Iterator. Права асоциативність /: може призвести до вкрай збентежливого коду, ціною збереження декількох символів.

Файли

Як правило, файли повинні містити єдиний логічний набір. Під "логічний” я маю на увазі клас, трейт або об'єкт. Одне виключення до цієї настанови є для класів або трейтів, що мають об'єкт-компанйон. Об'єкти-компанйони повинні групуватись зі своїми відповідними класами або трейтами в той же файл. Ці файл повинно іменувати відповідно до класа, трейта або об'єкта, що вони містять:

  1. package com.novell.coolness

  3. class Inbox { ... }

  5. // об'єкт-компанйон
  6. object Inbox { ... }

Ці одиниці компіляції повинно розташовувати в файі на ім'я nbox.scala в каталозі com/novell/coolness. Скорочено, треба схилятись до домовленостей іменування файлів Java, хоча, фактично, Scala дозволяє більшу гнучкість відносно цього.

Багато-об'єктні файли

Не зважаючи на те, що ми тільки що сказали, є деякі важливі ситуації, що підштовхують до вкладання декількох одиниць компіляції до одного файла. Одним важливим прикладом є прикріплений трейт, та декілька субкласів (що часто емулює можливість ADT мови, доступну в функціональних мовах):

  1. sealed trait Option[+A]

  3. case class Some[A](a: A) extends Option[A]

  5. case object None extends Option[Nothing]

Завдяки природі прив'язаних суперкласів (та трейтів), всі підтипи мають включатись в той же самий файл. Таким чином, така ситуація однозначно кваліфікується як приклад, коли принцип "один клас - один файл" має бути проігноровано. 

Інший випадок, коли декілька класів логічно формують одну, згуртовану групу, поділяючі концепції в тій мірі, коли підтримка дуже спрощується, якщо вони будуть розміщені в одному файлі. Ці ситуації складніше передбачити, ніж попередньо згадуване виключення для зв'язаних типів. Кажучи загалом, якщо простіше здійснювати довготривале обслуговування та розробку декількох одиниць в одному файлі, замість розділити на декілька, тоді така організаційна стратегія повинна використовуватись для ціх класів. Однак майте на увазі, що коли декілька одиниць містяться в одному файлі, часто більш складно знайти окремі одиниці, коли доходить до внесення змін.

Всі багато-об'єктні файли повинні іменуватись в верблюжомуРеєстрі, з першою малою літерою. Це дуже важлива домовленість. Це відрізняє багато- від одно-об'єктних файлів, що спрощує процес пошуку декларацій. Ці імена файлів можуть базуватись на важливому типі, якій вони містять, (як option.scala для прикладу вище), або можуть мати описове ім'я логічної властивості, що поділяють всі одиниці компіляції в файлі (як ast.scala).

Scaladoc

Важливо запровадити документацію для всіх пакунків, класів, трейтів, методів та інших членів. Scaladoc загалом слідує домовленостям Javadoc, однак є багато додаткових можливостей, що робить написання scaladoc простішим.

Загалом, вам слід більше турбуватись про наповнення та стиль написання, ніж про форматування. Scaladoc має бути корисною для нових користувачів кода, так само, як і для досвідчених. Досягти цього дуже просто: збільшуйте рівень деталей та пояснення, коли ви пишете, починаючи з небагатослівного підсумку (корисного для досвідчених користувачів в якості посилання), при цьому надаючи глибокі приклади в деталізованих розділах (що можуть бути проігноровані досвідченими користувачами, але неоціненні для початківців).

Головний формат коментарів Scaladoc повинен бути наступним:

  1. /** Це коротчкий опис того, що документується.
  2.   *
  3.   * Це подальша документація того, що документується. Вона повинна
  4.   * провадити більше деталей, як, та що, воно робить. 
  5.   */
  6. def myMethod = {}

Для методів та інших типів членів, де потрібна тільки проста документація, в вигляді короткого опису, можна використовувати такий формат:

  1. /** Does something very simple */
  2. def simple = {}

Зауваження, особливо для тих, хто прийщов з Java, що лівий відступ зірочок підпадає  під _третій_ рядок, не другий, як прийнято в Java.

Дивіться AuthorDocs на Scala wiki для додаткової інформації щодо форматування Scaladoc.

Загальний стиль

Важливо підтримувати узгоджений стиль за допомогою Scaladoc. Також важливо націлювати Scaladoc на обох, того, хто не знайомий з вашим кодом, та досвідчених користувачів, яким потрібне тільки швидке посилання. Ось деякі загальні настанови:

  • Діходьте до суті як можна швидше. Наприклад, скажіть "повертає true в разі такої умови" замість "якщо деяка умова справдиться".
  • Намагайтесь форматувати перше речення метода як "Повертає XXX”, як в "Повертає перший елемент в List", а не "цей метод повертає" або "отримує перший елемент", тощо. Методи типово те, що вони повертають.
  • Те ж саме підходить і для класів; уникайте "Цей клас робить XXX”; просто скажіть "Робить XXX”
  • Стрюйте посилання на використовувані класи Scala Library, з викристанням синтаксиса в прямокутних дужках, як[[scala.Option]]
  • Підсумуйте щодо повернутого значення метода в анотації @return, залишивши довше пояснення для головного Scaladoc.
  • Якщо документація метода є опис в один рядок, того, що повертає метод, не повторюйте те саме в анотації @return.
  • Документуйте, що робить метод, не те, що він повинен зробити. Іншими словами, скажіть "повертає результат застосування f до x", ніж "повертається резльтат застосування f до x”. Тонка матерія, але важливо. 
  • Коли посилаєтесь на примірник класа, використовуйте "цей XXX”, або "цей”, але не "XXX”. Для об'єктів кажіть "цей об'єкт”.
  • Робіть приклади узгодженими з ціма настановами.
  • Використовуйте синтаксис wiki-стилю, замість HTML, де це можливо.
  • Приклади повинні використовувати або повні лістинги кода, або REPL, в залежності від того, що потрібно (простіший спосіб включити код REPL є розробляти приклади в REPL, та скидати його в Scaladoc).
  • Вільно використовуйте @macro для посилання на загально повторювані значення, що потребують спеціального форматування.

Пакунки

Провадьте Scaladoc для кожного пакунку. Вона іде в файлі на ім'яpackage.scalaв вашому каталозі пакунку, та виглядає подібним чином (для пакунка  parent.package.name.mypackage):

  1. package parent.package.name

  3. /** Це Scaladoc для пакунку */
  4. package object mypackage {
  5. }

Документація пакунку повинна спочатку документувати, які типи класів є частиною пакунку. Друге, документувати загальні речі, що провадить сам пакунок. 

Хоча документація пакунка не має бути повноцінним туторіалом по використанню класів в пакунку, вона має давати огляд головним класам, з деякими базовими прикладами того, як використовувати класи в цьому пакунку. Переконайтесь, що посилаєтесь на класи за допомогою нотації прямокутних дужок:

  1. package my.package
  2. /** Провадить класи для роботи з комплексними числами. Також провадить
  3.   * неявні для перетворенням між комплексними та `Int`.
  4.   *
  5.   * ==Огляд==
  6.   * Головний клас для використання [[my.package.complex.Complex]]
  7.   * {{{
  8.   * scala> val complex = Complex(4,3)
  9.   * complex: my.package.complex.Complex = 4 + 3i
  10.   * }}}
  11.   *
  12.   * Якщо ви підключите [[my.package.complex.ComplexConversions]], ви 
  13.   * можете конвертувати числа більш прямолінійно
  14.   * {{{
  15.   * scala> import my.package.complex.ComplexConversions._
  16.   * scala> val complex = 4 + 3.i
  17.   * complex: my.package.complex.Complex = 4 + 3i
  18.   * }}} 
  19.   */
  20. package complex {}

Класи, об'єкти та трейти

Документуйте всі класи, об'єкти та трейти. Перше речення Scaladoc повинне провадити підсумок того, що робить клас. Документуйте всі параметри типів за допомогою@tparam.

Класи

Якщо клас повинно створювати за допомогою об'єкта-компанйона, вкажіть це після опису класа (але залиште деталі створення до об'єкта-компанйона). На жаль, на зараз немає способу сворити посилання на о'бєкт-компан'он прямо на місці, але зненерований Scaladoc створить посилання за вас при створенні документації класа.

Якщо клас повинен створюватись з використанням конструктора, задокументуйте це за допомогою синтаксису @constructor:

  1. /** Особистіть, що користується нашою програмою.
  2.   *
  3.   * @constructor створює нову особу з ім'м та віком.
  4.   * @param name ім'я 
  5.   * @param age вік в роках
  6.   */
  7. class Person(name: String, age: Int) {
  8. }

В залежності від складності вашого класа, надайте приклад загального використання.

Об'єкти

Оскільки об'єкти можуть використовуватись для декількох цілей, важливо документувати, як використовувати об'єкт (тобто, як фабрику, для неявних методів). Якщо об'єкт є фабрикою для інших об'єктів, вкажіть це тут, відклавши особистості до Scaladoc для метода(-дів) apply. Якщо ваш об'єкт не використовуєapply в якості метода-фабрики, переконайтесь, що вказали справжні імена методів:

  1. /** Factory for [[mypackage.Person]] instances. */
  2. object Person {
  3.   /** Створює особу з ім'ям та віком.
  4.     *
  5.     * @param name ім'я
  6.     * @param age вік особи
  7.     */
  8.   def apply(name: String, age: Int) = {}

  10.   /** Створює особу з ім'ям та датою народження
  11.     *
  12.     * @param name ім'я
  13.     * @param birthDate день народження
  14.     * @return новий примірник Person з обчисленим віком на базі поточної дати. 
  15.     */
  16.   def apply(name: String, birthDate: java.util.Date) = {}
  17. }

Якщо ваш об'єкт має наявні перетворення, запровадьте приклад в Scaladoc:

  1. /** Неявні перетворення та допоміжники для примірників [[mypackage.Complex]].
  2.   *
  3.   * {{{
  4.   * import ComplexImplicits._
  5.   * val c: Complex = 4 + 3.i
  6.   * }}} 
  7.   */
  8. object ComplexImplicits {}

Трейти

Після огляду того, що робить трейт, надайте огляд методів та типів, що мають бути вказані в класах, що підмішуватимуть цей трейт. Якщо відомі класи, що використовують цей трейт, дайте посилання на них.

Методи та інші члени

Документуйте всі методи. Як з іншими документованими сутностями, перше речення повинне підсумовувати те, що робить метод. Наступні речення пояснюють подальші деталі. Документуйте кожний параметр, так само, як кожний параметр типу (за допомогою @tparam). Для карованих функцій розгляньте надати більш детальні приклади, в залежності від очікуваного або ідеоматичного використання. Для неявних параметрів надайте окрему увагу поясненню, звідки ці параметри надходимуть, та чи користувач потребує виконати якусь додаткову роботу, щоб переконатись, що параметри будуть доступні.
