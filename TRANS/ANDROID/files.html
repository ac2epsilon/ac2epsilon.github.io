Андроїд використовує файлову систему, подібну до розташованої на диску для інших платформ. Цей урок описує, як робити з файловою системою Android, щоб записувати та читати файли за допомогою File API.

Об’єкт File підходить для читання та запису великих фрагментів даних в порядку від початку до кінця, без пропусків. Нвприклад, це добре для файлів зображень, або будь чого, чим обмінюються в мережі.

Цей урок показує, як виконувати базові, пов’язані з файлами, завдання в вашому застосуванні. Урок припускає, що ви добре знайомі з основами файлової системи Linux, та стадартним файловим вводом-виводом в java.io.

Обрання внутрішнього або зовнішнього сховища

Всі пристрої Android мають два місця зберігання: "внутрішнє" та  "зовнішнє" сховище. Ці імена походять з ранішніх днів Android, коли більшість пристроїв пропонували вбудовану незалежну пам’ять (внутрішнє сховище), плюс змінний носій, такий, як картка microSD (зовнішнє сховище). Деякі пристрої поділяють простір постійного зберігання на  "внутрішній" та "зовнішній" розділи, так що навіть без сховища на змінному носії завжди є два простори зберіганя, та поведінка API така ж сама, чи зовнішнє сховище дійсно на змінному носії, чи ні. Наступний список підсумовує факти щодо кожного простору зберігання.

Внутрішнє сховище:

  • Є завжди доступним.
  • Збережені тут файли доступні тількі для вашого застосування.
  • Коли користувач деінсталює ваше застосування, система видаляє всі файли вашого застосування з внутрішнього сховища.

Внутрішнє сховище є кращим, коли ви бажаєте бути впевненим, що ні користувач, ні другі застосування не зможуть отримати до них доступ.

Зовнішнє сховище:

  • Не завжди є доступним, оскількі користувач може змонтувати зовнішній пристрій як USB сховище, та в деяких випадках видаляти його з пристрою.
  • Воно читається усім світом, так що файли, збережені тут, можуть бути прочитані зовні без вашого контролю.
  • Коли користувач видаляє ваше застосування, система видаляє файли вашого застосування звідси, тільки якщо ви зберігаєте їх в каталозі від getExternalFilesDir().

Зовнішнє сховище є кращим місцем для файлів, що не потребують обмежень на доступ та для файлів, які ви бажаєте поділяти з іншими застосуваннями, або дозволити користувачеві отримати доступ за допомогою компьютера. 

Підказка: Хоча застосування встановлюються у внутрішнє сховище по замовчанню, ви можете вказати атрибут android:installLocation в вашому маніфесті, так що ваше застосування може бути встановлено на зовнішній носій. Користувачі  змінюють цю опцію, коли розмір APK дуже великий, та вони мають зовнішнє сходище, що більше за внутрішнє. Для додаткової інформації дивіться App Install Location.

Отримання дозволу для зовнішнього сховища

Щоб записувати в зовнішнє сховище ви маєте запросити дозвіл  WRITE_EXTERNAL_STORAGE в вашому  файлі маніфесту:

<manifest ...>
    <uses-permission android:name="android.permission.WRITE_EXTERNAL_STORAGE" />
    ...
</manifest>

Застереження: Наразі всі застосування мають здібність читати зовнішнє сховище без спеціального дозволу. Однак, це може бути змінено в подальних версіях. Якщо ваше застосування потребує читати зовнішнє сховище (але не писати до нього), тоді вам потрібно задекларувати дозвіл READ_EXTERNAL_STORAGE. Щоб впевнитись, що ваше застосування робить як очікується, вам треба задекларувати цей дозвіл зараз, перед тим, як зміни наберуть чинності.

<manifest ...>
    <uses-permission android:name="android.permission.READ_EXTERNAL_STORAGE" />
    ...
</manifest>

Однак якщо застосування використовує дозвіл WRITE_EXTERNAL_STORAGE, тоді воно неявно має дозвіл на читання з зовнішнього сховища.

Вам не треба жодних дозволів щоб зберігати файли у внутрішньому сховищі. Ваше застосування має дозвіл читати та писати файли в каталозі внутрішнього сховища.

Зберігання файла у внутрішньому сховищі

Коли файл зберігається у внутрішньому сховищі, ви можете отримати відповідний каталог як  File, викликавши один з наступних методів:

getFilesDir()
Повертає  File, що представляє внутрішній каталог для вашого застосування.
getCacheDir()
Повертає  File, що представляє внутрішній каталог для тимчасових файлів кешу вашого застосування. Впевніться, що видалили кожний файл, коли він більше не потрібен, та реалізуйте логічне обмеження розміру пам’яті, що ви можете використовувати в кожний окремий момент часу, такий як 1MB. Якщо система починає вичерпувати пам’ять, вона може видалити ваші файли в кеші без попередження.

Щоб створити новий файл в одному з ціх каталогів, ви можете використати конструктор  File(), передаючи  File, що провадить один з подальших методів, що задають каталог вашого внутрішнього сховища. Наприклад:

File file = new File(context.getFilesDir(), filename);

Альтернативно, ви  можете викликати  openFileOutput() для отримання FileOutputStream, що записує файл в ваше внутрішнє сховище. Наприклад, ось як записати деякий текст в файл:

String filename = "myfile";
String string = "Hello world!";
FileOutputStream outputStream;

try {
  outputStream = openFileOutput(filename, Context.MODE_PRIVATE);
  outputStream.write(string.getBytes());
  outputStream.close();
} catch (Exception e) {
  e.printStackTrace();
}

Або, якщо ви бажаєте кешувати деякі файли,  замість цього вам слідує використовувати  createTempFile(). Наприклад, наступний метод отримує ім’я файла з  URL, та створює файл з ім’ям в директорії вашого внутрішнього сховища:

public File getTempFile(Context context, String url) {
    File file;
    try {
        String fileName = Uri.parse(url).getLastPathSegment();
        file = File.createTempFile(fileName, null, context.getCacheDir());
    catch (IOException e) {
        // Error while creating file
    }
    return file;
}

Зауваження: Ваше внутрішнє сховище застосування визначається на основі імені пакунку вашого застосування, та розташовується в спеціальному місці файлової системи Android. Технічно, інше застосування може читати ваші внутрішні файли, якщо ви встановили режим, що дозволяє читання. Однак інше застосування також має знати ім’я пакунку вашого застосування, та імена файлів. Інші застосування не можуть переглядати ваші внутрішні каталоги, та не можуть отримати дозвіл на читання або запис, якщо ви явно не зробите файли доступними для читання або запису. Таким чином, доки ви використовуєте MODE_PRIVATE для ваших файлів у внутрішньому сховищі, вони ніколи не будуть доступними для інших застосувань.

Зберігання файла в зовнішньому сховищі

Оскільки зовнішнє сховище може бути недоступним -  як от коли користувач змонтував сховище в PC, або видалив SDcard, що містить зовнішнє сховище — вам належить завжди перевіряти, що том доступний, перед доступом до нього. Ви можете запросити зовнішнє сховище, викликавши getExternalStorageState(). Якщо він повертає стан MEDIA_MOUNTED, тоді ви можете читати та писати ваші файли. Наприклад, наступні методи корисні для визначення доступності сховища:

/* Перевіряє, чи зовнішнє сховище доступне для читання та запису */
public boolean isExternalStorageWritable() {
    String state = Environment.getExternalStorageState();
    if (Environment.MEDIA_MOUNTED.equals(state)) {
        return true;
    }
    return false;
}

/* Перевіряє, чи зовнішнє сховище доступне щонайменш для читання */
public boolean isExternalStorageReadable() {
    String state = Environment.getExternalStorageState();
    if (Environment.MEDIA_MOUNTED.equals(state) ||
        Environment.MEDIA_MOUNTED_READ_ONLY.equals(state)) {
        return true;
    }
    return false;
}

Хоча зовнішнє сховище модифікується користувачем та іншими застосуваннями, є дві категорії файлів, що ви можете тут зберігати:

Публічні файли
Файли, що можуть бути вільно доступні для інших застосувань, та для користувача. Коли користувач деінсталює ваше застосування, ці файли повинні залишитися доступними для користувача.

Наприклад, світлини, захвачені вашим застосуванням, або інші завантажені файли.

Приватні файли
Файли, що належать виключно до вашого застосування, та мають бути видалені, коли користувач деінсталює застосування. Хоча ці файли технічно доступні для користувача та інших застосувань, оскільки вони на зовнішньому сховищі, ці файли, насправді, не мають значення для користувача за межами вашого застосуваня. Коли користувач деінсталює ваше застосування, система видаляє всі файли в приватньому каталозі зовнішнього сховища.

Наприклад, додаткові ресурси, завантажені вашим застосування, або тимчасові медіа файли. 

Якщо ви бажаєте зберігати файли у зовнішньому сховищі, використовуйте метод  getExternalStoragePublicDirectory(), щоб отримати File, що представляє відповідний каталог на зовнішньому сховищі. Метод приймає аргумент, що задає тип файлу, який ви бажаєте зберігти, так що вони будуть логічно організовані з другими публічними файлами. такі як  DIRECTORY_MUSIC або DIRECTORY_PICTURES. Наприклад:

public File getAlbumStorageDir(String albumName) {
    // Отримати каталог для публічного каталогу зображень. 
    File file = new File(Environment.getExternalStoragePublicDirectory(
            Environment.DIRECTORY_PICTURES), albumName);
    if (!file.mkdirs()) {
        Log.e(LOG_TAG, "Directory not created");
    }
    return file;
}

Якщо ви бажаєте зберегти файли, що є приватними до вашого застосування, ви можете отримати відповідний каталог, викликавши  getExternalFilesDir(), та передаючи ім’я, що вказує на тип каталогу, що вам потрібен. Кожний каталог, створений таким чином, додається в батьківській каталог, що заключає всі файл вашого зовнішнього сховища, та що система видалить, коли користувач деінсталює ваше застосування.

Наприклад, осі метод, що ви можете використовувати для створення каталогу для індивідуального  альбому світлин:

public File getAlbumStorageDir(Context context, String albumName) {
    // Отримати каталог для приватних зображень вашого застосування. 
    File file = new File(context.getExternalFilesDir(
            Environment.DIRECTORY_PICTURES), albumName);
    if (!file.mkdirs()) {
        Log.e(LOG_TAG, "Directory not created");
    }
    return file;
}

Якщо жодне з визначених імен каталогів не задовільняє вашим потребам, ви можете замість цього викликати  getExternalFilesDir() та передати null. Це повертає кореневий каталог вашого приватного каталогу на зовнішньому сховищі.

Пам’ятайте, що getExternalFilesDir() створює підкаталог в каталозі, що буде видалений, коли користувач деінсталюватиме ваше застосування. Якщо файли, що ви зберігали, повинні залишатись доступними після того, як користувач деінсталює ваше застосування — наприклад, ваше застосування є камерою, та користувач буде бажати зберігти світлини — вам треба замість цього використовувати getExternalStoragePublicDirectory().

Безвідносно, чи ви використовуєте getExternalStoragePublicDirectory() для файлів, що ви використовуєте спільно, чи getExternalFilesDir() для файлів, що є приватними для вашого застосування, є важливим, щоб ви використовували імена каталогів, наданих константами API, на кшталт DIRECTORY_PICTURES. Ці імена каталогів запевняють, що файли будуть трактуватись системою вірно. Наприклад, файли, збережені в DIRECTORY_RINGTONES будуть категорізовані системним медіа сканером як рінгтони, замість музики.

Запит вільного місця

Якщо ви знаєте наперед, якільки даних ви будете зберігати, ви можете винайти, скільки доступного місця є, без спрацювання IOException, викликаючи getFreeSpace() або getTotalSpace(). Ці методи провадять поточне доступне місце та загальний розмір місця на томі зберігання, відповідно. Ця інформація також корисна, щоб запобігти заповненню сховища більше визначеної межі.

Однак система не гарантує, що ви можете записати стільки байтів, як вказано в getFreeSpace(). Якщо число, що повертається є на декілька Мб більшою, ніж дані, що ви бажаєте зберегти, або якщо файлова система менше ніж на 90% повна, тоді, вирогідно, що ви можете безпечно продовжувати. Інакше вам, скоріше, не слід записувати в сховище.

Зауваження: Вам не слід перевіряти розмір доступного простору перед тим, як ви зберігаєте свій файл. Замість цього ви можете спробувати записувати файл напряму, та потів відсліджувати, чи виникне IOException. Ви можете потребувати цього, коли ви не знаєте заздалегідь, скільки простору вам треба. Наприклад, якщо ви змінюєте кодування файлів перед збереженням, перетворюючи зображення PNG до JPEG, ви не знаєте розмір файлу наперед.

Видалення файла

Ви завжди можете видалити файли, що вам більше не треба. Найбільш прямий спосіб видалити файл, це зробити виклик delete() для відкритого файла.

myFile.delete();

Якщо файл збережений у внутрішньому сховищі, ви також можете запитати Context, щоб знайти та видалити файл викликом deleteFile():

myContext.deleteFile(fileName);

Зауваження: Коли користувач деінсталює ваше застосування, система Android видаляє наступне:

  • Всі файли, збережені у внутрішньому сховищі
  • Всі файли, що ви зберегли на зовнішньому сховищі з використанням getExternalFilesDir().

Однак, вам треба вручну видалити всі кешовані файли, створені за допомогою getCacheDir() звичайним чином, а також звичайно видалити інші файли, що вам більше не треба.
