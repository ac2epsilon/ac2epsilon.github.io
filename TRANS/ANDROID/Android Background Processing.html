Інструкція з фонової обробки

Цей розділ допоможе вам обрати кращий спосіб для вашого застосування, щоб виконати операції в фоновому режимі, без завдання шкоди продуктивності.

Кожне застосування  Android має головний потік, що відповідає за обробку UI (включачи обчислення та відображення елементів керування), координування взаємодії з користувачем та отримання подій життєвого циклу. Якщо в цьому потоці відбувається багато обробки, застосування виглядає завислим або уповільненим, що закінчується небажаним користувацьким досвідом. Любі довгі обчислення та операції, такі, як тривалі обчислення, такі як декодування бітової карти, доступ до диску або виконання мережевих запитів, повинно виконуватись в окремому фоновому потоці. Загалом, будь-що, що займає більш ніж декілька мілісекунд, повинно бути делеговане до фонового потоку.

Коли ви вирішуєте робити роботу за межами головного потоку, розгляньте три фактори:

  • Чи може робота бути відкладеною, або вона має виконуватись саме коли вона заплановкана? Наприклад, якщо вам треба підтягнути деякі дані з мережі у відповідь на натискання користувачем кнопки, ця робота має бути виконана прямо зараз. Однак, якщо ви бажаєте завантажити ваші логотипи на сервер, ця робота може бути відкладена без впливу на продуктивність вашого застосування або на очікування користувача.

  • Коли ваша робота розпочала викоання, чи повинна OS намагатись утримувати процес застосування живим? Наприклад, декодування та відображення бітової мапи є дечим, що має траплятись, тільки якщо застосування на передньому плані та процес живий. Однак, музичний плеєр повинен продовжувати грати музику, навіть коли застосування на задньому плані, та не використовується активно.

  • Чи робота розпочинається у відповідь на спрацювання системних тригерів? Ці тригери можуть бути такими речами, як мережевий статус, стан батареї, рівні сховища, і таке інше. Наприклад, ви можете забажати очікувати комунікації з вашим сервером, доки ви не вийдете з режиму політу. В цьому випадку, якщо процес застосування буде мертвим, ви можете побажати відтворити його, коли застосування знову буде онлайн, так що ви зможете виконати вашу роботу.

ThreadPools

Для роботи, що має бути виконана тільки коли застосування на передньому плані, використовуйте ThreadPools. ThreadPools провадить групу фонових потоків, що приймають та ставлять в чергу надіслану роботу. Якщо вам треба моніторити системні тригери під час цього, використовуйте динамічно зареєстровані широкополосні броадкаст отримувачі для моніторінгу стану та OS тригерів.
Зауваження: Групуйте ваші ThreadPools (для мережі, дискового I/O та обчислень). Ці типи роботи загалом незалежні, та не повинні блокувати один одного.

Фонові сервіси

Для роботи, що має виконуватись до завершення, якщо вам треба виконати роботу безпосередньо, використовуйте  фонові сервіси. Використання фонового сервісу каже системі, що застосування робить щось важливе, і вони не повинні убиватись. Фонові сервіси видимі для користувачів через невідкладні нотифікації в треї нотифікацій.

WorkManager

Для роботи, що має виконуватись до завершення, та її можна відкласти, використовуйте WorkManager. WorkManager є бібліотекою Android, що люб'язно виконує відкладувану фонову роботу, коли задовільняються тригери роботи (як відповідний стан мережі та стан батареї). WorkManager використовує фреймворк JobScheduler, коли це можливо, щоб допомогти виживанню батареї та складати завдання в пакети. На пристроях нижче Android 6.0 (API level 23),  WorkManager намагається використовувати Firebase JobDispatcher, якщо він вже включений як залежність до вашого застосування; в іншому випадку WorkManager скочується до власної реалізації AlarmManager, та люб'язно обробляє вашу фонову роботу.

Обмеження на фонову обрбоку

Щоб максимізувати ресурс батареї та примусити застосування до гарної поведінки, Android обмежує фонову роботу, коли застосування (або нотифікація фонового сервісу) невидимий користувачеві.

  • Android 6.0 (API level 23) вводить Режим дрімоти та очікування застосування. Режим дрімоти обмежує поведінку застосування, коли екран вимкнений, та пристрій не рухається. Режим очікування застосування переводить невикористані застосування в особливий стан, що обмежує їх доступ до мережі, завдання та синхронізацію.
  • Android 7.0 (API level 24) обмежує неявні броадкасти та вводить Дрімоту-на-ходу.
  • Android 8.0 (API level 26) надалі обмежує фонову поведінку, таку, як отримання розташування в фоні, та надає кешовані блокування просинання.

WorkManager розроблений для надання найкращої з можливих поведінок під ціма обмеженнями.

Надсилання операцій в декілька потоків

Швидкість та ефективність довготривалих, дані-інтенсивних операцій часто покращується, коли ви розбиваєте їх на декілька потоків. На пристрої, що має CPU з багатьма процесорами (ядрами), система може використовувати потоки паралельно, замість щоб кожна операція очікувала на шанс виконання. Наприклад, декодування декількох файлів зображень по порядку для відображення їх на екрані мініатюр робить швидше, коли ви виконуєте кожне доекодування в окремому потоці.

Цей гайд показує вам, як встановити та використовувати декілька потоків в застосуванні Android, використовувачі об'єкт пулу потоків. Ви також вивчите, як визначати код для виконання на потоці, та як комунікувати між ціма потоками та UI потоком.

Задання коду для роботі на потоці

Цей гайд покаже вам, як реалізувати клас Runnable, що виконує код в своєму методі Runnable.run()на окремому потоці. Ви також можете передати Runnable до іншого об'єкту, що потім може приєднати його до потоку, та виконати його. Один або більше об'єктів Runnable, що виконують окрему операцію, іноді називаються завданнями.

Thread та Runnable є базовими класами, та самі по особі мають лише обмежену потужність. Замість цього вони є базою для потужних класів Android, як HandlerThread, AsyncTask та IntentService. Thread та Runnable є також базисом для класу ThreadPoolExecutor. Цей клас автоматично керує потоками та чергами завдань, та може навіть виконувати декілька потоків паралельно.

Визначення класу, що реалізує Runnable

Створення класу, що реалізує Runnable, є прямолінійним. Наприклад:

public class PhotoDecodeRunnable implements Runnable {
    ...
    @Override
    public void run() {
        /*
         * Код, що ви бажаєте виконувати на потоці
         */
        ...
    }
    ...
}

Реалізація методу run()

В класі метод Runnable.run() містить код, що буде виконаний. Зазвичай в Runnableдозволено все. Однак, пам'ятайте, що Runnable не може виконуватись на UI потоці, так що він не може модифікувати UI об'єкти, такі, як об'єктиView. Для комунікації з UI потоком, ви маєте використовувати прийоми, описвані в уроці Комуникація з  потоком.

Напочатку метода run()встановіть для потоку використання фонового приоритету, викликавши Process.setThreadPriority() з параметром THREAD_PRIORITY_BACKGROUND. Цей підхід зменшує конкуренцію за ресурси між потоком об'єкту Runnable, та потоком UI.

Ви також повинні зберігти посилання на Thread об'єкту Runnable в самому об'єкті Runnable, викликавши Thread.currentThread().

Наступний фрагмент показує, як налаштувати метод run():

class PhotoDecodeRunnable implements Runnable {
...
    /*
     * Визначає код для виконання для цього завдання.
     */
    @Override
    public void run() {
        // Переміщує поточний Thread в фоновий режим
        android.os.Process.setThreadPriority(android.os.Process.THREAD_PRIORITY_BACKGROUND);
        ...
        /*
         * Зберігає поточний Thread в примірнику PhotoTask, 
	 * так що примірник може перервати Thread.
         */
        mPhotoTask.setImageDecodeThread(Thread.currentThread());
        ...
    }
...
}


Створення менеджера для декількох потоків

Попередній гайд, що навчає, як налаштувати код для виконання на потоці, показує, як визначити завдання, що виконується в окремому потоці. Якщо ви бажаєте виконати це завдання тільки один раз, це може бути все, що вам треба. Якщо ви бажаєте виконувати завдання постійно з різними наборами даних, але в кожний час вам треба тільки одне робоче виконання, вашим потребам задовільняє IntentService. Щоб автоматично виконувати завдання по мірі появи доступних ресурсів, або якщо ви бажаєте дозволити виконання декількох завдань одночасно (або обоє разом), вам треба впровадити керовану колекцію потоків. Щоб зробити це, використовуйте примірник ThreadPoolExecutor, що виконує завдання з черги, коли поток в пулі стає вільним. Щоб виконати завдання, все, що вам потрібно зробити, це додати завдання до черги.

Пул потоків може виконувати декілька паралельних примірників завдань, так що ви повинні переконатись, що виконуєте потік-безпечний код. Оточуйте змінні, до яких можливий доступ з декількох потоків, в блокsynchronized. Цей підхід буде запобігати читанню змінної одним потоком, в той самий час, коли інший записує до неї. Типово ця ситуація виникає зі статичними змінними, але вона також виникає в любому об'єкті, який існує в одниму екземплярі. Щоб дізнатись більше про це, прочитайте гайд Огляд процесів та потоків.

Визначення класу пулу потоків

Створіть ThreadPoolExecutor в своєму власному класі. В цьому класі зробіть наступне:

Використовуйте статичні змінні для пулів потоків
Ви маєте створювати один примірник пулу потоків в вашому застосуванні, щоб мати єдину точку контролю для обмежених ресурсів CPU або мережі. Якщо ви маєте різні типи Runnable, ви можете побажати мати пул потоків для кожного типу, але кожний з них має бути в одному примірнику. Наприклад, ви можете додати його як частину ваших глобальних декларацій: 
public class PhotoManager {
    ...
    static  {
        ...
        // Створює єдиний статичний примірник PhotoManager
        sInstance = new PhotoManager();
    }
    ...
Використовуйте приватний конструктор
Створення приватного конструктора запевняє, що це синглтон. Це означає, що ви не моєте заточувати доступ до класу в блок synchronized: 
public class PhotoManager {
    ...
    /**
     * Будує робочі черги та пули потоків для завантаження та декодування
     * зображень. Оскільки конструктор відмічений як приватний, він недоступний для
     * інших класів, навіть в тому самому пакунку.
     */
    private PhotoManager() {
    ...
    }

Запустіть ваші завдання через виклик методів в класі пулу потоків.
Визначте метод в класі пулу потоків, що додає завдання по черги пулу потоків. Наприклад: 
public class PhotoManager {
    ...
    // Викликається PhotoView для отримання світлин
    static public PhotoTask startDownload(
        PhotoView imageView,
        boolean cacheFlag) {
        ...
        // Додає завданя завантаження до пулу потоків для виконання
        sInstance.
                mDownloadThreadPool.
                execute(downloadTask.getHTTPDownloadRunnable());
        ...
    }
Створіть примірник Handler в конструкторі, та додайте його до UI потоку застосування.
Handler дозволяє вашому застосуванню безпечно викликати методи UI об'єктів, таких, як об'єкти View. Більшість UI об'єктів можуть бути безпечно змінені тільки з потоку UI. Цей підхід описаний більш детально в уроці Комунікація з UI потоком. Наприклад: 
    private PhotoManager() {
    ...
        // Визначає об'єкт Handler, що приєднаний до UI потоку
        mHandler = new Handler(Looper.getMainLooper()) {
            /*
             * handleMessage() визначає операції до виконання, коли
             * Handler отримує нове Message для обробки.
             */
            @Override
            public void handleMessage(Message inputMessage) {
                ...
            }
        ...
        }
    }

Визначення параметрів пулу потоків

Як тільки ви матимете загальну структуру класу, ви можете почати визначати пул потоків. Щоб створити примірник об'єкту ThreadPoolExecutor, вам треба наступні значення:

Первинний розмір пулу та максимальний розмір пулу
Початкове число потоків, що треба розташувати в пулі, та максимальне можливе значення. Число потоків, що ви можете мати в пулі, залежить від числа ядер, доступних для вашого пристрою. Це число доступне з системного оточення: 
public class PhotoManager {
...
    /*
     * Отримує число доступних ядер (не завжди те саме, що максимальне 
     * число ядер)
     */
    private static int NUMBER_OF_CORES =
            Runtime.getRuntime().availableProcessors();
}
Це число може не відзеркалювати число фізичних ядер в вашому пристрої; деякі пристрої мають CPU, що деактивують одне або більше ядер в залежності від навантаження на систему. Для ціх пристроїв availableProcessors() повертає число активних ядер, що може бути менше, ніж загальне число ядер.
Час підтримки життя та одиниця часу
Відтінок часу, який потік має простоювати, перед тим, як він завершиться. ВІдтінок інтерпретується як значення в одиницях часу, одної з констант, визначених в TimeUnit.
Черга завдань
Вхідна черга, з якої ThreadPoolExecutor бере об'єкти Runnable. Щоб запустити код на потоці, менеджер пулу бере об'єкт Runnable з черги перший-увійшов перший-вийшов, та приєднує його до потоку. Ви провадите цю чергу об'єктів, коли створюєте пул потоків, використовуючи любий клас черги, що реалізує інтерфейс BlockingQueue. Для відповідності до потреб вашого застосування, ви можете обрати з доступних реалізацій черги; для додаткової інформації дивіться огляд класу ThreadPoolExecutor. Цей приклад використовує клас LinkedBlockingQueue: 
public class PhotoManager {
    ...
    private PhotoManager() {
        ...
        // Черга з Runnables
        private final BlockingQueue<Runnable> mDecodeWorkQueue;
        ...
        // Ініціює чергу Runnables як LinkedBlockingQueue
        mDecodeWorkQueue = new LinkedBlockingQueue<Runnable>();
        ...
    }
    ...
}

Щоб створити пул потоків створіть менеджер пулу потоків, через виклик ThreadPoolExecutor(). Він створює та керує обмежену групу потоків. Оскільки початковий розмір пулу, та максимальний розмір пулу ті самі, ThreadPoolExecutor створює всі потоки при створенні примірника. Наприклад:

Run code on a thread pool thread

The previous guide, which teaches you to Create a manager for multiple threads, shows you how to define a class that manages thread pools and the tasks that run on them. This lesson shows you how to run a task on a thread pool. To do this, you add the task to the pool's work queue. When a thread becomes available, the ThreadPoolExecutor takes a task from the queue and runs it on the thread.

This lesson also shows you how to stop a task that's running. You might want to do this if a task starts, but then discovers that its work isn't necessary. Rather than wasting processor time, you can cancel the thread the task is running on. For example, if you are downloading images from the network and using a cache, you probably want to stop a task if it detects that an image is already present in the cache. Depending on how you write your app, you may not be able to detect this before you start the download.

Run a task on a thread in the thread pool

To start a task object on a thread in a particular thread pool, pass the Runnable to ThreadPoolExecutor.execute(). This call adds the task to the thread pool's work queue. When an idle thread becomes available, the manager takes the task that has been waiting the longest and runs it on the thread:

public class PhotoManager {
    public void handleState(PhotoTask photoTask, int state) {
        switch (state) {
            // The task finished downloading the image
            case DOWNLOAD_COMPLETE:
            // Decodes the image
                mDecodeThreadPool.execute(
                        photoTask.getPhotoDecodeRunnable());
            ...
        }
        ...
    }
    ...
}

When ThreadPoolExecutor starts a Runnable on a thread, it automatically calls the object's run()method.

Interrupt running code

To stop a task, you need to interrupt the task's thread. To prepare to do this, you need to store a handle to the task's thread when you create the task. For example:

class PhotoDecodeRunnable implements Runnable {
    // Defines the code to run for this task
    public void run() {
        /*
         * Stores the current Thread in the
         * object that contains PhotoDecodeRunnable
         */
        mPhotoTask.setImageDecodeThread(Thread.currentThread());
        ...
    }
    ...
}

To interrupt a thread, call Thread.interrupt(). Notice that Thread objects are controlled by the system, which can modify them outside of your app's process. For this reason, you need to lock access on a thread before you interrupt it, by placing the access in a synchronized block. For example:

public class PhotoManager {
    public static void cancelAll() {
        /*
         * Creates an array of Runnables that's the same size as the
         * thread pool work queue
         */
        Runnable[] runnableArray = new Runnable[mDecodeWorkQueue.size()];
        // Populates the array with the Runnables in the queue
        mDecodeWorkQueue.toArray(runnableArray);
        // Stores the array length in order to iterate over the array
        int len = runnableArray.length;
        /*
         * Iterates over the array of Runnables and interrupts each one's Thread.
         */
        synchronized (sInstance) {
            // Iterates over the array of tasks
            for (int runnableIndex = 0; runnableIndex < len; runnableIndex++) {
                // Gets the current thread
                Thread thread = runnableArray[taskArrayIndex].mThread;
                // if the Thread exists, post an interrupt to it
                if (null != thread) {
                    thread.interrupt();
                }
            }
        }
    }
    ...
}

In most cases, Thread.interrupt() stops the thread immediately. However, it only stops threads that are waiting, and will not interrupt CPU or network-intensive tasks. To avoid slowing down or locking up the system, you should test for any pending interrupt requests before attempting an operation :

/*
 * Before continuing, checks to see that the Thread hasn't
 * been interrupted
 */
if (Thread.interrupted()) {
    return;
}

Communicate with the UI thread

The previous guide, which teaches you how to Run code on a thread pool thread, shows you how to start a task on a thread managed by ThreadPoolExecutor. This final lesson shows you how to send data from the task to objects running on the user interface (UI) thread. This feature allows your tasks to do background work and then move the results to UI elements such as bitmaps.

Every app has its own special thread that runs UI objects such as View objects; this thread is called the UI thread. Only objects running on the UI thread have access to other objects on that thread. Because tasks that you run on a thread from a thread pool aren't running on your UI thread, they don't have access to UI objects. To move data from a background thread to the UI thread, use a Handler that's running on the UI thread.

Define a handler on the UI thread

Handler is part of the Android system's framework for managing threads. A Handler object receives messages and runs code to handle the messages. Normally, you create a Handler for a new thread, but you can also create a Handler that's connected to an existing thread. When you connect a Handler to your UI thread, the code that handles messages runs on the UI thread.

Instantiate the Handler object in the constructor for the class that creates your thread pools, and store the object in a global variable. Connect it to the UI thread by instantiating it with the Handler(Looper)constructor. This constructor uses a Looper object, which is another part of the Android system's thread management framework. When you instantiate a Handler based on a particular Looperinstance, the Handler runs on the same thread as the Looper. For example:

private PhotoManager() {
...
    // Defines a Handler object that's attached to the UI thread
    mHandler = new Handler(Looper.getMainLooper()) {
    ...

Inside the Handler, override the handleMessage() method. The Android system invokes this method when it receives a new message for a thread it's managing; all of the Handler objects for a particular thread receive the same message. For example:

        /*
         * handleMessage() defines the operations to perform when
         * the Handler receives a new Message to process.
         */
        @Override
        public void handleMessage(Message inputMessage) {
            // Gets the image task from the incoming Message object.
            PhotoTask photoTask = (PhotoTask) inputMessage.obj;
            ...
        }
    ...
    }
}

The next section shows how to tell the Handler to move data.

Move data from a task to the UI thread

To move data from a task object running on a background thread to an object on the UI thread, start by storing references to the data and the UI object in the task object. Next, pass the task object and a status code to the object that instantiated the Handler. In this object, send a Message containing the status and the task object to the Handler. Because Handler is running on the UI thread, it can move the data to the UI object.

Store data in the task object

For example, here's a Runnable, running on a background thread, that decodes a Bitmap and stores it in its parent object PhotoTask. The Runnable also stores the status code DECODE_STATE_COMPLETED.

// A class that decodes photo files into Bitmaps
class PhotoDecodeRunnable implements Runnable {
    ...
    PhotoDecodeRunnable(PhotoTask downloadTask) {
        mPhotoTask = downloadTask;
    }
    ...
    // Gets the downloaded byte array
    byte[] imageBuffer = mPhotoTask.getByteBuffer();
    ...
    // Runs the code for this task
    public void run() {
        ...
        // Tries to decode the image buffer
        returnBitmap = BitmapFactory.decodeByteArray(
                imageBuffer,
                0,
                imageBuffer.length,
                bitmapOptions
        );
        ...
        // Sets the ImageView Bitmap
        mPhotoTask.setImage(returnBitmap);
        // Reports a status of "completed"
        mPhotoTask.handleDecodeState(DECODE_STATE_COMPLETED);
        ...
    }
    ...
}

PhotoTask also contains a handle to the ImageView that displays the Bitmap. Even though references to the Bitmap and ImageView are in the same object, you can't assign the Bitmap to the ImageView, because you're not currently running on the UI thread.

Instead, the next step is to send this status to the PhotoTask object.

Send status up the object hierarchy

PhotoTask is the next higher object in the hierarchy. It maintains references to the decoded data and the View object that will show the data. It receives a status code from PhotoDecodeRunnable and passes it along to the object that maintains thread pools and instantiates Handler:

public class PhotoTask {
    ...
    // Gets a handle to the object that creates the thread pools
    sPhotoManager = PhotoManager.getInstance();
    ...
    public void handleDecodeState(int state) {
        int outState;
        // Converts the decode state to the overall state.
        switch(state) {
            case PhotoDecodeRunnable.DECODE_STATE_COMPLETED:
                outState = PhotoManager.TASK_COMPLETE;
                break;
            ...
        }
        ...
        // Calls the generalized state method
        handleState(outState);
    }
    ...
    // Passes the state to PhotoManager
    void handleState(int state) {
        /*
         * Passes a handle to this task and the
         * current state to the class that created
         * the thread pools
         */
        sPhotoManager.handleState(this, state);
    }
    ...

Move data to the UI

From the PhotoTask object, the PhotoManager object receives a status code and a handle to the PhotoTask object. Because the status is TASK_COMPLETE, creates a Message containing the state and task object and sends it to the Handler:

public class PhotoManager {
    ...
    // Handle status messages from tasks
    public void handleState(PhotoTask photoTask, int state) {
        switch (state) {
            ...
            // The task finished downloading and decoding the image
            case TASK_COMPLETE:
                /*
                 * Creates a message for the Handler
                 * with the state and the task object
                 */
                Message completeMessage =
                        mHandler.obtainMessage(state, photoTask);
                completeMessage.sendToTarget();
                break;
            ...
        }
        ...
    }

If the status code is TASK_COMPLETE, then the task is finished, and the PhotoTask object in the Message contains both a Bitmap and an ImageView.
