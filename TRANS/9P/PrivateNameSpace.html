Abstract. As part of our metacomputing research we have built a private name space system for Unix. The current system runs completely in user mode, requiring no kernel changes of any kind. The name space can be built by one program and then is inherited via fork by child processes. The system is a ‘clean-room’ implementation of the proto- col described on the Plan 9 WEB pages and is ‘copylefted’ and available under the terms of the GNU Programming License.

We created this system in order to support our distributed computing environment in a large computer network. Our target environment for this system is a network of 1000

1.0 Introduction

Distributed computing is the use of many computers for one application. Typically the computers are part of a campus or wide area network. For small networks (100 or 200 nodes) one can assume that all the computers are under the control of one organization; larger collections of computers are typically under the control of more than one organi- zation. Often the most difficult challenges involved in distributed computing involve crossing organizational boundaries.

There are a number of software support systems for smaller collections of nodes, most notably Condor[7] and PVM[8]. The use of larger collections of nodes, especially groups of nodes that span multilple administrative domains, has not been well addressed by these systems. For the most part distributed computing support systems provide a scheduler or remote task startup library and depend on the existing environment for the rest of the support, particularly file system services.

An early problem encountered by users of these systems was gaining access to a com- mon file server. A user might start up a program on a remote node only to find that the file systems on the remote node did not have the same names or that the two nodes did not share common servers. A trip to the system administrator became necessary, in order to plead for a new file system mount point on the target machine. The difficulty of setting up new mounts points, especially across organizational boundaries, led to the development in Condor of the Remote System Call library.

With the advent of the automounters[4][5][6] this problem largely disappeared: all file systems from all servers, and in some cases from all workstations, became accessible via the automounter moint point (usually /net), as shown in Figure 1. A user on machine ‘B’ wishes to start a process on machine ‘A’. Because the system is running an auto- mounter, and ‘B’ is exporting file systems, the process on machine ‘A’ can access ‘B’s disks as /net/b/bdisk1 and /net/b/bdisk2. The advent of the automounter led many users of Condor to compile out the Remote System Call library, since it seemed superfluous for most cases.

The automounter solution becomes an automounter problem in large environments, especially if workstations are exporting file systems. Consider a network of 400 machines, with a large number of servers and desktop workstations which export file systems. A single user on a desktop system may only access a very small number of servers, and will almost certainly not access other desktop workstation file systems. Distributed programs result in quite different file access behavior for every desktop. The set of distributed programs in most cases will access every file system that may be accessed by any user at any time, and because the schedulers are trying to use as many machines as possible, these programs will access all these file systems from every desk- top. As a result, once distributed computing enters the picture, desktop workstations may end up with mountpoints to unexpected places. In the worst case (which we have seen in practice) the system ends up with ‘spaghetti mounts’, with many workstations mounting each other and even circular dependencies. The entire collection of machines can become vulnerable to a single desktop machine going down. This increased proba- bility of failure can make distributed computing very unpopular, and in some environ- ments has led to its demise.

The real problem is not that distributed programs are accessing many file systems. The problem is that when a program accesses a remote file system, it makes changes to the global state of the operating system, and these changes can affect the operating system and all the processes running on it. In fact the problem is even worse if ‘A’ and ‘B’ are in different administrative domains, since the process from ‘B’ has now gained access to all of the servers that ‘A’ can access. This single security prob- lem alone can make it impossible to convince different organizations to share com- pute resources, since the adminstrators are unwilling to leave their file system environment wide open to other organizations.

The chroot system call has sometimes been proposed as a solution to these problems. Remote programs could run in a seperate root partition, restricted to a subset of the file system. In order to gain access to home file systems, however, processes still need a way to access the automounter. System administrators would need to create and main- tain a set of special automounter tables for the restricted programs. Furthermore, for true separation of different organization’s programs there must be a separate chroot and automount table for each different organization. Use of chroot and special automount tables brings the problem full circle to the pre-automounter era, creating more work for system adminstrators than they can likely handle, especially in very large environments. Finally, the chroot solution only solves points 1 and 3 of the problems mentioned above. Problem 2, in which processes on ‘A’s machine can easily peruse ‘B’s file systems, remains and can not be solved by the simple chroot solution.

The type of name space supported by Unix is known as a global name space. In global name spaces, the file system name space is common to all processes and traversable by any process on the machine. The only exception is that processes which have had their name space pruned by the chroot system call are somewhat more restricted. As new file systems are mounted they become accessible to all the processes.

The combination of global name spaces and distributed computing create problems that can not be solved by manipulations of directory permissions, automount tables, or judi- cious use of the chroot system call. A new type of file system structure is needed, one not based on global name spaces. Fortunately, such a structure has been designed and reduced to practice in the Plan 9 operating system. The structure is known as private name spaces.

Private name spaces allow a process to build its own file system name space, indepen- dent of any other process running on the system. We illustrate the difference in Figure 2, which shows the same remote execution as scenario Figure 1. This time the process run- ning on ‘B’ has constructed a private name space consisting solely of /bdisk1 from ‘B’. The process has no need of /bdisk2 so has not mounted it. When the process moves to ‘A’, it will restart with only /bdisk1 in its name space, and will have no access to any of ‘A’s disks. No process on ‘A’ can access /bdisk1 without communicating with ‘B’ and mounting the disk; ‘A’s processes can not access /bdisk1 just because a process from ‘B’ happens to be running on’A’. If ‘B’ goes down, then the only process which will hang or fail is the remote process started from ‘B’. The failure of ‘B’ has no impact on any other process on ‘A’, or on ‘A’s operating system. Private name spaces remedy the problems which we have experienced in large-scale distributed computing environments.

The lowest-level layer, called the t9 (transport 9) layer, handles basic client-server mes- sages. This layer formats messages (defined by structures as documented on the Plan 9 Web page) and sends and receives these messages on TCP/IP. This layer also has func- tions for connection from client to server.

The middle layer, called u9 (user 9), supports the standard Plan9 library function com- patibility . This layer consists of a Plan 9 system call compatible layer, for functions documented on the Plan 9 Web pages. Many of these functions are similar in nature to their Unix counterparts, with slightly different semantics. Any user program can use the mount function, for example. Directories can be opened and read as files. File status information has a different format from that used on Unix. The semantics of how file status is changed is different. Directories are created as a special case of file creation. File creation is atomic. All of these differences are captured in this library.

The u9 layer on the server side is responsible for mapping the file system t9 messages to operations. Currently, the only u9 server we have is for the file system. We plan to have other servers eventually.

The upper layer provides a C library of Unix compatible functions including the usual open/close/read/write/stat and so on. This library can be compiled to be used as part of a program or as a shared library to be layered the standard C library (this works with the GNU ld.so dynamic linker which is available for Linux, SunOS, Free and NetBSD, and many other systems).

It is essential that this system be usable by any program without recompiling the pro- gram. Furthermore when a program calls fork the name space needs to be passed on to the child process. It is also useful in some cases to allow a union of the global and pri- vate name spaces. We support these goals by use of a stub library as shown in the Fig- ure. The library preempts some of the basic C library functions to add any functionality needed by the private name space library.

There are several components not shown in the Figure. First are the dump/restore func- tions which support private name space inheritance transparently (in the C library they are integrated into fork()). The dump function, called when fork() is called, dumps the private name space state into a text file in /var/tmp, using a name generated by mktemp(). The restore function is called the first time any of the private name space functions are called. It uses the file created by the dump function to restore the private name space.

We have also built additional programs for testing the libraries and building the name space which is inherited by unmodified Unix programs. The first is a TCL interface to the C library, U9 and T9 layers for testing. The TCL interface allows a user to com- pletely exercise each layer together or independently. The second is a TCL-based pro- gram which allows a user to build a name space using TCL commands then start up a program (such as a shell) which will inherit the name space. In this way users can con- struct a name space to be used to other programs. We discuss these programs in more detail below.

2.0 Related Work

There has been no work on private name spaces on Unix of which we are aware. The work most similar to ours was done for Condor[7]. Condor supports a remote system call library which allows a program to access its home file systems from remote nodes. To use the library a program has to be statically linked with the library. Almost every system call is preempted, and results in an RPC-like call to a system call server on the home machine. The system included no user-mount protocol as the Plan 9 system does. Processes can not tailor a name space as needed and pass that name space on to child processes. System calls such as fork are disabled. Finally, the protocol and system are not as flexible as the Plan 9 protocol.

Quick overview of the Plan 9 protocol

WebNFS is a very recent attempt to extend NFS to the Web[9][10]. This extension requires a user-mode mount protocol. The WebNFS mount protocol is not very secure, in contrast to the Plan 9 mount protocol, and WebNFS itself is primarily intended forread-only access to remote files. The specification clearly states that any security must be added on by mutual agreement between client and server.

3.0 Quick overview of the Plan 9 protocol

In this section we present a brief overview Plan9[11] name space ideas and the 9p proto- col. We do not have space to do it justice. Readers looking for more detail should see the Plan9 web site at plan.bell-labs.com.

The Plan 9 protocol is simple, regular, but sophisticated in the design decisions. The protocol uses stream-oriented communications services to connect clients to servers. Communications are via T messages and R messages. T messages are requests. R mes- sages are replies.

Each T message has a unique 16-bit tag, a T ID or TID. The tag may not be used again until an R message with the same tag has been sent, which retires that T message. The tags allow concurrency in the message transmission, processing, and reply. Requests may be aborted via a T_FLUSH message.

Sessions are established via a T_SESSION message. Authentication is done at this time. Mount points are established via attach operations. Both mount points and open files are referred to by File IDs, or FIDs, but unlike NFS or other remote file system protocols, FIDs are specified by the client, not the server. This client-side control of the FID space has useful implications we describe later.

Once the attach points have been set up, the client side can start opening files. To open files, a client first clones (duplicates) the mount point, then sends a series of walk mes- sages to walk the file system tree. Interestingly enough as a FID is walked it does not change, but the server side information about that FID changes. FID invariance coupled with the pipelining allowed by the TID system potentially allows for having a sequence of walk requests sent at one time, thus reducing the dead time that often accompanies remote file name lookup. We do not exploit this potential pipelining, and do not know if Plan 9 does either.

To open a file the client starts with a FID and simply sends an open message. At that point read or write messages may be issued. Again, due to the TID mechanism many concurrent reads or writes can be posted, allowing for read-ahead to be supported.

When a client is done with a file it sends a T_CLUNK message to indicate that the server can forget about the file. A clunk for an attach point is equivalent to an unmount.

Status for a file is accessed via a STAT message, and the status may be changed via a WSTAT message. Several other message types exists for manipulation of file state.

This short overview should give some flavor of the nature of the protocol. Based on our earlier extensive work with NFS, we feel the protocol compares very favorably to NFS. In contrast to the many variable-length fields in NFS, the fields in the 9p messages are

4.0 Implementation

We now go into more detail on the implementation of the private name space system. We will describe the layers in the order in which we presented them above. First, though, we need to discuss one limitation in this implementation.

4.1Single-threaded, synchronous implementation

Our first implementation of the Plan 9 protocol is single-threaded and synchronous. The reason for this decision is simple: we wanted to focus on the semantics of the Plan 9 protocol, on getting the mapping between the protocol and the Unix file system correct, and finally on making the system work in user mode in a transparent manner. For these reasons, we decided not to try to build a multi-threaded asynchronous server on the first try. Having a simple, single-thread, synchronous implementation provides us with a ref- erence platform against which to measure later multi-threaded implementations. For now, at the server side, each transaction must be completed before the next one is ser- viced. Clients block on each transaction. A multi-threaded version is in the works, how- ever.

4.2T9 layer

The t9 layer is responsible for transport. It currently only uses TCP/IP for transport, but we have begun to change is so that the way in which it is written allows for any send and receive functions to be used which match the prototypes shown in Figure 4. Currently only the client side has been modified. The read function is required to return complete messages, however: functions which return a message in pieces (such as reading from a TCP socket) must be called from a wrapper which reads the whole message. The func- tions return an indicating a success or failure (after the style of read and write); they accept a pointer to a function returning int, a void *, and a size. Typically the fp_send uses write(), and the fp_recv uses a wrapper for read().

The T9 layers are quite different for the client and the server, due to the many differ- ences in function. They do share many common functions and include files, but the parts which we discuss in this paper are very different. In addition, the server side implements both the T9 and U9 layers in an integrated way, even though they are in concept different layers. For these reasons we will cover the client and server separately. Because of the amount of code involved, we will use only two functions from each side as examples for the implementation.

4.4Client Side T9 layer

For the client side we will also show the clunk call. Shown in Figure 8 is the code for the client-side call to clunk. This function illustrates the essential features of all client- side functions for all T9 layer functions.

Most client-side functions take four basic parameters, as does this one: pointer to a send function, pointer to a receive function, integer file descriptor, and a pointer to an ename_t. An ename_t is a structure containing a fixed-size character array for error messages. All T9 layer functions return error messages by reference as well as error numbers by value. On the client side, all the basic elements such as open files, attach points, and sessions are referenced by an index number. This number-based scheme makes dump and restore very simple and fast, and makes moving to other machines very simple.

The first three steps are to turn the file number into a file structure pointer, get a session pointer, and get an attach pointer. With these three components we can proceed to oper- ate on the file. The next step is to see if we are freeing the attach point itself (this is the EQ_FID test). If so, we must first clunk all the files that are opened for this attach point. We walk through the open file table and clunk any files using this attach point. Finally, we set up the arguments and use the SEND_RECV macro to send the request and receive the return message. We test the error return and if it is the same as R_CLUNK then the operation succeeded: we free the file structure so it can be reused. The result of a failed clunk is not well specified in the documentation, so if the operation failed we free the file anyway.

4.7dump/restore functions

Dump and restore is fairly simple due to our use of tables. Implementing these functions has still had its problems, however. We use an environment variable (P9RESTART) to indicate to child processes the name of the restart file. We have found that we need to change this environment variable very early in the process startup; if we change the variable before calling fork, it will not be set. There is also a lingering problem with fscanf not correctly reading the last text-only line of the restart file, and hence returning an empty current working directory. Finally, ntol incorrectly parses 0x80000000 as 0x7fffffff, which has required us to use a work-around.

4.8Testing the libraries via the t9 program

We have two programs for testing each of these levels, one called t9 (for test 9) and one called t9d (for test 9 dynamic). Both programs are TCL interpreters. The use of TCL makes building debugging tests and procedures very easy.

T9 allows independent testing of each of the levels. T9 is statically linked to the plan 9 library, and hence does not exercise any of the capabilities of clib_libc.so. T9 is useful for chasing down basic bugs in the libraries themselves.

The t9d program is a much more comprehensive exercise of the libraries and of clib_libc.so. T9d is used to set up private name spaces, and then run standard Unix pro- grams which inherit those name spaces. Users start t9d, build the name space via mount commands, and then start up a command shell. The name space built via t9d commands is an indivisible part of the shell’s and shell’s child process name spaces. If there is a need to prune the name space for some reason, t9d can be run under the shell and the name space modified.

We have a number of tests which run under t9, and interactive usage can be used to flush out bugs. But for real bug-hunting t9d is the tool of choice. Tests which we write for T9 can not exercise the libraries nearly as thoroughly as real programs can.

To run t9d the user uses a simple shell script called rund which sets the environment variables correctly for the GNU ld.so program. We show this script in Figure 12. The user invokes this program by typing:

./rund t9d

5.0 Using the private name space system

There are several ways to use this system. Because the system is currently only sup- ported in user mode the only way to make a name space available to another process is to dump the state into a file and then have another program reload that file. It is not cur- rently possible to have a command modify another process state so as to augment that other process name space. We currently perform the dump implicitly via fork; restore is performed when the first p9 library function is called.

The most common way we currently use the system is via the t9d program described above. We build the name space and then run a shell command.

6.0 Additional Uses of Private Name Spaces

6.1Union File Systems

The Plan 9 union semantics make stacking and union file systems trivial. The union file system found in some BSD kernels is still buggy and requires root access to make it work at all. Unions are integral to the Plan 9 model and have been working in our imple- mentation since the beginning. We show an example of a union operation in Figure 13.

One useful idea for metacomputing is having a unified process id space across the com- puters that are in use by the application, so that the activity on the computers can more easily be observed. Having a process id space that covers all computers in the network is not practical, since the meaning of “a ll the computers” ca n be highly dynamic as com- puters come and go (especially in a two-week or one-month run); and also because showing all the computers process id spaces to an application which may only be using 100 quickly results in information overload. With private name spaces an application can easily build a process name space for those computers which it is using. Setting up the unified name space is trivial: we show the TCL commands for building a union / proc file system in Figure 14. Note that this unified /proc will work with FreeBSD, Linux, and other operating systems capable of supporting the Plan 9 server code.

6.3Securing files such as /etc/passwd

Suppose we want to only allow certain programs (e.g. login) to access key files such as / etc/passwd.

In current systems this goal is achievable only be renaming the file and hoping that no one can figure out all the permutations (e.g. master.passwd, passwd.shadow, etc.) or by restricting access via a chroot system call or permissions on the file. The chroot system call requires that the system administrator build a complete clone of the root file system somewhere on the computer. There are many difficulties with administering programs that require chroot support, and use of chroot still dodges the real problem: while only a very small number of programs need access to the files, only a small number of pro- grams can function well in a chroot environment. FTP daemons for example have no problem in a constricted environment. General user programs, however, can not operate in such environments, or they would not be started on the machine in the first place.

In the private name space model limiting access to a set of files is easy: the login pro- gram builds a name space which has a portion of /etc which any program may access, and a different separate mount point which has certain special files. /etc is thus a union mount point, of the general access and restricted access files. Before login actually starts up the user shell, it unmounts the special part, and user programs can only see files they should see. We show this sequence in Figure 15. This technique is much more powerful than the current one of using chroot, since it allows very fine control over how the name space is built.

6.4Versioning File Systems

Over the years many users have requested versioning file systems for Unix. Many pro- posals have been put forth, but the major stumbling block of integrating such an activity into the kernel has always which version system to use, how to build such capabilities into the kernel in a reasonable way, how to manage the permissions, and so on. With theuser-mode servers these problems for the most part disappear: the server runs as the user, and many types of servers can be made available which use the different file ver- sion software available.

7.0Future Work

The next step is to integrate the code into a Virtual File System for the client side and a kernel server. The VFS work will come first. Part of this will involve augmenting the proc structure so that per-process private name space information can be maintained, and another part will involve changing namei so that the private name space is searched first. In the long term, it would be useful to completely eliminate global name spaces on Unix to see if private name spaces are as useful as they seem. The kernel server should not be too difficult, since we can use existing NFS servers as a model for supporting the complications of remote communications.

Conclusions

8.0 Conclusions

We have described our first implementation of private file system name spaces for Unix. The system runs in user mode, and is single-threaded and synchronous. Nevertheless, it works and can be used with unmodified Unix binaries. Users can set up a private name space using our program and then run a command interpreter which inherits that name space. All child processes of that command interpreter will also inherit the name space. The name space can be inherited across the network as well, supporting migration.

Next steps involve integration in the FreeBSD and Linux kernels. In the long term, we would like to see if it is possible to run with only the private name space enabled.
